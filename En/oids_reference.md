# oids вҖ” Module Reference

> **Nim standard library** | `import std/oids`

## What Is an OID?

An **OID (Object Identifier)** is a 12-byte globally unique identifier designed to be generated independently on many machines without coordination, yet still produce values that never collide. This module implements a scheme inspired by MongoDB's BSON ObjectId.

Each OID is assembled from three independent components stored in a fixed binary layout:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ   time (8 bytes) в”Ӯ  fuzz (4 bytes) в”Ӯ  count (4 bytes)  в”Ӯ
в”Ӯ  Unix timestamp  в”Ӯ random per-proc в”Ӯ atomic counter    в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                     Total: 16 bytes (128 bits)
                     String representation: 24 hex characters
```

- **`time`** вҖ” seconds since the Unix epoch at the moment of generation. Encodes *when* the OID was created and causes IDs generated later to sort lexicographically after earlier ones.
- **`fuzz`** вҖ” a random 32-bit value chosen once when the module is first loaded. Serves as a process/machine fingerprint, ensuring IDs generated by different processes (or on different machines at the same second) do not collide.
- **`count`** вҖ” an atomic monotonically increasing 32-bit integer. Guarantees uniqueness within the same process in the same second, even across threads.

The combination of all three makes collision practically impossible: two OIDs would have to be generated in the same process, at the same second, and with the counter wrapping around to the same value вҖ” an astronomically unlikely event.

**Wire format:** When serialised to a string via `$`, the OID becomes a 24-character lowercase hexadecimal string, e.g. `"5fc7f546ddbbc84800006aaf"`.

Nim's oid takes up **16 bytes of memory** because it consists of three fields: `time` (8 bytes), `fuzz` (4 bytes), and `count` (4 bytes). However, when converted to a string (`$oid`), not the entire object is used, but only **12 bytes**. This is achieved by serializing starting from the 4th byte of the structure: the first 4 bytes of `time` are skipped, and the last 4 bytes of the time, plus all the bytes of `fuzz`, and all the bytes of `count` are taken. Thus, the string representation includes the low-order 4 bytes of the time, the full `fuzz`, and the full `count`. These 12 bytes are encoded into a hexadecimal string of length **24 characters**. It turns out that the object in memory is larger than its serialization, because some of the data (the high-order 4 bytes of the time) remains inside the structure but is not included in the string. This scheme corresponds to the classic MongoDB-OID format, where the identifier is always 12 bytes. As a result, `Oid` = 16 bytes in memory, `$oid` = 24 characters, and serialization only covers the lower half of the field along with the remaining fields.

> If the top 4 bytes of 64-bit Unix time are discarded, only the bottom 32 bits remain. This means that time is stored with a precision of seconds, but the range is limited to 2^32 - 1 second вүҲ 4.29 billion seconds. Converted to years, this is вүҲ 136 years.

> This means that the bottom 32 bits of Unix time allow dates to be encoded within a range of 136 years. The precision is incremental: seconds are stored (as in full Unix time). And the range is reduced: from 1970 to approximately 2106.

**Initialisation:** The random seed and the `fuzz` value are initialised automatically the first time the module is loaded. You do not need to call any setup procedure.

## Types

### `Oid`

```nim
type Oid* = object
```

A 12-byte value type (stored inline, not heap-allocated) holding the three components that make up an OID. Because `Oid` is an `object` (not a `ref`), assignment copies the value вҖ” there is no aliasing.

The fields `time`, `fuzz`, and `count` are private. You interact with the OID exclusively through the procedures in this module. The canonical external representation is the 24-character hex string produced by `$`.

```nim
var a = genOid()   # value is on the stack, no allocation
var b = a          # b is an independent copy
```

## Generation

### `genOid`

```nim
proc genOid*(): Oid
```

Creates and returns a new, globally unique OID. This is the only procedure you need for most use cases.

The function captures the current Unix timestamp, atomically increments a module-level counter, and combines both with the fixed `fuzz` value computed at module load time. Thread-safe: the counter increment uses an atomic CPU instruction, so concurrent calls from multiple threads produce distinct OIDs without locking.

```nim
let id = genOid()
echo $id           # e.g. "5fc7f546ddbbc84800006aaf"
echo $id.len == 24 # true вҖ” always exactly 24 characters

# Each call produces a unique value
let id1 = genOid()
let id2 = genOid()
assert id1 != id2
```

```nim
# Generating many IDs in a loop вҖ” safe and fast
var ids: seq[Oid]
for i in 1..1000:
  ids.add(genOid())
```

## Serialisation and Parsing

### `$` (to string)

```nim
proc `$`*(oid: Oid): string
```

Converts an OID to its canonical 24-character lowercase hexadecimal string representation. The bytes are serialised in big-endian order, so the string preserves the timestamp-first ordering: OIDs generated earlier sort lexicographically before OIDs generated later.

The result is always exactly 24 characters long and contains only the characters `0вҖ“9` and `aвҖ“f`.

```nim
let id = genOid()
let s = $id
echo s          # e.g. "5fc7f546ddbbc84800006aaf"
echo s.len      # 24

# Useful for storing in databases, JSON, log files, HTTP headers, etc.
let json = """{"_id": """ & "\"" & $id & "\"" & "}"
```

---

### `parseOid`

```nim
proc parseOid*(str: cstring): Oid
```

Parses a 24-character hexadecimal string back into an `Oid`. This is the inverse of `$`. The input must be a valid 24-character hex string вҖ” the function does not validate the input and will silently produce a garbage `Oid` if given malformed data.

The parameter type is `cstring` (a C-style null-terminated string). Nim `string` values are implicitly convertible to `cstring`, so you can pass a Nim string directly.

```nim
let original = genOid()
let s = $original              # "5fc7f546ddbbc84800006aaf"

let restored = parseOid(s)     # pass Nim string вҖ” implicit cstring conversion
assert original == restored    # round-trip is lossless

# Parsing a hard-coded string (e.g. from a database or config file):
let known = parseOid("5fc7f546ddbbc84800006aaf")
echo $known  # 5fc7f546ddbbc84800006aaf
```

> **Important:** `parseOid` does not validate its input. Passing a string shorter than 24 characters, or one containing non-hex characters, results in undefined behaviour. Always validate the string's length and character set before calling `parseOid` on user-supplied data.

## Inspection

### `generatedTime`

```nim
proc generatedTime*(oid: Oid): Time
```

Extracts and returns the Unix timestamp embedded in the OID as a Nim `Time` value (from `std/times`). This tells you *when* the OID was created, accurate to the second.

This is useful for:
- Displaying human-readable creation times without storing a separate timestamp field in your data model.
- Filtering or sorting records by creation time using only the OID.
- Debugging вҖ” quickly checking when an OID was generated.

```nim
import std/[oids, times]

let id = genOid()
let t = id.generatedTime()
echo t.format("yyyy-MM-dd HH:mm:ss")  # e.g. "2024-01-15 14:32:07"

# The time is accurate to the second
let before = getTime()
let id2 = genOid()
let after = getTime()
assert id2.generatedTime() >= before
assert id2.generatedTime() <= after
```

## Equality and Hashing

### `==` operator

```nim
proc `==`*(oid1: Oid, oid2: Oid): bool
```

Returns `true` if and only if all three components (`time`, `fuzz`, `count`) are identical. Two OIDs are equal if and only if they are the exact same OID вҖ” there are no "equivalent but different" representations, because the round-trip through `$` and `parseOid` is lossless.

```nim
let a = genOid()
let b = a            # copy
let c = genOid()     # a new, different OID

assert a == b        # same value
assert a != c        # different OIDs
assert not (a == c)
```

---

### `hash`

```nim
proc hash*(oid: Oid): Hash
```

Computes a hash of the OID for use in Nim's `Table`, `HashSet`, and other hash-based containers. The hash combines all three components to produce a well-distributed integer.

You never call `hash` directly вҖ” it is picked up automatically by the container when you use an `Oid` as a key or set element.

```nim
import std/[oids, tables, sets]

# OIDs as table keys
var registry = initTable[Oid, string]()
let id1 = genOid()
let id2 = genOid()
registry[id1] = "first record"
registry[id2] = "second record"
echo registry[id1]  # first record

# OIDs in a hash set
var seen = initHashSet[Oid]()
seen.incl(id1)
assert id1 in seen
assert id2 notin seen
```

## Low-level Helper

### `hexbyte`

```nim
proc hexbyte*(hex: char): int
```

Converts a single hexadecimal character (`'0'`вҖ“`'9'`, `'a'`вҖ“`'f'`, `'A'`вҖ“`'F'`) to its integer value (0вҖ“15). Used internally by `parseOid` to decode the hex string byte by byte.

This function is exported but rarely needed in application code. It is available if you are implementing custom OID serialisation or parsing logic.

```nim
assert hexbyte('0') == 0
assert hexbyte('9') == 9
assert hexbyte('a') == 10
assert hexbyte('f') == 15
assert hexbyte('A') == 10
assert hexbyte('F') == 15
```

## Complete Example

The following example demonstrates the typical lifecycle of an OID: generation, storage as a string, round-trip parsing, creation-time inspection, and use as a table key.

```nim
import std/[oids, times, tables]

# --- Generation ---
let id = genOid()
echo "New OID:  ", $id                                       # 24-char hex
echo "Created:  ", id.generatedTime().format("yyyy-MM-dd")   # creation date
echo "Length:   ", ($id).len                                 # 24

# --- Round-trip serialisation ---
let serialised = $id                   # store this in a DB, JSON, log, etc.
let restored   = parseOid(serialised)  # recover the Oid later
assert id == restored
echo "Round-trip OK: ", id == restored  # true

# --- Use as a table key ---
var store = initTable[Oid, string]()
for name in ["Alice", "Bob", "Carol"]:
  store[genOid()] = name

for id, name in store:
  echo $id, "  ->  ", name

# --- Sorting by creation time ---
# Because $oid is big-endian timestamp-first, lexicographic order
# equals chronological order for OIDs generated in the same process.
var ids: seq[Oid]
for _ in 1..5:
  ids.add(genOid())

let times = ids.mapIt(it.generatedTime())
# All timestamps are equal or increasing (same-second granularity)
for i in 1 ..< times.len:
  assert times[i] >= times[i-1]
```
