# `tasks` Module Reference

> **Module:** `std/tasks`  
> **Language:** Nim  
> **Purpose:** Lightweight primitives for creating deferred, transferable units of work — tasks — that can be queued, sent to worker threads, and executed later.

---

## Core Idea

A `Task` is a self-contained bundle that wraps a procedure call together with a snapshot of all its arguments. Once created, a `Task` can be stored in a queue, sent across a channel to a worker thread, and executed at any point later by calling `invoke`. The original call site does not need to be running at the time of execution.

This design is the foundation of thread pool implementations, work-stealing schedulers, and any system where work is described in one place and executed in another.

**The single most important rule:** a `Task` has exactly one owner at a time and **cannot be copied** — the copy constructor is disabled at the type level. You move a task, you do not share it.

---

## Exported Type: `Task`

```nim
type Task* = object
  callback: proc (args, res: pointer) {.nimcall, gcsafe.}
  args: pointer
  destroy: proc (args: pointer) {.nimcall, gcsafe.}
```

`Task` is a plain object (not a ref) with three internal fields:

- **`callback`** — a GC-safe, `nimcall`-convention function pointer. This is the wrapper procedure generated by `toTask` that unpacks the frozen arguments and calls the original function.
- **`args`** — a raw pointer to a heap-allocated "scratch object" holding all the frozen argument values. Allocated with `allocShared0` so it is suitable for cross-thread use.
- **`destroy`** — a GC-safe destructor function pointer that runs `=destroy` on the scratch object's fields before the shared memory is freed. This ensures that any Nim managed objects stored as arguments (strings, seqs, refs) are properly cleaned up.

### Copy is intentionally disabled

```nim
proc `=copy`*(x: var Task, y: Task) {.error.}
```

Attempting to copy a `Task` is a **compile-time error**. This is a deliberate design choice: if a task could be copied, two copies could both be given to different threads, and both would free the same underlying arguments memory — a double-free. Ownership must be singular and transferred via `move`, not shared via copy.

### Destruction is automatic

When a `Task` goes out of scope (under ARC/ORC garbage collectors), `=destroy` is called automatically. It:
1. Runs the `destroy` callback to clean up the arguments' managed fields.
2. Calls `deallocShared` to free the shared heap allocation.

If `invoke` was never called, the task's arguments are still cleaned up correctly.

## Macro: `toTask`

```nim
macro toTask*(e: typed{nkCall | nkInfix | nkPrefix | nkPostfix | nkCommand | nkCallStrLit}): Task
```

**What it does:** The central macro of this module. It takes a **call expression** — a function call with all its arguments written out — and transforms it at compile time into a `Task` object. The macro captures and freezes the argument values at the moment `toTask` is called, wrapping everything into a heap-allocated bundle that can be executed later by `invoke`.

**What the macro generates:** For a call like `toTask hello(42, "nim")`, the macro produces roughly:

1. A private struct type (`ScratchObj`) with one field per argument.
2. Allocation of the scratch object on the shared heap (`allocShared0`).
3. Isolation and assignment of each argument value into the scratch object (via `isolate`/`extract` from `std/isolation` — this is what enforces thread-safety of the captured data).
4. A wrapper `callback` procedure that casts the opaque pointer back to the scratch struct, unpacks each field into a local variable, and calls the original function.
5. A `destroy` procedure that runs `=destroy` on the scratch struct.
6. Construction and return of a `Task` object tying all three together.

**Restrictions — what `toTask` cannot wrap:**

- **Closure calls** — functions that capture variables from an enclosing scope cannot be used. A task must be self-contained; all data must be passed as explicit arguments.
- **`var` parameters** — the wrapped function cannot have `var` (mutable reference) parameters, because the argument would have to outlive the call site, which a task cannot guarantee.
- **`typed`/`untyped` parameters** — these are compile-time-only constructs and cannot be frozen into a runtime value.
- **Non-GC-safe functions** — when compiling with `--threads:on`, the target function must be GC-safe (i.e. it must not access global GC-managed state without synchronization). This is checked at compile time.

**Arguments must be isolatable:** Each argument value is passed through `isolate`, which verifies at compile time that the value does not share mutable state with anything outside the task. If an argument refers to a `ref` object that is also reachable from somewhere else, the compiler will refuse with an "expression cannot be isolated" error.

```nim
import std/tasks

proc greet(name: string) =
  echo "Hello, ", name

# Create a task. "Alice" is captured and frozen inside the task.
let t = toTask greet("Alice")

# t can now be stored, sent to another thread, etc.
# When ready to run:
t.invoke()  # prints: Hello, Alice
```

**Functions that return a value:** `toTask` also works with non-void functions. The return value is written via the `res: pointer` parameter of the callback. You must pass a correctly typed pointer to `invoke` to retrieve the result.

```nim
import std/tasks

proc add(a, b: int): int = a + b

let t = toTask add(10, 32)

var result: int
t.invoke(addr result)
assert result == 42
```

**`varargs` / `openArray` parameters:** When the wrapped function takes a `varargs` or `openArray` parameter, `toTask` automatically converts the passed sequence to a `seq` and stores it in the scratch object, since `openArray` cannot be heap-allocated directly.

**Static / literal parameters:** Parameters that are literals (`42`, `"hello"`, `nil`, etc.) or static compile-time values are passed directly into the generated call without going through the scratch object, since they do not need heap storage.

## Procedure: `invoke`

```nim
proc invoke*(task: Task; res: pointer = nil) {.inline, gcsafe.}
```

**What it does:** Executes the task — calls the frozen procedure with the frozen arguments. This is the only way to run a `Task`; there is no automatic execution.

The `res` parameter is an optional output pointer. When the wrapped procedure returns a value, you pass the address of a variable of the correct type here, and `invoke` will write the return value into it. When the wrapped procedure is `void`, simply omit `res` or pass `nil`.

**A task can be invoked multiple times.** Each call executes the callback again with the same frozen arguments. This is useful for repeated background tasks, retrying failed work, or benchmarking. Note that if the wrapped procedure mutates its arguments (e.g. the task captures a `ref` object and modifies its fields), those mutations accumulate across invocations — the scratch object is not reset between calls.

```nim
import std/tasks

var count = 0
proc increment(n: int) = count += n

let t = toTask increment(5)

t.invoke()   # count == 5
t.invoke()   # count == 10
t.invoke()   # count == 15
```

**With a return value:**

```nim
import std/tasks

proc multiply(a, b: int): int = a * b

let t = toTask multiply(6, 7)
var result: int

t.invoke(addr result)
assert result == 42

# Calling again is fine:
t.invoke(addr result)
assert result == 42
```

**Important:** `invoke` asserts that `task.callback` is not nil. A default-initialized (zero) `Task` will cause a runtime assertion failure if you attempt to invoke it.

## The `isolation` Module (re-exported)

`tasks` re-exports `std/isolation`. Understanding isolation is essential to using `toTask` correctly.

The `isolate` procedure wraps a value and statically verifies that it does not share mutable heap state with any other reachable value. The `extract` procedure retrieves the value out of the isolated wrapper. Together they form the compile-time safety gate that ensures captured arguments are safe to hand off to another thread.

If you see the error **"expression cannot be isolated"**, it means the argument you are trying to capture has aliases that make it unsafe to hand to a task. The fix is usually to create the value as a fresh expression directly in the `toTask` call rather than passing a pre-existing variable that might be shared.

```nim
import std/tasks

type Node = ref object
  value: int

# This would fail — x is a ref, potentially shared:
# let x = Node(value: 5)
# let t = toTask myProc(x)   # error: cannot be isolated

# This works — the ref is created fresh, not reachable from anywhere else:
let t = toTask myProc(Node(value: 5))
```

## Typical Usage Pattern

The intended workflow for `Task` in a thread-pool or producer-consumer scenario:

```nim
import std/tasks

# 1. Define the work procedure.
proc processItem(id: int, value: string) =
  echo "Processing [", id, "]: ", value

# 2. Create tasks on the producer side and send them through a channel.
#    (Channel[Task] requires move semantics — use send/recv with move.)
var taskQueue: seq[Task]  # simplified — use Channel[Task] in real code

taskQueue.add toTask processItem(1, "alpha")
taskQueue.add toTask processItem(2, "beta")
taskQueue.add toTask processItem(3, "gamma")

# 3. On the worker side, receive and execute each task.
for t in mitems(taskQueue):
  t.invoke()
```

## What `toTask` Cannot Do — Summary

| Restriction | Reason |
|---|---|
| No closure captures | Tasks must be self-contained; closures reference external state |
| No `var` parameters | The referenced location might be gone by execution time |
| No `typed`/`untyped` parameters | These are compile-time only; cannot be frozen as runtime values |
| Non-GC-safe functions rejected (with `--threads`) | Would be unsafe to call from another thread |
| Arguments must be isolatable | Prevents accidental sharing of mutable heap state across threads |

## Design Notes

**Why shared heap allocation?** Arguments are allocated with `allocShared0` (zero-initialized shared memory) rather than the thread-local GC heap. This means the memory is accessible from any thread without triggering the GC — essential for cross-thread hand-off.

**Why copy is disabled?** Double-ownership of the same `args` pointer would lead to double-free on destruction. The type system enforces single ownership statically.

**Why `nimcall` convention?** The callback and destroy pointers must use a known, stable calling convention that is compatible across thread boundaries and does not involve a GC frame pointer. `nimcall` is Nim's C-compatible calling convention for this purpose.

**Debug mode:** Compiling with `-d:nimTasksDebug` causes `toTask` to print the generated AST and code to stdout at compile time, which is invaluable for understanding what the macro produces.
