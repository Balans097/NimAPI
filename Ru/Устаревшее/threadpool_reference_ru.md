# Модуль Threadpool — Справочник

> ⚠️ **Уведомление об устаревании.** Этот модуль официально помечен как устаревший. Команда Nim рекомендует использовать вместо него пакеты Nimble: **[malebolgia](https://github.com/araq/malebolgia)**, **[taskpools](https://github.com/status-im/nim-taskpools)** или **[weave](https://github.com/mratsim/weave)**. API помечен как нестабильный.

> 🔧 **Требование к компиляции.** Модуль требует передачи флага `--threads:on` компилятору. Без него компиляция завершается явной ошибкой.

---

## Что делает этот модуль

`threadpool` реализует высокоуровневый параллелизм на основе задач для Nim. Вместо того чтобы вручную управлять потоками (создавать, соединять, синхронизировать), вы передаёте отдельные вызовы функций пулу рабочих потоков и получаете обратно специальные объекты — **FlowVar**, представляющие будущие результаты этих вызовов.

Центральная идея — **параллелизм потоков данных**: порождённая задача выполняется параллельно, а тот, кому нужен её результат, просто ждёт завершения FlowVar. Пул сам управляет количеством потоков, увеличивая и уменьшая его в пределах настраиваемых границ в зависимости от нагрузки.

---

## Основные понятия

### Пул потоков

При запуске программы модуль автоматически вызывает `setup()`, который определяет количество ядер процессора и запускает столько же рабочих потоков (не более `MaxThreadPoolSize`). Эти потоки образуют пул. Задачи, переданные через `spawn`, распределяются между ними.

Размер пула эластичен: он может вырасти до `maxPoolSize`, когда задачи накапливаются, и сократиться обратно до `minPoolSize`, когда система простаивает.

### FlowVar

`FlowVar[T]` — это типизированный «заполнитель» для значения, которого ещё не существует: оно будет вычислено рабочим потоком в какой-то момент в будущем. Можно думать о нём как об одноразовом канале или обещании (promise). Как только рабочий поток завершает работу, он записывает результат во FlowVar; любой поток, ожидающий этот FlowVar, тут же разблокируется.

### Выделенные потоки (distinguished threads)

Небольшой набор потоков (до `MaxDistinguishedThread`, по умолчанию 32) можно «закрепить»: при использовании `pinnedSpawn` задача отправляется конкретному фиксированному рабочему потоку, идентифицируемому по `ThreadId`. Это полезно, когда часть состояния является локальной для потока и не должна мигрировать между потоками.

## Экспортируемое API

---

### `spawn`

```nim
proc spawn*(call: sink typed) {.magic: "Spawn".}
```

#### Описание

`spawn` — основной способ передачи работы в пул потоков. Он принимает выражение вызова процедуры и организует выполнение этого вызова на рабочем потоке — никогда не на вызывающем потоке. Компилятор переписывает вызов во внутреннюю диспетчеризацию через `nimSpawn3`.

Если процедура возвращает значение, `spawn` возвращает `FlowVar[T]`, содержащий будущий результат. Если процедура возвращает `void`, FlowVar не создаётся.

Передаваемая в `spawn` процедура **обязана** быть помечена `{.gcsafe.}` (или быть выведена как таковая), потому что она будет выполняться в отдельном потоке, который не разделяет корни сборщика мусора с вызывающим потоком.

Стратегия диспетчеризации жёсткая: `spawn` всегда помещает задачу в пул и никогда не выполняет её встроенно в вызывающем потоке (для такого поведения см. `spawnX`).

#### Что происходит, когда все рабочие потоки заняты

Если ни один рабочий поток не свободен немедленно, реализация входит в цикл ожидания. В этом цикле она может:
- Расширить пул, активировав новый поток (если ниже `maxPoolSize`).
- Если вызывающий поток сам является рабочим — выполнить задачу встроенно, чтобы избежать взаимоблокировки.
- Заблокироваться на семафоре до освобождения рабочего потока.

#### Параметры

| Параметр | Описание |
|----------|----------|
| `call` | Выражение вызова процедуры `p(арг1, арг2, ...)`. Процедура должна быть `gcsafe`. |

#### Возвращаемое значение

`FlowVar[T]`, если `p` возвращает `T`; ничего, если `p` возвращает `void`.

#### Примеры

```nim
import threadpool

proc heavyComputation(n: int): int {.gcsafe.} =
  result = 0
  for i in 1..n: result += i

# Порождаем задачу; получаем FlowVar обратно
let fv = spawn heavyComputation(1_000_000)

# Здесь можно делать другую работу, пока задача выполняется...

# Блокируемся до готовности результата и читаем его
echo ^fv   # → 500000500000
```

```nim
import threadpool

proc printMessage(msg: string) {.gcsafe.} =
  echo msg

# Процедура void — FlowVar не возвращается
spawn printMessage("Привет из рабочего потока!")
sync()
```

### `spawnX`

```nim
template spawnX*(call)
```

#### Описание

`spawnX` — более умная, адаптивная версия `spawn`. Сначала она вызывает `preferSpawn()`, чтобы проверить, свободен ли хоть один рабочий поток. Если да — порождает задачу на этом потоке (`spawn call`). Если нет — выполняет вызов **встроенно в вызывающем потоке**.

Это делает `spawnX` оптимизацией производительности для ситуаций, когда гранулярность работы настолько мала, что накладные расходы на планирование задачи в занятом пуле превышают выгоду от параллельного выполнения. Вместо того чтобы производящий поток ждал, он просто делает работу сам.

Компромисс в том, что вы теряете гарантию параллельного выполнения вызова. Если пул перегружен, `spawnX` деградирует до последовательного выполнения. Это делает `spawnX` непригодным, когда нужен асинхронный результат или гарантированно неблокирующее добавление в очередь.

#### Примеры

```nim
import threadpool

proc processChunk(data: int): int {.gcsafe.} =
  result = data * data

var results: array[10, FlowVar[int]]

for i in 0..<10:
  # Порождаем только если поток свободен; иначе вычисляем встроенно
  results[i] = spawnX processChunk(i)

sync()
for fv in results:
  echo ^fv
```

### `pinnedSpawn`

```nim
proc pinnedSpawn*(id: ThreadId; call: sink typed) {.magic: "Spawn".}
```

#### Описание

Работает как `spawn`, но гарантирует выполнение вызова на конкретном «выделенном» рабочем потоке, идентифицируемом по `id`. Выделенные потоки — это отдельный фиксированный пул (до `MaxDistinguishedThread` = 32 по умолчанию), существующий параллельно с основным динамическим пулом.

Поскольку данный `id` всегда соответствует одному и тому же потоку, можно безопасно хранить в нём локальное для потока состояние (например, OpenGL-контекст, соединение с базой данных или любой ресурс, не безопасный с точки зрения GC, который не должен перемещаться между потоками ОС) и всегда направлять связанную работу именно на него.

Вызов блокируется, если выбранный выделенный поток в данный момент занят, дожидаясь его освобождения.

#### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `id` | `ThreadId` | Целое число в диапазоне `0 .. MaxDistinguishedThread-1`. Идентифицирует выделенный поток. |
| `call` | typed | Выражение вызова `gcsafe`-процедуры. |

#### Примеры

```nim
import threadpool

# Поток 0 «владеет» ресурсом, который нельзя делить
proc renderFrame(frameId: int) {.gcsafe.} =
  echo "Рендеринг кадра ", frameId, " на выделенном потоке"

# Всегда отправляется на выделенный поток 0
pinnedSpawn 0, renderFrame(1)
pinnedSpawn 0, renderFrame(2)
sync()
```

### `parallel`

```nim
proc parallel*(body: untyped) {.magic: "Parallel".}
```

#### Описание

`parallel` вводит **параллельный блок** — структурированный регион кода, в котором компилятор может распараллелить ограниченное подмножество операторов Nim. Внутри блока `parallel` можно использовать `spawn` и итерировать диапазоны с неявным параллелизмом.

Компилятор анализирует тело блока, доказывает независимость данных между итерациями и автоматически генерирует необходимую синхронизацию. Когда блок `parallel` завершается, все порождённые в нём задачи гарантированно выполнены — вызывать `sync()` явно не нужно.

Тело должно соответствовать специфическому DSL, описанному в [экспериментальном руководстве](https://nim-lang.org/docs/manual_experimental.html#parallel-amp-spawn). Не все конструкции Nim допустимы внутри `parallel`.

#### Примеры

```nim
import threadpool

proc processItem(x: int): int {.gcsafe.} = x * x

var data = @[1, 2, 3, 4, 5, 6, 7, 8]
var results = newSeq[int](data.len)

parallel:
  for i in 0 ..< data.len:
    results[i] = spawn processItem(data[i])

# Все задачи завершены здесь; sync() не нужен
echo results  # → @[1, 4, 9, 16, 25, 36, 49, 64]
```

### `sync`

```nim
proc sync*()
```

#### Описание

`sync` — простой **глобальный барьер**: блокирует вызывающий поток до тех пор, пока каждый рабочий поток в пуле не завершит свою текущую задачу и не вернётся в состояние ожидания.

Внутри реализация опрашивает все рабочие потоки каждые `threadpoolWaitMs` миллисекунд (по умолчанию 100 мс, настраивается через `--define:threadpoolWaitMs=N`), пока все не будут помечены как готовые. Подход с опросом (а не блокировкой на семафоре) сделан намеренно: он избегает гонки, при которой рабочий поток мог бы завершиться прежде, чем успеет просигнализировать семафор.

**Когда использовать `sync`:**
- После нескольких `spawn` void-задач нужно убедиться, что все они завершились.
- Нужна контрольная точка после пакета задач, прежде чем продолжить.

**Когда не использовать `sync`:**
- Нужен возвращаемый результат порождённой задачи — используйте `^` или `awaitAndThen`.
- Вы находитесь внутри блока `parallel` — он сам обрабатывает синхронизацию.

#### Примеры

```nim
import threadpool

proc writeToLog(msg: string) {.gcsafe.} =
  # Представьте, что это запись в файл или базу данных
  echo "[LOG] ", msg

spawn writeToLog("Задача A")
spawn writeToLog("Задача B")
spawn writeToLog("Задача C")

sync()  # Ждём завершения всех трёх операций записи
echo "Все задачи выполнены."
```

### `^` (оператор «шляпа» / разыменование)

```nim
proc `^`*[T](fv: FlowVar[T]): T
```

#### Описание

Оператор `^` — стандартный способ **извлечь результат** из `FlowVar[T]`. Он блокирует вызывающий поток до тех пор, пока рабочий поток не вычислит значение, после чего возвращает его.

Для типов-значений (целые числа, числа с плавающей запятой, простые объекты) значение возвращается из внутреннего поля (`blob`). Для размещённых в куче типов (`string`, `seq`, `ref`) выполняется глубокое копирование, чтобы безопасно передать владение из кучи GC рабочего потока вызывающему. В модели памяти ARC/ORC Nim (`nimV2`) глубокое копирование избегается и значение перемещается напрямую.

После возврата `^` FlowVar помечается как использованный (`finished` вызывается внутренне). Не вызывайте `^` дважды на одном FlowVar.

#### Примеры

```nim
import threadpool

proc fibonacci(n: int): int {.gcsafe.} =
  if n <= 1: return n
  return fibonacci(n-1) + fibonacci(n-2)

let fv = spawn fibonacci(30)

# Вызывающий поток свободен делать другую работу
echo "Ожидаем числа Фибоначчи..."

let result = ^fv   # Блокируется до готовности результата
echo "fibonacci(30) = ", result   # → 832040
```

```nim
import threadpool

proc buildString(n: int): string {.gcsafe.} =
  result = "Результат: " & $n

let fv = spawn buildString(42)
let s = ^fv   # Глубокая копия обеспечивает безопасную передачу владения
echo s        # → "Результат: 42"
```

### `awaitAndThen`

```nim
proc awaitAndThen*[T](fv: FlowVar[T]; action: proc (x: T) {.closure.})
```

#### Описание

`awaitAndThen` — альтернатива `^`, которая избегает лишней копии для крупных типов-значений. Она блокируется до готовности значения FlowVar, затем вызывает `action` с этим значением **с семантикой передачи по ссылке** (используя правила передачи параметров Nim). Когда `action` возвращается, FlowVar финализируется.

Для типов вроде `string` или `seq` использование `awaitAndThen` может быть эффективнее `^`, потому что значение не нужно глубоко копировать во фрейм вызывающего; вместо этого action получает прямой доступ к результату рабочего потока.

**Ограничение:** `awaitAndThen` нельзя использовать с `FlowVar[ref T]` — это принудительно проверяется как ошибка компиляции. Для ссылочных типов используйте `unsafeRead`.

#### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `fv` | `FlowVar[T]` | FlowVar, результата которого нужно дождаться. |
| `action` | `proc (x: T) {.closure.}` | Замыкание, вызываемое с результатом, как только он готов. |

#### Примеры

```nim
import threadpool

proc buildReport(n: int): string {.gcsafe.} =
  result = "Отчёт для элемента " & $n & ": " & $(n * n)

let fv = spawn buildReport(7)

awaitAndThen(fv) do (report: string):
  # 'report' доступен здесь без лишнего глубокого копирования
  echo report   # → "Отчёт для элемента 7: 49"
```

### `unsafeRead`

```nim
proc unsafeRead*[T](fv: FlowVar[ref T]): ptr T
```

#### Описание

`unsafeRead` — лазейка для `FlowVar[ref T]`, то есть для FlowVar, содержащих ссылочные типы. Обычный оператор `^` для ссылочных типов выполняет глубокое копирование, что зачастую нежелательно. `unsafeRead` блокируется до готовности значения, затем возвращает **сырой указатель** прямо на размещённый в куче объект рабочего потока.

Это называется **небезопасным** не зря: получив указатель, вы обходите отслеживание сборщика мусора. Объект, на который указывает указатель, должен оставаться живым всё время его использования. На практике это означает, что указатель нужно использовать немедленно и не сохранять за пределами текущей области видимости.

Используйте `unsafeRead` только тогда, когда вам нужен доступ к большому размещённому в куче результату без копирования, и вы понимаете последствия для времени жизни объекта.

#### Примеры

```nim
import threadpool

type BigData = ref object
  values: seq[int]

proc computeBigData(): BigData {.gcsafe.} =
  result = BigData(values: @[1, 2, 3, 4, 5])

let fv = spawn computeBigData()
let p = unsafeRead(fv)   # ptr BigData — без глубокого копирования
echo p[].values           # → @[1, 2, 3, 4, 5]
# НЕ храните 'p' после этой точки
```

### `blockUntil` (для FlowVarBase)

```nim
proc blockUntil*(fv: var FlowVarBaseObj)
```

#### Описание

`blockUntil` — низкоуровневый примитив ожидания, лежащий в основе `^`, `awaitAndThen` и `unsafeRead`. Он блокирует вызывающий поток до тех пор, пока семафор FlowVar не будет просигнализирован (то есть пока рабочий поток не запишет результат).

В большинстве прикладного кода вы не будете вызывать его напрямую — используйте `^` или `awaitAndThen`. `blockUntil` полезен прежде всего при прямой работе с `FlowVarBase` (нетипизированным базовым классом) или при создании высокоуровневых абстракций.

Вызов `blockUntil` на FlowVar, ожидание которого уже было выполнено, является пустой операцией (флаг `awaited` защищает от двойной блокировки).

### `blockUntilAny`

```nim
proc blockUntilAny*(flowVars: openArray[FlowVarBase]): int
```

#### Описание

`blockUntilAny` принимает массив объектов `FlowVarBase` и блокируется до тех пор, пока **хотя бы один из них** не будет готов. Возвращает **индекс** первого готового FlowVar.

Это аналог `select` или `poll` из сетевого программирования: вместо ожидания конкретного результата вы ждёте того, какая из набора параллельных задач завершится первой.

**Важные оговорки:**
- Если FlowVar уже готов в момент вызова `blockUntilAny`, его индекс возвращается немедленно без блокировки.
- Каждый FlowVar может участвовать только в **одном** вызове `blockUntilAny` одновременно. Если вызвать `blockUntilAny([a, b])` и `blockUntilAny([b, c])` параллельно, второй вызов отнесётся к `b` так, как если бы его не было в списке (обнаружение конфликта).
- Если все FlowVar конфликтуют (за ними уже следит кто-то другой), возвращается `-1`.
- Документация явно предупреждает, что это приводит к **недетерминированному поведению**, и рекомендует избегать этого подхода.

#### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `flowVars` | `openArray[FlowVarBase]` | Набор FlowVar для наблюдения. Должны быть приведены к `FlowVarBase` из `FlowVar[T]`. |

#### Возвращаемое значение

Индекс первого готового FlowVar или `-1`, если ожидать было нечего.

#### Примеры

```nim
import threadpool

proc slowTask(id, ms: int): int {.gcsafe.} =
  sleep(ms)
  result = id

let fv0: FlowVarBase = spawn slowTask(0, 300)
let fv1: FlowVarBase = spawn slowTask(1, 100)
let fv2: FlowVarBase = spawn slowTask(2, 200)

let first = blockUntilAny([fv0, fv1, fv2])
echo "Первым завершился: task ", first   # → скорее всего 1 (100 мс)
```

### `isReady`

```nim
proc isReady*(fv: FlowVarBase): bool
```

#### Описание

`isReady` — **неблокирующая** проверка, которая возвращает `true`, если FlowVar уже содержит готовое значение, и `false`, если рабочий поток ещё вычисляет его. Вызов `^` или `awaitAndThen` на FlowVar, для которого `isReady` вернул `true`, завершится немедленно без блокировки.

Это полезно для паттернов опроса, при которых нужно проверять несколько FlowVar в цикле и обрабатывать те, что уже готовы, не привязываясь к ожиданию какого-то конкретного.

Обратите внимание: то, что `isReady` вернул `false` в один момент, не означает, что он останется `false` мгновение спустя. Это мгновенный снимок состояния.

#### Примеры

```nim
import threadpool

proc longTask(n: int): int {.gcsafe.} =
  sleep(500)
  result = n * 2

let fv = spawn longTask(21)

while not isReady(fv):
  echo "Всё ещё вычисляется..."
  sleep(100)

echo "Результат: ", ^fv   # → 42
```

### `setMinPoolSize`

```nim
proc setMinPoolSize*(size: range[1..MaxThreadPoolSize])
```

#### Описание

Устанавливает минимальное количество потоков, которое пул будет поддерживать. По умолчанию — `4`. Пул не сократится ниже этого значения, даже когда потоки простаивают.

Увеличение минимума полезно в приложениях, чувствительных к задержке, где хочется, чтобы потоки всегда были готовы без накладных расходов на их создание по требованию. Уменьшение полезно в средах с ограниченной памятью.

Нужно вызывать до порождения задач, чтобы оказать значимый эффект на начальную конфигурацию пула.

#### Примеры

```nim
import threadpool

setMinPoolSize(8)   # Держать не менее 8 потоков живыми постоянно
spawn myTask()
```

### `setMaxPoolSize`

```nim
proc setMaxPoolSize*(size: range[1..MaxThreadPoolSize])
```

#### Описание

Устанавливает максимальное количество потоков, до которого пул может вырасти. По умолчанию — `MaxThreadPoolSize` (256). Пул никогда не превысит это число потоков.

Если вызвать `setMaxPoolSize` со значением, меньшим текущего размера пула, лишние потоки помечаются для завершения. Они не прерываются принудительно — каждому позволяют завершить текущую задачу и затем корректно выйти.

Используйте это для ограничения потребления ресурсов в средах, где большое количество параллельных потоков вредно (например, ограниченная память, общий хостинг или предотвращение перегрузки на системе с малым числом ядер).

#### Примеры

```nim
import threadpool

# На 4-ядерной машине ограничиваем пул, чтобы избежать перегрузки
setMaxPoolSize(4)
```

### `preferSpawn`

```nim
proc preferSpawn*(): bool
```

#### Описание

`preferSpawn` — быстрая неблокирующая подсказка, сообщающая, приведёт ли порождение новой задачи к реальному параллелизму. Возвращает `true`, если хотя бы один рабочий поток в данный момент простаивает (счётчик глобального семафора готовности положителен), и `false`, если все рабочие потоки кажутся занятыми.

Вызывать напрямую нужно редко. Это механизм, лежащий в основе шаблона `spawnX`. Используйте его, если реализуете собственную адаптивную логику диспетчеризации.

#### Примеры

```nim
import threadpool

proc myWork(n: int): int {.gcsafe.} = n * n

if preferSpawn():
  let fv = spawn myWork(10)
  echo ^fv
else:
  echo myWork(10)  # Выполнить встроенно; пул занят
```

## Экспортируемые константы и типы

### `MaxThreadPoolSize`

```nim
const MaxThreadPoolSize* {.intdefine.} = 256
```

Жёсткий верхний предел того, сколько рабочих потоков пул может когда-либо содержать. Можно переопределить во время компиляции: `--define:MaxThreadPoolSize=N`.

---

### `MaxDistinguishedThread`

```nim
const MaxDistinguishedThread* {.intdefine.} = 32
```

Максимальное количество «выделенных» (закреплённых) потоков, доступных для `pinnedSpawn`. Можно переопределить: `--define:MaxDistinguishedThread=N`.

---

### `ThreadId`

```nim
type ThreadId* = range[0..MaxDistinguishedThread-1]
```

Диапазонный тип для идентификации выделенного потока при `pinnedSpawn`. Допустимые значения: от `0` до `MaxDistinguishedThread - 1`.

---

### `FlowVarBase`

```nim
type FlowVarBase* = ref FlowVarBaseObj
```

Нетипизированный базовый класс для всех экземпляров `FlowVar[T]`. Используется в API, которым нужно работать с FlowVar вне зависимости от типа результата: `blockUntilAny`, `isReady`.

---

### `FlowVar[T]`

```nim
type FlowVar*[T] = ref FlowVarObj[T]
```

Типизированная переменная потока данных. Возвращается `spawn`, когда порождённая процедура имеет ненулевой тип возврата. Предоставляет типизированный интерфейс для получения результата через `^` или `awaitAndThen`. Наследуется от `FlowVarBase`.

## Настройка времени компиляции

| Флаг | По умолчанию | Эффект |
|------|--------------|--------|
| `--threads:on` | *(обязателен)* | Обязателен. Без него модуль отказывается компилироваться. |
| `--define:MaxThreadPoolSize=N` | `256` | Максимальное общее число потоков в пуле. |
| `--define:MaxDistinguishedThread=N` | `32` | Максимальное число закреплённых/выделенных потоков. |
| `--define:threadpoolWaitMs=N` | `100` | Интервал опроса (мс), используемый `sync` и логикой завершения. |
| `--define:nimRecursiveSpawn` | *(выключен)* | Включает рекурсивный spawn: рабочий поток, который ждёт, может сам выполнить ожидающую задачу, предотвращая взаимоблокировку в рекурсивных параллельных алгоритмах. |
| `--define:nimPinToCpu` | *(выключен)* | Включает привязку потоков к ядрам CPU: каждый рабочий поток закрепляется за конкретным ядром. |

## Типичные паттерны и подводные камни

**Паттерн: разветвление / слияние (fan-out / fan-in)**

Порождаем много задач, собираем все FlowVar в последовательность, затем читаем их все.

```nim
import threadpool

proc square(n: int): int {.gcsafe.} = n * n

var fvs = newSeq[FlowVar[int]](10)
for i in 0..<10:
  fvs[i] = spawn square(i)

var results = newSeq[int](10)
for i in 0..<10:
  results[i] = ^fvs[i]

echo results   # → @[0, 1, 4, 9, 16, 25, 36, 49, 64, 81]
```

**Подводный камень: двойное чтение FlowVar**

Вызов `^` второй раз на одном FlowVar — неопределённое поведение. Внутреннее состояние потребляется при первом чтении.

**Подводный камень: безопасность GC**

Все процедуры, передаваемые в `spawn` или `pinnedSpawn`, должны быть `{.gcsafe.}`. Разделение изменяемых данных под управлением GC между потоками без явной блокировки приведёт к повреждению памяти или аварийному завершению.

**Подводный камень: `sync` — грубый инструмент**

`sync()` ждёт, пока каждый поток в пуле не станет свободным. Если другая часть вашей программы тоже порождала работу, `sync` будет ждать и её. Предпочитайте чтение результатов через FlowVar, когда нужна тонкозернистая синхронизация.
