# Модуль Cpuload — Справочник

> ⚠️ **Нестабильное API.** Модуль входит в стандартную библиотеку Nim, но явно помечен как имеющий нестабильный интерфейс. Сигнатуры и поведение могут меняться между версиями Nim.

---

## Что делает этот модуль

`cpuload` отвечает на единственный вопрос, который пул потоков должен периодически задавать себе: с учётом текущей нагрузки на систему, стоит ли пулу **вырасти** (добавить поток), **сократиться** (завершить поток) или **остаться прежним**?

Это не универсальная библиотека мониторинга CPU. Это узконаправленный советник, предназначенный для вызова изнутри цикла планирования пула потоков — конкретно из `nimSpawn3` модуля `threadpool`. Весь его дизайн ориентирован на одну точку принятия решения: поможет ли запуск большего числа задач прямо сейчас, или система уже насыщена?

Модуль намеренно минимален. Он предоставляет два типа и одну процедуру. Всё остальное — платформозависимые внутренности.

> **Текущий охват платформ:** Адаптивная логика полностью реализована только на **Windows** и **Linux**. На всех остальных платформах `advice` всегда возвращает `doNothing` — пул не растёт и не сокращается на основе нагрузки; потоки поддерживаются живыми только минимальным размером пула.

---

## Ключевая концепция: состоятельная инкрементальная выборка

Главное конструктивное ограничение: `advice` вызывается **внутри горячего цикла планирования** — цикла, срабатывающего каждый раз, когда нужно отправить новую задачу, но ни один рабочий поток не свободен немедленно. Этот цикл должен быть быстрым.

Чтобы избежать дорогих системных вызовов на каждой итерации, `advice` использует **счётчик вызовов** (`state.calls`) для пропуска тяжёлого измерения большую часть времени. Реальная проверка выполняется лишь когда `calls` меньше `maxPoolSize` (фаза «прогрева») или когда `(calls and 127) == 0` (периодическая проверка примерно раз в 128 вызовов). Этим управляет вызывающий код (`nimSpawn3` из `threadpool`), а не сам `cpuload`.

На Windows измерение инкрементальное: `ThreadPoolState` хранит показания CPU-времени с предыдущего вызова, и `advice` вычисляет **разницу** с тех пор. Это означает, что первый вызов после инициализации состояния всегда является холостым (только записывает базовую линию), а второй вызов даёт первое реальное измерение.

## Экспортируемые типы

### `ThreadPoolAdvice`

```nim
type ThreadPoolAdvice* = enum
  doNothing,
  doCreateThread,
  doShutdownThread
```

#### Описание

Перечисление, представляющее три возможные рекомендации, которые может вернуть `advice`. Это выходной тип всего модуля — всё остальное существует ради производства одного из этих трёх значений.

| Значение | Смысл |
|----------|-------|
| `doNothing` | Текущий размер пула подходящий. Не расти и не сокращаться. |
| `doCreateThread` | Нагрузка на систему достаточно низкая, чтобы добавление рабочего потока, вероятно, улучшило пропускную способность. |
| `doShutdownThread` | Слишком много активных потоков относительно доступной работы; завершение одного снизит конкуренцию и накладные расходы. |

Получатель рекомендации (`threadpool`) не обязан следовать ей безоговорочно. Он проверяет дополнительные условия (например, `currentPoolSize < maxPoolSize`) перед выполнением `doCreateThread`, и аналогично для `doShutdownThread`. Рекомендация — подсказка, а не команда.

---

### `ThreadPoolState`

```nim
type ThreadPoolState* = object
  when defined(windows):
    prevSysKernel, prevSysUser, prevProcKernel, prevProcUser: FILETIME
  calls*: int
```

#### Описание

Небольшой объект состояния, который `advice` использует для передачи информации между вызовами. Он должен быть создан вызывающим кодом, инициализирован нулевым значением (что обеспечивает как `default(ThreadPoolState)`, так и объявление `var` на уровне модуля) и передан по `var`-ссылке в каждый вызов `advice`.

Объект имеет два слоя:

**`calls`** (экспортируется, все платформы): простой счётчик, увеличивающийся на единицу при каждом вызове `advice`. Вызывающий код (или сам модуль) использует это число, чтобы решить, как часто должно реально выполняться дорогостоящее системное измерение.

**`prevSys*` / `prevProc*`** (только Windows, не экспортируются): четыре значения `FILETIME`, фиксирующие время ядра и пользовательское время системы и текущего процесса на момент последнего измерения. Используются для вычисления дельт CPU-времени между последовательными вызовами.

На не-Windows платформах блок `when defined(windows)` отсутствует, поэтому `ThreadPoolState` содержит только `calls`. Объект остаётся корректным и по-прежнему должен передаваться в `advice`; он просто не несёт данных о времени.

**Время жизни:** единственный экземпляр `ThreadPoolState` должен существовать в течение всего времени жизни пула потоков. Повторная инициализация (например, присвоение нового `default(ThreadPoolState)`) сбрасывает счётчик вызовов и уничтожает базовую линию тайминга на Windows, из-за чего следующее Windows-измерение будет пропущено (первый вызов после инициализации всегда только записывает базовую линию).

## Экспортируемая процедура

### `advice`

```nim
proc advice*(s: var ThreadPoolState): ThreadPoolAdvice
```

#### Описание

Инспектирует текущую нагрузку CPU системы и возвращает значение `ThreadPoolAdvice`, показывающее, должен ли пул потоков расти, сокращаться или оставаться прежним. Объект состояния `s` обновляется при каждом вызове: как минимум, инкрементируется `s.calls`.

#### Как работает на каждой платформе

**Windows:**

Использует Win32-функции `GetSystemTimes` и `GetProcessTimes` для получения CPU-времени в единицах 100 наносекунд:
- `GetSystemTimes` возвращает три значения `FILETIME`: время простоя, время ядра (включает простой) и пользовательское время.
- `GetProcessTimes` возвращает четыре значения `FILETIME` для текущего процесса: время создания, завершения, ядра и пользовательское.

При первом вызове (`s.calls == 0`) функция записывает текущие значения как базовую линию и возвращает `doNothing`.

При последующих вызовах вычисляется:
- `sysTotal` = (sysKernel − prevSysKernel) + (sysUser − prevSysUser) — суммарное CPU-время, затраченное системой (по всем ядрам) с момента последнего измерения.
- `procTotal` = (procKernel − prevProcKernel) + (procUser − prevProcUser) — CPU-время, потреблённое **текущим процессом** с момента последнего измерения.

Если `procTotal / sysTotal < 0.85` (процесс использовал менее 85% суммарного CPU-времени), возвращается `doCreateThread`. Порог 85% выбран потому, что практические измерения показывают: настоящая 100%-ная утилизация почти никогда не регистрируется, даже когда все ядра заняты.

Если любой снимок времени не удаётся (Win32-вызовы возвращают 0), функция безопасно возвращает `doNothing`.

**Linux:**

Читает `/proc/loadavg`, который ядро Linux обновляет примерно раз в 5 секунд. Файл имеет формат:

```
<1-мин> <5-мин> <15-мин> <выполняющихся>/<всего> <последний-pid>
```

Функция читает поля `<выполняющихся>/<всего>`: `busy` — количество процессов, готовых к выполнению прямо сейчас (на CPU или в очереди готовности), `total` — общее количество процессов/потоков в системе.

Решение принимается на основе сравнения `busy - 1` (вычитание 1 для исключения самого текущего процесса) с количеством логических процессоров (`countProcessors()`):

| Условие | Возвращаемая рекомендация |
|---------|--------------------------|
| `busy - 1 < cpus` | `doCreateThread` — задач меньше, чем ядер; ядра недозагружены |
| `busy - 1 >= cpus * 2` | `doShutdownThread` — задач вдвое больше ядер; система перегружена |
| иначе | `doNothing` — нагрузка в допустимом диапазоне |

Если `/proc/loadavg` не удаётся открыть, функция безопасно возвращает `doNothing`.

**Все остальные платформы:**

Всегда возвращает `doNothing`. Пул не растёт и не сокращается на основе нагрузки. Размер пула определяется исключительно `minPoolSize` и `maxPoolSize` и тем, есть ли свободные рабочие потоки.

#### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `s` | `var ThreadPoolState` | Объект состояния. Должен сохраняться между вызовами. Всегда изменяется: как минимум инкрементируется `s.calls`. |

#### Возвращаемое значение

Значение `ThreadPoolAdvice`. Смысл каждого значения — в описании типа выше.

#### Примеры

```nim
import cpuload

var state: ThreadPoolState
# state инициализирован нулями; state.calls == 0

# Первый вызов на Windows: записывает базовую линию, возвращает doNothing
# Первый вызов на Linux: читает /proc/loadavg немедленно
echo advice(state)   # → doNothing (базовая линия Windows) или реальное значение (Linux)
echo state.calls     # → 1

# Последующие вызовы дают реальные рекомендации
for i in 1..10:
  let rec = advice(state)
  case rec
  of doNothing:        echo "Размер пула в норме"
  of doCreateThread:   echo "Рассмотреть добавление рабочего потока"
  of doShutdownThread: echo "Рассмотреть завершение рабочего потока"
```

```nim
# Типичный паттерн использования, отражающий то, что threadpool делает внутри:
import cpuload, os

var state: ThreadPoolState

proc schedulingLoop() =
  while true:
    # ... пробуем отправить задачу существующим рабочим потокам ...

    # Проверяем нагрузку только периодически (каждые ~128 вызовов или при прогреве)
    if state.calls < maxPoolSize or (state.calls and 127) == 0:
      case advice(state)
      of doCreateThread:
        if currentPoolSize < maxPoolSize:
          activateNewWorker()
      of doShutdownThread:
        if currentPoolSize > minPoolSize:
          signalWorkerToStop()
      of doNothing:
        discard

    os.sleep(10)
```

#### Предостережения

**Задержка `/proc/loadavg` на Linux.** Ядро обновляет `/proc/loadavg` примерно раз в 5 секунд. В промежутке между обновлениями повторные вызовы `advice` на Linux читают устаревшее значение. Это означает, что решения о нагрузке на Linux имеют врождённую задержку в 5 секунд. Для пула потоков это как правило приемлемо — создание и уничтожение потоков само по себе медленная операция.

**Пропуск первого вызова на Windows.** На Windows самый первый вызов после инициализации `ThreadPoolState` всегда возвращает `doNothing`, потому что дельту ещё не из чего вычислить. Это намеренно: базовая линия должна быть установлена прежде, чем можно провести осмысленное сравнение.

**Не автономный инструмент мониторинга.** `advice` предназначен для вызова из конкретного контекста планирования внутри `threadpool`. Использование его в отрыве (например, для построения CPU-дашборда) технически сработает, но проектные предположения (короткие интервалы между вызовами, ограничения размера пула на стороне вызывающего) не будут выполняться, и рекомендации могут вводить в заблуждение.

**Пробел в поддержке платформ.** macOS, BSD и другие платформы всегда получают `doNothing`. На этих системах размер пула определяется исключительно `minPoolSize`, `maxPoolSize` и наличием свободных рабочих потоков.

## Связь с модулем `threadpool`

`cpuload` — приватная зависимость `threadpool`. Полное взаимодействие выглядит так (из `nimSpawn3` модуля `threadpool.nim`):

```nim
# Внутри nimSpawn3 (упрощённо):
if state.calls < maxPoolSize or (state.calls and 127) == 0:
  if tryAcquire(stateLock):
    case advice(state)          # <-- здесь вызывается cpuload.advice
    of doNothing: discard
    of doCreateThread:
      if currentPoolSize < maxPoolSize:
        activateWorkerThread(currentPoolSize)
        atomicInc currentPoolSize
    of doShutdownThread:
      if currentPoolSize > minPoolSize:
        workersData[currentPoolSize-1].shutdown = true
    release(stateLock)
```

Ключевые наблюдения:
- `advice` вызывается только под `stateLock`, поэтому он никогда не вызывается конкурентно из нескольких потоков.
- Вызывающий код применяет собственные проверки (`currentPoolSize < maxPoolSize` и т.д.) перед выполнением рекомендации.
- `state` — это `var` уровня модуля в `threadpool`, существующий всё время жизни программы.
