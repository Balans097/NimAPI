# Справочник модуля `importutils`

> **Модуль:** `std/importutils` (стандартная библиотека Nim)  
> **Статус:** ⚠️ Экспериментальный API — может изменяться в будущих версиях Nim.  
> **Назначение:** Утилиты для работы с импортом и разрешением символов; в первую очередь — контролируемый доступ к закрытым (приватным) полям типов.

---

## Контекст: какую проблему решает этот модуль?

В Nim типы и их поля могут быть публичными (экспортированы с `*`) или приватными (не экспортированы). Приватные поля, по правилам языка, недоступны за пределами модуля, в котором определён тип — это намеренный механизм инкапсуляции.

Тем не менее существуют законные ситуации, когда нужно получить доступ к приватным полям извне:

- **Тестирование:** вы хотите проверить внутреннее состояние объекта, не добавляя публичный аксессор только ради тестов.
- **Совместимость:** вы работаете со сторонним типом, автор которого не предусмотрел нужный вам публичный интерфейс.
- **Инструменты отладки и инспекции:** нужно сериализовать или распечатать полное состояние объекта.
- **FFI и низкоуровневый код:** вы интегрируете Nim с внешним кодом и вам необходим полный доступ к полям.

Модуль `importutils` предоставляет ровно один инструмент для этого: `privateAccess`.

---

## Экспортируемые функции

---

### `privateAccess`

```nim
proc privateAccess*(t: typedesc) {.magic: "PrivateAccess".}
```

**Что делает**

`privateAccess` открывает доступ к приватным полям заданного типа в **текущем лексическом блоке**. После вызова поля `t`, которые в обычных условиях вызвали бы ошибку компиляции, становятся доступны как для чтения, так и для записи — но только внутри того блока, где был сделан вызов.

Это **конструкция времени компиляции**, а не времени выполнения. Прагма `{.magic: "PrivateAccess".}` означает, что процедура обрабатывается непосредственно компилятором Nim как встроенная — она не генерирует исполняемый код. Накладных расходов во время выполнения нет.

Ограничение по области видимости — намеренная мера безопасности: предоставленное разрешение на доступ не «вытекает» за пределы блока. Как только блок лексически завершается, компилятор снова считает приватные поля недоступными.

**Параметры**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `t` | `typedesc` | Тип, к приватным полям которого нужен доступ. Можно передать имя типа напрямую (например, `Goo`) или вывести тип из существующей переменной (например, `myVar.type`). |

**Возвращает:** Ничего. Эффект — исключительно разрешение на уровне компилятора.

---

### Как работает область видимости

Ключевое понятие: `privateAccess` работает **в пределах блока**. Рассмотрим структуру:

```nim
block внешний:
  # f.f0 здесь НЕ доступно

  block внутренний:
    privateAccess(MyType)
    # f.f0 здесь ДОСТУПНО
    # вложенные блоки тоже наследуют доступ:
    block глубже:
      # f.f0 здесь ТОЖЕ доступно

  # f.f0 снова НЕ доступно
```

Это поведение аналогично тому, как работают объявления `var` в Nim: объявление внутри `block` видно до конца этого блока. `privateAccess` следует тем же правилам.

---

### Два способа передать тип

**Через переменную** — удобно, когда у вас уже есть экземпляр объекта:

```nim
var f = initFoo()
privateAccess(f.type)   # f.type вычисляется в Foo во время компиляции
```

**Через имя типа напрямую** — удобно для обобщённых типов или когда экземпляра ещё нет:

```nim
privateAccess(Goo)           # конкретный тип
privateAccess(Goo[float])    # инстанцированный обобщённый тип
```

Оба способа эквивалентны по результату.

---

### Пример: доступ к приватным полям конкретного типа

```nim
# Предположим, в другом модуле определено:
# type
#   Foo = object
#     f0: int  # приватное — не экспортировано
# proc initFoo*(): auto = Foo()

import std/importutils
import mymodule  # предоставляет initFoo

var f = initFoo()

# Без privateAccess — ошибка компиляции:
# f.f0 = 1  # Error: undeclared field 'f0'

block:
  privateAccess(f.type)   # открываем доступ к приватным полям Foo
  f.f0 = 1                # теперь это компилируется и работает
  assert f.f0 == 1        # доступ на чтение тоже работает

# За пределами блока — доступ снова закрыт:
# f.f0 = 2  # снова ошибка компиляции
assert not compiles(f.f0)
```

Строка `assert not compiles(f.f0)` — это утверждение времени компиляции: она проверяет, что доступ *действительно* не работает за пределами блока. Такой паттерн используется в собственных тестах модуля для документирования гарантии области видимости.

---

### Пример: доступ к приватным полям обобщённого типа

Обобщённые типы работают так же — нужно передать имя типа (с параметрами типа или без):

```nim
# Предположим:
# type
#   Goo*[T] = object
#     g0: int  # приватное

import std/importutils

privateAccess(Goo)                          # открываем доступ для любого T
assert Goo[float](g0: 1).g0 == 1           # создаём и читаем приватное поле

privateAccess(Goo[int])                     # альтернативно — конкретная инстанция
var x = Goo[int](g0: 42)
assert x.g0 == 42
```

Обратите внимание: хотя сам тип `Goo` экспортирован (имеет `*`), его поле `g0` — нет. `privateAccess` устраняет это ограничение для обобщённых типов так же легко, как и для конкретных.

---

### Пример: использование `privateAccess` в тестах

Очень распространённый и идиоматичный сценарий — «белый ящик»-тестирование, когда нужно проверить внутреннее состояние без загрязнения публичного API аксессорами, нужными только тестам:

```nim
# tests/test_foo.nim
import std/importutils
import mymodule

proc testFooInternals() =
  var f = initFoo()
  doSomethingWith(f)

  block проверкаВнутреннегоСостояния:
    privateAccess(Foo)
    # Проверяем, что внутренний счётчик обновился корректно:
    assert f.f0 == 42, "Ожидалось f0 == 42 после обработки"

testFooInternals()
```

Тестовый файл — это отдельный модуль, поэтому в обычных условиях он не видит `f0`. С `privateAccess` доступ открыт — но только внутри тестового блока, и остальной код остаётся чистым.

## Важные замечания и предостережения

**`privateAccess` — инструмент для намеренных, контролируемых исключений, а не способ игнорировать инкапсуляцию.** Использование этой функции в логике приложения — признак архитектурной проблемы. Поле, к которому вы обращаетесь, может быть изменено, переименовано или удалено в будущей версии библиотеки, и никакого предупреждения об устаревании не последует — ведь оно никогда не было частью публичного контракта.

**Область видимости — лексическая, а не динамическая.** Разрешение на доступ не передаётся вызовам функций из блока. Если внутри блока вы вызываете `someProc()`, тело `someProc` автоматически не получает доступа к приватным полям. Только код, *текстуально* находящийся внутри блока, пользуется разрешением.

**Никаких накладных расходов во время выполнения.** `privateAccess` — полностью директива времени компиляции. Она не порождает исполняемого кода и не влияет на производительность.

**Экспериментальный статус.** Заголовок модуля помечает этот API как экспериментальный. В будущем в модуль могут быть добавлены новые утилиты для работы с символами (см. комментарии в исходнике о `whichModule`, `getCurrentPkgDir` и других запланированных функциях).

## Итоговая шпаргалка

| Аспект | Подробности |
|--------|-------------|
| Что открывает | Приватные (неэкспортированные) поля любого объектного типа Nim |
| Когда действует доступ | Только внутри лексического блока, где был вызван `privateAccess` |
| Работает с обобщёнными типами? | Да — передайте имя типа с параметрами типа или без |
| Накладные расходы | Нулевые — исключительно время компиляции |
| Типичные сценарии использования | Тестирование, отладка, интероп, низкоуровневые утилиты |
| Риск злоупотребления | Нарушает инкапсуляцию; использовать только там, где действительно необходимо |
| Стабильность API | Экспериментальный — может измениться |
