# `compilation.nim` — справочник модуля

> **Часть стандартной библиотеки Nim (модуль `system`)**  
> Все символы доступны без каких-либо импортов — они внедряются компилятором напрямую.

---

## Общее описание

`compilation.nim` предоставляет **интерфейс компилятора для интроспекции и метапрограммирования на этапе компиляции**. Каждый символ в этом модуле вычисляется при компиляции, а не во время выполнения — они позволяют коду задавать компилятору вопросы о самом себе: *Какая версия Nim используется? Определён ли этот флаг? Умеет ли этот тип выполнять такую операцию? Что содержит этот файл?*

Это фундамент условной компиляции и генерации кода во время сборки в Nim. Эти символы используются каждый раз, когда вы пишете `when defined(...)`, встраиваете файл в бинарник как константу, вызываете внешний инструмент во время сборки или пишете примеры в документации, которые автоматически проверяются.

Два важных момента, которые нужно знать с самого начала:

1. **Ничто в этом модуле не выполняется во время работы программы.** Каждая процедура и каждая константа здесь вычисляются компилятором во время прохода компиляции. В итоговом бинарнике не остаётся следов этих вызовов — только их результаты (`bool`, `string`, вычисленная константа).
2. **Многие из этих символов являются «магическими».** Прагма `{.magic: "...".}` означает, что компилятор встроил знание об этих символах на уровне собственной реализации. Тела их Nim-процедур (если они есть) — это заглушки; компилятор подставляет вместо них собственную логику.

## Константы версии

### `NimMajor*`

```nim
const NimMajor* {.intdefine.}: int = 2
```

**Мажорный** номер версии используемого компилятора Nim. Прагма `{.intdefine.}` означает, что значение можно переопределить при компиляции через `-d:NimMajor=X`, но на практике этого делать не следует — компилятор выставляет его корректно самостоятельно.

Используйте в связке с `NimMinor` и `NimPatch`, чтобы защитить код, требующий определённой минимальной версии Nim:

```nim
when (NimMajor, NimMinor, NimPatch) >= (2, 0, 0):
  echo "Запускается на Nim 2.x или новее"
else:
  echo "Запускается на Nim 1.x"
```

Сравнение кортежей в Nim лексикографическое, поэтому одна строка корректно проверяет полную трёхчастную версию.

---

### `NimMinor*`

```nim
const NimMinor* {.intdefine.}: int = 2
```

**Минорный** номер версии. По соглашению Nim: **чётные** минорные номера — стабильные релизы; **нечётные** — сборки в разработке. То есть `2.0`, `2.2`, `2.4` — релизы, а `2.1`, `2.3` — незавершённые версии.

```nim
when NimMinor mod 2 == 1:
  {.warning: "Вы собираете с разрабатываемой версией Nim.".}
```

---

### `NimPatch*`

```nim
const NimPatch* {.intdefine.}: int = 8
```

**Патч**-номер версии. Та же чётность что и у `NimMinor`: чётное = релиз, нечётное = в разработке.

```nim
# Требуем минимум 2.0.4 для конкретного исправления:
when (NimMajor, NimMinor, NimPatch) < (2, 0, 4):
  {.error: "Эта библиотека требует Nim >= 2.0.4".}
```

## Состояние компиляции

### `nimvm*`

```nim
let nimvm* {.magic: "Nimvm", compileTime.}: bool = false
```

Специальный булев флаг, который равен **`true` внутри виртуальной машины Nim** (то есть внутри блоков `static:`, инициализаторов `const` и вызовов процедур времени компиляции) и **`false` в обычном компилируемом коде**. Может использоваться только в выражениях `when` — никогда в обычных `if` времени выполнения.

Это стандартный способ написать код, который ведёт себя по-разному в зависимости от того, выполняется ли он при компиляции или во время работы программы:

```nim
proc serialize(x: int): string =
  when nimvm:
    # Путь времени компиляции: системные вызовы недоступны
    $x
  else:
    # Путь времени выполнения: можно использовать полное форматирование
    formatInt(x, padding = 10)
```

Прагма `{.profiler: off.}` вокруг объявления гарантирует, что этот символ не появится в выводе профилировщика.

---

### `isMainModule*`

```nim
const isMainModule* {.magic: "IsMainModule".}: bool = false
```

Равно `true` **только в том модуле, который был передан компилятору напрямую** — то есть в точке входа. В любом импортированном модуле равно `false`.

Это идиоматический аналог Python-конструкции `if __name__ == "__main__":` в Nim. Самое частое применение — встраивание самотестирования прямо в библиотечный модуль, чтобы тесты запускались при самостоятельной компиляции модуля, но безмолвно пропускались при его импорте:

```nim
# В файле mylib.nim:
proc add(a, b: int): int = a + b

when isMainModule:
  assert add(2, 3) == 5
  echo "Все тесты прошли"
  # Этот блок полностью отсутствует в бинарнике,
  # когда mylib импортируется другим модулем.
```

---

### `CompileDate*`

```nim
const CompileDate* {.magic: "CompileDate".}: string = "0000-00-00"
```

**Дата компиляции бинарника по UTC** в виде строки формата `YYYY-MM-DD`. Устанавливается компилятором во время сборки. Значение-заглушка `"0000-00-00"` на практике никогда не встречается — компилятор заменяет его.

```nim
const buildInfo = "Собрано: " & CompileDate
echo buildInfo   # например: "Собрано: 2025-07-15"
```

Полезно для встраивания метаданных сборки в исполняемые файлы, строки версий или вывод логов.

---

### `CompileTime*`

```nim
const CompileTime* {.magic: "CompileTime".}: string = "00:00:00"
```

**Время компиляции бинарника по UTC** в виде строки формата `HH:MM:SS`.

```nim
const versionBanner = "v1.0 собрано " & CompileDate & " в " & CompileTime
```

Вместе `CompileDate` и `CompileTime` позволяют встроить точную метку времени сборки в любой бинарник. Обратите внимание: их использование делает сборки невоспроизводимыми (бинарник меняется при каждой компиляции) — это осознанный компромисс.

## Процедуры интроспекции времени компиляции

### `defined`

```nim
proc defined*(x: untyped): bool {.magic: "Defined", noSideEffect, compileTime.}
```

Проверяет, был ли **символ** `x` определён через флаг компилятора `-d:x`, прагму `{.define.}` или является встроенным символом платформы/ОС.

Это основной механизм **условной компиляции** в Nim. Имя символа передаётся не строкой — а как голый идентификатор прямо компилятору:

```nim
when defined(release):
  # Оптимизированная сборка — пропускаем дорогие проверки
  discard
else:
  # Отладочная сборка — полная валидация
  doExpensiveCheck()
```

Компилятор предоставляет множество встроенных символов, которые можно проверять без какого-либо флага `-d:`:

| Символ | Истина когда… |
|---|---|
| `windows` | Сборка под Windows |
| `linux` | Сборка под Linux |
| `macosx` | Сборка под macOS |
| `posix` | Сборка под любую POSIX-систему |
| `amd64` | 64-битная архитектура x86 |
| `i386` | 32-битная архитектура x86 |
| `arm` | Архитектура ARM |
| `release` | Передан флаг `-d:release` |
| `debug` | Без флагов release/danger |
| `danger` | Передан флаг `-d:danger` |
| `gcArc` | Используется управление памятью ARC |
| `gcOrc` | Используется управление памятью ORC |
| `nimscript` | Выполнение внутри NimScript |

```nim
when defined(linux) and defined(amd64):
  echo "64-битный Linux"

# Пользовательские флаги:
# Компиляция: nim c -d:myFeature app.nim
when defined(myFeature):
  includeMyFeature()
```

### `declared`

```nim
proc declared*(x: untyped): bool {.magic: "Declared", noSideEffect, compileTime.}
```

Проверяет, был ли идентификатор `x` **объявлен в любом видимом месте** от текущей точки (то есть в любом импортированном модуле или любой охватывающей области видимости). `x` должен быть идентификатором или квалифицированным именем вида `модуль.символ`.

Канонический сценарий использования — написание кода, который корректно адаптируется к тому, что предоставляет библиотека, без жёстких ошибок компиляции при отсутствии символа:

```nim
when not declared(strutils.toUpper):
  proc toUpper(s: string): string =
    # Предоставляем собственную реализацию-заглушку
    result = s
    for c in result.mitems: c = c.toUpperAscii()
```

Ещё один распространённый паттерн — обнаружение возможностей в кроссплатформенном коде:

```nim
when declared(os.getEnv):
  let home = os.getEnv("HOME")
else:
  let home = ""
```

---

### `declaredInScope`

```nim
proc declaredInScope*(x: untyped): bool {.magic: "DeclaredInScope", noSideEffect, compileTime.}
```

Как `declared`, но **строже**: возвращает `true` только если `x` объявлен именно в **текущей лексической области видимости** — не в охватывающей и не в импортированном модуле. `x` должен быть простым (неквалифицированным) идентификатором.

Полезно в макросах и обобщённом коде, когда нужно проверить, установлена ли конкретная локальная привязка именно в точке вычисления:

```nim
block:
  when declaredInScope(result):
    echo "мы внутри процедуры, 'result' доступен"

# Вне процедуры:
when not declaredInScope(result):
  echo "здесь нет неявного 'result'"
```

### `compiles`

```nim
proc compiles*(x: untyped): bool {.magic: "Compiles", noSideEffect, compileTime.}
```

Пытается **проверить типы и скомпилировать выражение `x`** без генерации какого-либо реального кода и возвращает `true`, если это удаётся без ошибок, `false` — в противном случае. Выражение никогда не исполняется.

Это основной инструмент Nim для **проверок в стиле утиной типизации** — тестирование того, поддерживает ли тип определённую операцию, до того как её фактически использовать:

```nim
proc makeNoise[T](x: T) =
  when compiles(x.speak()):
    x.speak()
  elif compiles(x.sound):
    echo x.sound
  else:
    echo "..."
```

Практическое применение с обобщёнными типами — проверяем до вызова, а не после:

```nim
proc tryAdd[T](a, b: T): T =
  when compiles(a + b):
    a + b
  else:
    {.error: "Тип " & $T & " не поддерживает '+'".}
```

Тело `compiles` — это `discard`-заглушка; вся логика скрыта в «магии» компилятора.

### `astToStr`

```nim
proc astToStr*[T](x: T): string {.magic: "AstToStr", noSideEffect.}
```

Преобразует **исходный текст AST-представления выражения `x`** в строку — в том виде, в каком оно записано в исходном коде. Это чисто отладочный и диагностический инструмент: он показывает то, что видит компилятор, а не значение `x` во время выполнения.

```nim
let a = 42
echo astToStr(a)           # "a"
echo astToStr(1 + 2 * 3)   # "1 + 2 * 3"
echo astToStr(isMainModule) # "isMainModule"
```

Наиболее полезно внутри макросов и прагм `{.error.}` или `{.warning.}`, когда нужно включить оригинальный текст выражения пользователя в диагностическое сообщение:

```nim
macro mustBePositive(x: int): untyped =
  if x.intVal <= 0:
    error("Ожидалось положительное значение, получено: " & astToStr(x))
```

## Документационная утилита

### `runnableExamples`

```nim
proc runnableExamples*(rdoccmd = "", body: untyped) {.magic: "RunnableExamples".}
```

Помечает блок кода как **запускаемый пример документации**. Поведение различается в зависимости от контекста:

- **В обычных сборках (debug/release):** тело полностью игнорируется и исключается из кода — нулевые накладные расходы во время выполнения.
- **При генерации документации `nim doc`:** каждый блок `runnableExamples` извлекается, помещается в отдельный временный `.nim`-файл, компилируется и запускается. Если пример падает — сборка документации завершается с ошибкой. Это гарантирует, что примеры в документации всегда корректны и актуальны.

Необязательный параметр `rdoccmd` передаёт дополнительные флаги компилятора при сборке конкретного примера:

```nim
proc myProc*(x: int): string =
  ## Преобразует целое число в строку.
  ##
  runnableExamples:
    # Базовое использование — компилируется и запускается в `nim doc`
    assert myProc(42) == "42"
    assert myProc(0)  == "0"

  runnableExamples "-d:myFlag":
    # Компилируется только при установленном -d:myFlag
    when defined(myFlag):
      assert myProc(1) == "1"

  runnableExamples "-r:off":
    # Компилируется, но НЕ запускается (полезно для примеров, открывающих
    # браузер, запускающих процессы или имеющих побочные эффекты):
    import std/browsers
    openDefaultBrowser("https://nim-lang.org")

  $x
```

Примеры помещаются в собственную область видимости модуля, поэтому не могут случайно обращаться к неэкспортированным символам — это позволяет обнаруживать ошибки в документации, где пример работает только из-за внутреннего доступа.

## Запросы параметров компилятора

### `compileOption` (вкл/выкл)

```nim
proc compileOption*(option: string): bool {.magic: "CompileOption", noSideEffect.}
```

Запрашивает текущее значение **булевой опции компилятора (`on`/`off`)** по имени. Возвращает `true`, если опция в данный момент включена (`on`). Имя опции — строчная версия переключателя компилятора (без `--`).

Это полезно, когда нужно адаптировать код в зависимости от того, какие флаги оптимизации или безопасности активны в месте вызова — опции можно даже локально включать и выключать через `{.push.}` / `{.pop.}`:

```nim
proc safeDivide(a, b: float): float =
  when compileOption("infchecks"):
    if b == 0.0: raise newException(DivByZeroDefect, "деление на ноль")
  a / b
```

Тестирование локальных изменений опций:

```nim
static: doAssert not compileOption("floatchecks")

{.push floatChecks: on.}
static: doAssert compileOption("floatchecks")
{.pop.}

static: doAssert not compileOption("floatchecks")
```

Часто запрашиваемые булевы опции: `assertions`, `boundchecks`, `floatchecks`, `infchecks`, `nanchecks`, `overflowchecks`, `rangechecks`, `stacktrace`, `linetrace`, `threads`.

---

### `compileOption` (перечисление)

```nim
proc compileOption*(option, arg: string): bool {.magic: "CompileOptionArg", noSideEffect.}
```

Запрашивает **многозначную (перечислимую) опцию компилятора** — проверяет, установлена ли опция `option` в значение `arg`. Оба аргумента являются строками.

```nim
when compileOption("opt", "size"):
  echo "Оптимизация по размеру бинарника"

when compileOption("opt", "speed"):
  echo "Оптимизация по скорости"

when compileOption("gc", "orc"):
  echo "Используется сборщик мусора ORC"
```

Это правильный инструмент, когда `compileOption(option)` (вкл/выкл) недостаточно, потому что опция имеет более двух возможных значений. Часто запрашиваемые перечислимые опции: `opt` (`none`, `speed`, `size`), `gc` (`refc`, `arc`, `orc`, `boehm`, `go`, `none`), `backend` (`c`, `cpp`, `js`, `objc`), `exceptionSystem` (`setjmp`, `cpp`, `goto`, `quirky`).

## Интроспекция путей

### `currentSourcePath`

```nim
template currentSourcePath*: string = instantiationInfo(-1, true).filename
```

Раскрывается в **полный путь в файловой системе** к исходному файлу, в котором вызван этот шаблон — на этапе компиляции. Поскольку это шаблон (а не процедура), он инстанциируется в месте вызова, поэтому путь всегда соответствует вызывающему файлу, а не `compilation.nim`.

```nim
echo currentSourcePath
# например: "/home/user/myproject/mylib.nim"

# Получить директорию текущего исходного файла:
import std/os
const thisDir = currentSourcePath.parentDir()
const dataFile = thisDir / "data" / "config.json"
```

Основное применение — построение **путей времени компиляции относительно исходного файла**, чтобы встраиваемые ресурсы находились корректно независимо от того, откуда запускается компилятор. Это надёжнее, чем жёстко прописывать абсолютный путь или полагаться на рабочую директорию.

## Встраивание файлов и процессов

### `staticRead` / `slurp`

```nim
proc staticRead*(filename: string): string {.magic: "Slurp".}
proc slurp*(filename: string): string {.magic: "Slurp".}
```

Читает файл **во время компиляции** и возвращает его содержимое как строковую константу. Файл встраивается непосредственно в скомпилированный бинарник — итоговая программа не имеет зависимости от этого файла во время выполнения. `slurp` — это просто псевдоним для `staticRead`; они идентичны.

Имя файла разрешается относительно **исходного файла**, содержащего вызов (не рабочей директории на момент компиляции).

```nim
const htmlTemplate = staticRead("templates/index.html")
const privateKey    = slurp("certs/server.pem")
const wordList      = staticRead("data/words.txt")
```

Весь файл становится строковой константой времени компиляции. Максимальный размер встраиваемого файла ограничен объёмом свободной памяти на сборочной машине — для очень больших файлов это может быть практическим ограничением.

```nim
# В связке с currentSourcePath для надёжности:
const schema = staticRead(currentSourcePath.parentDir() / "schema.sql")
```

### `staticExec` / `gorge`

```nim
proc staticExec*(command: string, input = "", cache = ""): string {.magic: "StaticExec".}
proc gorge*(command: string, input = "", cache = ""): string {.magic: "StaticExec".}
```

Выполняет **внешнюю команду оболочки во время компиляции** и захватывает объединённый вывод stdout+stderr как строку. `gorge` — псевдоним для `staticExec`; они идентичны.

Это мощный инструмент для встраивания динамической информации о сборке или запуска генераторов кода как части компиляции:

```nim
const gitRevision  = staticExec("git rev-parse --short HEAD")
const buildHost    = staticExec("hostname")
const platformInfo = staticExec("uname -srm")

const versionString = "v1.0 rev:" & gitRevision & " собрано на " & buildHost
```

**Параметр `input`** передаёт строку в стандартный ввод команды:

```nim
const processed = staticExec("awk '{print toupper($0)}'", input = "hello world")
# processed == "HELLO WORLD\n"
```

**Параметр `cache`** включает кэширование: вывод сохраняется в `nimcache/` и повторно используется при последующих компиляциях, пока объединённый ключ `command & input & cache` остаётся неизменным. Используйте строку версии как ключ кэша, чтобы инвалидировать его при изменении инструмента:

```nim
const optimisedDFA = staticExec("dfaoptimizer", "my_grammar", "tool_v2.1")
# Перезапускает dfaoptimizer только при изменении входных данных или ключа версии.
# Используйте --forceBuild для полного обхода кэша.
```

Можно также использовать внутри прагм:

```nim
{.passc: staticExec("pkg-config --cflags openssl").}
{.passl: staticExec("pkg-config --libs openssl").}
```

---

### `gorgeEx`

```nim
proc gorgeEx*(command: string, input = "", cache = ""): tuple[output: string, exitCode: int]
```

Аналог `staticExec` / `gorge`, но дополнительно возвращает **код завершения** команды. Полезно, когда нужно обнаружить сбой вместо того, чтобы предполагать всегда успешное выполнение.

```nim
let (output, code) = gorgeEx("git rev-parse --short HEAD")
const gitRevision =
  if code == 0: output.strip()
  else: "unknown"
```

```nim
let (libFlags, err) = gorgeEx("pkg-config --libs libpng")
when err != 0:
  {.error: "libpng не найден — пожалуйста, установите его.".}
else:
  {.passl: libFlags.}
```

Прагма `{.noinit.}` на возвращаемом кортеже означает, что компилятор не будет обнулять его перед присваиванием — незначительная деталь производительности, не влияющая на корректность.
