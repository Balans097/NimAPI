# Справочник по модулю `ioselectors_kqueue`

> **Модуль:** `std/ioselectors` (бэкенд BSD kqueue)  
> **Язык:** Nim  
> **Платформа:** macOS, FreeBSD, NetBSD, OpenBSD, DragonFly BSD  
> **Назначение:** Реализует API `ioselectors` поверх механизма `kqueue`/`kevent` ядра BSD, обеспечивая эффективное мультиплексирование I/O-событий — одновременное наблюдение за множеством файловых дескрипторов без циклического опроса.

---

## Контекст: что такое kqueue?

Прежде чем смотреть на API, полезно понять механизм, который под ним работает.

**kqueue** — это интерфейс ядра BSD для уведомления о событиях. Вместо того чтобы спрашивать «готов ли этот сокет?» в цикле (polling), ваша программа регистрирует интерес к определённым событиям в ядре, а затем вызывает `kevent()`, который *блокируется* до тех пор, пока одно или несколько таких событий не произойдут. Ядро ждёт за вас; процесс спит и не потребляет процессорное время.

По сравнению с POSIX `select`/`poll`, kqueue умеет следить за практически неограниченным числом дескрипторов одновременно, и — что принципиально важно — следить не только за сокетами, но и за таймерами, сигналами, завершением дочерних процессов и изменениями файловой системы через единый интерфейс.

Этот модуль — BSD-реализация API Nim `std/ioselectors`. Обычно его не импортируют напрямую: `std/ioselectors` сам выбирает нужный бэкенд в зависимости от целевой ОС.

### Как изменения отправляются в ядро

Каждая функция, регистрирующая или удаляющая интерес, ставит в очередь дескриптор изменения `KEvent` (через внутренний шаблон `modifyKQueue`). Эти изменения отправляются ядру в одном из двух режимов:

- **Немедленный режим** (по умолчанию): изменения сбрасываются в ядро через `kevent()` сразу же после постановки в очередь, посредством `flushKQueue`.
- **Режим кэша** (флаг компиляции `-d:kqcache`): изменения накапливаются и передаются в ядро единым пакетным вызовом внутри `selectInto`. Это снижает накладные расходы на системные вызовы, когда перед следующим ожиданием происходит много регистраций.

## Типы

### `Selector[T]`

```nim
# однопоточная сборка:
type Selector*[T] = ref SelectorImpl[T]
# потокобезопасная сборка (--threads:on):
type Selector*[T] = ptr SelectorImpl[T]
```

Центральный объект. Оборачивает файловый дескриптор kqueue и поддерживает таблицу, отображающую каждый наблюдаемый файловый дескриптор на `SelectorKey[T]`, который хранит зарегистрированные события и пользовательское значение типа `T`. Параметр типа `T` позволяет прикрепить к каждому дескриптору произвольное Nim-значение — состояние соединения, колбэк, указатель на собственный контекст и т.д.

В потокобезопасной сборке (`--threads:on`) `Selector` является сырым `ptr` в разделяемой памяти, а весь доступ к буферу ожидающих изменений защищён мьютексом `Lock`.

### `SelectEvent`

```nim
type SelectEvent* = ptr SelectEventImpl
```

Событие, запускаемое вручную. Внутри это OS-пайп (два конца: читаемый и записываемый). Когда `trigger` записывает байт в записывающий конец, kqueue срабатывает на читающем конце. Поскольку объект живёт в разделяемой памяти (`allocShared0`), один и тот же `SelectEvent` можно безопасно передавать между потоками — это стандартный способ разбудить спящий вызов `select`/`selectInto` из другого потока.

## Функции жизненного цикла

### `newSelector`

```nim
proc newSelector*[T](): owned(Selector[T])
```

#### Что делает

Создаёт и инициализирует новый селектор. Внутри:

1. Запрашивает у ядра через `sysctl` максимальное количество файловых дескрипторов, которые процесс может открыть, затем заранее выделяет таблицу `fds` нужного размера. Именно поэтому поиск наблюдаемого дескриптора работает за O(1) — это прямая индексация массива.
2. Открывает файловый дескриптор kqueue системным вызовом `kqueue()`.
3. Создаёт фиктивный TCP-сокет (`usock`), единственное назначение которого — дублирование через `F_DUPFD_CLOEXEC` функцией `getUnique` в будущем. Поскольку у событий таймера, сигнала и процесса нет настоящих файловых дескрипторов в kqueue, модуль фабрикует уникальные fd-подобные целые числа путём дублирования этого сокета. Продублированные числа не связаны ни с каким реальным I/O — они существуют исключительно как ключи в таблице `fds`.
4. В потокобезопасной сборке выделяет всё в разделяемой памяти и инициализирует мьютекс.

#### Возвращаемое значение

`owned`-`Selector[T]`, готовый к работе. Вызовите `close`, когда он больше не нужен.

#### Пример

```nim
import std/ioselectors

type ConnState = object
  id: int

let sel = newSelector[ConnState]()
defer: sel.close()
```

---

### `close` (Selector)

```nim
proc close*[T](s: Selector[T])
```

#### Что делает

Освобождает все ресурсы, удерживаемые селектором:

- Закрывает файловый дескриптор kqueue (`kqFD`).
- Закрывает фиктивный сокет, использовавшийся для генерации уникальных индексов.
- В потокобезопасных сборках: уничтожает мьютекс и освобождает разделяемую память.

Бросает `IOSelectorsError`, если любой из системных вызовов `close` завершается неудачей. Ответственность за отмену регистрации всех дескрипторов *перед* вызовом этой функции лежит на вызывающем коде, поскольку `close` не выполняет автоматическую отмену регистрации наблюдателей.

#### Пример

```nim
let sel = newSelector[int]()
# ... работа с sel ...
sel.close()
```

---

### `newSelectEvent`

```nim
proc newSelectEvent*(): SelectEvent
```

#### Что делает

Создаёт `SelectEvent` на основе OS-пайпа. Оба конца пайпа переводятся в неблокирующий режим. Объект выделяется в разделяемой памяти, чтобы его можно было безопасно передавать между потоками.

Типичный паттерн: один поток вызывает `select`/`selectInto` и блокируется; другой поток вызывает `trigger` на этом событии, чтобы разбудить первый.

#### Пример

```nim
import std/ioselectors

let wakeup = newSelectEvent()
defer: wakeup.close()
```

---

### `trigger`

```nim
proc trigger*(ev: SelectEvent)
```

#### Что делает

Сигнализирует о событии, записывая 8 байт (значение `uint64` равное `1`) в записывающий конец внутреннего пайпа. Это заставляет вызов `kevent()` в селекторе вернуться с результатом `{Event.User}` для дескриптора, связанного с этим событием.

`trigger` безопасно вызывать из любого потока. Он спроектирован для вызова из потока, отличного от того, что заблокирован в `select`/`selectInto`.

Бросает `IOSelectorsError`, если запись завершилась ошибкой по любой причине, кроме переполнения буфера пайпа.

#### Пример

```nim
# в рабочем потоке:
wakeupEvent.trigger()

# в потоке основного цикла, selectInto возвращается с Event.User:
let ready = sel.select(timeout = -1)
for key in ready:
  if Event.User in key.events:
    echo "разбужен другим потоком"
```

---

### `close` (SelectEvent)

```nim
proc close*(ev: SelectEvent)
```

#### Что делает

Закрывает оба конца внутреннего пайпа, затем освобождает блок разделяемой памяти. Всегда вызывайте это, когда `SelectEvent` больше не нужен. Убедитесь, что сначала вызвали `unregister` на селекторе, и только потом — `close` на событии.

Бросает `IOSelectorsError`, если любой из системных вызовов `close` завершается неудачей.

## Функции регистрации

Все функции `register*` прикрепляют файловый дескриптор (или виртуальный идентификатор) к селектору и сообщают ядру, каких событий следует ждать. Все они:

- Проверяют, что номер дескриптора находится в допустимых пределах (`checkFd`).
- Сохраняют `SelectorKey` в таблице `fds` с набором событий и пользовательскими данными.
- Ставят в очередь один или несколько дескрипторов изменений `KEvent`.
- Сбрасывают изменения в ядро немедленно (если не установлен флаг `-d:kqcache`).
- Увеличивают `s.count`.

### `registerHandle`

```nim
proc registerHandle*[T](s: Selector[T], fd: int | SocketHandle,
                        events: set[Event], data: T)
```

#### Что делает

Регистрирует существующий файловый дескриптор — сокет, конец пайпа, `stdin`, любой читаемый/записываемый OS-ресурс — в селекторе. Вы указываете, какие события важны: `{Event.Read}`, `{Event.Write}` или оба.

Это основная рабочая лошадка для сетевых серверов: каждый принятый клиентский сокет регистрируется здесь, чтобы селектор мог сообщить, у какого из них есть ожидающие данные или кто готов принять запись.

`events` может быть `{}`, чтобы зарегистрировать дескриптор без фактического наблюдения за событиями; используйте `updateHandle` позднее, чтобы активировать наблюдение.

#### Параметры

| Параметр | Описание |
|----------|----------|
| `fd` | OS-дескриптор или дескриптор сокета для наблюдения |
| `events` | `{Event.Read}` и/или `{Event.Write}` |
| `data` | Пользовательское значение, хранящееся рядом с дескриптором |

#### Пример

```nim
import std/[ioselectors, nativesockets]

let sel = newSelector[string]()
let sock = createNativeSocket()
sock.setBlocking(false)
sel.registerHandle(sock, {Event.Read}, "клиент-1")
```

---

### `updateHandle`

```nim
proc updateHandle*[T](s: Selector[T], fd: int | SocketHandle,
                      events: set[Event])
```

#### Что делает

Изменяет набор наблюдаемых событий для уже зарегистрированного файлового дескриптора. Можно добавить `Event.Read` или `Event.Write`, убрать одно из них, переключиться или передать `{}`, чтобы прекратить наблюдение без отмены регистрации.

Работает только для обычных I/O-дескрипторов (сокетов, пайпов). Явно запрещает дескрипторы, зарегистрированные с событиями таймера, сигнала, процесса, vnode, пользователя, oneshot или ошибки — у них есть собственные процедуры регистрации, и `updateHandle` их не касается.

Именно так правильно реализуется избирательное наблюдение за готовностью к записи: регистрируем с `{Event.Read}` изначально, затем вызываем `updateHandle` для добавления `Event.Write` только тогда, когда есть данные для отправки, и снова убираем его, как только буфер записи опустел.

#### Пример

```nim
# Изначально смотрим только на чтение
sel.registerHandle(sock, {Event.Read}, myData)

# Позже, когда нужно что-то отправить:
sel.updateHandle(sock, {Event.Read, Event.Write})

# После того как запись осушена:
sel.updateHandle(sock, {Event.Read})
```

### `registerTimer`

```nim
proc registerTimer*[T](s: Selector[T], timeout: int, oneshot: bool,
                       data: T): int {.discardable.}
```

#### Что делает

Регистрирует таймер ядра, который срабатывает каждые `timeout` миллисекунд. Настоящий файловый дескриптор не нужен — модуль создаёт синтетический уникальный целочисленный идентификатор (дублируя внутренний фиктивный сокет) и использует `EVFILT_TIMER` kqueue.

- `oneshot = false`: таймер срабатывает многократно, пока не будет отменён. Каждое срабатывание появляется в результатах select как `Event.Timer`.
- `oneshot = true`: таймер срабатывает ровно один раз. После срабатывания к набору событий ключа добавляется `Event.Finished`, а `s.count` уменьшается внутри, но ключ остаётся в `fds`, чтобы можно было прочитать связанные данные. Вызовите `unregister` для очистки.

Возвращает синтетический целочисленный дескриптор, который потребуется, если позднее нужно вызвать `unregister`.

#### Пример

```nim
# Повторяющийся таймер: срабатывает каждые 500 мс
let timerFd = sel.registerTimer(500, oneshot = false, data = "heartbeat")

# Одноразовый таймер: срабатывает через 2 секунды
let onceFd = sel.registerTimer(2000, oneshot = true, data = "startup-delay")
```

### `registerSignal`

```nim
proc registerSignal*[T](s: Selector[T], signal: int,
                        data: T): int {.discardable.}
```

#### Что делает

Регистрирует номер POSIX-сигнала так, чтобы селектор сообщал о нём как о `Event.Signal` вместо доставки обработчику сигнала. Внутри:

1. Сигнал **блокируется** в маске сигналов процесса (чтобы обработчик по умолчанию не сработал).
2. Обработчик сигнала устанавливается в `SIG_IGN` (семантика «поглощения» сигналов, совместимая с Linux).
3. `EVFILT_SIGNAL` kqueue используется для доставки сигнала как обычного события.

Это превращает традиционно неудобную модель обработки сигналов в тот же событийный цикл, что используется для I/O, полностью исключая небезопасные для сигналов колбэки.

Возвращает синтетический fd-целое для последующего `unregister`.

#### Пример

```nim
import std/posix
let sigFd = sel.registerSignal(SIGTERM.int, data = "got-sigterm")
```

---

### `registerProcess`

```nim
proc registerProcess*[T](s: Selector[T], pid: int,
                         data: T): int {.discardable.}
```

#### Что делает

Регистрирует дочерний процесс по его PID так, чтобы селектор сигнализировал о `Event.Process` при завершении этого процесса (`NOTE_EXIT`). Это всегда одноразовое событие: после завершения процесса ключ помечается `Event.Finished`, `s.count` уменьшается, но ключ остаётся в `fds` до вызова `unregister`.

Позволяет интегрировать сбор дочерних процессов в тот же событийный цикл, что и I/O, без обработчиков `SIGCHLD`.

Возвращает синтетический fd-целое для последующего `unregister`.

#### Пример

```nim
let childPid = startProcess("worker")
let procFd = sel.registerProcess(childPid.processID, data = "worker-proc")
```

### `registerEvent`

```nim
proc registerEvent*[T](s: Selector[T], ev: SelectEvent, data: T)
```

#### Что делает

Регистрирует `SelectEvent` (вручную запускаемое событие на основе пайпа) в селекторе. После регистрации любой вызов `ev.trigger()` заставит `selectInto`/`select` вернуться с результатом `{Event.User}` для этого дескриптора.

В отличие от `registerHandle`, использует fd читающего конца пайпа напрямую как ключ — синтетический идентификатор не нужен.

#### Пример

```nim
let wakeup = newSelectEvent()
sel.registerEvent(wakeup, data = "сигнал-пробуждения")
```

### `registerVnode`

```nim
proc registerVnode*[T](s: Selector[T], fd: cint, events: set[Event], data: T)
```

#### Что делает

Регистрирует файловый дескриптор (как правило, открытый файл или директорию) для получения уведомлений об изменениях файловой системы через `EVFILT_VNODE` kqueue. Вы указываете, за какими изменениями следить:

| Событие | Флаг kqueue | Значение |
|---------|-------------|---------|
| `Event.VnodeDelete` | `NOTE_DELETE` | Файл удалён |
| `Event.VnodeWrite` | `NOTE_WRITE` | В файл были записаны данные |
| `Event.VnodeExtend` | `NOTE_EXTEND` | Файл увеличился в размере |
| `Event.VnodeAttrib` | `NOTE_ATTRIB` | Метаданные изменились (chmod, chown…) |
| `Event.VnodeLink` | `NOTE_LINK` | Число жёстких ссылок изменилось |
| `Event.VnodeRename` | `NOTE_RENAME` | Файл переименован |
| `Event.VnodeRevoke` | `NOTE_REVOKE` | Доступ к файлу отозван |

Флаг `EV_CLEAR` всегда установлен, поэтому событие срабатывает по одному разу на каждое изменение, а не непрерывно.

Именно так строятся файловые наблюдатели без опроса: откройте целевой файл, передайте его fd сюда, и селектор разбудит вас при любом из указанных изменений.

#### Пример

```nim
import std/posix
let fileFd = posix.open("config.json", O_RDONLY)
sel.registerVnode(fileFd, {Event.VnodeWrite, Event.VnodeDelete},
                  data = "config-watcher")
```

## Отмена регистрации

### `unregister` (fd)

```nim
proc unregister*[T](s: Selector[T], fd: int | SocketHandle)
```

#### Что делает

Удаляет файловый дескриптор из селектора. Конкретные действия по очистке зависят от типа события, с которым дескриптор был зарегистрирован:

- **Read/Write**: удаляет `EVFILT_READ` и/или `EVFILT_WRITE` из kqueue.
- **Timer**: удаляет `EVFILT_TIMER` и закрывает синтетический fd (если не уже `Finished`).
- **Signal**: разблокирует сигнал, восстанавливает `SIG_DFL`, удаляет `EVFILT_SIGNAL`, закрывает синтетический fd.
- **Process**: удаляет `EVFILT_PROC` (если не уже `Finished`), закрывает синтетический fd.
- **Vnode**: удаляет `EVFILT_VNODE`.
- **User**: удаляет `EVFILT_READ` (fd на основе пайпа).

Во всех случаях запись `SelectorKey` очищается, а `s.count` уменьшается (если он не был уменьшен раньше при событии `Finished`).

---

### `unregister` (SelectEvent)

```nim
proc unregister*[T](s: Selector[T], ev: SelectEvent)
```

#### Что делает

Удаляет `SelectEvent` из селектора. Убирает фильтр `EVFILT_READ` для читающего конца пайпа и очищает ключ. Всегда вызывайте это перед вызовом `ev.close()`.

## Ожидание событий

### `selectInto`

```nim
proc selectInto*[T](s: Selector[T], timeout: int,
                    results: var openArray[ReadyKey]): int
```

#### Что делает

Блокируется до тех пор, пока не произойдёт одно или несколько зарегистрированных событий или не истечёт `timeout`, и записывает результаты в переданный вызывающим кодом массив `results`. Это вариант без выделения памяти: буфер принадлежит вам, поэтому на каждый вызов не происходит ни одного выделения в куче.

Возвращаемое значение — количество готовых событий, записанных в `results`. Индексы `0 ..< возвращаемое_значение` корректны; остаток `results` не тронут.

**Семантика timeout:**

| Значение | Смысл |
|----------|-------|
| `0` | Вернуться немедленно (неблокирующая проверка) |
| `> 0` | Ждать не дольше указанного числа миллисекунд |
| `-1` | Ждать бесконечно |

Если вызов прерывается сигналом (`EINTR`), возвращается `0` без исключения.

В режиме кэша (`-d:kqcache`) ожидающие изменения регистраций отправляются в ядро *атомарно в том же вызове `kevent()`*, что ждёт результатов, что снижает количество системных вызовов.

**Демультиплексирование событий:** для каждого возвращённого `KEvent` процедура проверяет поле filter и транслирует его обратно в соответствующий набор Nim-событий `Event`:

- `EVFILT_READ` → `Event.Read` (или `Event.User`, если fd зарегистрирован как пользовательское событие — при этом читается из пайпа для его опустошения)
- `EVFILT_WRITE` → `Event.Write`
- `EVFILT_TIMER` → `Event.Timer`
- `EVFILT_VNODE` → `Event.Vnode` + детальные vnode-события
- `EVFILT_SIGNAL` → `Event.Signal`
- `EVFILT_PROC` → `Event.Process`
- Флаг `EV_ERROR` → `Event.Error` с кодом ошибки
- Флаг `EV_EOF` → `Event.Error` с `ECONNRESET` (для сокетов, удалённый конец которых закрылся)

#### Пример

```nim
var ready: array[64, ReadyKey]
while true:
  let n = sel.selectInto(timeout = 1000, ready)
  for i in 0 ..< n:
    let key = ready[i]
    if Event.Read in key.events:
      echo "fd ", key.fd, " имеет данные"
    if Event.Error in key.events:
      echo "fd ", key.fd, " ошибка: ", key.errorCode
```

---

### `select`

```nim
proc select*[T](s: Selector[T], timeout: int): seq[ReadyKey]
```

#### Что делает

Удобная обёртка над `selectInto`, которая выделяет и возвращает `seq[ReadyKey]`. Внутри предварительно выделяет последовательность из `MAX_KQUEUE_EVENTS` (64) слотов, вызывает `selectInto`, затем обрезает последовательность до реального количества.

Используйте `selectInto` в горячих циклах, где важны выделения памяти. Используйте `select`, когда удобство важнее производительности.

#### Пример

```nim
let ready = sel.select(timeout = 500)
for key in ready:
  echo "событие на fd: ", key.fd, " события: ", key.events
```

## Функции доступа к данным

### `getData`

```nim
proc getData*[T](s: Selector[T], fd: SocketHandle | int): var T
```

#### Что делает

Возвращает изменяемую ссылку на пользовательские данные `T`, связанные с указанным файловым дескриптором. Возвращаемая ссылка указывает непосредственно в таблицу `fds`, поэтому изменение через неё изменяет хранимое значение на месте.

Если `fd` не зарегистрирован, поведение не определено (на данные нет проверки границ, только на диапазон fd). Используйте `contains` для предварительной проверки.

#### Пример

```nim
type Conn = object
  bytesSent: int

sel.registerHandle(sock, {Event.Write}, Conn(bytesSent: 0))
sel.getData(sock).bytesSent += 1024
```

---

### `setData`

```nim
proc setData*[T](s: Selector[T], fd: SocketHandle | int, data: T): bool
```

#### Что делает

Заменяет пользовательские данные для зарегистрированного дескриптора. Возвращает `true`, если дескриптор был зарегистрирован и данные обновлены; `false`, если дескриптор не найден (не зарегистрирован).

#### Пример

```nim
let updated = sel.setData(sock, Conn(bytesSent: 0))
if not updated:
  echo "sock не был зарегистрирован"
```

---

### `withData` (ветка «найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body: untyped)
```

#### Что делает

Выполняет `body` с `value`, связанным как указатель на пользовательские данные для `fd`, если `fd` зарегистрирован. Если `fd` не зарегистрирован, `body` не выполняется. Это идиоматический способ доступа к данным без отдельной проверки `contains` — проверка регистрации и доступ к данным составляют один атомарный шаг.

`value` является изменяемым указателем (`var T`), поэтому хранимые данные можно изменять на месте.

#### Пример

```nim
sel.withData(sock, connPtr):
  connPtr.bytesSent += 512
  echo "обновлено количество отправленных байт"
```

---

### `withData` (ветки «найдено» и «не найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body1, body2: untyped)
```

#### Что делает

Двухветочная версия: `body1` выполняется (с привязанным `value`), если `fd` зарегистрирован; `body2` выполняется, если нет. Воспроизводит паттерн `if/else`, объединяя проверку регистрации и доступ к данным в одну конструкцию.

#### Пример

```nim
sel.withData(sock, connPtr):
  connPtr.bytesSent += 512
do:
  echo "fd не зарегистрирован — игнорируем"
```

## Функции запросов

### `isEmpty`

```nim
template isEmpty*[T](s: Selector[T]): bool
```

#### Что делает

Возвращает `true`, если в данный момент в селекторе нет ни одного активного зарегистрированного дескриптора (`s.count == 0`). Полезно как условие выхода из цикла: если ничего не наблюдается, вызывать `select` бессмысленно.

Важно: `count` уменьшается для одноразовых событий (таймеров, завершений процессов) при их срабатывании — ещё до вызова `unregister`. Поэтому `isEmpty` может стать `true`, пока соответствующие ключи всё ещё существуют в `fds`.

#### Пример

```nim
while not sel.isEmpty:
  let ready = sel.select(timeout = -1)
  # обрабатываем события...
```

---

### `contains`

```nim
proc contains*[T](s: Selector[T], fd: SocketHandle | int): bool {.inline.}
```

#### Что делает

Возвращает `true`, если указанный файловый дескриптор в данный момент зарегистрирован в селекторе. Проверка — прямой поиск по массиву, O(1).

#### Пример

```nim
if sock in sel:
  sel.updateHandle(sock, {Event.Read, Event.Write})
```

---

### `getFd`

```nim
proc getFd*[T](s: Selector[T]): int
```

#### Что делает

Возвращает OS-дескриптор самого экземпляра kqueue (`kqFD`). В обычном использовании это редко нужно, но может быть полезно для вложенных селекторов (наблюдение одного kqueue из другого), передачи fd kqueue в библиотеку, ожидающую сырой fd, или при отладке.

#### Пример

```nim
echo "fd kqueue = ", sel.getFd()
```

## Краткий справочник по типам событий

| Значение `Event` | Источник | Смысл в `ReadyKey.events` |
|------------------|----------|--------------------------|
| `Event.Read` | `EVFILT_READ` | Доступны данные для чтения |
| `Event.Write` | `EVFILT_WRITE` | В буфере есть место, запись не заблокируется |
| `Event.Timer` | `EVFILT_TIMER` | Интервал таймера истёк |
| `Event.Signal` | `EVFILT_SIGNAL` | Получен POSIX-сигнал |
| `Event.Process` | `EVFILT_PROC` | Дочерний процесс завершился |
| `Event.Vnode` | `EVFILT_VNODE` | Произошло изменение файловой системы |
| `Event.VnodeDelete` | `NOTE_DELETE` | Наблюдаемый файл удалён |
| `Event.VnodeWrite` | `NOTE_WRITE` | В наблюдаемый файл записаны данные |
| `Event.VnodeExtend` | `NOTE_EXTEND` | Наблюдаемый файл вырос |
| `Event.VnodeAttrib` | `NOTE_ATTRIB` | Метаданные наблюдаемого файла изменились |
| `Event.VnodeLink` | `NOTE_LINK` | Число жёстких ссылок изменилось |
| `Event.VnodeRename` | `NOTE_RENAME` | Наблюдаемый файл переименован |
| `Event.VnodeRevoke` | `NOTE_REVOKE` | Доступ к файлу отозван |
| `Event.User` | чтение из пайпа | Был вызван `trigger()` |
| `Event.Error` | `EV_ERROR`/`EV_EOF` | Ошибка на дескрипторе |
| `Event.Finished` | внутреннее | Одноразовое событие уже сработало (пользователю не возвращается) |

## Полный пример: минимальный событийный цикл

```nim
import std/[ioselectors, nativesockets, posix]

type Client = object
  id: int

let sel = newSelector[Client]()
defer: sel.close()

# Следим за stdin
sel.registerHandle(0, {Event.Read}, Client(id: 0))

# Повторяющийся таймер каждую секунду
let hbFd = sel.registerTimer(1000, oneshot = false, Client(id: -1))

# Следим за SIGINT для корректного завершения
let sigFd = sel.registerSignal(SIGINT.int, Client(id: -2))

while not sel.isEmpty:
  let ready = sel.select(timeout = 5000)
  for key in ready:
    if Event.Read in key.events and key.fd == 0:
      echo "stdin содержит данные"
    if Event.Timer in key.events:
      echo "heartbeat"
    if Event.Signal in key.events:
      echo "получен SIGINT, завершаем работу"
      sel.unregister(hbFd)
      sel.unregister(sigFd)
      sel.unregister(0)
      break
```

## Флаги компиляции

| Флаг | Эффект |
|------|--------|
| `--threads:on` | Переключает на выделение в разделяемой памяти и добавляет `Lock` вокруг буфера изменений |
| `-d:kqcache` | Включает кэширование изменений: изменения накапливаются и отправляются в том же вызове `kevent()`, что и ожидание |

## Примечания и подводные камни

- **Синтетические fd для таймеров/сигналов/процессов.** У этих событий нет реального OS-дескриптора. Модуль дублирует внутренний фиктивный сокет через `F_DUPFD_CLOEXEC`, чтобы получить уникальные целые числа, пригодные как ключи таблицы `fds`. Эти синтетические числа закрываются в `unregister`, а не вызывающим кодом.
- **Всегда вызывайте `unregister` перед `close`.** Закрытие OS-ресурса, пока он всё ещё зарегистрирован в селекторе, оставляет устаревшие записи в таблице `fds` и может приводить к сбоям `doAssert` или молча некорректному поведению при последующей регистрации того же номера fd.
- **`MAX_KQUEUE_EVENTS = 64`** ограничивает количество событий, которые один вызов `select`/`selectInto` может вернуть. В очень высоконагруженном сервере может потребоваться цикл `selectInto` до тех пор, пока `result < maxres`, или `selectInto` с буфером большего размера.
- **`EV_EOF` на сокетах** транслируется в `Event.Error` с `ECONNRESET`. В самом модуле отмечено, что это упрощение (поле fflags может нести другие коды ошибок) и в будущем может быть доработано.
- **Одноразовые события остаются в `fds`.** После срабатывания одноразовый таймер или событие завершения процесса помечается `Event.Finished` и больше не учитывается в count, но занимает слот в `fds` до вызова `unregister`. Отсутствие вызова `unregister` — это утечка ресурсов.
- **`EINTR` молча игнорируется.** Если `kevent()` прерывается сигналом во время ожидания, `selectInto` возвращает `0` вместо исключения. Это корректное поведение — сигнал появится как результат `Event.Signal` при следующем вызове, если он был зарегистрирован.
