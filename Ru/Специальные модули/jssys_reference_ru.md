# `jssys.nim` — справочник модуля

> **Модуль:** `jssys` — Стандартная библиотека Nim, системный слой для JavaScript-бэкенда  
> **Цель:** только JavaScript (`--backend:js` / `--target:js`). На нативных платформах не компилируется.  
> **Стабильность:** Большинство модуля составляют символы `compilerproc` — внутренние помощники, которые вызываются исключительно кодогенератором Nim для JS. Пользовательский API — только пять символов, отмеченных `*`, описанных в этом справочнике.

---

## Обзор

`jssys.nim` — это JavaScript-аналог пары `excpt.nim` / `system.nim` для нативных платформ. При компиляции программы Nim в JavaScript этот модуль обеспечивает:

- **Рантайм исключений**: возбуждение, повторное возбуждение и перехват исключений; обёртку нативных JS-ошибок в Nim-исключения.
- **Сбор и форматирование трассировки стека** (при активном `-d:nimStackTrace`).
- **Арифметические примитивы** с проверкой переполнения и деления на ноль (вызываются генерируемым JS-кодом, не напрямую пользователем).
- **Строковые и множественные примитивы**, адаптированные к модели памяти JavaScript (строки Nim в JS — это `Array` из числовых кодов символов).
- Тонкий **мост к консоли** (`log`).

Поскольку в JavaScript нет адресной арифметики, `setjmp` и принципиально иная модель памяти по сравнению с C, каждый примитив в этом файле — тонкая Nim-обёртка вокруг встроенного JavaScript через `{.emit: …}` или `{.importc: …}`.

---

## Архитектурная заметка: как работают исключения в JS-бэкенде

В JS-бэкенде нет `setjmp`/`longjmp`. Обработка исключений использует нативный JavaScript `try/catch`. Генерируемый компилятором JS-код оборачивает блоки в `try { … } catch(e) { lastJSError = e; … }`. Глобальная переменная `lastJSError` всегда хранит последний пойманный JS-объект ошибки.

Когда пойманный объект является исключением Nim (он несёт свойство `m_type`, установленное кодогенератором), `lastJSError` просто приводится к типу `ref Exception`. Когда это *нативная* JS-ошибка `Error` (выброшенная JS-окружением, например `TypeError`), модуль читает свойство `.message` и представляет его как строку.

Такой подход гарантирует корректную работу `getCurrentException` и `getCurrentExceptionMsg` независимо от того, откуда пришло исключение — из Nim или из JavaScript.

## Экспортируемые функции

---

### `log`

```nim
proc log*(s: cstring) {.importc: "console.log", varargs, nodecl.}
```

**Что делает:**  
Прямая привязка к функции `console.log` браузера или Node.js. Принимает `cstring` первым аргументом и объявлена `varargs`, поэтому при вызове из встроенного JS или генерируемого кода можно передавать дополнительные аргументы.

**Почему экспортирована:**  
Это самый низкоуровневый примитив вывода на JS-бэкенде — `echo` и `writeStackTrace` в конечном счёте вызывают `console.log`. Экспорт через `*` позволяет Nim-коду в том же юните компиляции вызывать `log` напрямую, минуя накладные расходы форматирования `echo`.

**Практическое применение:** Полезна, когда нужно вывести что-то в самом начале инициализации (до того, как строковая машинерия Nim полностью готова), или когда намеренно нужна семантика `console.log` (например, передать JS-объект прямо в DevTools браузера).

```nim
# Вывод прямо в консоль — без преобразований Nim-строк
log("Hello from Nim")
```

> **Примечание:** Для обычного вывода из программы используйте `echo`. `log` — только когда нужен сырой JS-примитив или гарантированно доступный путь вывода в `{.emit:…}`-коде.

### `getCurrentException`

```nim
proc getCurrentException*(): ref Exception {.compilerRtl, gcsafe.}
```

**Что делает:**  
Возвращает исключение, которое *в данный момент обрабатывается* — то есть то, которое вызвало вход в ближайший активный блок `except`. Если пойманный объект является настоящим исключением Nim (имеет поле `m_type`), он приводится к `ref Exception` и возвращается. Если активного исключения нет — возвращает `nil`.

**Поведение на JS-бэкенде:**  
В отличие от нативных платформ, «текущее исключение» не хранится в потокозависимом связанном списке. Вместо этого оно читается напрямую из `lastJSError` — JS-переменной, хранящей последний пойманный объект ошибки. Возвращаются только объекты с маркером `m_type` от Nim. Для чистых JS-ошибок `Error` функция вернёт `nil` — в этом случае используйте `getCurrentExceptionMsg`.

**Стандартное использование:** Эта функция неявно вызывается при использовании `getCurrentException()` внутри блока `except`. Напрямую вызывать её редко нужно — идиоматичный способ получить объект исключения — конструкция `except IOError as e`.

```nim
try:
  riskyOperation()
except:
  let e = getCurrentException()
  if e != nil:
    echo "Nim-исключение: ", e.msg
  else:
    echo "Нативная JS-ошибка: ", getCurrentExceptionMsg()
```

```nim
# Универсальный логгер ошибок, работающий с любыми исключениями:
proc logAnyException() =
  let e = getCurrentException()
  if e != nil:
    echo "[", e.name, "] ", e.msg
  else:
    echo "[JS Error] ", getCurrentExceptionMsg()
```

### `getCurrentExceptionMsg`

```nim
proc getCurrentExceptionMsg*(): string
```

**Что делает:**  
Возвращает читаемое сообщение текущего активного исключения в виде Nim-строки `string`. Обрабатывает оба вида исключений на JS-бэкенде:

1. **Nim-исключения** — возвращает `e.msg`, строку, заданную при создании через `newException(…, "сообщение")`.
2. **Нативные JS-ошибки** — читает свойство `.message` объекта JavaScript `Error` (например, `TypeError: cannot read property 'x' of undefined`, выброшенный JS-движком).

Возвращает пустую строку `""`, если активного исключения нет.

**Почему это надёжнее, чем `getCurrentException().msg`:**  
Когда JS-движок выбрасывает нативную ошибку (не Nim-исключение), `getCurrentException()` возвращает `nil`, и обращение к `.msg` вызовет краш. `getCurrentExceptionMsg()` безопасно покрывает оба пути и всегда даёт что-то пригодное для вывода.

```nim
try:
  callJSAPIThatMightThrow()
except:
  # Работает и для Nim-исключений, и для нативных JS-ошибок
  echo "Ошибка: ", getCurrentExceptionMsg()
```

```nim
# Единообразная обработка ошибок при вызове JS API:
proc safeCall(f: proc()) =
  try:
    f()
  except:
    let msg = getCurrentExceptionMsg()
    reportError(msg)  # отправить в систему трекинга ошибок
```

### `setCurrentException`

```nim
proc setCurrentException*(exc: ref Exception)
```

**Что делает:**  
Вручную устанавливает «текущее исключение» в `exc`, приводя его к `PJSError` и сохраняя в `lastJSError`. После этого вызова `getCurrentException()` и `getCurrentExceptionMsg()` будут возвращать данные из этого объекта.

**Когда вызывать:**  
В прикладном коде — редко. Основные сценарии использования:

- **Пропаганда исключения через async/корутины**: вручную перенести объект исключения из одного контекста выполнения в другой.
- **Тестирование**: внедрить синтетическое исключение в состояние рантайма, чтобы проверить код обработки ошибок без реального возбуждения и поимки.
- **Интероп**: если вы получили `ref Exception` из обратного вызова и хотите сделать его «текущим» перед входом в обработчик, вызывающий `getCurrentException()`.

**Важная оговорка:** Это **не** возбуждает исключение. Оно лишь записывается в слот «последнего пойманного» в рантайме. Оно будет видно как текущее исключение до следующего цикла `try/catch`, перезапишущего `lastJSError`.

```nim
# Сохранить исключение в обратном вызове и использовать позже:
var savedExc: ref Exception

proc asyncCallback() =
  try:
    doWork()
  except Exception as e:
    savedExc = e  # захватить до возврата callback

# Позже, в основном потоке:
if savedExc != nil:
  setCurrentException(savedExc)
  handleCurrentException()
```

```nim
# Юнит-тест: проверить, что обработчик правильно читает сообщение
let fakeExc = newException(ValueError, "тестовое сообщение")
setCurrentException(fakeExc)
assert getCurrentExceptionMsg() == "тестовое сообщение"
```

### `getStackTrace` *(без аргумента)*

```nim
proc getStackTrace*(): string
```

**Что делает:**  
Возвращает текущий стек вызовов в виде форматированной многострочной строки — последовательности вызовов процедур, приведшей к точке, в которой вызван `getStackTrace()`.

**Формат** (при активированных трассировках):
```
Traceback (most recent call last)
myfile.nim(42) at myProc
myfile.nim(10) at main
```

Возвращает `"No stack traceback available\n"` когда трассировки отключены (программа скомпилирована без `-d:nimStackTrace`, что является поведением по умолчанию для release-сборок через `--opt:speed`).

**Особенность JS-бэкенда:** Трассировка строится из собственного связанного списка `CallFrame` Nim, а **не** из нативного `Error.stack` JS-движка. Это означает, что вы получаете информацию на уровне Nim-исходников (файл, строка, имя процедуры), а не JS-уровень. С другой стороны, это работает только при активной инструментации стека Nim.

```nim
// Залогировать текущее местоположение без возбуждения исключения:
echo getStackTrace()
```

```nim
// Прикрепить снимок стека к диагностическому отчёту:
proc createReport(msg: string): string =
  result = msg & "\n" & getStackTrace()
```

### `getStackTrace` *(с исключением)*

```nim
proc getStackTrace*(e: ref Exception): string
```

**Что делает:**  
Возвращает трассировку стека, захваченную **в момент возбуждения исключения `e`**, хранящуюся внутри объекта исключения как `e.trace`. Это замороженная трассировка из момента `raise`, а не текущей точки вызова.

Возвращает пустую строку `""`, если `e` равно `nil` или трассировка не была сохранена (release-сборки).

**Ключевое различие с вариантом без аргумента:**  
- `getStackTrace()` — снимок того, *где вы находитесь прямо сейчас*.  
- `getStackTrace(e)` — снимок того, *откуда было выброшено исключение*.

Это различие важно при вызове `getStackTrace` из блока `finally`, из подпрограммы формирования отчётов или из любого места, далеко удалённого по стеку от оригинального `raise`.

```nim
try:
  riskyOperation()
except IOError as e:
  # Вывести место возникновения ошибки, а не место её поимки:
  echo getStackTrace(e)
```

```nim
// Структурированное форматирование ошибки с исходным местом возникновения:
proc formatError(e: ref Exception): string =
  result = "Ошибка: " & e.msg & "\n"
  let trace = getStackTrace(e)
  if trace.len > 0:
    result &= "Трассировка стека:\n" & trace
```

## Символы `compilerproc` — обзор

В таблице ниже перечислены внутренние помощники, **не входящие** в пользовательский API, но видимые в генерируемом JS-коде. Понимание их назначения помогает при отладке.

| Символ | Назначение |
|---|---|
| `raiseException` | Вызывается генерируемым кодом для каждого оператора `raise` |
| `reraiseException` | Вызывается генерируемым кодом при голом `raise` (повторное возбуждение) |
| `raiseOverflow` | Возбуждается при переполнении целочисленной арифметики |
| `raiseDivByZero` | Возбуждается при целочисленном делении или остатке от деления на ноль |
| `raiseRangeError` | Возбуждается при выходе значения за диапазон (подтиповые типы) |
| `raiseIndexError` | Возбуждается при выходе за границы массива/последовательности |
| `raiseFieldError2` | Возбуждается при некорректном доступе к дискриминантному полю вариантного объекта |
| `addInt` / `subInt` / `mulInt` / `divInt` / `modInt` | Проверяемая 32-битная целочисленная арифметика |
| `addInt64` … `modInt64` | Проверяемая 64-битная арифметика (использует JS `BigInt`) |
| `cmpStrings` / `eqStrings` | Сравнение строк (строки Nim в JS — `Array` числовых кодов символов) |
| `nimCopy` | Семантика глубокого копирования значений для типов-значений Nim |
| `cstrToNimstr` / `toJSStr` | Конвертация между нативными JS-строками и внутренним форматом Nim (массив кодов) |
| `SetCard`, `SetEq`, `SetLe`, `SetLt`, `SetMul`, `SetPlus`, `SetMinus`, `SetXor` | Операции над типом `set` Nim (представлен JS-объектом) |
| `nimParseBiggestFloat` | Разбор вещественных чисел с полной семантикой Nim (NaN, Inf, подчёркивания, знак) |
| `chckIndx` / `chckRange` / `chckObj` / `isObj` | Проверки границ и типов во время выполнения |

## Краткая шпаргалка

| Символ | Вид | Когда использовать |
|---|---|---|
| `log` | proc | Сырой `console.log` — низкоуровневый или ранний вывод |
| `getCurrentException` | proc | Получить объект активного Nim-исключения внутри `except` |
| `getCurrentExceptionMsg` | proc | Получить строку сообщения — работает и для JS-ошибок |
| `setCurrentException` | proc | Вручную вставить исключение в состояние рантайма |
| `getStackTrace()` | proc | Снимок текущего стека вызовов в виде строки |
| `getStackTrace(e)` | proc | Замороженная трассировка, хранящаяся внутри исключения `e` |
