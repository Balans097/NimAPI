# Справочник модуля `arc`

> **Модуль:** `arc.nim` — стандартная библиотека Nim  
> **Назначение:** Основная реализация управления памятью **ARC** (Automatic Reference Counting) в Nim, с опциональными расширениями для **ORC** (сборщик циклов поверх ARC) и **Atomic ARC** (потокобезопасный счётчик ссылок). Каждое выделение памяти под `ref`, каждое копирование `ref` и каждое освобождение `ref` при выходе из области видимости проходит через процедуры, определённые здесь.

---

## Общая картина

ARC — детерминированная стратегия управления памятью в Nim, пришедшая на смену классическому сборщику мусора. Она работает путём встраивания счётчика ссылок в небольшой заголовок (`RefHeader`), расположенный непосредственно перед каждым объектом на куче. Когда счётчик достигает нуля, объект уничтожается немедленно — без пауз GC, без фоновых потоков, без непредсказуемых задержек.

Модуль определяет:
- Структуру `RefHeader` и псевдоним `Cell` для указателя на неё.
- Низкоуровневые процедуры выделения памяти, вызываемые напрямую из генерируемого компилятором кода.
- Манипуляции со счётчиком ссылок (инкремент, декремент, проверка последней ссылки).
- Оператор `=dispose` для `owned ref`.
- Вспомогательные процедуры для отладки, потокобезопасности и проверок типов во время исполнения.

Все процедуры, помеченные `compilerRtl`, вызываются кодом, генерируемым компилятором, — вы никогда не вызываете их из своего Nim-кода напрямую. Небольшое число процедур, помеченных `*`, образует публичный API для авторов библиотек и опытных пользователей.

## Внутренние типы и константы

### `RefHeader`

```nim
type RefHeader = object
  rc: int
  # (только ORC)            rootIdx: int
  # (отладка/ID)            refId: int
  # (ORC + детектор утечек) filename: cstring; line: int
```

Заголовок, расположенный **непосредственно перед** каждым объектом `ref` на куче. Его ключевое поле — `rc`, упакованный счётчик ссылок.

Счётчик ссылок хранится не как простое целое число. Три младших бита (ARC) или четыре (ORC) зарезервированы под флаги среды исполнения, а сам счётчик — в старших битах. Такая упаковка позволяет флагам и счётчику сосуществовать в одном машинном слове без дополнительной памяти.

| Режим | `rcIncrement` | `rcMask` | `rcShift` | Зарезервировано |
|---|---|---|---|---|
| ARC | `0b1000` (8) | `0b111` | 3 | 3 младших бита |
| ORC | `0b10000` (16) | `0b1111` | 4 | 4 младших бита |

**Опциональные поля** (включаются только при наличии соответствующего define):
- `rootIdx` — только ORC. Индекс в списке кандидатов на корни циклов; позволяет удалять потенциальный корень цикла за O(1) без двусвязных списков.
- `refId` — режим отладки/ID. Монотонно возрастающее целое, уникально идентифицирующее данное выделение памяти. Удобно для условных точек останова: «остановить при выделении объекта №42».
- `filename` / `line` — детектор утечек ORC. Записывает место в исходном коде, где произошло выделение, для отчётов об утечках.

### `Cell`

```nim
type Cell = ptr RefHeader
```

Типизированный псевдоним для «указателя на заголовок ref-объекта». Используется по всему модулю для удобочитаемости при прямых манипуляциях с заголовками.

### Шаблон `head`

```nim
template head(p: pointer): Cell
```

По пользовательскому указателю `p` (на данные объекта) отступает назад в памяти ровно на `sizeof(RefHeader)` байт, достигая `RefHeader`. Это фундаментальный переход между пользовательским взглядом на ref и взглядом среды исполнения на его заголовок.

```
Расположение в памяти:
[ RefHeader | ... данные объекта ... ]
^            ^
Cell         p  (то, что держит программист)
head(p) ──►  Cell
```

## Экспортируемые процедуры (публичный API)

---

### `isUniqueRef`

```nim
proc isUniqueRef*[T](x: ref T): bool {.inline.}
```

**Что делает:**  
Возвращает `true`, если `x` — **единственная** живая ссылка на объект, то есть внутренний счётчик ссылок равен нулю. В ARC-конвенции «начинаем с нуля» значение `rc = 0` означает ровно одного владельца: `rc` инкрементируется только когда появляется вторая копия. Таким образом, `rc == 0` действительно означает «уникальный».

**Для чего полезно:**

*Проверка потокобезопасности.* Перед передачей `ref` в другой поток убедитесь, что никто другой его не держит:
```nim
assert isUniqueRef(myObj), "объект должен иметь единственного владельца перед передачей в поток"
channel.send(myObj)
```

*Оптимизация мутации на месте.* Некоторые API могут выбрать более быстрый путь, зная, что объект не разделяется:
```nim
if isUniqueRef(buf):
  mutateInPlace(buf)   # безопасно: никто другой изменение не увидит
else:
  buf = buf.deepCopy()
  mutateInPlace(buf)
```

*Контракт владения как проверяемый инвариант.* `assert isUniqueRef(x)` превращает неформальное требование единственного владельца в явную проверку во время исполнения.

**Важная оговорка:**  
Компилятор и оптимизатор вправе устранять или переупорядочивать операции со счётчиком ссылок. Результат `isUniqueRef` может меняться в зависимости от версии компилятора и уровня оптимизации. Никогда не используйте её в логике корректности — только для отладки и утверждений.

### `GC_ref`

```nim
proc GC_ref*[T](x: ref T)
```

**Что делает:**  
Вручную инкрементирует счётчик ссылок `x` на единицу. Если `x` — `nil`, ничего не делает.

При ARC/ORC классического сборщика мусора не существует, поэтому `GC_ref` — прокладка совместимости для кода, написанного под старые GC-рантаймы Nim. Внутри вызывает `nimIncRef`.

**Когда нужна:**  
При взаимодействии с C-кодом или другой средой исполнения, которая держит сырой указатель на Nim-объект `ref` в обход обычной семантики переменных. Если внешний код удерживает указатель дольше, чем все Nim-переменные, указывающие на тот же объект, необходимо вручную поднять счётчик, чтобы ARC не уничтожил объект раньше времени.

```nim
var obj: ref MyStruct = new MyStruct
GC_ref(obj)                          # вручную повышаем счётчик
passToCCallback(cast[pointer](obj))  # C-код может пережить переменную 'obj'
# ... C-колбэк завершается ...
GC_unref(obj)                        # парный вызов; объект освобождается когда надо
```

---

### `GC_unref`

```nim
proc GC_unref*[T](x: ref T)
```

**Что делает:**  
Вручную декрементирует счётчик ссылок `x` на единицу, запуская уничтожение объекта, если счётчик достигает нуля. Точный аналог `GC_ref`. Внутри вызывает `=destroy` на курсоре (невладеющем представлении) `x`, что задействует стандартный ARC-путь разрушения.

**Предупреждение:** вызов `GC_unref` без предшествующего парного `GC_ref` приведёт к чрезмерному декременту, преждевременному уничтожению и использованию освобождённой памяти. Всегда используйте `GC_ref` и `GC_unref` парами.

```nim
GC_ref(obj)
passToCCallback(cast[pointer](obj))
GC_unref(obj)   # парный вызов к GC_ref выше
```

---

### `=dispose` (оператор)

```nim
template `=dispose`*[T](x: owned(ref T))
```

**Что делает:**  
Немедленно и безусловно освобождает объект, на который указывает `x`, минуя счётчик ссылок. Это ARC-замена вызова `dealloc` на owned-ссылках.

Если вы владеете `owned ref` — эксклюзивное владение доказано компилятором — можно явно освободить его в нужный момент через `=dispose`, не дожидаясь конца области видимости.

**Защитная проверка** (при `nimOwnedEnabled`): перед освобождением убеждается, что нет других владельцев. Если проверка не проходит, программа завершается с фатальным сообщением.

```nim
var buf: owned(ref BigBuffer) = new BigBuffer
fillBuffer(buf)
processBuffer(buf)
`=dispose`(buf)    # явно освобождаем здесь, не дожидаясь конца области видимости
                   # любое использование 'buf' после этого — неопределённое поведение
```

### `GC_fullCollect`

```nim
template GC_fullCollect*
```

**Что делает:**  
**Заглушка** при `--mm:arc`. Присутствует исключительно ради совместимости на уровне исходного кода с программами, написанными под старые GC-рантаймы Nim. При ARC память освобождается в момент, когда счётчик ссылок достигает нуля — откладывать нечего.

```nim
# Старый код компилируется без изменений:
GC_fullCollect()  # тихо ничего не делает под --mm:arc
```

---

### `setupForeignThreadGc` / `tearDownForeignThreadGc`

```nim
template setupForeignThreadGc*
template tearDownForeignThreadGc*
```

**Что делают:**  
Оба — **заглушки** при ARC/ORC. В старых GC-рантаймах Nim каждый поток, взаимодействующий с GC-управляемыми данными, должен был явно регистрироваться и дерегистрироваться у сборщика. При ARC такая регистрация не нужна — подсчёт ссылок ведётся на уровне объекта.

```nim
# Унаследованный многопоточный код компилируется без изменений:
setupForeignThreadGc()
doWorkWithNimObjects()
tearDownForeignThreadGc()
```

### `deallocatedRefId` (только отладка)

```nim
proc deallocatedRefId*(p: pointer): int
```

**Доступно при:** компиляции с `-d:nimArcDebug`

**Что делает:**  
Проверяет, был ли объект по сырому указателю `p` **уже освобождён**. Если да — возвращает уникальный `refId` этого объекта (обнаружено использование освобождённой памяти), иначе — `0` (объект жив).

В режиме `nimArcDebug` функция `nimRawDispose` **не освобождает** память на самом деле — вместо этого добавляет ячейку в набор `freedCells`. Такая намеренная «утечка во время отладки» означает, что адрес памяти остаётся читаемым после логического уничтожения, поэтому `deallocatedRefId` способна однозначно сказать: «да, вы держите висячий указатель на объект №N».

```nim
# Компилировать с: nim c -d:nimArcDebug myfile.nim

var p: ref int = new int
let raw = cast[pointer](p)
p = nil   # освобождаем последнюю ссылку; объект «уничтожен» (добавлен в freedCells)

let id = deallocatedRefId(raw)
if id != 0:
  echo "Использование освобождённой памяти! refId объекта = ", id
```

## Компилятор-внутренние процедуры

Вызываются исключительно кодом, генерируемым компилятором. Знание их полезно при чтении сгенерированного C-кода, изучении поведения ARC или отладке проблем с памятью на низком уровне.

---

### `nimNewObj`

```nim
proc nimNewObj(size, alignment: int): pointer {.compilerRtl.}
```

**Что делает:**  
Выделяет `size + hdrSize` байт **обнулённой** памяти с указанным выравниванием, возвращает указатель, смещённый за заголовок (на данные объекта). Поле `rc` нового заголовка равно `0` — один владелец, копий нет.

Вызывается для каждого выделения `new T` или `ref T`, где память должна начинаться с нулей, — что является умолчанием для всех `object`-типов в Nim.

```
После nimNewObj:
[ RefHeader (rc=0) | обнулённые данные объекта ]
                    ^
                    возвращаемый указатель (то, что держит ref-переменная)
```

**Отладочные возможности:** при `nimArcDebug` или `nimArcIds` присваивает уникальный `refId` из глобального счётчика `gRefId`. Если `refId` совпадает с compile-time константой `traceId`, выводится трассировка стека — адресная точка останова без интерактивного отладчика.

```nim
# Ваш Nim-код:
var x: ref MyObj = new MyObj
# Компилятор генерирует примерно следующее:
#   x = nimNewObj(sizeof(MyObj), alignOf(MyObj))
```

---

### `nimNewObjUninit`

```nim
proc nimNewObjUninit(size, alignment: int): pointer {.compilerRtl.}
```

**Что делает:**  
Аналогична `nimNewObj`, но выделяет **неинициализированную** память. Компилятор использует её только когда анализ потока данных доказывает, что каждое поле будет записано до первого чтения. Поля заголовка `rc` и `rootIdx` (ORC) всё равно явно обнуляются, так как инварианты среды исполнения их требуют.

**Почему важно:** для крупных объектов или часто вызываемых путей выделения памяти пропуск `memset` может дать ощутимый прирост производительности. Компилятор принимает это решение автоматически.

---

### `nimIncRef`

```nim
proc nimIncRef(p: pointer) {.compilerRtl, inl.}
```

**Что делает:**  
Инкрементирует счётчик ссылок объекта по адресу `p` на одно `rcIncrement`. Вызывается каждый раз, когда создаётся новая копия `ref`: при присваивании другой переменной, передаче как аргумент функции или сохранении в структуре данных.

При `--mm:atomicArc` с поддержкой потоков использует атомарный инкремент (`atomicInc`). В противном случае — обычная целочисленная арифметика.

---

### `nimDecWeakRef`

```nim
proc nimDecWeakRef(p: pointer) {.compilerRtl, inl.}
```

**Что делает:**  
Декрементирует счётчик ссылок объекта по адресу `p` на одно `rcIncrement` **без** запуска уничтожения, даже если счётчик достигает нуля. Используется для слабых ссылок — ссылок, наблюдающих за объектом без владения им.

После декремента слабой ссылки необходимо убедиться, что объект ещё жив, прежде чем его использовать.

---

### `nimDecRefIsLast`

```nim
proc nimDecRefIsLast(p: pointer): bool {.compilerRtl, inl.}
```

**Что делает:**  
Декрементирует счётчик ссылок `p` и возвращает `true`, если этот декремент обнулил счётчик — то есть это была **последняя** ссылка и объект должен быть уничтожен.

Компилятор вставляет вызов этой процедуры в каждую точку, где `ref` выходит из области видимости или перезаписывается. Окружающий паттерн в генерируемом C-коде:

```c
if (nimDecRefIsLast(p)) {
    nimDestroyAndDispose(p);
}
```

При Atomic ARC декремент использует `atomicDec` и проверяет сигнальное значение `-rcIncrement`.

---

### `nimRawDispose`

```nim
proc nimRawDispose(p: pointer, alignment: int) {.compilerRtl.}
```

**Что делает:**  
Безусловно освобождает память, занятую объектом по адресу `p`, включая его `RefHeader`. Вычисляет смещение заголовка, отступает назад до начала оригинальной аллокации и вызывает `alignedDealloc`.

При `nimArcDebug` — вместо освобождения добавляет ячейку в `freedCells`.
При `nimOwnedEnabled` — проверяет отсутствие других владельцев перед освобождением.

---

### `nimDestroyAndDispose`

```nim
proc nimDestroyAndDispose(p: pointer) {.compilerRtl, quirky, raises: [].}
```

**Что делает:**  
Двухфазное завершение, выполняемое когда счётчик ссылок ref-объекта достигает нуля:

1. **Уничтожение** — читает `PNimTypeV2`-информацию о типе и вызывает деструктор (если есть). Деструктор рекурсивно декрементирует ссылки, хранящиеся в полях объекта.
2. **Освобождение** — вызывает `nimRawDispose` для возврата сырой памяти.

Разделение фаз важно: деструктор может обращаться к `self` и другим объектам. Освобождение памяти до завершения деструктора было бы использованием освобождённой памяти.

---

### `isObjDisplayCheck`

```nim
proc isObjDisplayCheck(source: PNimTypeV2, targetDepth: int16, token: uint32): bool {.compilerRtl, inl.}
```

**Что делает:**  
Быстрая O(1)-реализация оператора `of` в Nim (проверка принадлежности типу). Каждый тип в иерархии наследования имеет массив `display` и поле `depth` в своём дескрипторе `PNimTypeV2`. Для проверки `x of T`:

1. `targetDepth` — известная во время компиляции глубина `T` в иерархии (корень = 0).
2. `token` — уникальный идентификационный токен `T`.
3. Проверка читает `source.display[targetDepth]` и сравнивает с `token`.

Если `source` является подтипом `T`, токен `T` будет храниться по индексу `targetDepth` в массиве display `source` — одна проверка границ плюс одно чтение из массива, независимо от глубины иерархии.

```nim
type
  Animal = ref object of RootObj
  Dog    = ref object of Animal
  Cat    = ref object of Animal

var d: Animal = Dog()
echo d of Dog     # true  — isObjDisplayCheck(тип Dog, Dog.depth, Dog.token)
echo d of Animal  # true  — display Dog содержит и токен Animal
echo d of Cat     # false — токена Cat нет в display Dog
```

### `nimGetVTable`

```nim
proc nimGetVTable(p: pointer, index: int): pointer {.compilerRtl, inline, raises: [].}
```

**Доступно при:** компиляции с `-d:gcDestructors`

**Что делает:**  
Извлекает указатель на функцию по индексу `index` из таблицы виртуальных методов (vtable) объекта по адресу `p`. Vtable встроена в дескриптор типа `PNimTypeV2`, находящийся по смещению 0 каждого объекта.

Это механизм, лежащий в основе диспетчеризации `method` в Nim: компилятор транслирует вызов виртуального метода в поиск через `nimGetVTable` с последующим косвенным вызовом.

## Ключевые принципы проектирования

**Расположение заголовка перед данными.** `RefHeader` находится непосредственно перед пользовательскими данными объекта в памяти. Единственный шаблон `head(p)` осуществляет переход между двумя представлениями. Не-ref объекты вообще не несут заголовка — накладные расходы равны нулю в общем случае.

**Упакованный счётчик ссылок.** Поле `rc` хранит и счётчик ссылок (старшие биты), и флаги состояния (младшие биты) в одном `int`. Инкремент и декремент на `rcIncrement` (а не на 1) не затрагивает биты флагов — операции подсчёта и операции с флагами всегда независимы.

**Конвенция «начинаем с нуля».** Только что выделенный объект имеет `rc = 0`, что означает одного владельца. В подавляющем большинстве случаев `ref` имеет единственного владельца и `rc` вообще не нужно трогать. `rc` становится ненулевым только при появлении второго владельца.

**Атомарный и неатомарный режимы.** При `--mm:atomicArc` с `hasThreadSupport` инкремент/декремент используют атомарные инструкции CPU. В противном случае — обычная целочисленная арифметика: быстрее, но небезопасна при совместном использовании `ref` между потоками. Используйте `--mm:atomicArc` только когда ref-объекты действительно передаются между потоками.

**Расширение ORC для циклов.** При `--mm:orc` поле `rootIdx` в `RefHeader` позволяет сборщику циклов находить и удалять кандидатов на корни циклов за O(1). Сборщик циклов (реализован в `orc.nim`, включается условно) обрабатывает то, с чем чистый ARC не справляется: циклические графы объектов, где ни один счётчик ссылок не достигает нуля через обычный декремент.
