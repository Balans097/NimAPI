# Справочник модуля `lexbase`

> **Модуль:** `std/lexbase`  
> **Назначение:** Базовый объект и инфраструктура буферизации для построения эффективных лексеров (токенизаторов), читающих из `Stream`.

---

## Общее описание

При написании лексера с нуля возникают две проблемы, не имеющие отношения к самой распознаваемой грамматике: эффективная буферизация ввода и отслеживание номеров строк и столбцов для сообщений об ошибках. `lexbase` берёт обе проблемы на себя, позволяя конкретным лексерам (таким как в `std/json`, `std/xmlparser` и т. д.) сосредоточиться исключительно на распознавании токенов.

### Основная идея: буферизация с сентинелом

Читать ввод из потока посимвольно — слишком медленно: каждое чтение — системный вызов. `lexbase` вместо этого сразу читает большой блок данных во внутренний буфер (`buf`). Лексер затем сканирует буфер на полной скорости работы с памятью.

Ключевое решение — механизм **сентинела** (sentinel). Вместо того чтобы проверять `bufpos < bufLen` на каждый символ (дополнительное ветвление в самом внутреннем цикле), `lexbase` помещает специальный завершающий символ в заранее выбранную позицию буфера — *сентинел*. Сентинел всегда является одним из символов `refillChars` (по умолчанию — символ новой строки). Когда сканер достигает позиции сентинела, он знает: нужно либо дочитать буфер из потока, либо обработать конец строки — оба события всё равно требуют специальной обработки. Таким образом, **в главном цикле обработки символов нет ни одной дополнительной проверки границ**.

```
buf: "Example Text\n ha!"   bufLen = 17
      ^pos = 0     ^sentinel = 12 ('\n')
```

Когда `bufpos` достигает позиции сентинела, вызывается процедура дозаполнения: она копирует необработанный хвост буфера в его начало, читает новый блок из потока и расставляет следующий сентинел.

### Связь с конкретными лексерами

`BaseLexer` объявлен как `object of RootObj`, что означает: вы создаёте собственный тип лексера путём наследования:

```nim
import std/lexbase, std/streams

type
  MyLexer = object of BaseLexer
    token: string   # ваши дополнительные поля
```

Все публичные поля `BaseLexer` (`buf`, `bufpos`, `lineNumber` и др.) напрямую доступны в вашем подтипе.

## Экспортируемые константы

---

### `EndOfFile`

```nim
const EndOfFile* = '\0'
```

Символ-маркер конца файла внутри буфера. Когда поток исчерпан, `lexbase` записывает `'\0'` в текущую позицию буфера — лексерные циклы могут обнаружить EOF тем же механизмом сравнения символов, что и для всего остального, без отдельного булевого флага или проверки длины.

```nim
# Типичное использование в ручном цикле лексера:
while L.buf[L.bufpos] != EndOfFile:
  processChar(L.buf[L.bufpos])
  inc(L.bufpos)
```

---

### `NewLines`

```nim
const NewLines* = {'\c', '\L'}
```

Множество, содержащее два символа перевода строки:

| Символ | Литерал | Смысл |
|---|---|---|
| `'\c'` | CR, `\r`, 0x0D | Возврат каретки (старый Mac OS; первый байт `\r\n` в Windows) |
| `'\L'` | LF, `\n`, 0x0A | Перевод строки (Unix-стандарт; второй байт `\r\n` в Windows) |

Это множество является значением по умолчанию для параметра `refillChars` в `open`. Также удобно использовать в условиях завершения лексерных циклов и классификации символов.

```nim
# Проверка: является ли символ концом строки
if L.buf[L.bufpos] in NewLines:
  # обработать конец строки
```

## Экспортируемые типы

---

### `BaseLexer`

```nim
type BaseLexer* = object of RootObj
  bufpos*:     int    # текущая позиция чтения в buf
  buf*:        string # внутренний буфер
  lineNumber*: int    # текущий номер строки (с единицы)
  offsetBase*: int    # смещение buf[0] от начала потока
  # (приватные: input, sentinel, lineStart, refillChars)
```

Базовый тип для всех лексеров, построенных на этом модуле. Наследуйтесь от него; не создавайте экземпляры напрямую.

#### Публичные поля

| Поле | Тип | Описание |
|---|---|---|
| `bufpos` | `int` | Индекс в `buf` следующего символа для обработки. Увеличивайте его по мере чтения символов. |
| `buf` | `string` | Сам символьный буфер. Читайте символы из него; не изменяйте его размер вручную. |
| `lineNumber` | `int` | Текущий номер строки, начиная с 1. Обновляется процедурами `handleCR` и `handleLF`. |
| `offsetBase` | `int` | Смещение потока для `buf[0]`. Прибавьте `bufpos`, чтобы получить абсолютное смещение от начала ввода: `offsetBase + bufpos`. |

#### Приватные поля (недоступны напрямую)

| Поле | Назначение |
|---|---|
| `input` | Читаемый `Stream`. |
| `sentinel` | Индекс символа-сентинела в `buf`. |
| `lineStart` | Индекс в `buf`, где началась текущая строка; используется для вычисления номера столбца. |
| `refillChars` | Множество символов, при достижении которых проверяется необходимость дозаполнения буфера; по умолчанию `NewLines`. |

## Экспортируемые процедуры

---

### `open`

```nim
proc open*(L: var BaseLexer, input: Stream, bufLen: int = 8192;
           refillChars: set[char] = NewLines)
```

#### Что делает

Инициализирует `BaseLexer` (или любой наследуемый тип) и подготавливает его к чтению из заданного `Stream`. Это первый вызов, который необходимо сделать перед использованием любой другой процедуры `BaseLexer`. Процедура:

1. Сохраняет ссылку на поток.
2. Выделяет внутренний буфер размером `bufLen` символов.
3. Выполняет первое заполнение буфера из потока.
4. Обнаруживает и молча пропускает UTF-8 BOM (`\xEF\xBB\xBF`) в самом начале ввода, чтобы нижестоящие парсеры его не видели.
5. Устанавливает `lineNumber` в 1 (строки нумеруются с единицы).

#### Параметры

| Имя | Тип | Умолч. | Описание |
|---|---|---|---|
| `L` | `var BaseLexer` | — | Объект лексера для инициализации. Объявляется вызывающим кодом. |
| `input` | `Stream` | — | Любой `Stream` (файловый, строковый, в памяти и т. д.). Не должен быть `nil`. |
| `bufLen` | `int` | `8192` | Размер внутреннего буфера в символах. Большие значения сокращают число операций дозаполнения; минимальное разумное значение — 1. |
| `refillChars` | `set[char]` | `NewLines` | Символы-кандидаты в сентинелы. Дозаполнение буфера происходит при достижении одного из этих символов. Обычно оставляйте по умолчанию; пользовательские значения полезны для форматов, где переводы строк не имеют особого смысла, зато значимы другие символы (например, `\0`, `>`). |

#### Примеры

```nim
import std/lexbase, std/streams

var L: BaseLexer
let s = newStringStream("hello\nworld")
L.open(s)
# L готов к работе; L.buf содержит данные, L.lineNumber == 1
```

```nim
import std/lexbase, std/streams

# Чтение из файла
var L: BaseLexer
let fs = openFileStream("data.csv")
L.open(fs, bufLen = 32768)  # большой буфер для крупных файлов
defer: L.close()
```

```nim
import std/lexbase, std/streams

# Пользовательские символы дозаполнения: полезно для формата, где '>'
# является единственным структурным разделителем, а переводы строк — нет
var L: BaseLexer
L.open(newStringStream("<tag>value>"), refillChars = {'>'})
```

### `close`

```nim
proc close*(L: var BaseLexer)
```

#### Что делает

Закрывает `BaseLexer` и его нижестоящий `Stream`. После вызова `close` содержимое `buf` теряет смысл, никакие дальнейшие операции чтения выполнять нельзя. Всегда вызывайте эту процедуру по завершении лексирования, или используйте `defer: L.close()` сразу после `open`, чтобы гарантировать освобождение ресурсов.

#### Пример

```nim
import std/lexbase, std/streams

var L: BaseLexer
L.open(newStringStream("data"))
defer: L.close()
# ... читаем токены ...
```

### `handleCR`

```nim
proc handleCR*(L: var BaseLexer, pos: int): int
```

#### Что делает

Должна вызываться всякий раз, когда сканер встречает символ возврата каретки (`'\c'`, `\r`) в буфере. Выполняет три действия:

1. **Увеличивает `lineNumber`** — CR означает конец строки.
2. **Инициирует дозаполнение буфера при необходимости** — CR является одним из `refillChars` и, следовательно, выступает в роли сентинела.
3. **Поглощает следующий LF, если он есть** — перевод строки в Windows — это `\r\n`. Если сразу после CR идёт `\n`, `handleCR` автоматически пропускает его, так что лексер видит единственный логический конец строки независимо от платформенного соглашения.
4. **Обновляет `lineStart`** — фиксирует начало новой строки, что обеспечивает корректный подсчёт номера столбца.

Возвращает позицию в буфере, с которой следует **продолжить** сканирование после конца строки.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `pos` | `int` | Позиция символа `'\c'` в `buf`. `L.buf[pos]` должен быть равен `'\c'`. |

#### Возвращаемое значение

Новое значение `bufpos`, указывающее на первый символ следующей строки.

#### Пример

```nim
# Внутри сканирующего цикла лексера:
case L.buf[L.bufpos]
of '\c':
  L.bufpos = handleCR(L, L.bufpos)
  # L.bufpos теперь указывает на первый символ следующей строки
  # L.lineNumber увеличен на 1
of '\L':
  L.bufpos = handleLF(L, L.bufpos)
else:
  inc(L.bufpos)
```

### `handleLF`

```nim
proc handleLF*(L: var BaseLexer, pos: int): int
```

#### Что делает

Должна вызываться всякий раз, когда сканер встречает символ перевода строки (`'\L'`, `\n`) в буфере. Это Unix-аналог `handleCR`. Процедура:

1. **Увеличивает `lineNumber`**.
2. **Инициирует дозаполнение буфера при необходимости**.
3. **Обновляет `lineStart`**, указывая на начало новой строки.

В отличие от `handleCR`, она **не** заглядывает вперёд в поисках второго символа новой строки — самостоятельный `\n` является полноценным Unix-окончанием строки.

Возвращает позицию в буфере, с которой следует продолжить сканирование.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `pos` | `int` | Позиция символа `'\L'` в `buf`. `L.buf[pos]` должен быть равен `'\L'`. |

#### Возвращаемое значение

Новое значение `bufpos`, указывающее на первый символ следующей строки.

#### Пример

```nim
of '\L':
  L.bufpos = handleLF(L, L.bufpos)
  # L.lineNumber++, L.lineStart обновлён, bufpos на первом символе следующей строки
```

### `handleRefillChar`

```nim
proc handleRefillChar*(L: var BaseLexer, pos: int): int
```

#### Что делает

Обобщённая версия `handleCR` / `handleLF` для **символов дозаполнения, не являющихся переводами строк**. Если при вызове `open` в `refillChars` были включены символы помимо `'\c'` и `'\L'`, вызывайте эту процедуру, когда сканер наталкивается на один из них. Она инициирует дозаполнение буфера, но **не** увеличивает `lineNumber` и **не** обновляет `lineStart` — это сугубо строко-учётные вещи.

Полезно для форматов (бинарные протоколы, некоторые DSL), где определённый символ-разделитель должен выступать сентинелом, не означая при этом новую строку.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `pos` | `int` | Позиция символа дозаполнения. `L.buf[pos]` должен принадлежать `L.refillChars`. |

#### Возвращаемое значение

Новое значение `bufpos`, с которого следует продолжить сканирование.

#### Пример

```nim
import std/lexbase, std/streams

# Предположим, открыто с refillChars = {'|', '\c', '\L'}
# При встрече '|' дозаполняем буфер, но не считаем новую строку:
of '|':
  L.bufpos = handleRefillChar(L, L.bufpos)
  # буфер дозаполнен; bufpos стоит после '|'
```

### `getColNumber`

```nim
proc getColNumber*(L: BaseLexer, pos: int): int
```

#### Что делает

Возвращает **номер столбца (начиная с 0)** символа по позиции `pos` внутри текущей строки. Столбец 0 — самый левый символ. Результат вычисляется как `abs(pos - L.lineStart)`.

Это ключевая функция для генерации точных сообщений об ошибках: обнаружив проблему на `L.bufpos`, вызовите `getColNumber(L, L.bufpos)`, чтобы узнать, в каком столбце находится некорректный токен.

Заметьте: столбец измеряется в позициях буфера (символах), а не в визуальных колонках экрана. Для ASCII-ввода они совпадают; для многобайтовых UTF-8 последовательностей визуальный столбец может отличаться.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `pos` | `int` | Любая позиция в буфере, обычно `L.bufpos`. |

#### Возвращаемое значение

Номер столбца, начиная с 0.

#### Примеры

```nim
import std/lexbase, std/streams

var L: BaseLexer
L.open(newStringStream("hello world\ngoodbye"))
# Симулируем, что сканер дошёл до позиции 6 ('w' в "world")
L.bufpos = 6

echo getColNumber(L, L.bufpos)  # → 6 (с нуля)
```

```nim
# В процедуре сообщения об ошибке:
proc lexError(L: BaseLexer, msg: string) =
  let line = L.lineNumber
  let col  = getColNumber(L, L.bufpos) + 1  # перевод в счёт с единицы для вывода
  echo msg & " на строке " & $line & ", столбец " & $col
```

### `getCurrentLine`

```nim
proc getCurrentLine*(L: BaseLexer, marker: bool = true): string
```

#### Что делает

Возвращает **полный текст текущей сканируемой строки** в виде строки, заканчивающейся `"\n"`. По умолчанию добавляет вторую строку с символом каретки (`^`), расположенным под `L.bufpos` — визуально указывающим на тот самый символ, который обрабатывался в момент вызова.

Функция спроектирована специально для сообщений об ошибках. Вывод выглядит так:

```
let x = @bad_token
         ^
```

Функция читает вперёд начиная с `L.lineStart` до первого из `'\c'`, `'\L'` или `EndOfFile`, собирая все символы между ними. Состояние лексера при этом не изменяется.

#### Параметры

| Имя | Тип | Умолч. | Описание |
|---|---|---|---|
| `marker` | `bool` | `true` | При `true` добавляет строку с индикатором `^` под текущей позицией `bufpos`. Передайте `false`, чтобы получить только текст строки. |

#### Возвращаемое значение

`string`, содержащая текущую строку исходника (плюс `"\n"`), и — если `marker` равен `true` — вторую строку из пробелов, завершающихся `"^\n"`.

#### Примеры

```nim
import std/lexbase, std/streams

var L: BaseLexer
L.open(newStringStream("let x = 42\necho x"))
L.bufpos = 8  # указываем на '4' в "42"

echo getCurrentLine(L)
# Вывод:
# let x = 42
#         ^
```

```nim
# Минимальный репортер ошибок для ручного лексера:
proc reportError(L: BaseLexer, msg: string) =
  echo "Ошибка (строка ", L.lineNumber, ", столбец ",
       getColNumber(L, L.bufpos) + 1, "): ", msg
  echo getCurrentLine(L)   # показывает строку с маркером ^

# Если лексер наткнулся на неожиданный '#' в столбце 5, строка 3:
# Ошибка (строка 3, столбец 5): неожиданный символ '#'
# foo = #bar
#     ^
```

```nim
# Без маркера — полезно, когда нужен только текст строки:
let lineText = getCurrentLine(L, marker = false)
echo "Проблемная строка: ", lineText
```

## Полный минимальный пример лексера

Ниже показано, как все части складываются в настоящий, пусть и крошечный, лексер, который токенизирует поток на слова и подсчитывает строки:

```nim
import std/lexbase, std/streams

type
  WordLexer = object of BaseLexer

proc initWordLexer(src: string): WordLexer =
  result.open(newStringStream(src))

proc nextToken(L: var WordLexer): string =
  result = ""
  # Пропускаем пробельные символы, обрабатывая переводы строк корректно
  while true:
    case L.buf[L.bufpos]
    of ' ', '\t':
      inc(L.bufpos)
    of '\c':
      L.bufpos = handleCR(L, L.bufpos)
    of '\L':
      L.bufpos = handleLF(L, L.bufpos)
    of EndOfFile:
      return ""
    else:
      break

  # Собираем слово
  while L.buf[L.bufpos] notin {' ', '\t', '\c', '\L', EndOfFile}:
    result.add(L.buf[L.bufpos])
    inc(L.bufpos)

var lex = initWordLexer("hello world\nfoo bar")
defer: lex.close()

var tok = lex.nextToken()
while tok.len > 0:
  echo "строка ", lex.lineNumber, ": ", tok
  tok = lex.nextToken()

# Вывод:
# строка 1: hello
# строка 1: world
# строка 2: foo
# строка 2: bar
```

## Замечания по дизайну и типичные ловушки

**1. Всегда присваивайте возвращаемое значение процедур `handle*`**  
`handleCR`, `handleLF` и `handleRefillChar` возвращают новое значение `bufpos`. Если забыть написать `L.bufpos = handleCR(L, L.bufpos)`, дозаполнение буфера произойдёт, но `bufpos` не обновится — лексер будет вечно перечитывать одну и ту же позицию.

**2. Позиция сентинела нестабильна между дозаполнениями**  
После дозаполнения буфера содержимое `buf` сдвигается: хвост старого буфера становится началом нового. Любой ранее сохранённый индекс в `buf` становится недействительным. Единственный индекс, который корректно переживает дозаполнение, — сам `bufpos` (который `fillBaseLexer` сбрасывает в 0).

**3. `offsetBase + bufpos` даёт абсолютное смещение в потоке**  
Если нужно зафиксировать позицию начала токена в потоке (например, для повторного чтения или точных сообщений об ошибках), сохраняйте `L.offsetBase + L.bufpos` в начале токена, а не просто `L.bufpos`.

**4. `lineNumber` обновляется только в `handleCR` / `handleLF`**  
Если лексер пропускает переводы строк, не вызывая один из этих обработчиков, `lineNumber` будет неверным и `getCurrentLine` будет выдавать некорректные позиции.

**5. `getColNumber` считает с нуля; в сообщениях для пользователей обычно нужен счёт с единицы**  
Прибавляйте 1 к результату `getColNumber` перед выводом пользователю: `getColNumber(L, L.bufpos) + 1`.

**6. UTF-8 BOM пропускается автоматически**  
`open` молча перешагивает через ведущую последовательность `\xEF\xBB\xBF`. Если вы реализуете лексер для формата, в котором BOM имеет смысл — учтите, что `BaseLexer` уже потребил его к моменту, когда начинает работать ваш код.
