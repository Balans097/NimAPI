# chains — Справочное руководство

## Общее описание

Модуль `chains` предоставляет три **обобщённых шаблона** для работы с односвязными и двусвязными списками. В отличие от большинства контейнерных модулей Nim, `chains` **не определяет никаких собственных типов**. Вместо этого он работает с любыми структурами, которые вы предоставляете, при условии что у них есть нужные имена полей. Это подход *intrusive linked list* (интрузивный список): поля связи (`next`, `prev`) живут непосредственно внутри ваших узлов, а отдельный объект-заголовок хранит указатели `head`/`tail`.

### Паттерн интрузивного связного списка

В классической библиотеке связных списков библиотека владеет типом узла, а вы храните данные внутри него. В интрузивном списке всё наоборот: **вы владеете типом узла, а библиотека манипулирует вашими объектами напрямую**. Преимущество — отсутствие лишних аллокаций: каждый узел и есть сам объект данных, и один и тот же объект может участвовать в нескольких списках, имея несколько наборов полей связи.

`chains` использует проверки `when compiles(...)` во время компиляции, чтобы автоматически адаптироваться к любой комбинации полей вашего типа:

| Поля заголовка | Поля узла | Вид списка |
|---|---|---|
| только `head` | только `next` | Односвязный, только вперёд |
| `head` + `tail` | только `next` | Односвязный, с отслеживанием хвоста |
| `head` + `tail` | `next` + `prev` | Двусвязный (полный) |

Параметр типа не передаётся никогда — шаблоны проверяют ваши конкретные объекты во время компиляции и генерируют соответствующий код.

### Минимальная настройка

```nim
# Тип узла — имя произвольное, поля обязаны называться next/prev
type
  Node = ref object
    value: int
    next: Node      # требуется для всех операций со списком
    prev: Node      # опционально — включает двусвязные операции

# Тип заголовка — имя произвольное, поля обязаны называться head/tail
type
  List = object
    head: Node      # требуется
    tail: Node      # опционально — включает O(1) append и отслеживание хвоста
```

Типы полей должны быть указателеподобными (ref или pointer), чтобы `nil` означал отсутствие соседа. Других ограничений нет.

## Экспортируемый API

### `prepend`

```nim
template prepend*(header, node)
```

**Назначение:** Вставляет `node` в **начало** списка (перед текущим головным элементом). После вызова `node` становится новой головой. Это операция O(1) вне зависимости от длины списка.

**Что делает шаблон, шаг за шагом:**

1. Если у заголовка есть поле `head`:
   - Если у `node` есть поле `prev` и список не пуст — устанавливает `prev` старой головы на `node`.
   - Устанавливает `node.next` на старую голову.
   - Устанавливает `header.head` на `node`.
2. Если у заголовка есть поле `tail` и список был пустым — устанавливает `header.tail` на `node` (новый узел одновременно является и головой, и хвостом).

**Пример — построение стека (LIFO) с помощью prepend**

```nim
type
  Node = ref object
    val: int
    next: Node
    prev: Node

type
  List = object
    head: Node
    tail: Node

var list: List
var a = Node(val: 1)
var b = Node(val: 2)
var c = Node(val: 3)

list.prepend(a)   # список: [1]
list.prepend(b)   # список: [2, 1]
list.prepend(c)   # список: [3, 2, 1]

# Обход с головы даёт обратный порядок вставки: 3, 2, 1
var cur = list.head
while cur != nil:
  echo cur.val    # 3, затем 2, затем 1
  cur = cur.next
```

**Когда использовать `prepend`:** когда вы наполняете список и итоговый порядок неважен, или когда нужен именно последний добавленный элемент первым (семантика стека). Всегда O(1).

### `append`

```nim
template append*(header, node)
```

**Назначение:** Вставляет `node` в **конец** списка (после текущего хвостового элемента). После вызова `node` становится новым хвостом. Это операция O(1), если у заголовка есть поле `tail`. Без поля `tail` шаблон обновляет только `head` (если список пуст) — достичь текущего хвоста без обхода невозможно, поэтому новый узел не присоединяется к предыдущему хвосту. Для корректного O(1) append нужно всегда включать `tail` в заголовок.

**Что делает шаблон, шаг за шагом:**

1. Если у заголовка есть поле `head` и список пуст — устанавливает `header.head` на `node`.
2. Если у заголовка есть поле `tail`:
   - Если у `node` есть поле `prev` — устанавливает `node.prev` на старый хвост.
   - Если список был непустым — устанавливает `next` старого хвоста на `node`.
   - Устанавливает `header.tail` на `node`.

**Пример — построение очереди (FIFO) с помощью append**

```nim
type
  Node = ref object
    val: int
    next: Node
    prev: Node

type
  List = object
    head: Node
    tail: Node

var list: List
var a = Node(val: 1)
var b = Node(val: 2)
var c = Node(val: 3)

list.append(a)   # список: [1]
list.append(b)   # список: [1, 2]
list.append(c)   # список: [1, 2, 3]

# Обход с головы даёт порядок вставки: 1, 2, 3
var cur = list.head
while cur != nil:
  echo cur.val    # 1, затем 2, затем 3
  cur = cur.next
```

**Когда использовать `append`:** когда нужно сохранить порядок вставки (семантика очереди, список событий, очередь задач). Всегда включайте `tail` в заголовок для корректной и эффективной работы.

### `unlink`

```nim
template unlink*(header, node)
```

**Назначение:** Удаляет `node` из списка, перевязывая его соседей мимо него. После вызова `node` отсоединён — его собственные поля `next` и `prev` не обнуляются (они по-прежнему указывают на бывших соседей), но ни один сосед больше не указывает обратно на `node`. Указатели `head` и `tail` заголовка обновляются, если `node` оказался первым или последним элементом. Операция O(1).

**Что делает шаблон, шаг за шагом:**

1. Если `node.next` не равен nil — устанавливает `node.next.prev` на `node.prev` (перешагиваем через `node` назад).
2. Если `node.prev` не равен nil — устанавливает `node.prev.next` на `node.next` (перешагиваем через `node` вперёд).
3. Если `header.head == node` — устанавливает `header.head` на `node.prev`.
4. Если `header.tail == node` — устанавливает `header.tail` на `node.next`.

**Замечание о логике обновления head/tail:** присваивания `header.head = node.prev` и `header.tail = node.next` выглядят нелогично (ожидаешь `node.next` для новой головы). Это отражает предположение, что список может обходиться в *обоих* направлениях — `prev` головы является «предыдущим в порядке обхода», что в типичном двусвязном кольце было бы предшественником. Для стандартного двусвязного списка, где `prev` первого узла равен `nil`, удаление головы корректно устанавливает `header.head = nil` именно потому, что `node.prev == nil`. Учитывайте это, если структура ваших указателей отличается от канонической.

**Пример — удаление узла из середины**

```nim
type
  Node = ref object
    val: int
    next: Node
    prev: Node

type
  List = object
    head: Node
    tail: Node

var list: List
var a = Node(val: 1)
var b = Node(val: 2)
var c = Node(val: 3)

list.append(a)
list.append(b)
list.append(c)
# список: [1] <-> [2] <-> [3]

list.unlink(b)
# список: [1] <-> [3]
# b.next по-прежнему указывает на c, b.prev — на a (устаревшие, игнорировать)

assert list.head.val == 1
assert list.tail.val == 3
assert list.head.next.val == 3
```

**Пример — удаление головы**

```nim
list.unlink(a)
# список: [3]
assert list.head.val == 3
assert list.tail.val == 3
```

**Когда использовать `unlink`:** когда нужно удалить произвольный известный узел из середины, начала или конца списка за O(1). Поскольку для корректной перевязки соседей требуется поле `prev`, `unlink` фактически требует наличия `prev` в типе узла. Вызов `unlink` для узла без поля `prev` (односвязного) скомпилируется, но даст неверный результат или ошибку компиляции, так как `node.next.prev` не существует.

## Совместное использование шаблонов: полный пример

Ниже реализована простая двусвязная очередь задач, где задачи можно добавлять в начало или конец и удалять произвольные задачи по ссылке.

```nim
type
  Task = ref object
    name: string
    next: Task
    prev: Task

type
  TaskQueue = object
    head: Task
    tail: Task

proc newTask(name: string): Task =
  Task(name: name)

var q: TaskQueue

let t1 = newTask("фоновая задача")
let t2 = newTask("срочная задача")
let t3 = newTask("обычная задача")

q.append(t1)    # [фоновая]
q.prepend(t2)   # [срочная, фоновая]
q.append(t3)    # [срочная, фоновая, обычная]

# Срочная обрабатывается первой (с головы)
echo q.head.name    # "срочная задача"

# Отменяем обычную задачу, пока она ещё в очереди
q.unlink(t3)        # [срочная, фоновая]

# Опустошаем очередь
var cur = q.head
while cur != nil:
  echo "обрабатываем: ", cur.name
  let nxt = cur.next
  q.unlink(cur)
  cur = nxt
```

## Особенности и ограничения

**Узел нельзя одновременно включать в два списка (с теми же полями).** Поскольку `prev` и `next` — поля самого узла, узел может отслеживать только один набор соседей. Если нужно, чтобы узел участвовал в двух списках, дайте ему два отдельных набора полей связи (например, `nextByTime`, `prevByTime`, `nextByPriority`, `prevByPriority`) и создайте два заголовка.

**`unlink` не обнуляет поля отсоединённого узла.** После `unlink(header, node)` поля `node.next` и `node.prev` по-прежнему хранят старые значения. Не обходите список через поля отсоединённого узла — они устарели. Если в вашем коде это возможно случайно, обнулите поля вручную после отсоединения.

**Нет проверок границ или nil за пределами того, что обеспечивает шаблон.** Шаблоны не защищают от попытки отсоединить узел, которого нет в списке, от передачи `nil` в `prepend`/`append`, или от двойного отсоединения. Такое некорректное использование молча испортит структуру списка.

**Односвязные списки:** если в типе узла есть только `next` (нет `prev`), `prepend` и `append` всё равно скомпилируются и корректно обновят `head`/`tail`. Однако `unlink` требует `prev` для правильной перевязки соседей — без `prev` присваивание `node.next.prev = ...` либо не скомпилируется, либо даст неверный результат.
