# Справочник по модулю `tasks`

> **Модуль:** `std/tasks`  
> **Язык:** Nim  
> **Назначение:** Лёгкие примитивы для создания отложенных, передаваемых между потоками единиц работы — задач, — которые можно ставить в очередь, отправлять рабочим потокам и выполнять позднее.

---

## Ключевая идея

`Task` — это самодостаточный пакет, оборачивающий вызов процедуры вместе со снимком всех её аргументов. После создания `Task` можно положить в очередь, отправить через канал рабочему потоку и выполнить в любой момент позднее с помощью `invoke`. Исходное место вызова не обязано быть активным в момент выполнения.

Такая архитектура лежит в основе реализаций пулов потоков, планировщиков с похищением работы и любых систем, где работа описывается в одном месте, а выполняется в другом.

**Самое важное правило:** у `Task` ровно один владелец в каждый момент времени, и его **нельзя скопировать** — конструктор копирования отключён на уровне типа. Задачу перемещают, а не разделяют.

---

## Экспортируемый тип: `Task`

```nim
type Task* = object
  callback: proc (args, res: pointer) {.nimcall, gcsafe.}
  args: pointer
  destroy: proc (args: pointer) {.nimcall, gcsafe.}
```

`Task` — это обычный объект (не ref) с тремя внутренними полями:

- **`callback`** — GC-безопасный указатель на функцию с соглашением о вызове `nimcall`. Это обёрточная процедура, сгенерированная макросом `toTask`: она распаковывает замороженные аргументы и вызывает исходную функцию.
- **`args`** — сырой указатель на выделенный в разделяемой куче «scratch-объект», хранящий все замороженные значения аргументов. Выделяется через `allocShared0`, что делает его пригодным для межпоточного использования.
- **`destroy`** — GC-безопасный указатель на деструктор, запускающий `=destroy` над полями scratch-объекта перед освобождением разделяемой памяти. Это гарантирует корректную очистку любых управляемых Nim-объектов, хранящихся как аргументы (строк, seq, ref).

### Копирование намеренно запрещено

```nim
proc `=copy`*(x: var Task, y: Task) {.error.}
```

Попытка скопировать `Task` — это **ошибка времени компиляции**. Это осознанное проектное решение: если бы задачу можно было копировать, две копии могли бы попасть к разным потокам и обе попытались бы освободить одну и ту же память аргументов — двойное освобождение. Владение должно быть единственным и передаётся через `move`, а не через копирование.

### Уничтожение происходит автоматически

Когда `Task` выходит из области видимости (при использовании сборщиков мусора ARC/ORC), `=destroy` вызывается автоматически. Он:
1. Выполняет обратный вызов `destroy` для очистки управляемых полей аргументов.
2. Вызывает `deallocShared` для освобождения выделенной разделяемой памяти.

Если `invoke` так и не был вызван, аргументы задачи всё равно будут корректно очищены.

## Макрос: `toTask`

```nim
macro toTask*(e: typed{nkCall | nkInfix | nkPrefix | nkPostfix | nkCommand | nkCallStrLit}): Task
```

**Что делает:** Центральный макрос модуля. Принимает **выражение вызова** — вызов функции со всеми аргументами — и на этапе компиляции преобразует его в объект `Task`. Макрос захватывает и замораживает значения аргументов в момент вызова `toTask`, упаковывая всё в кучевой пакет, который можно выполнить позже через `invoke`.

**Что генерирует макрос:** Для вызова вида `toTask hello(42, "nim")` макрос примерно производит:

1. Приватный тип-структуру (`ScratchObj`) с одним полем на каждый аргумент.
2. Выделение scratch-объекта на разделяемой куче (`allocShared0`).
3. Изоляцию и присвоение каждого значения аргумента в scratch-объект (через `isolate`/`extract` из `std/isolation` — именно это обеспечивает потокобезопасность захваченных данных).
4. Обёрточную процедуру `callback`, которая приводит непрозрачный указатель обратно к типу scratch-структуры, распаковывает каждое поле в локальную переменную и вызывает исходную функцию.
5. Процедуру `destroy`, запускающую `=destroy` на scratch-структуре.
6. Конструирование и возврат объекта `Task`, связывающего все три части вместе.

**Ограничения — что `toTask` не может обернуть:**

- **Вызовы замыканий** — функции, захватывающие переменные из окружающей области видимости, использовать нельзя. Задача должна быть самодостаточной; все данные обязаны передаваться как явные аргументы.
- **Параметры типа `var`** — оборачиваемая функция не может иметь параметры `var` (изменяемые ссылки), потому что такая переменная должна пережить место вызова, чего задача гарантировать не может.
- **Параметры `typed`/`untyped`** — это конструкции исключительно времени компиляции; их нельзя заморозить как значение времени выполнения.
- **Не GC-безопасные функции** — при компиляции с `--threads:on` целевая функция обязана быть GC-безопасной (то есть не должна обращаться к глобальному GC-управляемому состоянию без синхронизации). Это проверяется на этапе компиляции.

**Аргументы должны быть изолируемыми:** Каждое значение аргумента пропускается через `isolate`, который на этапе компиляции проверяет, что значение не разделяет изменяемое состояние ни с чем за пределами задачи. Если аргумент ссылается на `ref`-объект, доступный также откуда-то ещё, компилятор откажет с ошибкой «expression cannot be isolated».

```nim
import std/tasks

proc greet(name: string) =
  echo "Привет, ", name

# Создаём задачу. "Alice" захвачена и заморожена внутри задачи.
let t = toTask greet("Alice")

# t теперь можно хранить, отправлять другому потоку и т.д.
# Когда нужно выполнить:
t.invoke()  # выводит: Привет, Alice
```

**Функции, возвращающие значение:** `toTask` также работает с не-void функциями. Возвращаемое значение записывается через параметр `res: pointer` обратного вызова. Для получения результата нужно передать в `invoke` указатель на переменную правильного типа.

```nim
import std/tasks

proc add(a, b: int): int = a + b

let t = toTask add(10, 32)

var result: int
t.invoke(addr result)
assert result == 42
```

**Параметры `varargs` / `openArray`:** Когда оборачиваемая функция принимает `varargs` или `openArray`, `toTask` автоматически преобразует переданную последовательность в `seq` и сохраняет её в scratch-объекте, поскольку `openArray` не может быть размещён в куче напрямую.

**Статические / литеральные параметры:** Параметры, являющиеся литералами (`42`, `"hello"`, `nil` и т.д.) или статическими значениями времени компиляции, передаются напрямую в сгенерированный вызов без помещения в scratch-объект, поскольку им не нужна кучевая память.

## Процедура: `invoke`

```nim
proc invoke*(task: Task; res: pointer = nil) {.inline, gcsafe.}
```

**Что делает:** Выполняет задачу — вызывает замороженную процедуру с замороженными аргументами. Это единственный способ запустить `Task`; автоматического выполнения нет.

Параметр `res` — необязательный указатель на выходное значение. Когда оборачиваемая процедура возвращает значение, сюда передаётся адрес переменной правильного типа, и `invoke` запишет в неё результат. Когда оборачиваемая процедура `void` — просто опустите `res` или передайте `nil`.

**Задача может быть вызвана несколько раз.** Каждый вызов выполняет обратный вызов заново с теми же замороженными аргументами. Это полезно для повторяющихся фоновых задач, повторного выполнения после сбоя или бенчмаркинга. Обратите внимание: если оборачиваемая процедура изменяет свои аргументы (например, задача захватила `ref`-объект и модифицирует его поля), эти изменения накапливаются между вызовами — scratch-объект не сбрасывается между запусками.

```nim
import std/tasks

var count = 0
proc increment(n: int) = count += n

let t = toTask increment(5)

t.invoke()   # count == 5
t.invoke()   # count == 10
t.invoke()   # count == 15
```

**С возвращаемым значением:**

```nim
import std/tasks

proc multiply(a, b: int): int = a * b

let t = toTask multiply(6, 7)
var result: int

t.invoke(addr result)
assert result == 42

# Повторный вызов допустим:
t.invoke(addr result)
assert result == 42
```

**Важно:** `invoke` проверяет утверждением, что `task.callback` не равен nil. Задача, инициализированная по умолчанию (нулями), вызовет сбой утверждения во время выполнения при попытке её запустить.

## Модуль `isolation` (реэкспортируется)

`tasks` реэкспортирует `std/isolation`. Понимание изоляции критически важно для правильного использования `toTask`.

Процедура `isolate` оборачивает значение и статически проверяет, что оно не разделяет изменяемое кучевое состояние ни с каким другим доступным значением. Процедура `extract` извлекает значение из изолированной обёртки. Вместе они образуют статический защитный шлюз безопасности, гарантирующий, что захваченные аргументы безопасно передавать другому потоку.

Если вы видите ошибку **«expression cannot be isolated»**, это означает, что аргумент, который вы пытаетесь захватить, имеет псевдонимы, делающие небезопасной его передачу задаче. Исправление обычно состоит в создании значения как свежего выражения прямо в вызове `toTask`, а не в передаче уже существующей переменной, которая может быть разделена.

```nim
import std/tasks

type Node = ref object
  value: int

proc processNode(n: Node) = echo n.value

# Это не сработает — x является ref, потенциально разделённым:
# let x = Node(value: 5)
# let t = toTask processNode(x)   # ошибка: cannot be isolated

# Это работает — ref создаётся заново, нигде больше не доступен:
let t = toTask processNode(Node(value: 5))
```

## Типичный паттерн использования

Предполагаемый рабочий процесс для `Task` в сценарии пула потоков или производитель-потребитель:

```nim
import std/tasks

# 1. Определяем рабочую процедуру.
proc processItem(id: int, value: string) =
  echo "Обработка [", id, "]: ", value

# 2. Создаём задачи на стороне производителя и отправляем через канал.
#    (Channel[Task] требует семантики перемещения — используйте send/recv с move.)
var taskQueue: seq[Task]  # упрощённо; в реальном коде используйте Channel[Task]

taskQueue.add toTask processItem(1, "альфа")
taskQueue.add toTask processItem(2, "бета")
taskQueue.add toTask processItem(3, "гамма")

# 3. На стороне рабочего потока получаем и выполняем каждую задачу.
for t in mitems(taskQueue):
  t.invoke()
```

## Что `toTask` не может сделать — сводка

| Ограничение | Причина |
|---|---|
| Нельзя замыкания | Задачи должны быть самодостаточными; замыкания ссылаются на внешнее состояние |
| Нельзя параметры `var` | Место, на которое указывает ссылка, может исчезнуть к моменту выполнения |
| Нельзя параметры `typed`/`untyped` | Только для времени компиляции; нельзя заморозить как значение времени выполнения |
| Не GC-безопасные функции отклоняются (с `--threads`) | Небезопасно вызывать из другого потока |
| Аргументы должны быть изолируемыми | Предотвращает случайное разделение изменяемого кучевого состояния между потоками |

## Архитектурные заметки

**Почему выделение в разделяемой куче?** Аргументы выделяются через `allocShared0` (обнуляемая разделяемая память), а не в потоковой локальной GC-куче. Это означает, что память доступна из любого потока без активации GC — что критично для межпоточной передачи.

**Почему копирование запрещено?** Двойное владение одним и тем же указателем `args` приведёт к двойному освобождению при уничтожении. Система типов статически обеспечивает единственное владение.

**Почему соглашение о вызове `nimcall`?** Указатели на callback и destroy должны использовать известное, стабильное соглашение о вызове, совместимое между потоками и не предполагающее GC-фрейм. `nimcall` — это C-совместимое соглашение Nim, предназначенное именно для этого.

**Режим отладки:** Компиляция с флагом `-d:nimTasksDebug` заставляет `toTask` выводить сгенерированный AST и код в stdout во время компиляции. Это бесценно для понимания того, что именно производит макрос.

## Внутреннее устройство: что генерирует `toTask`

Понимание сгенерированного кода помогает правильно использовать модуль и разбираться в ошибках. Для вызова:

```nim
proc hello(a: int, b: string) = echo $a & b
let literal = "Nim"
let t = toTask(hello(521, literal))
```

Макрос генерирует примерно следующее:

```nim
# 1. Тип scratch-объекта для хранения замороженных аргументов
type ScratchObj_xyz = object
  a: int
  b: string

# 2. Выделение на разделяемой куче (нулевая инициализация)
let scratch_xyz = cast[ptr ScratchObj_xyz](allocShared0(sizeof(ScratchObj_xyz)))

# 3. Изоляция и захват каждого аргумента
block:
  var isoTemp1 = isolate(521)
  scratch_xyz.a = extract(isoTemp1)
  var isoTemp2 = isolate(literal)
  scratch_xyz.b = extract(isoTemp2)

# 4. Обёрточная процедура — будет вызвана из invoke()
proc hello_xyz(args, res: pointer) {.gcsafe, nimcall.} =
  let objTemp = cast[ptr ScratchObj_xyz](args)
  let tmp_a = objTemp.a
  let tmp_b = objTemp.b
  hello(a = tmp_a, b = tmp_b)

# 5. Деструктор — освобождает управляемые поля scratch-объекта
proc destroyScratch_xyz(args: pointer) {.gcsafe, nimcall.} =
  let obj = cast[ptr ScratchObj_xyz](args)
  `=destroy`(obj[])

# 6. Итоговая задача
let t = Task(callback: hello_xyz, args: scratch_xyz, destroy: destroyScratch_xyz)
```

Каждый раз при вызове `t.invoke()` выполняется `hello_xyz(scratch_xyz, nil)` — те же замороженные данные, снова и снова.
