# Справочник по модулю `sysrand`

> **Модуль:** `std/sysrand`  
> **Язык:** Nim  
> **Добавлен в:** Nim 1.6  
> **Назначение:** Криптографически стойкие случайные байты, получаемые непосредственно от операционной системы — для генерации ключей, nonce-значений, токенов, солей и других целей, связанных с безопасностью.

> ⚠️ **Предупреждение о безопасности:** Этот модуль не проходил аудит у специалистов по криптографии. Он предоставляется «как есть», без гарантий. Для продакшн-систем, работающих с чувствительными данными, рекомендуется провести независимый аудит безопасности перед использованием.

---

## Ключевая идея: криптографическая vs. статистическая случайность

Большинство генераторов случайных чисел общего назначения — включая `std/random` из стандартной библиотеки Nim — являются *псевдослучайными*: они производят детерминированные последовательности на основе начального значения (seed). Это вполне подходит для симуляций, игр, перемешивания или любых задач, требующих лишь статистической непредсказуемости. Но для задач безопасности (пароли, токены сессий, ключи шифрования, значения соли) детерминизм — это уязвимость: если злоумышленник сможет определить или повлиять на seed, он сможет предсказать все будущие выходные значения.

`std/sysrand` решает эту проблему, делегируя работу собственному источнику энтропии операционной системы. ОС накапливает истинную случайность из аппаратных событий — тайминга клавиатуры, прерываний диска, тайминга сетевых пакетов, аппаратного шума — и предоставляет её через системный вызов. Результат недетерминирован и, что критически важно, **не засевается ничем, что контролирует программа**, что делает его пригодным для криптографического использования.

## Реализации по платформам

Модуль автоматически выбирает наилучший доступный OS-примитив на этапе компиляции. Выбор делается за вас — для вашей целевой платформы используется правильный вариант:

| Платформа | Механизм |
|---|---|
| **Windows** | `BCryptGenRandom` — криптографический API Windows, использует системный ГСЧ |
| **Linux** | системный вызов `getrandom(2)` — блокируется до инициализации пула энтропии |
| **macOS / iOS** | `SecRandomCopyBytes` — фреймворк Apple Security |
| **OpenBSD** | `getentropy(2)` — высококачественный, максимум 256 байт за вызов |
| **FreeBSD** | `getrandom(2)` — аналогично Linux, максимум 256 байт за вызов |
| **Zephyr RTOS** | `sys_csrand_get` — аппаратный ГСЧ для встроенных систем |
| **Браузер (JS)** | `window.crypto.getRandomValues` — Web Crypto API, максимум 65536 байт за вызов |
| **Node.js** | `crypto.randomFillSync` — криптографический модуль Node.js |
| **Прочие POSIX** | `/dev/urandom` — чтение из случайного устройства ядра |

### Linux: отказ от `getrandom`

На Linux, если вы нацеливаетесь на ядра старше 3.17 (где системный вызов `getrandom` отсутствует), компилируйте с флагом:

```
nim c -d:nimNoGetRandom yourprogram.nim
```

Это откатится к чтению из `/dev/urandom`, который работает на всех POSIX-совместимых системах.

### Почему `/dev/urandom`, а не `/dev/random`?

Резервный POSIX-вариант использует `/dev/urandom`, а не `/dev/random`. На современных ядрах Linux практически нет разницы в безопасности между ними после инициализации пула энтропии — `/dev/random` будет блокироваться бесконечно долго в ожидании большего количества аппаратной энтропии, тогда как `/dev/urandom` — нет. Резервная реализация также проверяет, что открытый файл является символьным устройством (`S_ISCHR`), прежде чем читать из него — для защиты от атаки, при которой злоумышленник подменяет его обычным файлом.

## Экспортируемые процедуры

Экспортируются ровно две процедуры, обе названы `urandom`. Они отличаются только тем, как вы предоставляете выходной буфер.

---

### `urandom` — заполнить существующий буфер

```nim
proc urandom*(dest: var openArray[byte]): bool
```

**Что делает:** Заполняет каждый байт буфера `dest` криптографически стойкими случайными байтами из источника ОС. Возвращает `true` в случае успеха и `false`, если вызов ОС завершился ошибкой.

Это вариант с заполнением на месте (in-place). Вы передаёте предварительно выделенный буфер — любой `array`, `seq` или `openArray` из `byte` — и процедура заполняет его. Внутри вызова новой памяти не выделяется.

**Пустой буфер:** Если длина `dest` равна 0, `urandom` немедленно возвращает `true`, не делая никакого вызова ОС.

**Обработка ошибок:** Булево возвращаемое значение **обязательно** нужно проверять. При неудаче `dest` может быть заполнен частично — не используйте его содержимое, если возвращён `false`. Случаи неудачи редки, но возможны: недостаточная энтропия ОС (в начале загрузки), исчерпание ресурсов ОС или системный вызов, прерванный невосстановимым образом.

На JavaScript-платформах ошибки игнорируются молча (JS-API криптографии не предоставляют индикацию ошибки таким же образом), и всегда возвращается `true`.

```nim
import std/sysrand

# Заполнить массив фиксированного размера (на стеке, без кучи):
var key: array[32, byte]
if not urandom(key):
  raise newException(IOError, "Не удалось сгенерировать случайный ключ")
echo "Байты ключа: ", key

# Заполнить seq:
var token = newSeq[byte](16)
if not urandom(token):
  raise newException(IOError, "Не удалось сгенерировать токен")

# Нулевая длина — всегда успех:
var empty: array[0, byte]
assert urandom(empty) == true
```

**Когда предпочесть эту перегрузку:** Когда у вас уже есть буфер, который нужно заполнить; когда хотите избежать лишнего выделения памяти; когда записываете в поля существующей структуры данных; или при интеграции с библиотекой, предоставляющей собственный буфер.

---

### `urandom` — выделить и вернуть новую последовательность

```nim
proc urandom*(size: Natural): seq[byte]
```

**Что делает:** Выделяет новый `seq[byte]` длины `size`, полностью заполняет его криптографически стойкими случайными байтами из ОС и возвращает его.

В отличие от варианта с заполнением на месте, эта версия **выбрасывает `OSError`** при ошибке, а не возвращает булево значение. Если возвращённый `seq` непустой, каждый его байт является надёжными случайными данными.

**Нулевой размер:** Запрос `urandom(0)` немедленно возвращает пустой `seq[byte]` без каких-либо вызовов ОС.

**Обработка ошибок:** Поскольку ошибки сигнализируются через исключение, проверять возвращаемое значение не нужно. Перехватывайте `OSError`, если требуется мягкая обработка отказов. В большинстве приложений отказ источника случайных чисел ОС невосстановим, и аварийное завершение является правильной реакцией.

```nim
import std/sysrand

# Сгенерировать 32-байтовый (256-битный) ключ шифрования:
let key = urandom(32)
assert key.len == 32

# Сгенерировать 128-битный токен сессии:
let token = urandom(16)

# Сгенерировать nonce типа UUID (16 байт, затем форматировать по необходимости):
let nonce = urandom(16)

# Нулевая длина:
assert urandom(0).len == 0

# Два вызова почти наверняка дадут разные результаты:
assert urandom(32) != urandom(32)
```

**Когда предпочесть эту перегрузку:** Когда нужен максимально простой API; когда байты всё равно нужны в новом `seq`; когда исключения — предпочтительный стиль обработки ошибок.

## Свойства безопасности и гарантии

### Что `urandom` гарантирует

- **Недетерминизм:** Байты нельзя предсказать, наблюдая предыдущий вывод, даже зная алгоритм.
- **Равномерное распределение:** Каждое значение байта 0–255 равновероятно.
- **Независимость вызовов:** Последовательные вызовы дают статистически независимые результаты.
- **Пул энтропии ОС:** Источник засевается ОС из аппаратных событий и никак не зависит от того, что делает ваша программа.

### Что `urandom` НЕ гарантирует

- **Аудит безопасности:** Код Nim-обёртки не проверялся криптографами. Лежащие в основе примитивы ОС — проверялись, обёрточный код — нет.
- **Квантовая устойчивость:** Как и все классические криптографические источники, этот предоставляет биты, которые достаточно мощный квантовый компьютер потенциально может эксплуатировать в некоторых алгоритмах. Сами по себе сырые байты — не проблема; то, что вы с ними строите, может быть или не быть квантово-устойчивым.
- **Прямая секретность состояния:** Если состояние пула энтропии ОС скомпрометировано (например, на виртуальной машине с клонированным снимком), прошлые и будущие результаты могут оказаться скоррелированы.

## Практические паттерны использования

### Генерация криптографического ключа

```nim
import std/sysrand

proc generateAesKey(): array[32, byte] =
  ## Генерирует 256-битный ключ AES.
  if not urandom(result):
    raise newException(IOError, "Источник случайных чисел ОС недоступен")
```

### Генерация URL-безопасного токена

```nim
import std/sysrand, std/base64

proc generateToken(byteLen = 24): string =
  ## Генерирует случайный токен в кодировке base64url.
  ## 24 случайных байта → 32 символа base64.
  let bytes = urandom(byteLen)
  result = encode(bytes, safe = true)

echo generateToken()  # например: "aB3kZ9mQpR2xYvLwNsT4cU7d"
```

### Генерация случайной соли для хеширования паролей

```nim
import std/sysrand

proc generateSalt(): array[16, byte] =
  ## 128-битная соль для использования с функцией хеширования паролей.
  if not urandom(result):
    raise newException(IOError, "Не удалось сгенерировать соль")
```

### Заполнение части большой структуры

```nim
import std/sysrand

type Packet = object
  nonce: array[12, byte]   # 96-битный nonce для AES-GCM
  payload: seq[byte]

proc initPacket(data: seq[byte]): Packet =
  result.payload = data
  if not urandom(result.nonce):
    raise newException(IOError, "Не удалось сгенерировать nonce")
```

## Сравнение: `sysrand` vs `random`

| Свойство | `std/sysrand` | `std/random` |
|---|---|---|
| Источник | Аппаратная энтропия ОС | Алгоритмический ГПСЧ (xoshiro256) |
| Можно задать seed | Нет | Да |
| Детерминированный | Нет | Да (при одинаковом seed) |
| Пригоден для крипто | Да (с оговоркой об аудите) | **Нет** |
| Пригоден для симуляций | Избыточен | Да |
| Скорость | Медленнее (системный вызов) | Быстро |
| API | Только байты | Богатый (целые, float, перемешивание…) |
| Риск блокировки | Возможен при загрузке (Linux) | Никогда |

Используйте `std/sysrand`, когда случайность будет применяться в целях безопасности. Используйте `std/random` для всего остального.

## Итоговая таблица по обработке ошибок

| Перегрузка | При успехе | При ошибке |
|---|---|---|
| `urandom(dest: var openArray[byte])` | Возвращает `true` | Возвращает `false` |
| `urandom(size: Natural): seq[byte]` | Возвращает заполненный `seq` | Выбрасывает `OSError` |

Всегда проверяйте возвращаемое значение первой перегрузки. Никогда не используйте частично заполненный буфер при неудаче. Дизайн второй перегрузки на основе исключений делает невозможным молчаливое игнорирование ошибки.
