# Модуль Cpuinfo — справочник

---

## Что делает этот модуль

`cpuinfo` решает одну конкретную задачу: сколько процессорных ядер есть у машины, на которой работает программа? Ответ нужен почти любой программе, распределяющей работу по потокам — хочется создать примерно столько рабочих потоков, сколько ядер, не больше и не меньше.

Модуль предоставляет ровно одну процедуру — `countProcessors`, — скрывая за ней всю сложность опроса операционной системы на каждой платформе, поддерживаемой Nim. У каждой ОС есть собственный API для этого, и они существенно различаются по соглашениям о вызовах, необходимым заголовочным файлам и смыслу возвращаемого значения.

---

## Что означает «количество процессоров»

Термин намеренно широкий. `countProcessors` возвращает количество **логических процессоров**, видимых операционной системе — количество вычислительных единиц, которым планировщик может независимо назначать потоки. На машине с гиперпоточностью (Hyper-Threading) или SMT (Simultaneous Multi-Threading) каждое физическое ядро выглядит как два логических процессора. На машине без гиперпоточности логические и физические ядра совпадают.

Для большинства параллельных задач именно количество логических процессоров — правильное число для определения размера пула потоков. Это количество потоков, которые реально могут выполняться параллельно в любой момент времени.

---

## Детали реализации по платформам

Реализация целиком выбирается во время компиляции через ветки `when defined(...)`. Обнаружения платформы во время выполнения нет. Понимание того, какая ветка активируется на вашей целевой платформе, помогает предвидеть граничные случаи.

| Платформа | Используемый API | Примечания |
|-----------|-----------------|------------|
| **Windows** | `GetSystemInfo` (Win32) | Читает `dwNumberOfProcessors` из структуры `SYSTEM_INFO`. Возвращает логические процессоры. |
| **macOS** | `sysctlbyname("hw.logicalcpu", ...)` | Запрашивает ключ sysctl `hw.logicalcpu`, являющийся псевдонимом `hw.activecpu`. При неудаче откатывается к `sysctl([CTL_HW, HW_NCPU])`. |
| **BSD** (FreeBSD, OpenBSD, NetBSD) | `sysctl([CTL_HW, HW_NCPU])` | Использует классическую константу sysctl `HW_NCPU`. |
| **Linux / POSIX** (по умолчанию) | `sysconf(SC_NPROCESSORS_ONLN)` | Возвращает только **онлайн**-процессоры — те, что в данный момент включены и доступны планировщику. Процессоры, существующие аппаратно, но отключённые (например, для экономии энергии), не учитываются. |
| **HP-UX** | `mpctl(MPC_GETNUMSPUS, ...)` | Использует интерфейс управления мультипроцессором HP-UX. |
| **IRIX** | `sysconf(_SC_NPROC_ONLN)` | Вариант вызова POSIX sysconf для SGI IRIX. |
| **Haiku** | `get_system_info` | Читает `cpu_count` из структуры `system_info` в `<OS.h>`. |
| **Genode** | `env.cpu().affinity_space().total()` | Запрашивает пространство аффинности CPU среды Genode. |
| **Node.js** | `require("os").cpus().length` | Использует модуль `os` Node.js. |
| **Браузер / Deno** | `navigator.hardwareConcurrency` | Использует Web API свойство, доступное в браузерах и Deno. |

## Экспортируемая процедура

### `countProcessors`

```nim
proc countProcessors*(): int
```

#### Описание

Опрашивает операционную систему и возвращает количество логических процессоров (ядер CPU), доступных на текущей машине. Вызов немедленно делегируется платформозависимой внутренней реализации, выбранной во время компиляции.

Функция не имеет состояния и побочных эффектов. При каждом вызове она напрямую обращается к ОС без кеширования, хотя на практике ОС обычно читает значение из структуры данных ядра и вызов выполняется очень быстро.

#### Возвращаемое значение

Неотрицательное целое число:
- **Положительное целое** — количество логических процессоров, о которых сообщает ОС.
- **Ноль** — количество не удалось определить. Это происходит либо потому что платформозависимый запрос вернул отрицательное значение (которое нормализуется до нуля), либо потому что вызов ОС завершился ошибкой. Ноль следует трактовать как «неизвестно», а не как «нет процессоров».

Функция **никогда не возвращает отрицательное число**.

#### Параметры

Отсутствуют.

#### Примеры

```nim
import cpuinfo

let cores = countProcessors()
echo "Логических процессоров: ", cores
# → например, "Логических процессоров: 8" на четырёхъядерном CPU с гиперпоточностью
```

```nim
import cpuinfo

# Определение размера пула потоков — наиболее частый сценарий
let poolSize = max(1, countProcessors())
# max(1, ...) защищает от случая нуля/неизвестного значения
```

```nim
import cpuinfo, threadpool

setMaxPoolSize(countProcessors())
```

```nim
import cpuinfo

# Решение, стоит ли параллельная обработка накладных расходов
let n = countProcessors()
if n >= 2:
  echo "Параллельный режим: ", n, " ядра"
else:
  echo "Одно ядро или неизвестно; последовательное выполнение"
```

#### Граничные случаи и предостережения

**Контейнеры и виртуальные машины.** В Docker-контейнерах и других средах с ограничением cgroup на Linux `sysconf(SC_NPROCESSORS_ONLN)` как правило возвращает количество логических процессоров **хост-машины**, а не выделенных контейнеру. Квота CPU cgroup (например, `--cpus=2`) не уменьшает значение, возвращаемое этим вызовом. Код, которому нужно количество ядер с учётом контейнера, должен читать файлы cgroup напрямую или использовать библиотеку, делающую это.

**Горячее подключение CPU.** На Linux процессоры можно включать и отключать во время работы системы. `SC_NPROCESSORS_ONLN` отражает количество на момент вызова. Если CPU отключаются динамически после запуска программы, последующие вызовы могут вернуть меньшее число.

**Нулевое возвращаемое значение.** Если `countProcessors()` вернул ноль, безопаснее всего откатиться к количеству потоков равному 1 (чисто последовательное выполнение). Пул с нулём потоков — ошибка программиста.

**Гиперпоточность.** Возвращаемое значение считает логические потоки, а не физические ядра. На четырёхъядерном Intel CPU с гиперпоточностью возвращается, как правило, 8. Использовать ли полное логическое количество или половину для CPU-bound задач — вопрос тонкой настройки производительности, однозначного ответа нет.

**NUMA-системы.** На многопроцессорных NUMA-машинах `countProcessors` возвращает суммарное количество по всем процессорам. Топологическую информацию (какое ядро на каком процессоре) он не предоставляет. Для NUMA-осведомлённого программирования используйте платформозависимые API.

## Настройка времени компиляции

Флагов времени компиляции, изменяющих поведение самой `countProcessors`, нет. Платформенная ветка выбирается автоматически компилятором Nim на основе целевой ОС.

Единственный релевантный флаг компиляции — `--threads:on`, который требуется при использовании результата `countProcessors` для определения размера пула потоков (например, через `setMaxPoolSize` из `threadpool`), но не требуется просто для импорта и вызова `countProcessors`.

---

## Связь с другими модулями

`countProcessors` используется внутри `threadpool` при вызове его `setup()` (который запускается автоматически при старте программы), чтобы определить начальный размер пула:

```nim
# Внутри threadpool.nim (упрощённо):
proc setup() =
  let p = countProcessors()
  currentPoolSize = min(p, MaxThreadPoolSize)
  for i in 0..<currentPoolSize:
    activateWorkerThread(i)
```

Это означает, что в большинстве программ, использующих `threadpool`, вызывать `countProcessors` самостоятельно не нужно — он уже был вызван за вас.
