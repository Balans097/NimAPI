# Справочник модуля `assertions`

> **Назначение модуля:** `assertions` предоставляет строительные блоки для проверки корректности программы во время выполнения. Он даёт инструменты для объявления инвариантов, которые обязаны выполняться, для проверки того, что определённые участки кода бросают ожидаемые исключения, и для настройки поведения при сбое утверждения — с точным контролем над тем, компилируются ли проверки в бинарный файл вообще.

## `raiseAssert`

```nim
proc raiseAssert*(msg: string) {.noinline, noreturn, nosinks.}
```

### Что делает

Самый низкоуровневый элемент модуля. `raiseAssert` безусловно бросает `AssertionDefect`, прикрепляя `msg` в качестве читаемой причины. Выполнение никогда не возвращается из этого вызова — прагма `noreturn` гарантирует это, позволяя компилятору рассуждать о том, что любой код, стоящий после вызова `raiseAssert`, недостижим.

Прагма `noinline` означает, что тело функции не дублируется на местах вызова, что позволяет предсказуемо контролировать размер бинарного файла даже при большом количестве утверждений по всей кодовой базе.

### Когда вызывать напрямую

Вы редко будете вызывать `raiseAssert` самостоятельно. Это общая точка завершения, к которой сходятся `assert`, `doAssert`, `failedAssertImpl` и `doAssertRaises`. Однако вы можете вызывать его явно, когда хотите пометить ветку кода как логически невозможную («этого никогда не должно произойти») без обёртывания в булево условие:

```nim
import std/assertions

proc classify(x: int): string =
  if x > 0: "положительное"
  elif x < 0: "отрицательное"
  elif x == 0: "ноль"
  else:
    # Математически недостижимо, но компилятор этого не знает.
    raiseAssert("невозможно: целое число не >, не < и не == 0")
```

## `failedAssertImpl`

```nim
proc failedAssertImpl*(msg: string) {.raises: [], tags: [].}
```

### Что делает

`failedAssertImpl` — это **брандмауэр системы эффектов** между вашим кодом и сбоями утверждений. Внутри он вызывает `raiseAssert`, но его сигнатура объявляет, что он ничего не бросает (`raises: []`) и не имеет побочных эффектов ввода-вывода или иных (`tags: []`). Это позволяет шаблонам `assert` и `doAssert` использоваться внутри процедур со строгими аннотациями эффектов, не вынуждая эти процедуры объявлять `{.raises: [AssertionDefect].}`.

Философия здесь такова: `AssertionDefect` представляет ошибку программирования, а не восстанавливаемое состояние времени выполнения. Он намеренно невидим для системы эффектов, чтобы случайно не стать частью публичного API-контракта.

### Взаимодействие с `onFailedAssert`

`failedAssertImpl` является намеренной **точкой расширения**. Шаблон `onFailedAssert` работает, локально затеняя этот процессор новым шаблоном с тем же именем. Именно этот механизм подстановки делает возможными пользовательские обработчики сбоев без какого-либо глобального состояния.

Вы почти никогда не будете вызывать `failedAssertImpl` напрямую, но понимание его роли является ключом к пониманию `onFailedAssert`.

```nim
import std/assertions

# Компилируется чисто, даже несмотря на то, что assert может бросать:
proc pureComputation(x: int): int {.raises: [].} =
  doAssert x >= 0, "x должен быть неотрицательным"
  x * 2
```

## `assert`

```nim
template assert*(cond: untyped, msg = "")
```

### Что делает

`assert` — это стандартная проверка корректности. Он вычисляет `cond` во время выполнения и, если `cond` ложно, бросает `AssertionDefect`, включающий местоположение в исходном коде, строковое представление выражения, которое не прошло проверку, и необязательное `msg`.

Ключевая особенность поведения: **`assert` полностью исключается из компиляции при активных `--assertions:off` или `-d:danger`**. Никакого кода не генерируется, условие не вычисляется, побочные эффекты `cond` не возникают. Это делает `assert` подходящим для проверок, стоимость которых важна в production-сборках, но которые нужны во время разработки и тестирования.

### Ключевые режимы поведения

| Ситуация | Поведение |
|---|---|
| `cond` истинно | Ничего не происходит, выполнение продолжается нормально |
| `cond` ложно (стандартная сборка) | Бросает `AssertionDefect` с местом в коде + выражение + `msg` |
| Сборка с `-d:danger` | Весь вызов `assert` исчезает из бинарного файла |
| Сборка с `--assertions:off` | То же, что `-d:danger` для утверждений |

### Примеры

```nim
import std/assertions

# Базовое использование — проверка логического инварианта
let items = @[1, 2, 3]
assert items.len > 0

# С полезным сообщением, которое появляется в ошибке
proc divide(a, b: float): float =
  assert b != 0.0, "деление на ноль не определено"
  a / b

echo divide(10.0, 2.0)   # 5.0
# divide(10.0, 0.0)      # AssertionDefect: ... `b != 0.0` деление на ноль не определено

# В danger-режиме утверждения не несут накладных расходов:
# nim c -d:danger myfile.nim  →  строки assert выше не производят ни одной инструкции
```

## `doAssert`

```nim
template doAssert*(cond: untyped, msg = "")
```

### Что делает

`doAssert` ведёт себя идентично `assert` во всех отношениях **кроме одного**: он **никогда не исключается из компиляции**. Проверка всегда присутствует в бинарном файле, независимо от `--assertions:off` или `-d:danger`. Это делает `doAssert` подходящим для проверок, которые должны выполняться даже в оптимизированных или боевых production-сборках — особенно в тестовом коде и критически важной валидации.

### Когда предпочесть `doAssert` вместо `assert`

- **В юнит-тестах:** вы хотите, чтобы тестовая логика всегда выполнялась, даже если сам тестовый бинарный файл собирается с `-d:release` или `-d:danger`.
- **Для постусловий на внешних данных:** если вы проверяете данные, целостность которых важна во время выполнения (а не только в процессе разработки), `doAssert` делает намерение явным.
- **Когда условие имеет побочные эффекты, которые нужны всегда:** поскольку `assert` может полностью устранить условие, любые побочные эффекты внутри него тоже исчезают. `doAssert` безусловно сохраняет их.

### Примеры

```nim
import std/assertions

# Проверяется всегда — даже в сборке с -d:danger
doAssert 2 + 2 == 4

proc parsePositive(s: string): int =
  result = parseInt(s)
  doAssert result > 0, "ожидалось положительное целое, получено: " & s

# В тестовом наборе: doAssert гарантирует, что проверка выполнится
proc testAdder() =
  let sum = 1 + 1
  doAssert sum == 2, "базовая арифметика сломана"
```

## `onFailedAssert`

```nim
template onFailedAssert*(msg, code: untyped): untyped {.dirty.}
```

### Что делает

`onFailedAssert` устанавливает **пользовательский обработчик сбоев утверждений**, который применяется ко всем вызовам `assert` и `doAssert` в той же области видимости начиная с этой точки. Когда утверждение не проходит, вместо броска `AssertionDefect` стандартным путём выполняется блок `code`. Внутри `code` параметр `msg` содержит полностью отформатированное сообщение о сбое (местоположение в исходнике + выражение, которое не прошло + пользовательское сообщение).

Механизм является **локальным для области видимости**: обработчик действует от вызова `onFailedAssert` до конца охватывающего блока. Он не влияет на утверждения в других областях видимости или модулях. Это позволяет разным частям кодовой базы использовать разные стратегии сбоев без глобального состояния.

### Зачем это нужно

По замыслу стандартный `AssertionDefect` является неперехватываемым дефектом. Но тестовые фреймворки, библиотеки отчётности об ошибках и специализированные подсистемы иногда нуждаются в том, чтобы сбои утверждений производили другой, перехватываемый тип исключения — такой, который несёт дополнительные метаданные: трассировку стека, запись в лог или идентификатор корреляции. `onFailedAssert` — это санкционированный хук для этого.

### Примеры

```nim
import std/assertions

# Пример 1: преобразование сбоя утверждения в перехватываемую ошибку с информацией о местоположении
type RichError = object of CatchableError
  location: tuple[filename: string, line: int, column: int]

block:
  onFailedAssert(msg):
    raise (ref RichError)(
      msg: msg,
      location: instantiationInfo(-2)
    )

  try:
    doAssert 1 == 2, "единица не равна двум"
  except RichError as e:
    echo "Перехвачено в: ", e.location.filename, ":", e.location.line
    echo "Сообщение:     ", e.msg

# Пример 2: логирование и продолжение (для некритических подсистем)
block:
  onFailedAssert(msg):
    echo "[ПРЕДУПРЕЖДЕНИЕ] Утверждение не выполнено: ", msg
    # нет raise — выполнение продолжается

  doAssert false, "это будет залогировано, но не упадёт"
  echo "всё ещё работаем"
```

## `doAssertRaises`

```nim
template doAssertRaises*(exception: typedesc, code: untyped)
```

### Что делает

`doAssertRaises` проверяет, что блок `code` бросает указанный тип `exception` (или его подтип). Если код бросает ожидаемое исключение, шаблон молча завершается успешно. Если код бросает *другое* исключение или завершается вообще без броска, сам `doAssertRaises` бросает `AssertionDefect` с точным сообщением, объясняющим, что пошло не так.

В отличие от `assert` и `doAssert`, этот шаблон **всегда компилируется** — флагов для его отключения нет. По своей природе это конструкция для тестирования.

### Три возможных исхода

| Что делает обёрнутый код | Результат |
|---|---|
| Бросает ожидаемое исключение (или подтип) | ✅ Успех — шаблон завершается нормально |
| Бросает совершенно другое исключение | ❌ `AssertionDefect`: «ожидалось X, вместо этого брошено Y» |
| Не бросает ничего | ❌ `AssertionDefect`: «ожидалось X, вместо этого ничего не брошено» |

### Работа с иерархией исключений

Поскольку исключения Nim образуют иерархию, можно использовать родительский тип, чтобы принять любого из его потомков:

```nim
import std/assertions

# Принимает любой подтип CatchableError:
doAssertRaises(CatchableError):
  raise newException(ValueError, "слишком мало")   # проходит

# Должно быть точным типом или подтипом:
doAssertRaises(ValueError):
  raise newException(ValueError, "плохое значение")   # проходит

# doAssertRaises(ValueError):
#   raise newException(IOError, "диск заполнен")   # упало бы — неверный тип
```

### Примеры

```nim
import std/assertions

# 1. Проверяем, что функция падает на плохих входных данных
proc mustBePositive(n: int): int =
  doAssert n > 0, "n должен быть положительным"
  n

doAssertRaises(AssertionDefect):
  discard mustBePositive(-1)

# 2. Проверяем, что парсер отвергает некорректный ввод
import std/strutils

doAssertRaises(ValueError):
  discard parseInt("не число")

# 3. Проверяем совпадение с иерархией
doAssertRaises(CatchableError):
  raise newException(KeyError, "ключ не найден")

# 4. Как выглядит сбой — если бы этот код выполнился, он бы произвёл:
# AssertionDefect: ожидалось 'ValueError', вместо этого ничего не брошено: echo "привет"
# doAssertRaises(ValueError):
#   echo "привет"
```

## Как выбрать нужный инструмент

| Я хочу… | Использую |
|---|---|
| Проверить инвариант, который можно отключить в production | `assert` |
| Проверить инвариант, который нельзя отключать никогда | `doAssert` |
| Обозначить логически недостижимый путь кода | `raiseAssert` |
| Убедиться, что вызов бросает ожидаемое исключение | `doAssertRaises` |
| Заменить стандартный `AssertionDefect` пользовательской ошибкой | `onFailedAssert` |

### Ментальная модель `-d:danger`

Думайте об этом как о двух уровнях защиты:

- **Уровень разработки** (`assert`): активен при `nim c` и `nim c -d:release`, исчезает при `nim c -d:danger`. Используйте для дорогостоящих проверок, отладочных охранников и всего, что навредило бы производительности в горячем цикле.
- **Постоянный уровень** (`doAssert`, `doAssertRaises`): активен всегда. Используйте для тестовых утверждений, постусловий на непроверенных входных данных и инвариантов, которыми вы не готовы жертвовать ради скорости.
