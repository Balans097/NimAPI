# Справочник модуля `nimprof`

**Модуль:** `nimprof` (часть стандартной библиотеки Nim)   
**Назначение:** Встроенный профилировщик стека вызовов — измеряет, где программа тратит процессорное время (профилирование по CPU) или какие точки вызовов выделяют больше всего памяти (профилирование памяти).

---

## Обзор

`nimprof` — это *встроенный* профилировщик: он запускается внутри самой программы, а не как отдельный внешний инструмент. Активация происходит исключительно через флаги компилятора — никаких внешних агентов или системных утилит не требуется.

Профилировщик перехватывает выполнение через равные промежутки времени, захватывает текущий стек вызовов и подсчитывает, сколько раз каждый уникальный стек был замечен. При завершении программы записывает удобочитаемый отчёт в файл `profile_results.txt`.

### Активация

```sh
# Профилирование CPU / времени (режим по умолчанию)
nim c --profiler:on --stackTrace:on myapp.nim

# Профилирование памяти / выделений
nim c --profiler:on --stackTrace:on -d:memProfiler myapp.nim
```

> **Важно:** Простой импорт `nimprof` без передачи `--profiler:on` компилятору вызывает ошибку компиляции. Модуль принудительно проверяет это через прагму `{.error.}`.

### Как это работает внутри

1. Среда выполнения Nim вызывает лёгкую *hook*-функцию при каждом входе в процедуру (или периодически, в зависимости от настройки интервала).
2. Каждый вызов hook хеширует текущий стек вызовов и сохраняет его в хеш-таблице фиксированного размера (`ProfileData`, 64 К слотов).
3. Когда таблица заполнена, наименее часто встречающаяся запись вытесняется, освобождая место для новых (вытеснение по принципу LRU).
4. При завершении программы `writeProfile` сортирует записи по счётчику попаданий, форматирует их и сохраняет файл.

## Экспортируемый API

Модуль экспортирует ровно **три** символа для публичного использования:

| Символ | Вид | Режим |
|---|---|---|
| `setSamplingFrequency` | процедура | только CPU-профилирование |
| `disableProfiling` | процедура | оба режима |
| `enableProfiling` | процедура | оба режима |

---

## `setSamplingFrequency`

```nim
proc setSamplingFrequency*(intervalInUs: int)
```

### Что делает

Управляет тем, *как часто* профилировщик снимает образцы стека вызовов.

- Параметр `intervalInUs` задаётся в **микросекундах** (мкс).
- Внутри значение переводится в наносекунды и хранится как целевой интервал между выборками.
- Вычитается небольшая поправочная константа (`tickCountCorrection = 50 000 нс`) для компенсации накладных расходов на чтение системного таймера.
- **По умолчанию:** 5 000 мкс (= 5 мс).

**Особый случай — `intervalInUs <= 0`:**  
Передача нуля или отрицательного значения переключает профилировщик с *временно́й* выборки на *счёт инструкций*. В этом режиме hook срабатывает при каждом вызове процедуры, не дожидаясь таймера. Это даёт больше данных, но существенно увеличивает накладные расходы; подходит для очень короткоживущих программ, где временно́й сэмплер не успел бы собрать ничего.

### Когда вызывать

Вызывайте `setSamplingFrequency` **до** того кода, который хотите измерить. Вызов после выполнения горячего участка не повлияет на уже собранные данные.

### Примеры

```nim
import nimprof

# Пример 1 — более редкая выборка: один образец каждые 20 мс.
# Снижает накладные расходы; подходит для длительных программ.
setSamplingFrequency(20_000)

# Пример 2 — более частая выборка: один образец каждую 1 мс.
# Больше деталей, выше накладные расходы; подходит для коротких бенчмарков.
setSamplingFrequency(1_000)

# Пример 3 — режим счёта инструкций.
# Каждый вызов процедуры считается. Очень высокие накладные расходы.
setSamplingFrequency(0)

runMyHotLoop()
```

> **Совет:** Если `profile_results.txt` содержит очень мало записей, программа завершается быстрее одного интервала выборки. Попробуйте `setSamplingFrequency(1_000)` или даже `setSamplingFrequency(0)`.

> **Замечание:** Эта процедура **недоступна** при компиляции с `-d:memProfiler`. Профилирование памяти использует фиксированный интервал — одна выборка на каждые 50 000 выделений — и не настраивается через этот API.

## `disableProfiling`

```nim
proc disableProfiling*()
```

### Что делает

Временно **приостанавливает** сбор данных, отсоединяя hook профилировщика от среды выполнения Nim. После этого вызова входы в процедуры больше не перехватываются и новые стеки вызовов не записываются — до следующего вызова `enableProfiling`.

Внутри используется атомарный счётчик (`disabled`), благодаря чему вложенные пары enable/disable корректно балансируются даже в многопоточном коде.

### Когда использовать

Применяйте `disableProfiling`, чтобы **исключить** из анализа участки кода, которые не представляют интереса. Типичные сценарии:

- Инициализация и завершение программы, которые исказили бы результаты.
- Тяжёлые по I/O участки (чтение конфигов, логирование), если вас интересует только CPU-работа.
- Инициализация сторонних библиотек, которую вы не контролируете.

### Пример

```nim
import nimprof

disableProfiling()           # не считать инициализацию программы
loadConfiguration("app.cfg") # неинтересная настройка
connectToDatabase()          # медленно, но не то, что мы измеряем
enableProfiling()            # начать измерение с этой точки

runCriticalAlgorithm()       # ← вот что нас интересует

disableProfiling()           # остановить перед завершением
closeDatabase()
```

> **Предупреждение:** Вызовы `disableProfiling` и `enableProfiling` учитываются как счётчик ссылок. Каждый `disableProfiling` должен иметь соответствующий `enableProfiling`. Несбалансированные вызовы навсегда отключат профилировщик на оставшееся время работы программы.

## `enableProfiling`

```nim
proc enableProfiling*()
```

### Что делает

Повторно подключает hook профилировщика после вызова `disableProfiling`, возобновляя сбор данных. Если внутренний счётчик показывает, что профилирование должно быть активно, hook заново устанавливается в `system.profilingRequestedHook`.

Проверка `atomicInc(disabled) >= 0` обеспечивает потокобезопасность: hook переустанавливается только тогда, когда счётчик возвращается к неотрицательному значению, предотвращая гонки между потоками, которые могут вызывать `enableProfiling` одновременно.

### Когда использовать

Всегда используйте в паре с `disableProfiling`. Допустима вложенность нескольких пар enable/disable вокруг разных участков кода, хотя на практике это редко нужно.

### Пример

```nim
import nimprof

# Профилировать только конкретную фазу многофазного конвейера.
disableProfiling()
phase1_dataIngest()          # неинтересно

enableProfiling()
phase2_transform()           # ← измеряем это

disableProfiling()
phase3_outputToDisk()        # I/O, неинтересно

enableProfiling()
phase4_postProcess()         # ← и это тоже измеряем

disableProfiling()           # чистый выход перед завершением
```

## Результат — `profile_results.txt`

Отчёт записывается автоматически при завершении программы (через `addExitProc`). Вызывать ничего вручную не нужно.

### Формат

```
total executions of each stack trace:
Entry: 1/42 Calls: 3812/10000 = 38.12% [sum: 3812; 3812/10000 = 38.12%]
  mymodule.nim: hotFunction  3812/10000 = 38.12%
  main.nim: main  10000/10000 = 100.00%
Entry: 2/42 Calls: 2100/10000 = 21.00% [sum: 5912; 5912/10000 = 59.12%]
  ...
```

- **Entry N/M** — это N-я по частоте трасса из M уникальных зарегистрированных трасс.
- **Calls X/Y = Z%** — данная трасса была выбрана X раз из Y всего сэмплов.
- **sum** — накопленное количество сэмплов по всем выведенным до сих пор записям.
- Каждая строка с отступом — один кадр стека, от самого внутреннего к самому внешнему.
- Процент после имени функции показывает, как часто *эта функция* встречалась во **всех** трассах (не только в трассе данной записи), давая вид плоского профиля.
- Записывается не более 100 записей, даже если было собрано больше.

## Полный рабочий пример

```nim
# скомпилировать с помощью:
#   nim c --profiler:on --stackTrace:on -r example.nim

import nimprof

proc fib(n: int): int =
  if n < 2: return n
  return fib(n - 1) + fib(n - 2)

proc heavyWork() =
  for i in 1..30:
    discard fib(i)

proc lightWork() =
  for i in 1..1_000_000:
    discard i * i

setSamplingFrequency(500)   # образец каждые 0,5 мс для большей детализации

disableProfiling()
echo "Запуск бенчмарка..."
enableProfiling()

heavyWork()
lightWork()

disableProfiling()
echo "Готово. Смотрите profile_results.txt"
```

После запуска `profile_results.txt` покажет `fib` в самом верху — из-за рекурсии она часто встречается в сэмплах.

## Ограничения и особенности

- Хеш-таблица вмещает максимум ~96 К уникальных трасс стека. При заполнении редкие трассы молча вытесняются.
- Трассы стека ограничены 21 кадром. Очень глубокий стек вызовов усекается.
- Профилировщик считает *попадания выборки*, а не реальное настенное время напрямую. Функция, вызываемая часто но кратко, может оказаться выше функции, вызываемой редко но надолго — если не используется временно́й сэмплинг.
- Потокобезопасность обеспечивается глобальным мьютексом (`profilingLock`). В сильно параллельных программах этот мьютекс может стать узким местом.
- `setSamplingFrequency` недоступна в режиме профилирования памяти (`-d:memProfiler`).
