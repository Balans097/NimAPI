# Справочник по модулю `sharedlist`

> **Модуль:** `std/sharedlist`  
> **Язык:** Nim  
> **Статус:** ⚠️ **Устарел — нестабильное API.** Модуль помечен `{.deprecated.}` и явно объявлен как имеющий нестабильное API. Не используйте его в новом коде. Для потокобезопасных разделяемых коллекций предпочтите `std/sharedtables` или архитектуру на основе передачи сообщений через `std/channels`.  
> **Назначение:** Обобщённый односвязный список в разделяемой памяти, доступный из нескольких потоков. Защищён единственным грубозернистым мьютексом `Lock`.

---

## Обзор

`SharedList[A]` — потокобезопасный список, преимущественно предназначенный для добавления элементов, где все данные хранятся в **разделяемой памяти** (выделенной через `allocShared0`). Это делает его доступным из любого потока независимо от сборщика мусора. Структура представляет собой односвязный список **блоковых узлов** фиксированного размера: каждый узел хранит до `ElemsPerNode = 100` элементов типа `A`.

### Модель хранения: блоковые узлы

Элементы хранятся не по одному на узел. Каждый `SharedListNode[A]` — это блок, содержащий массив из 100 элементов. Новые узлы выделяются только когда текущий хвостовой блок заполнен. Такой дизайн амортизирует накладные расходы на выделение памяти: вместо одного выделения на элемент — одно выделение на каждые 100 элементов.

```
head ──► [узел: d[0..99], dataLen=100] ──► [узел: d[0..5], dataLen=6, next=nil] ◄── tail
```

Обход всегда идёт от `head` к `tail`. Элементы внутри узла занимают позиции `d[0..dataLen-1]`.

### Модель потокобезопасности

Все экспортируемые процедуры (кроме `init`) используют встроенный `lock: Lock` через внутренний шаблон `withLock`, который захватывает мьютекс перед операцией и освобождает после. Мьютекс **грубозернистый** — каждая операция со списком, включая итерацию, удерживает единственный мьютекс на всём своём протяжении.

Заметное исключение — `iterAndMutate`, которая намеренно освобождает и перезахватывает мьютекс вокруг каждого вызова пользовательской процедуры `action`. Это проектное решение — и его последствия — самое важное, что нужно понять в этом модуле.

## Типы

### `SharedList[A]`

```nim
type SharedList*[A] = object
  head, tail: SharedListNode[A]
  lock*: Lock
```

Публичный тип. `A` — тип элемента. `head` и `tail` указывают на первый и последний блоковые узлы соответственно; оба равны `nil` в пустом списке. Поле `lock` экспортировано (`*`) — в принципе его можно захватить вручную, но это не рекомендуется.

`SharedList` — тип-значение (`object`, не `ref`). Чтобы разделять его между потоками, он должен находиться в разделяемой памяти — встройте его в структуру, выделенную через `ptr`, или объявите глобальной переменной:

```nim
var sharedGlobal: SharedList[int]  # глобальная — доступна из всех потоков
```

## Функции жизненного цикла

### `init`

```nim
proc init*[A](t: var SharedList[A])
```

#### Что делает

Инициализирует `SharedList` перед первым использованием. Вызывает `initLock` на встроенном мьютексе и устанавливает `head` и `tail` в `nil`. **Должна вызываться перед любой другой операцией** со списком — использование `SharedList` без предварительного вызова `init` оставляет мьютекс неинициализированным, что является неопределённым поведением.

`init` **не** захватывает мьютекс (он ещё не инициализирован в момент выполнения `init`), поэтому она должна вызываться из одного потока до того, как список станет общим.

#### Пример

```nim
import std/sharedlist

var list: SharedList[int]
list.init()
```

---

### `deinitSharedList`

```nim
proc deinitSharedList*[A](t: var SharedList[A])
```

#### Что делает

Полностью уничтожает список: вызывает `clear` для освобождения всех блоковых узлов, затем `deinitLock` для уничтожения мьютекса. После этого вызова объект `SharedList` нельзя использовать без нового вызова `init`.

Это правильная пара к `init`. Всегда вызывайте её, когда список больше не нужен, чтобы избежать утечек памяти и ресурсов мьютекса.

#### Пример

```nim
list.deinitSharedList()
```

---

### `clear`

```nim
proc clear*[A](t: var SharedList[A])
```

#### Что делает

Освобождает все блоковые узлы, устанавливая `head` и `tail` обратно в `nil`. Список становится пустым, но мьютекс остаётся нетронутым, а объект `SharedList` — пригодным для использования: после `clear` можно снова вызывать `add`.

Захватывает мьютекс на всё время обхода и освобождения.

Важно: `clear` **не** вызывает деструкторы или финализаторы для элементов, хранящихся в блоках. Если `A` содержит управляемые указатели или другие ресурсы, вы сами отвечаете за их очистку до вызова `clear`.

#### Пример

```nim
list.clear()
# список теперь пуст, но пригоден к использованию
list.add(42)
```

## Функции мутации

### `add`

```nim
proc add*[A](x: var SharedList[A]; y: A)
```

#### Что делает

Добавляет элемент `y` в конец списка. Захватывает мьютекс, затем:

- Если список пуст (`tail == nil`): выделяет новый блоковый узел, устанавливает его как `head` и `tail`, сохраняет `y` в позиции 0.
- Если хвостовой блок заполнен (`dataLen == ElemsPerNode`, то есть 100 элементов): выделяет новый блоковый узел, связывает его как новый `tail`, сохраняет `y` в позиции 0.
- Иначе: добавляет `y` в существующий хвостовой блок на позицию `dataLen` и инкрементирует `dataLen`.

Элементы хранятся в порядке добавления. Однако **`iterAndMutate` может нарушить этот порядок** при удалении элементов (см. ниже).

#### Пример

```nim
list.add(10)
list.add(20)
list.add(30)
```

## Функции итерации и запросов

### `items` (итератор)

```nim
iterator items*[A](x: var SharedList[A]): A
```

#### Что делает

Перебирает все элементы от `head` до `tail`, возвращая каждое значение в порядке хранения. Захватывает мьютекс на всё время итерации — ни один другой поток не может `add` или `clear` пока работает этот итератор.

Поскольку мьютекс удерживается на протяжении всей итерации, `items` предоставляет согласованный снимок списка, но **блокирует всех пишущих** на всё время обхода.

#### Пример

```nim
for val in list.items:
  echo val
```

---

### `iterAndMutate`

```nim
proc iterAndMutate*[A](x: var SharedList[A]; action: proc(x: A): bool)
```

#### Что делает

Перебирает все элементы, давая вызывающему возможность удалить каждый из них. Для каждого элемента вызывается `action` с его значением. Если `action` возвращает `true` — элемент удаляется.

Это самая сложная функция модуля, и её поведение относительно блокировки требует внимательного изучения.

**Стратегия удаления.** Когда элемент в позиции `i` узла `n` должен быть удалён, он заменяется **последним элементом хвостового узла** (`t.d[t.dataLen - 1]`), а `dataLen` хвоста уменьшается. Это операция «заменить последним из хвоста». Она O(1) и не требует сдвига элементов, но **нарушает порядок элементов**. Сам документирующий комментарий модуля предупреждает об этом.

**Паттерн освобождения мьютекса.** Для каждого элемента мьютекс **освобождается** перед вызовом `action` и **перезахватывается** после его возврата. Это намеренно: позволяет `action` вызывать `list.add` без дедлока (поскольку `add` тоже пытается захватить мьютекс). Однако это создаёт окно, в котором другой поток может изменить список между освобождением и перезахватом — итерация не атомарна.

В частности:
- Другой поток может добавить элементы в хвост во время выполнения `action`.
- Итерация посетит эти новые элементы, если они окажутся в узле, который ещё не был пройден (поскольку обход движется от `head` к `tail`, а новые узлы добавляются в хвост).
- Логика удаления (заменить последним из хвоста) может переместить элемент из хвоста на позицию, уже пройденную итератором, — и он не будет посещён в текущем проходе.

В исходном коде также есть известное дизайнерское ограничение: когда `dataLen` хвостового узла достигает 0 в результате удалений, пустой узел не освобождается и не отсоединяется. Это отмечено в коде как TODO, что означает: после интенсивного удаления в хвосте могут накапливаться пустые узлы.

#### Пример

```nim
# Удалить все чётные числа:
list.iterAndMutate(proc(x: int): bool = x mod 2 == 0)
```

#### Когда использовать `iterAndMutate` против `items`

| | `items` | `iterAndMutate` |
|-|---------|-----------------|
| Удаляет элементы? | Нет | Да, когда `action` возвращает `true` |
| Мьютекс удерживается во время колбэка? | Да (вся итерация) | Нет (освобождается на каждый элемент) |
| Порядок сохраняется? | Да | Нет (после удалений) |
| `action` может вызывать `list.add`? | Нет (дедлок) | Да (мьютекс освобождён) |
| Согласованный снимок? | Да | Нет |

## Сводная таблица

| Функция | Захватывает мьютекс | Примечания |
|---------|---------------------|------------|
| `init` | Нет | Должна вызываться первой; мьютекс ещё не инициализирован |
| `add` | Да, на всё время | Выделяет новый блок каждые 100 элементов |
| `items` | Да, на всю итерацию | Блокирует всех пишущих; согласованный снимок |
| `iterAndMutate` | Да, с освобождением на элемент | Неатомарна; допускает `add` из `action`; нарушает порядок при удалении |
| `clear` | Да, на всё время | Освобождает узлы; не вызывает деструкторы элементов |
| `deinitSharedList` | Через `clear` | Освобождает узлы + уничтожает мьютекс; нельзя использовать список после |

## Примечания и подводные камни

- **Устарел.** Модуль помечен как устаревший с нестабильным API. Используйте только для поддержки существующего кода.
- **`init` обязателен.** Если забыть вызвать `init` перед первым использованием, мьютекс останется неинициализированным, что вызовет неопределённое поведение — скорее всего, сбой или гонку данных.
- **`A` должен быть безопасен в разделяемой памяти.** Тип элемента `A` не должен содержать управляемых GC-указателей (ref, string со сборщиком мусора, seq), поскольку они недопустимы через границы потоков в GC-модели Nim. Используйте простые данные, `ptr`-типы или `cstring`.
- **`iterAndMutate` неатомарна.** Освобождение и перезахват мьютекса вокруг каждого вызова `action` означает, что список может измениться между посещениями элементов. Не рассчитывайте на то, что каждый элемент будет посещён ровно один раз.
- **Пустые хвостовые узлы после удаления.** Когда `dataLen` хвостового узла достигает 0 из-за удалений в `iterAndMutate`, пустой узел остаётся связанным как хвост. Последующие вызовы `add` будут использовать его (поскольку `dataLen < ElemsPerNode`), поэтому нового выделения не произойдёт — но нулевой `dataLen` хвостового узла является переходным состоянием, которое модуль не обрабатывает полностью.
- **Порядок не гарантируется после `iterAndMutate`.** Стратегия удаления «заменить последним из хвоста» перемещает элементы. Если порядок важен, не используйте `iterAndMutate` для удаления.
- **`lock` экспортирован, но не следует захватывать его вручную.** Захват `t.lock` вне функций модуля при одновременном вызове функций модуля из другого потока приведёт к дедлоку.
- **Трассировка стека отключена.** Модуль устанавливает `{.stackTrace: off.}`, поэтому исключения или ошибки внутри операций со списком будут производить менее информативные трассировки стека.
