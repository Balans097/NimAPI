# Модуль Atomics — Справочник

> ⚠️ **Нестабильное API.** Модуль входит в стандартную библиотеку Nim, но явно помечен как имеющий нестабильный интерфейс. Сигнатуры и поведение могут меняться между версиями Nim.

> 🔧 **Требование к компиляции.** Требует `--threads:on`. По умолчанию использует атомарные примитивы C11. Для переключения на C++ `std::atomic` скомпилируйте с `-d:nimUseCppAtomics`.

---

## Что делает этот модуль

Современным многоядерным программам нужен способ совместно использовать данные между потоками **без блокировок**. Наивный подход — просто читать и писать обычную переменную из нескольких потоков — небезопасен: и процессор, и компилятор свободны переупорядочивать операции с памятью так, что это невидимо для других ядер, порождая гонки данных и неопределённое поведение.

Этот модуль предоставляет **атомарные операции**: аппаратные инструкции чтения-изменения-записи, гарантированно неделимые и одинаково видимые всем потокам. Он также предоставляет элементы управления **порядком памяти**, позволяющие точно задать, сколько синхронизации вы платите — от абсолютного минимума (`moRelaxed`) до полной гарантии глобальной последовательности (`moSequentiallyConsistent`).

Два центральных типа: `Atomic[T]` — обобщённый контейнер, оборачивающий любой небольшой тип-значение и делающий каждую операцию с ним атомарной — и `AtomicFlag` — минимальный однобитный булев флаг, гарантированно реализованный без блокировок на каждой совместимой платформе.

### Как выбирается реализация

Модуль имеет три пути кода, выбираемых во время компиляции:

- **C++ с `std::atomic`** (`-d:nimUseCppAtomics`): операции напрямую отображаются на методы `std::atomic<T>`. Наиболее чистый путь.
- **C11 atomics (GCC/Clang)**: операции отображаются на типы `_Atomic` и свободные функции `atomic_*_explicit`. Используется по умолчанию для C-бэкенда.
- **MSVC**: C11 недоступен, поэтому модуль откатывается к intrinsics `_Interlocked*`.

Для типов, не умещающихся аккуратно в аппаратный атомарный объект (нетривиальные типы размером больше 8 байт), модуль автоматически откатывается к спин-блокировке, реализованной через `AtomicFlag`.

## Модель порядка памяти

Каждая операция в этом модуле принимает необязательный параметр `order: MemoryOrder`. Понимание его — ключ к правильному использованию модуля.

### `MemoryOrder`

```nim
type MemoryOrder* = enum
  moRelaxed
  moConsume
  moAcquire
  moRelease
  moAcquireRelease
  moSequentiallyConsistent
```

Порядок памяти управляет тем, какие гарантии даются о видимости **других** операций с памятью (не только самой атомарной) относительно данной атомарной операции. Это договор между вашим кодом и процессором/компилятором о том, какие оптимизации запрещены.

---

#### `moRelaxed`

Слабейший порядок. Единственная гарантия: сама атомарная операция неделима — она либо завершается полностью, либо не выполняется вовсе. Процессор и компилятор вправе переупорядочивать её относительно любых неатомарных загрузок и сохранений, и даже относительно других атомарных операций с другими адресами.

**Когда использовать:** Счётчики и статистика, где важно только итоговое значение, но не порядок, в котором инкременты происходят относительно другой работы. Например, потокобезопасный счётчик обращений.

```nim
var counter: Atomic[int]

# Потоки инкрементируют в любом порядке; нас интересует только сумма
counter.atomicInc(1)              # здесь moRelaxed был бы достаточен
echo counter.load(moRelaxed)
```

---

#### `moConsume`

Ослабленная форма acquire, предотвращающая переупорядочивание только операций, которые **зависят по данным** от загруженного значения. Сейчас не рекомендуется комитетом по стандарту C++, потому что компиляторы с трудом реализуют его корректно. В любом практическом коде предпочитайте `moAcquire`.

---

#### `moAcquire`

Применяется к **загрузке** (load). Гарантирует, что ни одно чтение или запись в текущем потоке, стоящие *после* этой загрузки в порядке исходного кода, не будут перемещены процессором или компилятором на выполнение *до* неё.

Иначе говоря: всё, что поток-потребитель делает после acquire-загрузки, гарантированно видит все записи, сделанные до парного release-сохранения в потоке-производителе.

**Когда использовать:** Сторона «потребителя» в паре производитель-потребитель. Потребитель делает acquire-загрузку флага, чтобы проверить готовность данных, а затем читает сами данные.

```nim
# Поток-потребитель
while not ready.load(moAcquire): discard  # крутимся в цикле
echo data  # гарантированно видит запись 'data' производителя
```

---

#### `moRelease`

Применяется к **сохранению** (store). Гарантирует, что ни одно чтение или запись в текущем потоке, стоящие *до* этого сохранения в порядке исходного кода, не будут перемещены на выполнение *после* него.

**Когда использовать:** Сторона «производителя» в паре производитель-потребитель. Производитель пишет данные, затем делает release-сохранение флага, сигнализируя о готовности.

```nim
# Поток-производитель
data = 42                     # сначала пишем данные
ready.store(true, moRelease)  # потом сигнализируем; запись data не может переместиться за эту точку
```

---

#### `moAcquireRelease`

Для операций чтения-изменения-записи (таких как `exchange`, `fetchAdd` и др.), которым нужно одновременно выступать как release для записей перед ними и как acquire для чтений после. Используйте для операций, атомарно и читающих, и записывающих значение.

---

#### `moSequentiallyConsistent`

Сильнейший порядок. В дополнение к гарантиям acquire/release он обеспечивает единый полный порядок всех `moSequentiallyConsistent`-операций в разных потоках. Каждый поток наблюдает их в одинаковой последовательности.

Это **значение по умолчанию** для всех операций в модуле. Наибезопаснейший выбор, наиболее сложный для некорректного использования. На большинстве архитектур (x86) практически не влияет на производительность; на архитектурах со слабым порядком памяти (ARM, POWER) может требовать полной инструкции барьера памяти.

**Когда использовать:** Если сомневаетесь. Также когда нужно, чтобы все потоки согласились с единым глобальным порядком событий — например, при реализации мьютекса или барьера с нуля.

## Экспортируемые типы

### `Atomic[T]`

```nim
type Atomic*[T] = object
```

Обобщённый атомарный контейнер. `T` может быть любым типом, но реализация различается:

- Для **тривиальных типов** (`SomeNumber`, `bool`, `enum`, `ptr`, `pointer`) размером 1, 2, 4 или 8 байт: значение хранится во внутреннем поле, подкреплённом аппаратным атомарным целым нужной ширины. Все операции компилируются в одну инструкцию процессора.
- Для **нетривиальных типов** (структуры, объекты, всё крупнее 8 байт): значение хранится в обычном поле вместе с `AtomicFlag`, выступающим спин-блокировкой. Операции захватывают эту блокировку, выполняют работу и освобождают её. Корректно, но не подлинно lock-free.

`Atomic[T]` всегда должен объявляться как `var` на уровне модуля, как поле объекта в куче или передаваться по `var`-ссылке. Никогда не копируйте его — копирование атомарного объекта не имеет смысла.

---

### `AtomicFlag`

```nim
type AtomicFlag* = object
```

Однобитный булев флаг, чьи две операции — `testAndSet` и `clear` — гарантированно реализованы без блокировок на каждой платформе, совместимой с C11/C++11. В отличие от `Atomic[bool]`, у `AtomicFlag` нет операции `load`; его можно только атомарно установить и атомарно сбросить.

`AtomicFlag` — строительный кирпич для спин-блокировок; внутри модуля он используется `Atomic[T]` для нетривиальных типов.

Свежеобъявленный `AtomicFlag` находится в неопределённом состоянии. Стандарт C гарантирует, что он начинается сброшенным при инициализации нулями (например, как глобальная переменная), но для стековых флагов это не гарантируется. Предпочитайте глобальные или размещённые в куче флаги или явно сбрасывайте их перед первым использованием.

## Операции доступа

Эти операции читают и/или записывают значение, хранящееся в `Atomic[T]`.

---

### `load`

```nim
proc load*[T](location: var Atomic[T]; order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно читает и возвращает текущее значение атомарного объекта. Ни один другой поток не может записать частичное значение — вы либо видите значение до параллельного сохранения, либо после него, но никогда не разорванную смесь байт от обоих.

Параметр `order` должен быть одним из: `moRelaxed`, `moConsume`, `moAcquire` или `moSequentiallyConsistent`. Порядки только для сохранения (`moRelease`, `moAcquireRelease`) недопустимы для загрузок — передавать их — неопределённое поведение.

#### Примеры

```nim
var x: Atomic[int]
x.store(42)

echo x.load()              # → 42  (последовательно согласованный)
echo x.load(moAcquire)     # → 42  (acquire; парный к release-сохранению)
echo x.load(moRelaxed)     # → 42  (relaxed; самый дешёвый, без гарантий порядка)
```

---

### `store`

```nim
proc store*[T](location: var Atomic[T]; desired: T; order: MemoryOrder = moSequentiallyConsistent)
```

#### Описание

Атомарно заменяет текущее значение атомарного объекта на `desired`. Ни один другой поток не может увидеть частичную запись.

Параметр `order` должен быть одним из: `moRelaxed`, `moRelease` или `moSequentiallyConsistent`. Порядки только для загрузки (`moAcquire`, `moConsume`) недопустимы для сохранений.

#### Примеры

```nim
var flag: Atomic[bool]

flag.store(true)                  # последовательно согласованный
flag.store(false, moRelease)      # release; парный к acquire-загрузке в другом потоке
flag.store(true, moRelaxed)       # самый дешёвый; использовать только когда порядок не важен
```

---

### `exchange`

```nim
proc exchange*[T](location: var Atomic[T]; desired: T; order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно заменяет значение атомарного объекта на `desired` и возвращает **старое** значение — то, что было там непосредственно до замены. Чтение и запись происходят как единый неделимый шаг.

Это операция чтения-изменения-записи, поэтому любое значение `MemoryOrder` допустимо для `order`.

#### Примеры

```nim
var state: Atomic[int]
state.store(10)

let old = state.exchange(20)
echo old           # → 10  (предыдущее значение)
echo state.load()  # → 20  (новое значение)
```

```nim
# Распространённый паттерн: получить владение общим указателем, заменив его на nil
var sharedPtr: Atomic[ptr MyObj]
let mine = sharedPtr.exchange(nil)
if mine != nil:
  # Теперь он наш; обрабатываем и освобождаем
  mine[].process()
```

---

### `compareExchange`

```nim
proc compareExchange*[T](location: var Atomic[T]; expected: var T; desired: T;
                          order: MemoryOrder = moSequentiallyConsistent): bool

proc compareExchange*[T](location: var Atomic[T]; expected: var T; desired: T;
                          success, failure: MemoryOrder): bool
```

#### Описание

Краеугольный камень lock-free программирования. Атомарно выполняет следующее:

1. Читает текущее значение `location`.
2. Сравнивает его с `expected`.
3. **Если они равны:** заменяет значение на `desired`, возвращает `true`. Переменная `expected` остаётся неизменной.
4. **Если они не равны:** записывает текущее значение `location` в `expected` (тем самым сообщая вызывающему, каково реальное значение), возвращает `false`. Атомарный объект остаётся неизменным.

Это «сильный» вариант, то есть он **никогда не завершается ложно** — если значения равны, обмен всегда успешен. См. `compareExchangeWeak` для варианта, допускающего ложные сбои, но потенциально более дешёвого в циклах на некоторых архитектурах.

Перегрузка с двумя `MemoryOrder` позволяет задать разные порядки для случая успеха (полная операция чтения-изменения-записи) и случая неудачи (только загрузка).

**Именно через эту операцию реализуется любая lock-free структура данных.** Типичный паттерн — цикл повторных попыток: читаем текущее значение, вычисляем новое на его основе, пробуем CAS, и если он не удался (другой поток изменил значение параллельно) — повторяем.

#### Примеры

```nim
var counter: Atomic[int]
counter.store(5)

# Пробуем изменить 5 → 10
var expected = 5
let ok = counter.compareExchange(expected, 10)
echo ok             # → true  (обмен удался)
echo expected       # → 5     (не изменилось при успехе)
echo counter.load() # → 10

# Пробуем снова; но counter теперь 10, а не 5
var expected2 = 5
let ok2 = counter.compareExchange(expected2, 99)
echo ok2            # → false  (не удалось; значения не совпали)
echo expected2      # → 10     (обновлено до реального значения)
echo counter.load() # → 10    (не изменилось)
```

```nim
# Классический lock-free инкремент через цикл CAS
proc lockFreeInc(counter: var Atomic[int]) =
  var current = counter.load(moRelaxed)
  while not counter.compareExchange(current, current + 1, moRelaxed, moRelaxed):
    discard  # current обновляется при каждой неудаче; просто повторяем
```

---

### `compareExchangeWeak`

```nim
proc compareExchangeWeak*[T](location: var Atomic[T]; expected: var T; desired: T;
                              order: MemoryOrder = moSequentiallyConsistent): bool

proc compareExchangeWeak*[T](location: var Atomic[T]; expected: var T; desired: T;
                              success, failure: MemoryOrder): bool
```

#### Описание

По семантике идентичен `compareExchange` с одним отличием: ему разрешено **завершаться ложно** — даже когда текущее значение совпадает с `expected`, операция может вернуть `false`. На некоторых архитектурах (особенно ARM с инструкциями load-linked/store-conditional) слабый вариант отображается на одну аппаратную инструкцию и дешевле сильного в цикле повторных попыток.

Используйте `compareExchangeWeak` вместо `compareExchange` **всякий раз, когда вы и так используете операцию внутри цикла**, потому что цикл уже обрабатывает ложные сбои.

Используйте `compareExchange` (сильный вариант), когда нужно отличить «значения не совпали» от «что-то пошло не так», то есть когда цикл невозможен.

#### Примеры

```nim
# Слабый вариант предпочтителен внутри цикла повторных попыток
proc lockFreeAdd(location: var Atomic[int]; delta: int) =
  var current = location.load(moRelaxed)
  while not location.compareExchangeWeak(current, current + delta, moRelaxed, moRelaxed):
    discard  # ложный или реальный сбой; в любом случае повторяем с обновлённым current
```

## Числовые операции

Эти операции доступны только для `Atomic[T]`, где `T` — `SomeInteger`. Все они возвращают **исходное значение**, которое было в атомарном объекте *до* операции, а не результат. Именно эта семантика «fetch-then-operate» отражена в названиях `fetch*`.

---

### `fetchAdd`

```nim
proc fetchAdd*[T: SomeInteger](location: var Atomic[T]; value: T;
                                order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно прибавляет `value` к атомарному целому и возвращает то, что было там **до** сложения. Это единая неделимая операция чтения-изменения-записи — ни один другой поток не может наблюдать промежуточное состояние.

#### Примеры

```nim
var n: Atomic[int]
n.store(10)

echo n.fetchAdd(3)  # → 10  (возвращает старое значение)
echo n.load()       # → 13  (новое значение)
echo n.fetchAdd(3)  # → 13
echo n.load()       # → 16
```

---

### `fetchSub`

```nim
proc fetchSub*[T: SomeInteger](location: var Atomic[T]; value: T;
                                order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно вычитает `value` из атомарного целого и возвращает **исходное значение**.

#### Примеры

```nim
var n: Atomic[int]
n.store(10)

echo n.fetchSub(4)  # → 10  (исходное)
echo n.load()       # → 6
```

---

### `fetchAnd`

```nim
proc fetchAnd*[T: SomeInteger](location: var Atomic[T]; value: T;
                                order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно заменяет атомарное целое его **побитовым И** с `value` и возвращает исходное значение. Полезно для атомарного сброса конкретных битов в слове флагов.

#### Примеры

```nim
var flags: Atomic[uint32]
flags.store(0b1111_1111'u32)

# Атомарно сбрасываем два младших бита
let before = flags.fetchAnd(0b1111_1100'u32)
echo before.toBin(8)       # → "11111111"
echo flags.load.toBin(8)   # → "11111100"
```

---

### `fetchOr`

```nim
proc fetchOr*[T: SomeInteger](location: var Atomic[T]; value: T;
                               order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно заменяет атомарное целое его **побитовым ИЛИ** с `value` и возвращает исходное значение. Полезно для атомарной установки конкретных битов в слове флагов.

#### Примеры

```nim
var flags: Atomic[uint32]
flags.store(0b0000_0000'u32)

# Атомарно устанавливаем бит 3
discard flags.fetchOr(0b0000_1000'u32)
echo flags.load.toBin(8)   # → "00001000"
```

---

### `fetchXor`

```nim
proc fetchXor*[T: SomeInteger](location: var Atomic[T]; value: T;
                                order: MemoryOrder = moSequentiallyConsistent): T
```

#### Описание

Атомарно заменяет атомарное целое его **побитовым исключающим ИЛИ** с `value` и возвращает исходное значение. Полезно для атомарного переключения конкретных битов.

#### Примеры

```nim
var flags: Atomic[uint32]
flags.store(0b1010_1010'u32)

# Инвертируем все биты
discard flags.fetchXor(0b1111_1111'u32)
echo flags.load.toBin(8)   # → "01010101"
```

## Удобные арифметические операции

Это тонкие обёртки над `fetchAdd` / `fetchSub`, отбрасывающие возвращаемое значение в случаях, когда вас интересует только побочный эффект, а не старое значение.

---

### `atomicInc`

```nim
proc atomicInc*[T: SomeInteger](location: var Atomic[T]; value: T = 1)
```

#### Описание

Атомарно увеличивает атомарное целое на `value` (по умолчанию 1). Старое значение отбрасывается. Эквивалентно `discard location.fetchAdd(value)`, но читается понятнее, когда предыдущее значение не нужно.

#### Примеры

```nim
var counter: Atomic[int]
counter.store(0)

counter.atomicInc()    # увеличить на 1
counter.atomicInc(5)   # увеличить на 5
echo counter.load()    # → 6
```

### `atomicDec`

```nim
proc atomicDec*[T: SomeInteger](location: var Atomic[T]; value: T = 1)
```

#### Описание

Атомарно уменьшает атомарное целое на `value` (по умолчанию 1). Старое значение отбрасывается.

#### Примеры

```nim
var counter: Atomic[int]
counter.store(10)

counter.atomicDec()    # → 9
counter.atomicDec(3)   # → 6
echo counter.load()    # → 6
```

### Оператор `+=`

```nim
proc `+=`*[T: SomeInteger](location: var Atomic[T]; value: T)
```

Синтаксический сахар для `atomicInc`. Позволяет использовать привычный синтаксис `x += n` для атомарных целых. Внутри вызывает `fetchAdd` и отбрасывает результат.

## Операции с флагами

---

### `testAndSet`

```nim
proc testAndSet*(location: var AtomicFlag; order: MemoryOrder = moSequentiallyConsistent): bool
```

#### Описание

Атомарно устанавливает `AtomicFlag` в `true` и возвращает его **предыдущее** значение. Это операция чтения-изменения-записи, гарантированно реализованная без блокировок.

Семантика: «был ли флаг уже установлен?» — если возвращаемое значение `false`, флаг был сброшен и вы только что его установили (вы «выиграли гонку»); если `true` — флаг уже был установлен кем-то другим.

`testAndSet` — фундаментальный строительный кирпич спин-блокировки: крутитесь в цикле, пока `testAndSet` возвращает `true` (блокировка удерживается кем-то), и останавливайтесь, когда вернёт `false` (вы захватили блокировку).

#### Примеры

```nim
var flag: AtomicFlag

echo flag.testAndSet()   # → false  (был сброшен; теперь установлен)
echo flag.testAndSet()   # → true   (уже был установлен)
echo flag.testAndSet()   # → true   (всё ещё установлен)
```

```nim
# Минимальная спин-блокировка
var lock: AtomicFlag

proc acquire(lock: var AtomicFlag) =
  while lock.testAndSet(moAcquire): discard  # крутимся, пока не установим

proc release(lock: var AtomicFlag) =
  lock.clear(moRelease)
```

---

### `clear`

```nim
proc clear*(location: var AtomicFlag; order: MemoryOrder = moSequentiallyConsistent)
```

#### Описание

Атомарно сбрасывает `AtomicFlag` в `false`. Это аналог `testAndSet` и гарантированно реализован без блокировок. После `clear` следующий вызов `testAndSet` из любого потока вернёт `false` и установит флаг.

Параметр `order` при использовании флага как спин-блокировки должен быть `moRelease` или `moSequentiallyConsistent`, чтобы все записи в память, сделанные пока блокировка удерживалась, были видны следующему её захватчику.

#### Примеры

```nim
var flag: AtomicFlag

discard flag.testAndSet()  # устанавливаем
flag.clear()               # сбрасываем
echo flag.testAndSet()     # → false  (снова был сброшен)
```

## Операции с барьерами памяти

Эти операции не действуют на какой-либо конкретный атомарный объект. Вместо этого они вставляют барьеры, управляющие тем, как компилятор и процессор вправе переупорядочивать операции с памятью глобально.

---

### `fence`

```nim
proc fence*(order: MemoryOrder)
```

#### Описание

Вставляет **аппаратный барьер памяти** — инструкцию (или набор инструкций) процессора, обеспечивающую ограничения порядка на все обращения к памяти, как атомарные, так и неатомарные, пересекающие границу барьера.

Барьер — «самостоятельное» ограничение порядка, не привязанное к какой-либо конкретной ячейке памяти. Это тяжелее, чем прикрепление порядка к атомарной операции, но иногда необходимо при координации с неатомарным кодом или аппаратными устройствами.

Смысл `order` тот же, что и для атомарных операций:
- `moAcquire`: запрещает перемещение загрузок выше барьера.
- `moRelease`: запрещает перемещение сохранений ниже барьера.
- `moAcquireRelease`: объединяет оба.
- `moSequentiallyConsistent`: полный барьер; все потоки видят одинаковый порядок.

Барьеры не запрещают компилятору кешировать значения в регистрах. Для взаимодействия с отображёнными в память регистрами аппаратного обеспечения используйте `volatile` (недоступен напрямую в стандартной библиотеке Nim).

#### Примеры

```nim
# Поток-производитель
data[0] = 1
data[1] = 2
fence(moRelease)      # все сохранения выше видны до любого сохранения ниже
readyFlag.store(true, moRelaxed)

# Поток-потребитель
while not readyFlag.load(moRelaxed): discard
fence(moAcquire)      # все загрузки ниже видят записи, сделанные производителем до его release-барьера
echo data[0]          # → 1
echo data[1]          # → 2
```

---

### `signalFence`

```nim
proc signalFence*(order: MemoryOrder)
```

#### Описание

Как `fence`, но **не генерирует инструкций процессора** — только предотвращает **переупорядочивание компилятором** обращений к памяти через границу барьера. Не влияет на аппаратное переупорядочивание между ядрами процессора.

`signalFence` предназначен специально для синхронизации между потоком и обработчиком сигналов (Unix-сигналы), работающим в том же потоке. Поскольку обработчик сигнала прерывает поток и выполняется в его контексте, аппаратный порядок памяти не является проблемой — важно только переупорядочивание компилятором. Использование `signalFence` вместо `fence` в этом сценарии позволяет избежать ненужных (и дорогостоящих) аппаратных инструкций барьера.

#### Примеры

```nim
# Синхронизация с обработчиком сигнала в том же потоке
var signalFlag: Atomic[bool]

proc mySignalHandler() {.noconv.} =
  signalFlag.store(true, moRelaxed)

# В главном потоке проверяем флаг, установленный обработчиком сигнала
signalFence(moAcquire)  # запрещаем компилятору кешировать чтение выше этой точки
if signalFlag.load(moRelaxed):
  echo "Сигнал получен"
```

## Быстрый справочник по выбору порядка памяти

| Сценарий | Порядок загрузки | Порядок сохранения |
|----------|-----------------|-------------------|
| Счётчик/статистика, порядок не важен | `moRelaxed` | `moRelaxed` |
| Публикация данных: производитель пишет, потом сигнализирует | — | `moRelease` |
| Потребление данных: потребитель проверяет сигнал, потом читает | `moAcquire` | — |
| Чтение-изменение-запись в lock-free структуре | `moAcquireRelease` | `moAcquireRelease` |
| Нужно, чтобы все потоки согласились с единым порядком | `moSequentiallyConsistent` | `moSequentiallyConsistent` |
| Неуверен / на первом месте корректность | `moSequentiallyConsistent` | `moSequentiallyConsistent` |

## Типичные паттерны

**Потокобезопасный счётчик ссылок:**
```nim
var refCount: Atomic[int]
refCount.store(1)

proc retain() = refCount.atomicInc()

proc release(): bool =
  # fetchSub возвращает старое значение; если было 1 — мы только что обнулили счётчик
  result = refCount.fetchSub(1, moAcquireRelease) == 1
```

**Спин-блокировка из AtomicFlag:**
```nim
var lock: AtomicFlag

template withSpinLock(body: untyped) =
  while lock.testAndSet(moAcquire): discard
  try: body
  finally: lock.clear(moRelease)

withSpinLock:
  sharedData.modify()
```

**Флаг однократной инициализации:**
```nim
var initialised: Atomic[bool]

proc ensureInit() =
  var expected = false
  if initialised.compareExchange(expected, true, moAcquireRelease):
    # Мы первые; выполняем реальную инициализацию
    setup()
  else:
    # Кто-то другой инициализирует или уже готово
    while not initialised.load(moAcquire): discard
```
