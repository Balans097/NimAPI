# Справочник по модулю `sysatomics`

> **Модуль:** `sysatomics` (внутренний модуль рантайма; современный публичный аналог — `std/atomics`)  
> **Язык:** Nim  
> **Назначение:** Низкоуровневые атомарные операции для lock-free конкурентного программирования — атомарные чтение, запись, обмен, сравнение-и-замена, арифметика, барьеры памяти и подсказки процессору.

> ⚠️ **Предупреждение об устаревании:** При компиляции с флагом `-d:nimPreviewSlimSystem` модуль выводит предупреждение об устаревании. Новый код должен использовать `std/atomics`.

---

## Зачем нужны атомарные операции?

В многопоточной программе обычные чтения и записи в разделяемые переменные небезопасны: компилятор и процессор свободны изменять порядок инструкций, кешировать значения в регистрах и выполнять записи частично. Результат — гонки данных, разорванные чтения и тонко сломанная логика, которую практически невозможно отладить.

Атомарные операции гарантированы аппаратно быть **неделимыми**: ни один другой поток не может наблюдать операцию в частично выполненном состоянии. Они являются фундаментом всех lock-free структур данных, подсчёта ссылок, флагов, счётчиков и примитивов синхронизации.

---

## Поддержка платформ

Модуль предоставляет две различные реализации:

- **GCC / Clang / LLVM-GCC / Nintendo Switch** — использует встроенные функции стиля C11 `__atomic_*`, дающие полный контроль над моделью памяти.
- **MSVC / Clang-CL (Windows)** — использует семейство Windows-интринсиков `Interlocked*`. Константы модели памяти существуют для совместимости API, но фактическое поведение барьеров отображается на соглашения Windows.
- **Прочие / однопоточные** — резервные реализации, которые мягко деградируют до обычных операций при отсутствии поддержки потоков.

## Экспортируемый тип: `AtomType`

```nim
type AtomType* = SomeNumber | pointer | ptr | char | bool
```

Ограничение типа-класса, используемое всеми обобщёнными атомарными процедурами. Только эти категории типов являются допустимыми аргументами. Более крупные составные типы (объекты, строки, последовательности) не могут использоваться атомарно через этот API — они не помещаются в регистр процессора и не могут быть обменяны неделимо.

## Константы модели памяти (`AtomMemModel`)

Понимание упорядочивания памяти — самая важная концепция в lock-free программировании. Эти константы указывают процессору и компилятору, насколько свободно они могут переставлять окружающие операции вокруг атомарной инструкции.

> **Доступно только на GCC/Clang.** На MSVC эквивалентные константы существуют для совместимости API, но реализация отображает их на соответствующие Windows-барьеры.

### `ATOMIC_RELAXED`

Самое слабое упорядочивание. Сама атомарная операция неделима, но компилятор и процессор свободно могут переставлять **любые** другие загрузки и сохранения вокруг неё. Используйте только тогда, когда вам нужна атомарность для корректности (например, счётчик, который никогда не используется для синхронизации), и вам не нужно устанавливать какое-либо отношение happens-before с другими потоками.

```
Поток A:                   Поток B:
flag.store(RELAXED)        if flag.load(RELAXED): ...
data = 42                  # data может оказаться 0!
```

### `ATOMIC_CONSUME`

Слабое упорядочивание типа acquire, привязанное к **цепочкам зависимостей данных**. Если поток B читает значение-указатель, которое было записано потоком A с зависимостью данных, процессор обязан сделать данные по указателю видимыми. На практике большинство компиляторов отображают это на `ACQUIRE` для безопасности.

### `ATOMIC_ACQUIRE`

Для **загрузок** (чтений). Никакое чтение или запись в текущем потоке, которые следуют за этой загрузкой в программном порядке, не могут быть переставлены так, чтобы оказаться перед ней. В паре с `RELEASE`-сохранением в другом потоке устанавливает точку синхронизации: всё, что поток A записал до своего `RELEASE`-сохранения, видимо потоку B после его `ACQUIRE`-загрузки.

```
Поток A (производитель):         Поток B (потребитель):
data = 42                        while not ready.load(ACQUIRE): discard
ready.store(RELEASE)             echo data  # гарантированно 42
```

### `ATOMIC_RELEASE`

Для **сохранений** (записей). Никакое чтение или запись в текущем потоке, которые предшествуют этому сохранению в программном порядке, не могут быть переставлены так, чтобы оказаться после него. Образует пару с `ACQUIRE`-загрузками в других потоках, формируя отношение happens-before.

### `ATOMIC_ACQ_REL`

Одновременно семантика acquire и release. Используется для операций **чтение-изменение-запись** (как обмен или сравнение-и-замена), которые и читают старое значение, и записывают новое, и нуждаются в синхронизации в обоих направлениях.

### `ATOMIC_SEQ_CST`

Самое сильное упорядочивание. Полный барьер памяти в обоих направлениях. Все последовательно согласованные атомарные операции во всех потоках выглядят так, как будто выполняются в единственном, глобально согласованном тотальном порядке. Это самый безопасный выбор и тот, который по умолчанию используют `atomicInc`, `atomicDec` и `cas`. На архитектурах со слабой моделью памяти (ARM, POWER) это также наиболее медленный вариант.

## Операции загрузки

### `atomicLoadN`

```nim
proc atomicLoadN*[T: AtomType](p: ptr T, mem: AtomMemModel): T
```

**Что делает:** Атомарно читает и возвращает значение, хранящееся по адресу `p`. Ни один другой поток не может выполнить запись в `*p` таким образом, чтобы получилось разорванное (частично записанное) чтение.

Это возвращающая значение версия («N» = «native/scalar», скалярная). Наиболее удобная форма для загрузки примитивных значений.

**Допустимые модели памяти:** `ATOMIC_RELAXED`, `ATOMIC_CONSUME`, `ATOMIC_ACQUIRE`, `ATOMIC_SEQ_CST`.

```nim
var counter: int = 0
let val = atomicLoadN(counter.addr, ATOMIC_ACQUIRE)
# val содержит то, что было последним атомарно сохранено в counter;
# все записи, выполненные до соответствующего RELEASE-сохранения,
# теперь видимы в этом потоке.
```

---

### `atomicLoad`

```nim
proc atomicLoad*[T: AtomType](p, ret: ptr T, mem: AtomMemModel)
```

**Что делает:** Обобщённая (указатель-к-указателю) версия `atomicLoadN`. Атомарно читает значение по адресу `p` и записывает его в `*ret`. Полезна, когда нужно работать с результатом через указатель, а не по значению.

```nim
var source: int = 99
var dest: int
atomicLoad(source.addr, dest.addr, ATOMIC_SEQ_CST)
# dest == 99
```

## Операции сохранения

### `atomicStoreN`

```nim
proc atomicStoreN*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel)
```

**Что делает:** Атомарно записывает `val` по адресу `p`. Запись неделима — ни один другой поток не увидит частичную запись.

**Допустимые модели памяти:** `ATOMIC_RELAXED`, `ATOMIC_RELEASE`, `ATOMIC_SEQ_CST`.  
(`ACQUIRE` и `CONSUME` недопустимы для сохранений — эти упорядочивания имеют смысл только для чтений.)

```nim
var ready: bool = false
# ... подготавливаем разделяемые данные ...
atomicStoreN(ready.addr, true, ATOMIC_RELEASE)
# Любой поток, который затем загрузит `ready` с ACQUIRE,
# гарантированно увидит все записи, выполненные до этого сохранения.
```

---

### `atomicStore`

```nim
proc atomicStore*[T: AtomType](p, val: ptr T, mem: AtomMemModel)
```

**Что делает:** Обобщённая версия `atomicStoreN`. Атомарно сохраняет значение `*val` по адресу `*p`.

```nim
var target: int = 0
var newVal: int = 42
atomicStore(target.addr, newVal.addr, ATOMIC_RELEASE)
```

---

## Операции обмена

### `atomicExchangeN`

```nim
proc atomicExchangeN*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делает:** Атомарно записывает `val` в `*p` и возвращает **старое** значение, которое там хранилось. Чтение и запись происходят как единая неделимая операция — ни один другой поток не может вклиниться между ними.

Это фундаментальный примитив обмена. Полезен для реализации спинлоков (записываем `true`, получаем обратно `false` → мы захватили блокировку) и передачи владения.

**Допустимые модели памяти:** все.

```nim
var lock: bool = false

# Попытка захватить спинлок:
let wasLocked = atomicExchangeN(lock.addr, true, ATOMIC_ACQ_REL)
if not wasLocked:
  echo "Блокировка захвачена!"
  # ... критическая секция ...
  atomicStoreN(lock.addr, false, ATOMIC_RELEASE)
```

---

### `atomicExchange`

```nim
proc atomicExchange*[T: AtomType](p, val, ret: ptr T, mem: AtomMemModel)
```

**Что делает:** Обобщённая версия `atomicExchangeN`. Атомарно сохраняет `*val` в `*p` и записывает старое значение `*p` в `*ret`.

## Сравнение-и-замена (CAS)

Сравнение-и-замена — самый мощный примитив в lock-free программировании. Это строительный блок практически всех lock-free алгоритмов: очередей, стеков, счётчиков, подсчёта ссылок.

### `atomicCompareExchangeN`

```nim
proc atomicCompareExchangeN*[T: AtomType](
    p, expected: ptr T,
    desired: T,
    weak: bool,
    success_memmodel: AtomMemModel,
    failure_memmodel: AtomMemModel): bool
```

**Что делает:** Сравнивает значение по адресу `*p` с `*expected`:

- Если они **равны**: записывает `desired` в `*p`, возвращает `true`. Упорядочивание памяти этого успешного обмена определяется `success_memmodel`.
- Если они **не равны**: записывает текущее значение `*p` обратно в `*expected` (чтобы можно было увидеть, что там реально находилось), возвращает `false`. Упорядочивание определяется `failure_memmodel`, который не может быть `RELEASE` или `ACQ_REL` и не должен быть сильнее `success_memmodel`.

Флаг `weak`: «слабый» CAS может ложно завершиться неудачей, даже когда значения совпадают (на некоторых архитектурах, для повышения эффективности). Всегда используйте `weak = false`, если только вы не находитесь в цикле повтора при неудаче (где ложные неудачи безвредны, а `weak` может быть быстрее).

**Когда использовать:** Реализация lock-free обновлений, когда нужно прочитать-вычислить-записать и гарантировать, что ни один другой поток не изменил значение между чтением и записью. Канонический паттерн — цикл с повтором.

```nim
var counter: int = 0

# Атомарно увеличиваем counter на 1, что бы ни делали другие потоки.
proc incrementSafely(p: ptr int) =
  while true:
    let old = atomicLoadN(p, ATOMIC_RELAXED)
    let newVal = old + 1
    if atomicCompareExchangeN(p, old.unsafeAddr, newVal, false,
                               ATOMIC_SEQ_CST, ATOMIC_RELAXED):
      break  # успех — наше новое значение на месте
    # иначе: другой поток изменил p раньше нас; повторяем со свежим old
```

---

### `atomicCompareExchange`

```nim
proc atomicCompareExchange*[T: AtomType](
    p, expected, desired: ptr T,
    weak: bool,
    success_memmodel: AtomMemModel,
    failure_memmodel: AtomMemModel): bool
```

**Что делает:** Идентично `atomicCompareExchangeN`, но `desired` передаётся как указатель, а не по значению. Используйте, когда желаемое значение велико или уже находится за указателем.

## Арифметические и битовые операции — «fetch-then-op» vs «op-then-fetch»

Существуют два семейства арифметических атомарных операций. Различие состоит только в том, **какое значение возвращается**:

| Семейство | Возвращает | Паттерн именования |
|---|---|---|
| `atomicFetch*` | **Старое** значение (до операции) | `atomicFetchAdd`, `atomicFetchSub`, … |
| `atomic*Fetch` | **Новое** значение (после операции) | `atomicAddFetch`, `atomicSubFetch`, … |

Все операции обоих семейств принимают любую модель памяти и применяют операцию атомарно. Промежуточное состояние не наблюдаемо другими потоками.

### `atomicFetchAdd` / `atomicAddFetch`

```nim
proc atomicFetchAdd*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T  # возвращает СТАРОЕ
proc atomicAddFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T  # возвращает НОВОЕ
```

**Что делают:** Атомарно добавляют `val` к `*p`.

```nim
var hits: int = 0

# Подсчёт посещения страницы; получаем счётчик ДО этого инкремента:
let before = atomicFetchAdd(hits.addr, 1, ATOMIC_RELAXED)

# Подсчёт посещения страницы; получаем счётчик ПОСЛЕ этого инкремента:
let after = atomicAddFetch(hits.addr, 1, ATOMIC_RELAXED)
```

---

### `atomicFetchSub` / `atomicSubFetch`

```nim
proc atomicFetchSub*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
proc atomicSubFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делают:** Атомарно вычитают `val` из `*p`. Классический случай использования — подсчёт ссылок: когда `atomicSubFetch` возвращает 0, текущий поток знает, что он последним освободил ссылку.

```nim
var refCount: int = 3

let remaining = atomicSubFetch(refCount.addr, 1, ATOMIC_ACQ_REL)
if remaining == 0:
  echo "Последняя ссылка освобождена — освобождаем объект."
```

---

### `atomicFetchOr` / `atomicOrFetch`

```nim
proc atomicFetchOr*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
proc atomicOrFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делают:** Атомарно применяют побитовое ИЛИ с `val` к `*p`. Полезно для установки отдельных битов в разделяемой битовой маске без блокировки.

```nim
var flags: int = 0b0000
const FLAG_READY = 0b0001

discard atomicOrFetch(flags.addr, FLAG_READY, ATOMIC_RELEASE)
# Бит FLAG_READY установлен, независимо от параллельных записей в другие биты
```

---

### `atomicFetchAnd` / `atomicAndFetch`

```nim
proc atomicFetchAnd*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
proc atomicAndFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делают:** Атомарно применяют побитовое И с `val` к `*p`. Полезно для сброса конкретных битов без воздействия на остальные.

```nim
var flags: int = 0b1111
const CLEAR_BUSY = 0b1101  # сбрасывает бит 1

discard atomicAndFetch(flags.addr, CLEAR_BUSY, ATOMIC_RELEASE)
```

---

### `atomicFetchXor` / `atomicXorFetch`

```nim
proc atomicFetchXor*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
proc atomicXorFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делают:** Атомарно применяют побитовое ИСКЛЮЧАЮЩЕЕ ИЛИ с `val` к `*p`. XOR с маской переключает биты: если бит был 0, становится 1, и наоборот.

```nim
var toggleState: int = 0
const TOGGLE_BIT = 0b0001

discard atomicXorFetch(toggleState.addr, TOGGLE_BIT, ATOMIC_SEQ_CST)
# бит 0 теперь равен 1
discard atomicXorFetch(toggleState.addr, TOGGLE_BIT, ATOMIC_SEQ_CST)
# бит 0 снова равен 0
```

---

### `atomicFetchNand` / `atomicNandFetch`

```nim
proc atomicFetchNand*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
proc atomicNandFetch*[T: AtomType](p: ptr T, val: T, mem: AtomMemModel): T
```

**Что делают:** Атомарно вычисляют `NOT ((*p) AND val)` и сохраняют результат. NAND — универсальная функция и может выразить любую булеву функцию, но практические применения редки. Иногда встречается в специализированном коде драйверов оборудования или манипуляциях с битовыми полями.

> ⚠️ Замечание: `__atomic_nand_fetch` в GCC до версии 4.4 реализовывался неверно (вычислял `~*p & val` вместо `~(*p & val)`). Проверяйте поведение при нацеливании на очень старые компиляторы.

## Test-and-Set / Clear

### `atomicTestAndSet`

```nim
proc atomicTestAndSet*(p: pointer, mem: AtomMemModel): bool
```

**Что делает:** Атомарно читает байт по адресу `p`, устанавливает его в некоторое ненулевое «установленное» значение (определяется реализацией, обычно `1`) и возвращает `true`, если байт уже был «установлен» до операции, и `false` — в противном случае.

Это традиционный примитив для реализации спинлоков: если `atomicTestAndSet` возвращает `false`, значит, флаг успешно установлен и блокировка захвачена. Если возвращает `true` — флаг уже установлен кем-то другим, нужно ждать и повторить попытку.

Обратите внимание: `p` имеет тип нетипизированного `pointer` — работает с сырыми байтами, а не с типизированным `bool` или `int`.

```nim
var spinlock: byte = 0

# Захват спинлока с busy-wait:
while atomicTestAndSet(spinlock.addr, ATOMIC_ACQUIRE):
  cpuRelax()  # подсказываем процессору, что мы в цикле ожидания

# ... критическая секция ...
atomicClear(spinlock.addr, ATOMIC_RELEASE)
```

---

### `atomicClear`

```nim
proc atomicClear*(p: pointer, mem: AtomMemModel)
```

**Что делает:** Атомарно записывает `0` в байт по адресу `p`. Это дополнение к `atomicTestAndSet` — используется для снятия блокировки или сброса флага, установленного с помощью `atomicTestAndSet`.

**Допустимые модели памяти:** `ATOMIC_RELAXED`, `ATOMIC_RELEASE`, `ATOMIC_SEQ_CST`.

## Барьеры памяти

Барьеры — это барьеры синхронизации, которые не работают с конкретным адресом памяти. Они ограничивают переупорядочивание *всех* операций с памятью относительно инструкции барьера.

### `atomicThreadFence`

```nim
proc atomicThreadFence*(mem: AtomMemModel)
```

**Что делает:** Вставляет барьер памяти между потоками. Все операции с памятью (загрузки и сохранения), которые предшествуют барьеру в программном порядке, не могут быть переставлены так, чтобы оказаться после него, и наоборот — в степени, определяемой `mem`. Это межпоточный барьер синхронизации.

```nim
# Гарантируем, что все предшествующие записи видимы перед последующими чтениями:
atomicThreadFence(ATOMIC_SEQ_CST)
```

---

### `atomicSignalFence`

```nim
proc atomicSignalFence*(mem: AtomMemModel)
```

**Что делает:** Похож на `atomicThreadFence`, но синхронизирует только между **потоком и обработчиками сигналов, выполняющимися в том же потоке**. Не выдаёт никаких инструкций CPU-барьера — только ограничивает переупорядочивание компилятором. Значительно дешевле `atomicThreadFence`, но бесполезен для межпоточной синхронизации.

---

### `fence` (шаблон / процедура)

```nim
template fence*()  # GCC: эквивалент atomicThreadFence(ATOMIC_SEQ_CST)
proc fence*()      # MSVC: вызывает _ReadWriteBarrier
```

**Что делает:** Удобный полный барьер. Эквивалент наиболее сильного возможного `atomicThreadFence`. Используйте это как простой, переносимый способ вставить полный барьер памяти, не думая о том, какую модель передать.

## Запросы о возможностях lock-free

### `atomicAlwaysLockFree`

```nim
proc atomicAlwaysLockFree*(size: int, p: pointer): bool
```

**Что делает:** Возвращает `true`, если атомарные операции над объектами размером `size` байт **всегда** генерируют lock-free инструкции для целевой архитектуры, как определено на **этапе компиляции**. Результат является константой времени компиляции.

«Lock-free» здесь означает, что процессор имеет нативные атомарные инструкции для данного размера и не нуждается во внутренних библиотечных блокировках. Объекты размером 1, 2, 4 и 8 байт, как правило, всегда lock-free на современных архитектурах. 16-байтные объекты (например, 128-битные целые или два указателя) lock-free на некоторых x86-64 с `CMPXCHG16B`, но не универсально.

`p` — необязательная подсказка выравнивания (передайте `nil` для типичного выравнивания).

```nim
echo atomicAlwaysLockFree(sizeof(int), nil)   # true на большинстве платформ
echo atomicAlwaysLockFree(16, nil)             # может быть false
```

---

### `atomicIsLockFree`

```nim
proc atomicIsLockFree*(size: int, p: pointer): bool
```

**Что делает:** Похож на `atomicAlwaysLockFree`, но может генерировать **вызов во время выполнения** для получения ответа. Возвращает `true`, если для объектов размером `size` байт будут использоваться lock-free атомарные инструкции. Если ответ не может быть определён на этапе компиляции, вызывается функция времени выполнения `__atomic_is_lock_free`.

Используйте `atomicAlwaysLockFree`, когда вам нужна константа времени компиляции; используйте `atomicIsLockFree`, когда допустима проверка во время выполнения.

## Высокоуровневые процедуры-утилиты

Эти процедуры доступны на всех платформах и автоматически выбирают правильную подлежащую реализацию (GCC-интринсики, MSVC Interlocked или последовательный запасной вариант).

### `atomicInc`

```nim
proc atomicInc*(memLoc: var int, x: int = 1): int {.discardable.}
```

**Что делает:** Атомарно добавляет `x` к `memLoc` и возвращает новое значение. По умолчанию `x = 1`, что делает это простым атомарным инкрементом. Использует `ATOMIC_SEQ_CST` для максимальной безопасности.

Это простейший атомарный примитив, доступный в этом модуле — никакого выбора модели памяти, никакой арифметики указателей, просто безопасный инкремент разделяемого целого числа на месте.

```nim
var visitors: int = 0

# В любом потоке:
let myCount = atomicInc(visitors)
echo "Посетитель номер: ", myCount
```

---

### `atomicDec`

```nim
proc atomicDec*(memLoc: var int, x: int = 1): int {.discardable.}
```

**Что делает:** Атомарно вычитает `x` из `memLoc` и возвращает новое значение. По умолчанию `x = 1`, что делает это простым атомарным декрементом. Использует `ATOMIC_SEQ_CST`.

```nim
var activeConnections: int = 0

atomicInc(activeConnections)   # соединение открыто
# ... обрабатываем соединение ...
let remaining = atomicDec(activeConnections)
if remaining == 0:
  echo "Все соединения закрыты."
```

---

### `addAndFetch`

```nim
proc addAndFetch*(p: ptr int, val: int): int
```

**Что делает:** Атомарно добавляет `val` к `*p` и возвращает **старое** значение (до сложения). Это более низкоуровневый примитив, используемый внутри `atomicInc`/`atomicDec` на MSVC.

На однопоточных или неподдерживаемых сборках деградирует до обычного инкремента без атомарности.

---

### `cas`

```nim
proc cas*[T: bool|int|ptr](p: ptr T; oldValue, newValue: T): bool
```

**Что делает:** Удобное, переносимое сравнение-и-замена. Атомарно проверяет, равно ли `*p` значению `oldValue`; если так — записывает `newValue` в `*p` и возвращает `true`. В противном случае возвращает `false` и оставляет `*p` неизменным.

Внутри использует `atomicCompareExchangeN` на GCC/Clang с `ATOMIC_SEQ_CST`, `Interlocked`-функции на MSVC, встроенный ассемблер на TCC или устаревший `__sync_bool_compare_and_swap` на старом GCC.

Принимает только типы `bool`, `int` и указатели (все размером с указатель или меньше на большинстве платформ).

```nim
var ticket: int = 0

# Только один поток успешно изменит ticket с 0 на 1:
if cas(ticket.addr, 0, 1):
  echo "Этот поток получил талон 1."
else:
  echo "Другой поток оказался первым."
```

## Подсказка процессору при спин-ожидании

### `cpuRelax`

```nim
proc cpuRelax*()
```

**Что делает:** Выдаёт инструкцию-подсказку процессору, что текущий поток находится в цикле busy-wait (спин-ожидании). На x86/x86-64 это выдаёт инструкцию `PAUSE`, которая снижает энергопотребление, предотвращает конвейерные опасности и может улучшить производительность других одновременных аппаратных потоков (гиперпоточность).

На других архитектурах может выдавать компиляторный барьер памяти (`asm volatile("" ::: "memory")`), предотвращающий полную оптимизацию цикла компилятором, или использовать `YieldProcessor` на Windows.

Эта процедура не влияет на корректность программы — это исключительно оптимизация производительности и энергопотребления. Однако отсутствие `cpuRelax` в спин-циклах на x86 может значительно снизить производительность как самого ожидающего потока, так и всех потоков, разделяющих тот же физический ядро процессора.

```nim
var flag: bool = false

# Ждём, пока другой поток установит флаг:
while not atomicLoadN(flag.addr, ATOMIC_ACQUIRE):
  cpuRelax()  # ведём себя хорошо во время ожидания

echo "Флаг установлен!"
```

## Краткий справочник: какую процедуру использовать?

| Цель | Рекомендуемая процедура |
|---|---|
| Чтение разделяемой переменной | `atomicLoadN` |
| Запись разделяемой переменной | `atomicStoreN` |
| Обмен значения, получить старое | `atomicExchangeN` |
| Условное обновление (lock-free цикл) | `cas` или `atomicCompareExchangeN` |
| Инкремент разделяемого счётчика | `atomicInc` или `atomicFetchAdd` |
| Декремент, обнаружить ноль | `atomicSubFetch` (возвращает новое значение) |
| Установить/сбросить отдельные биты | `atomicOrFetch` / `atomicAndFetch` |
| Захватить спинлок | `atomicTestAndSet`, затем `atomicClear` для освобождения |
| Полный барьер памяти | `fence()` |
| Эффективное спин-ожидание | `cpuRelax()` внутри цикла |
| Проверить, является ли тип lock-free | `atomicAlwaysLockFree` |

## Краткий справочник по модели памяти

| Модель | Для чего | Направление |
|---|---|---|
| `ATOMIC_RELAXED` | Счётчики без нужды в синхронизации | Любое |
| `ATOMIC_ACQUIRE` | Загрузки, начинающие критическую секцию | Чтение |
| `ATOMIC_RELEASE` | Сохранения, завершающие критическую секцию | Запись |
| `ATOMIC_ACQ_REL` | Чтение-изменение-запись (CAS, обмен) | Оба |
| `ATOMIC_SEQ_CST` | Безопасный выбор по умолчанию; глобальный порядок | Любое |
