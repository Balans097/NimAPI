# exitprocs — справочник по модулю

## Обзор

Модуль `std/exitprocs` позволяет регистрировать **функции очистки, автоматически вызываемые при завершении программы**. Это Nim-эквивалент функции `atexit` из стандартной библиотеки C, дополненный поддержкой обычных Nim-замыканий наряду с низкоуровневыми `noconv`-коллбэками, а также утилитами для чтения и установки кода возврата процесса.

Все зарегистрированные обработчики вызываются в порядке **«последним зарегистрирован — первым вызван» (LIFO)**. Регистрация защищена блокировкой, что делает вызов `addExitProc` из нескольких потоков безопасным. Диспетчер вызовов при завершении также работает под той же блокировкой.

**Поддерживаемые платформы:**

| Целевая платформа | Механизм подключения |
|---|---|
| Native (Linux, macOS, Windows, …) | `atexit()` из стандартной библиотеки C |
| Node.js | `process.on('exit', …)` |
| Браузерный JS | `window.onbeforeunload` |

**Важное ограничение.** Когда программа завершается из-за **необработанного исключения**, обработчики выхода намеренно *не* вызываются рантаймом автоматически. Если они должны выполниться даже при аварийном завершении, вызывайте их вручную — например, оборачивая тело `main` в `try/finally`.

## `addExitProc` — перегрузка для замыканий

```nim
proc addExitProc*(cl: proc() {.closure.})
```

### Описание

Регистрирует **процедуру-замыкание** для вызова при завершении программы. Замыканием является любой обычный Nim-`proc` или лямбда, способные захватывать переменные из окружающей области видимости. Это та перегрузка, которую вы будете использовать в подавляющем большинстве случаев.

При первом вызове `addExitProc` (любой из перегрузок) внутренний диспетчер `callClosures` регистрируется в платформенном механизме выхода. Последующие вызовы только добавляют запись во внутренний список — дополнительная платформенная регистрация не требуется.

Процедуры хранятся в последовательности и вызываются в **обратном порядке регистрации (LIFO)** при завершении программы. Это зеркалит поведение `atexit` в C и означает, что ресурсы, захваченные позже (зарегистрированные позже), освобождаются раньше ресурсов, захваченных раньше, — естественная дисциплина, напоминающая RAII.

### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `cl` | `proc() {.closure.}` | Регистрируемая процедура очистки |

### Примеры

**Базовая очистка — закрытие файла:**

```nim
import std/exitprocs

let f = open("output.log", fmWrite)
addExitProc(proc() =
  f.close()
  echo "Лог-файл закрыт.")

# f будет автоматически закрыт при завершении программы,
# даже если явный close() никогда не будет достигнут по потоку выполнения.
```

**Захват состояния в замыкании — логирование прошедшего времени:**

```nim
import std/exitprocs, std/times

let startTime = now()

addExitProc(proc() =
  let elapsed = now() - startTime
  echo "Общее время работы: ", elapsed)

# … остаток программы …
```

**Порядок LIFO — демонтаж в обратном порядке к монтажу:**

```nim
import std/exitprocs

addExitProc(proc() = echo "1: подключение к базе данных")  # зарегистрирован первым
addExitProc(proc() = echo "2: открытие кэша")              # зарегистрирован вторым
addExitProc(proc() = echo "3: запуск сервера")             # зарегистрирован третьим

# При завершении вывод будет:
# 3: запуск сервера
# 2: открытие кэша
# 1: подключение к базе данных
# то есть в обратном порядке — сервер останавливается до кэша,
# кэш — до базы данных.
```

**Несколько регистраций — каждый вызов добавляет независимо:**

```nim
import std/exitprocs

for i in 1..3:
  let n = i  # захватываем текущее значение
  addExitProc(proc() = echo "Шаг очистки ", n)

# При завершении:
# Шаг очистки 3
# Шаг очистки 2
# Шаг очистки 1
```

## `addExitProc` — перегрузка для noconv

```nim
proc addExitProc*(cl: proc() {.noconv.})
```

### Описание

Регистрирует процедуру с соглашением о вызове **`{.noconv.}`**. Эта перегрузка предназначена для совместимости с кодом на C или низкоуровневыми системными коллбэками, где соглашение о вызове должно точно соответствовать ABI платформы — например, функции, полученные через `importc`, или коллбэки, передаваемые через библиотеки C.

В прикладном коде эта перегрузка почти никогда не понадобится. Предпочитайте перегрузку для замыканий, если только вы не работаете именно с `noconv`-коллбэками, полученными из C-интерфейса.

Семантика — порядок LIFO, защищённая блокировкой регистрация, тот же диспетчер — идентична перегрузке для замыканий.

### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `cl` | `proc() {.noconv.}` | Регистрируемая `noconv`-процедура очистки |

### Пример

```nim
import std/exitprocs

proc myCleanup() {.noconv.} =
  # Может быть импортированной из C функцией или низкоуровневой процедурой остановки
  echo "Низкоуровневое завершение выполнено."

addExitProc(myCleanup)
```

## `getProgramResult`

```nim
proc getProgramResult*(): int
```

**Доступность:** только нативные платформы и Node.js. Недоступно на NimScript и браузерном JS.

### Описание

Возвращает **текущий код возврата**, который процесс сообщит операционной системе при нормальном завершении. По умолчанию код возврата равен `0` (успех). Если был вызван `setProgramResult`, процедура возвращает установленное значение.

Это полезно внутри обработчиков выхода, зарегистрированных через `addExitProc`, когда обработчику нужно принять решение, основываясь на том, завершилась ли программа успешно или с ошибкой, — например, пропустить определённые шаги очистки при ошибке или вывести разные сообщения в зависимости от статуса выхода.

### Возвращаемое значение

`int` — текущий код возврата программы. `0` означает успех; любое ненулевое значение по соглашению сигнализирует об ошибке.

### Примеры

**Чтение кода возврата внутри обработчика выхода:**

```nim
import std/exitprocs

addExitProc(proc() =
  let code = getProgramResult()
  if code == 0:
    echo "Программа завершилась успешно."
  else:
    echo "Программа завершилась с кодом ошибки: ", code)

# Где-то позднее:
setProgramResult(1)   # сигнализируем об ошибке
```

**Условная очистка в зависимости от статуса выхода:**

```nim
import std/exitprocs

var tmpFile = "/tmp/workfile.tmp"

addExitProc(proc() =
  if getProgramResult() != 0:
    echo "Оставляем ", tmpFile, " для посмертного анализа."
  else:
    removeFile(tmpFile)
    echo "Временный файл удалён.")
```

## `setProgramResult`

```nim
proc setProgramResult*(a: int)
```

**Доступность:** только нативные платформы и Node.js. Недоступно на NimScript и браузерном JS.

### Описание

Устанавливает **код возврата**, который процесс сообщит операционной системе при завершении. Вызов этой процедуры не завершает программу немедленно — он только записывает планируемый код возврата. Фактически код передаётся, когда процесс завершается: либо естественным образом (достигнув конца `main`), либо через `quit`.

На нативных платформах это записывается в глобальную переменную `programResult`. В Node.js устанавливается `process.exitCode`. Оба механизма учитываются стандартным путём завершения платформы, поэтому обработчики выхода, зарегистрированные через `addExitProc`, всё равно выполняются до того, как код будет передан.

Общепринятое значение успеха — `0`; любое ненулевое значение сигнализирует об ошибке. Конкретные ненулевые коды могут нести предметный смысл (например, `1` — общая ошибка, `2` — неправильное использование команды, `127` — команда не найдена в shell-соглашениях).

### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `a` | `int` | Устанавливаемый код возврата |

### Примеры

**Сигнализирование об ошибке без немедленного выхода:**

```nim
import std/exitprocs

proc processFile(path: string): bool =
  # … возвращает false при ошибке …
  false

if not processFile("data.csv"):
  setProgramResult(1)   # помечаем как ошибку; очистка всё равно выполнится при выходе
  # НЕ вызывайте quit здесь — дайте обработчикам выхода отработать нормально
```

**Использование кодов возврата для передачи результата shell-скрипту:**

```nim
import std/exitprocs

addExitProc(proc() =
  echo "Итоговый код возврата: ", getProgramResult())

let ok = runPipeline()
if not ok:
  setProgramResult(2)   # специфический код для ошибки конвейера
```

**Сброс к успеху после устранимой ошибки:**

```nim
import std/exitprocs

setProgramResult(1)  # предварительно помечаем как ошибку

if recover():
  setProgramResult(0)  # восстановление удалось — всё же сообщаем об успехе
```

## Взаимодействие процедур с необработанными исключениями

Тонкий, но важный аспект `addExitProc` — поведение при завершении программы из-за необработанного исключения:

```
Нормальное завершение        →  обработчики выхода вызываются автоматически
Необработанное исключение    →  обработчики выхода НЕ вызываются автоматически
```

Чтобы гарантировать выполнение обработчиков независимо от способа завершения программы, оберните код верхнего уровня в `try/finally`:

```nim
import std/exitprocs

var db: Database

addExitProc(proc() = db.close())

try:
  db = openDatabase("app.db")
  runApp()
finally:
  # Этот блок выполняется даже при необработанных исключениях, до завершения процесса.
  # Обработчик выхода также выполнится при нормальном завершении — значит, db.close()
  # может быть вызван дважды; сделайте его идемпотентным или используйте флаг.
  discard
```

Более чистый шаблон, когда обработчик должен выполниться в любом случае:

```nim
import std/exitprocs

proc cleanup() =
  echo "Завершаем работу…"
  # освобождаем ресурсы

addExitProc(cleanup)   # обрабатывает нормальные завершения

try:
  runApp()
except:
  cleanup()            # явно обрабатываем аварийное завершение
  raise               # перебрасываем, чтобы рантайм всё равно сообщил об ошибке
```

## Потокобезопасность

Внутренняя последовательность зарегистрированных коллбэков защищена `gFunsLock`. Это означает:

- Вызов `addExitProc` из нескольких потоков одновременно — безопасен.
- Диспетчер `callClosures` (запускаемый при завершении) также захватывает блокировку перед итерацией и очищает список после, предотвращая двойное выполнение.

Однако сами *коллбэки* по отдельности не синхронизированы. Если зарегистрированный обработчик обращается к разделяемому состоянию, это состояние должно быть защищено собственным механизмом синхронизации.
