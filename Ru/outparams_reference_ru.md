# std/outparams — Справочник модуля

> **Входит в стандартную библиотеку Nim** (`std/outparams`).

## Обзор

Модуль предоставляет единственный макрос — `outParamsAt` — чья единственная задача — **совместимость между версиями**: он позволяет помечать параметры процедур как `out` так, чтобы код компилировался корректно и на Nim 2.x (в котором тип параметра `out` введён), и на Nim 1.x (в котором `out` вообще не существует).

### Какую проблему решает

В Nim 2.0 появилось разграничение между `var`-параметрами и `out`-параметрами:

- **`var`-параметр** означает: «аргумент может быть *прочитан и записан* процедурой».
- **`out`-параметр** означает: «аргумент будет *только записан* процедурой — его значение до вызова не имеет значения, и вызывающий код не должен читать его до возврата из процедуры».

Разграничение `out` особенно важно при включённом экспериментальном режиме `strictDefs`: компилятор может доказать, что переменная *гарантированно инициализирована* после вызова, принимающего её как `out`, что позволяет писать:

```nim
{.experimental: "strictDefs".}
var x: int           # ещё не инициализирована — чтение x здесь — ошибка компиляции
myProc(x)            # x — out-параметр, значит после вызова x инициализирована
echo x               # OK: компилятор знает, что myProc присвоила значение x
```

Проблема в том, что запись `proc myProc(x: out int)` в исходном коде **не компилируется на Nim 1.x**, где конструктор типа `out` отсутствует. Макрос `outParamsAt` устраняет этот разрыв: на Nim 1.x каждый параметр остаётся `var` (ближайшая доступная замена), а на Nim 2.x выбранные параметры тихо переписываются с `var T` на `out T` во время компиляции.

## Макрос

### `outParamsAt`

```nim
macro outParamsAt*(positions: static openArray[int]; n: untyped): untyped
```

**Помечает выбранные `var`-параметры процедуры как `out`-параметры версионно-переносимым способом.**

Макрос применяется как прагма к объявлению процедуры. Он принимает один аргумент — массив позиций параметров с нумерацией с единицы — и переписывает эти параметры с `var T` на `out T` на компиляторах, поддерживающих `out`, оставляя объявление полностью без изменений на компиляторах, которые этого не поддерживают.

#### Параметры

| Параметр | Тип | Описание |
|---|---|---|
| `positions` | `static openArray[int]` | Индексы (с 1) параметров для перезаписи. Индекс `1` — первый параметр, `2` — второй и т.д. Индекс `0` — тип возвращаемого значения; никогда не передавайте `0`. |
| `n` | `untyped` | Объявление процедуры, к которому прикреплена прагма. Передаётся автоматически механизмом прагм Nim; вы никогда не пишете это явно. |

#### Возвращаемое значение

Возможно переписанный узел AST объявления процедуры. Поскольку это прагма-макрос, возвращающий `untyped`, он прозрачен: снаружи процедура выглядит и ведёт себя обычно.

#### Как работает переписывание

Внутри макрос проверяет флаг времени компиляции `nimHasOutParams` (автоматически устанавливаемый Nim 2.x):

- **На Nim 1.x** (`nimHasOutParams` *не* определён): макрос просто возвращает исходный узел без изменений. Каждый аннотированный параметр остаётся `var T`.
- **На Nim 2.x** (`nimHasOutParams` *определён*): для каждого индекса в `positions` макрос находит соответствующий узел параметра в AST, проверяет, что он типизирован как `var T`, и заменяет узел `nnkVarTy` на узел `nnkOutTy`, оборачивающий тот же внутренний тип `T`. Результат эквивалентен тому, как если бы вы написали `out T` непосредственно в исходном коде.

#### Шаблон использования

```nim
import std/outparams

proc myProc(result1: var int; result2: var string) {.outParamsAt: [1, 2].} =
  result1 = 42
  result2 = "hello"
```

Только параметры на позициях `1` и `2` являются кандидатами для переписывания; любые другие параметры, не вошедшие в список, остаются `var` (или тем, чем были).

## Примеры

### Базовый — один out-параметр

```nim
import std/outparams

proc parseInt(s: string; value: var int): bool {.outParamsAt: [2].} =
  ## Разбирает `s` как целое число, записывая результат в `value`.
  ## `value` — out-параметр: его предыдущее содержимое не имеет значения.
  try:
    value = system.parseInt(s)
    result = true
  except ValueError:
    result = false
```

На Nim 2.x это компилируется в эквивалент `proc parseInt(s: string; value: out int): bool`. На Nim 1.x компилируется с оригинальной сигнатурой `var int` — поведение идентично в обоих случаях, но компилятор Nim 2.x дополнительно может доказать, что `value` присваивается на ветке `true`.

---

### Несколько out-параметров

```nim
import std/outparams

proc splitPath(full: string; dir: var string; name: var string) {.outParamsAt: [2, 3].} =
  ## Разбивает путь к файлу на каталог и имя файла.
  ## Оба параметра `dir` и `name` — только для записи; они не обязаны быть
  ## инициализированы перед вызовом.
  let slash = full.rfind('/')
  if slash < 0:
    dir = ""
    name = full
  else:
    dir = full[0..slash]
    name = full[slash+1..^1]

var d, n: string
splitPath("/home/user/file.txt", d, n)
echo d   # "/home/user/"
echo n   # "file.txt"
```

### Взаимодействие со `strictDefs`

Наиболее весомая причина использовать `outParamsAt` — бесшовная работа со `strictDefs`, который требует, чтобы каждая переменная была гарантированно инициализирована перед использованием:

```nim
{.experimental: "strictDefs".}
import std/outparams

proc computeHash(data: string; hash: var uint64) {.outParamsAt: [2].} =
  hash = 0xcbf29ce484222325'u64
  for ch in data:
    hash = hash xor uint64(ord(ch))
    hash *= 0x100000001b3'u64

proc process(data: string) =
  var h: uint64           # не инициализирована — strictDefs отслеживает это
  computeHash(data, h)    # после этого вызова h гарантированно инициализирована (out)
  echo h                  # OK на Nim 2.x; предупреждения об «возможно неинициализированной» нет
```

Без `outParamsAt` переменная `h` была бы типизирована как `var uint64`, и `strictDefs` не смог бы доказать, что она инициализируется внутри вызова — потребовалось бы писать `var h: uint64 = 0` как обходное решение.

---

### Что происходит на каждой версии компилятора

```nim
import std/outparams

# Исходный код, который вы пишете:
proc fill(x: var int) {.outParamsAt: [1].} =
  x = 99

# На Nim 1.x — компилируется как:
proc fill(x: var int) = x = 99      # без изменений

# На Nim 2.x — компилируется как:
proc fill(x: out int) = x = 99      # переписывание var → out применено
```

Обе версии компилируются и выполняются идентично; разница только в том, что система типов может *доказать* о вызывающем коде.

## Важные ограничения

**Аннотируемые параметры должны уже быть `var T`.**  
Макрос вызывает `expectKind nnkVarTy` для каждого параметра из списка перед переписыванием. Если вы случайно укажете не-`var`-параметр, вы получите ошибку времени компиляции с описанием несоответствия.

**Позиции нумеруются с единицы.**  
Индекс `1` — первый параметр в списке формальных параметров процедуры. Это соответствует соглашению AST Nim, где индекс `0` — тип возвращаемого значения. Передача `0` означала бы попытку переписать тип возвращаемого значения, что бессмысленно — никогда так не делайте.

**Применяется только к объявлению, не к телу.**  
`outParamsAt` переписывает тип в сигнатуре. Внутри тела процедуры параметр по-прежнему используется как `var` — ему можно свободно присваивать значения. Различие `out` влияет только на то, что компилятор *вызывающего кода* может заключить об инициализации.

## Сводка

| Аспект | Подробности |
|--------|-------------|
| **Что это** | Прагма-макрос совместимости |
| **Эффект на Nim 1.x** | Никакого — процедура остаётся в точности как написана |
| **Эффект на Nim 2.x** | Выбранные `var T`-параметры переписываются в `out T` |
| **Формат аргумента** | Массив позиций с нумерацией с 1, например `[1]`, `[2, 3]` |
| **Требования** | Перечисленные параметры должны быть `var T` в исходнике |
| **Главная выгода** | Позволяет `strictDefs` доказывать инициализацию без потери совместимости с 1.x |
