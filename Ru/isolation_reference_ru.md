# Справочник модуля `isolation`

> **Модуль:** `std/isolation` (стандартная библиотека Nim)  
> **Статус:** ⚠️ Экспериментальный — интерфейс может измениться в будущих версиях Nim.  
> **Назначение:** Безопасное построение изолированных подграфов данных, которые можно передавать между потоками и каналами без гонок данных.

---

## Контекст: почему изоляция важна в конкурентном коде

Когда вы пишете конкурентные программы на Nim — с использованием `threads`, `channels` или структурированного параллелизма — вы сталкиваетесь с фундаментальной проблемой безопасности: если два потока одновременно обращаются к одному и тому же участку памяти в куче, возникает гонка данных — неопределённое поведение.

Классическое решение: данные, отправляемые между потоками, должны быть *изолированы* — ни один другой поток не должен держать на них ссылку. Но компилятор по умолчанию не может это проверить, и ответственность лежит на программисте. Одна ошибка — отправка значения при сохранении ссылки на него — приводит к трудновоспроизводимым багам.

Модуль `isolation` формализует это ограничение на уровне системы типов. Обёртка `Isolated[T]` делает контракт «эти данные принадлежат ровно одному владельцу» машинно-проверяемым:

- **Копирование запрещено на этапе компиляции** — нельзя случайно создать вторую ссылку.
- **Передача только через перемещение** — оригинал уничтожается в процессе.
- **Сама изоляция проверяется компилятором** (при использовании `isolate`) — компилятор доказывает, что у значения нет внешних псевдонимов.

Этот подход опирается на концепцию *линейных типов* из теории типов, практически реализованную поверх модели владения и деструкторов Nim.

## Экспортируемые символы

---

### `Isolated[T]`

```nim
type Isolated*[T] {.sendable.} = object
  value: T
```

**Что это такое**

`Isolated[T]` — обобщённый тип-обёртка, который устанавливает семантику «только перемещение» для любого значения типа `T`. Прагма `.sendable` сообщает компилятору Nim, что этот тип безопасно отправлять через каналы между потоками.

Внутреннее поле `value` намеренно приватное. Единственные способы поместить данные в `Isolated[T]` — через `isolate` или `unsafeIsolate`, а извлечь — только через `extract`. Такой контролируемый интерфейс гарантирует, что гарантия изоляции никогда не обходится молча.

**Три хука управления памятью**

Модуль определяет все три хука управления памятью для `Isolated[T]`. Понимание каждого из них объясняет фундаментальное поведение обёртки.

`=copy` **удалён** (помечен `{.error.}`). Любая попытка скопировать значение `Isolated[T]` — ошибка компиляции. Это краеугольный камень гарантии: нельзя создать второго владельца данных.

`=sink` **делегирует sink-у внутреннего значения**. Перемещение `Isolated[T]` передаёт владение содержимым получателю; источник остаётся в уничтоженном состоянии.

`=destroy` **делегирует деструктору внутреннего значения**. Когда `Isolated[T]` выходит из области видимости, содержимое корректно освобождается, предотвращая утечки памяти.

**Пример: запрет копирования в действии**

```nim
import std/isolation

let a = isolate(@[1, 2, 3])
# let b = a          # ошибка компиляции: '=copy' запрещён для Isolated[T]
let b = move(a)      # это допустимо — владение передаётся, a теперь пусто
```

### `isolate`

```nim
func isolate*[T](value: sink T): Isolated[T] {.magic: "Isolate".}
```

**Что делает**

`isolate` оборачивает значение в `Isolated[T]` с **проверкой безопасности на этапе компиляции**: компилятор Nim проверяет, что `value` образует истинный изолированный подграф — то есть ни одна существующая переменная за пределами этого вызова не держит ссылку на те же данные в куче.

Прагма `{.magic: "Isolate".}` означает, что компилятор не просто исполняет тело функции как написано — он перехватывает вызов и выполняет специальный анализ псевдонимов до продолжения работы. Если проверка не проходит, вы получаете ошибку компиляции, а не падение в рантайме.

Аннотация параметра `sink` означает, что функция забирает владение `value` у вызывающего кода. Исходная переменная потребляется: после `isolate(x)` переменная `x` более недействительна.

**Что такое «изолированный подграф»**

Подграф — это транзитивное замыкание всех объектов в куче, достижимых из значения. Для `seq[string]` — это буфер последовательности плюс буфер каждой строки. Для объекта с ref-полями — все объекты, на которые есть ссылки. Значение изолировано, когда *ничто другое* не указывает на этот подграф.

Простые значения — целые числа, объекты на стеке — всегда изолированы: у них нет ссылок на кучу. Интересный случай — структуры данных, выделенные в куче (ref, seq, string, замыкания), где псевдонимы возможны.

**Параметры**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `value` | `sink T` | Значение для оборачивания. Владение передаётся в `Isolated[T]`. Должно образовывать изолированный подграф — проверяется на этапе компиляции. |

**Возвращает:** `Isolated[T]` с обёрнутым значением.

**Пример: безопасная изоляция свежесозданного значения**

```nim
import std/isolation

# Только что созданный seq не имеет псевдонимов — проверка проходит:
let iso = isolate(@[10, 20, 30])

# Целое число тривиально изолировано:
let isoInt = isolate(42)

# Свежий объект изолирован:
type Config = object
  timeout: int
  retries: int

let isoConfig = isolate(Config(timeout: 5000, retries: 3))
```

**Пример: что компилятор отклоняет**

```nim
import std/isolation

var data = @[1, 2, 3]
var alias = data          # теперь `data` и `alias` разделяют один буфер seq

# isolate(data)           # ошибка компиляции: `data` не изолирована — `alias` является псевдонимом
```

Компилятор обнаруживает это потому, что после `var alias = data` две переменные указывают на один и тот же буфер. Обернуть `data` в `Isolated` при существующем `alias` нарушило бы инвариант «ровно один владелец».

**Пример: отправка изолированных данных в поток**

```nim
import std/isolation, std/tasks

# Подготовка данных в основном потоке:
let payload = isolate(@["hello", "world"])

# Передача в другой поток — копирования не происходит, владение перемещается:
channel.send(payload)
# `payload` теперь недействителен — канал владеет данными
```

### `unsafeIsolate`

```nim
func unsafeIsolate*[T](value: sink T): Isolated[T]
```

**Что делает**

`unsafeIsolate` делает ровно то же, что `isolate`, на *уровне типов* — оборачивает значение в `Isolated[T]` — но **пропускает проверку псевдонимов на этапе компиляции**. Ответственность за то, что значение действительно изолировано, полностью ложится на программиста.

Это аварийный выход для ситуаций, когда анализ компилятора слишком консервативен и отклоняет значение, которое на самом деле безопасно изолировать. Такое может происходить со сложными структурами данных или паттернами, которые статический анализ псевдонимов не может разобрать.

**Когда использовать**

Используйте `unsafeIsolate` только если:
1. Вы вручную убедились, что никакая другая переменная не является псевдонимом оборачиваемых данных.
2. `isolate` отклоняет значение из-за ограничения анализа компилятора, а не из-за реального псевдонима.
3. Вы понимаете, что ответственность за корректность полностью переходит к вам.

При некорректном использовании `unsafeIsolate` — если обернуть значение, у которого есть псевдоним — результатом будет гонка данных, то есть неопределённое поведение. Никакой проверки в рантайме нет.

**Параметры**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `value` | `sink T` | Значение для оборачивания. Владение передаётся. Проверка изоляции не выполняется. |

**Возвращает:** `Isolated[T]` с обёрнутым значением.

**Пример: использование unsafeIsolate когда вы уверены в безопасности**

```nim
import std/isolation

proc buildData(): seq[int] =
  result = newSeq[int](1000)
  for i in 0..<1000:
    result[i] = i * i
  # result — локальная переменная, которую вернут вызывающему;
  # после возврата вызывающий становится единственным владельцем

# Возвращённый seq не имеет псевдонимов — мы знаем это из конструкции,
# даже если компилятор в этом не уверен:
let iso = unsafeIsolate(buildData())
```

**Сравнение с `isolate`:**

```nim
# isolate проверяет статически:
let iso1 = isolate(buildData())        # нормально, если компилятор может доказать

# unsafeIsolate полностью пропускает проверку:
let iso2 = unsafeIsolate(buildData())  # нормально только если ВЫ знаете, что безопасно
```

### `extract`

```nim
func extract*[T](src: var Isolated[T]): T
```

**Что делает**

`extract` извлекает значение, хранящееся внутри `Isolated[T]`, потребляя содержимое обёртки в процессе. Внутри используется `move`, поэтому значение передаётся из `src` без копирования. После возврата `extract` переменная `src` находится в уничтоженном/пустом состоянии — дальнейшее обращение к ней некорректно.

Это зеркало `isolate`/`unsafeIsolate`: те помещают данные *внутрь*, `extract` достаёт данные *наружу*. Вместе три функции образуют полный жизненный цикл изолированного значения: **обернуть → передать → развернуть**.

`src` — параметр `var`, а не `sink`, потому что функция должна мутировать `src` (переместить из него данные), оставив саму оболочку `Isolated[T]` нетронутой для корректного вызова деструктора.

**Параметры**

| Параметр | Тип | Описание |
|----------|-----|----------|
| `src` | `var Isolated[T]` | Изолированная обёртка, из которой нужно извлечь значение. Должна быть изменяемой переменной. Содержимое перемещается наружу; `src` становится пустой. |

**Возвращает:** `T` — значение, которое находилось внутри обёртки.

**Пример: полный жизненный цикл**

```nim
import std/isolation

# Шаг 1: обернуть (на стороне производителя, например, основной поток)
var iso = isolate(@["a", "b", "c"])

# Шаг 2: передать (например, через канал — владение перемещается)
# channel.send(iso)
# ... на стороне потребителя receive возвращает Isolated[T] ...

# Шаг 3: развернуть (на стороне потребителя, например, рабочий поток)
let data = extract(iso)
echo data   # @["a", "b", "c"]
# iso теперь пуста — деструктор запустится, но освобождать нечего
```

**Пример: extract в контексте рабочего потока**

```nim
import std/isolation

proc worker(iso: var Isolated[seq[int]]) =
  let data = extract(iso)
  var sum = 0
  for x in data: sum += x
  echo "Сумма: ", sum

var payload = isolate(@[1, 2, 3, 4, 5])
worker(payload)
# payload теперь исчерпан
```

## Полная картина: поток данных между потоками

Три ключевые функции напрямую отображаются на три стадии конвейера производитель-потребитель:

```
[Поток-производитель]                  [Поток-потребитель]

 исходное значение
        │
        ▼
 isolate(value)          ─────────────────────────────►  Isolated[T]  (в канале)
 Isolated[T]                   channel.send(iso)                │
                                                                ▼
                                                          extract(iso)
                                                                │
                                                                ▼
                                                          исходное значение T
```

На каждом шаге система типов предотвращает случайное совместное использование:
- `Isolated[T]` нельзя скопировать — только переместить.
- Нельзя прочитать данные без вызова `extract`, который потребляет обёртку.
- Прагма `{.sendable.}` разрешает типу пересекать границы потоков в канальных операциях.

## Архитектурные замечания

**Почему `=copy` — ошибка компиляции, а не просто отсутствующая операция?** Жёсткая ошибка означает, что компилятор немедленно сообщает о случайной попытке копирования. Отсутствующий или подавленный `=copy` мог бы в некоторых бэкендах незаметно откатиться к побитовому копированию, что привело бы к дублированию указателей, двойному освобождению памяти или гонкам данных. Прагма `{.error.}` делает инвариант недвусмысленным.

**Почему `extract` принимает `var`, а не `sink`?** Если бы `extract` принимал `sink`, вызывающий код уничтожал бы объект-обёртку после вызова, что запустило бы `=destroy` на уже перемещённом значении. Параметр `var` позволяет `extract` переместить внутреннее значение наружу, оставив оболочку нетронутой — её деструктор затем корректно обработает пустое состояние.

**Связь с моделью владения Nim.** `Isolated[T]` — это применение инфраструктуры деструкторов ARC/ORC Nim. Сборщик мусора не требуется; тип эффективно работает в сборках `--gc:arc` и `--gc:orc`. Семантика перемещения имеет нулевую стоимость: при передаче владения никакого лишнего выделения памяти или копирования не происходит.

## Сводная таблица

| Символ | Вид | Назначение |
|--------|-----|------------|
| `Isolated[T]` | тип | Обёртка «только перемещение»; помечает данные как безопасные для отправки между потоками |
| `isolate(value)` | func | Обернуть с **проверкой изоляции** на этапе компиляции |
| `unsafeIsolate(value)` | func | Обернуть **без проверки** — ответственность программиста |
| `extract(src)` | func | Развернуть и получить владение содержимым |
| `=copy` | удалённый хук | Запрещает любое копирование — ошибка компиляции при попытке |
| `=sink` | хук жизненного цикла | Обеспечивает эффективное перемещение значений `Isolated[T]` |
| `=destroy` | хук жизненного цикла | Освобождает содержимое, когда обёртка выходит из области видимости |
