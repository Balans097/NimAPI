# Справочник по модулю `ioselectors_poll`

> **Модуль:** `std/ioselectors` (бэкенд POSIX poll)  
> **Язык:** Nim  
> **Платформа:** Любая POSIX-система с поддержкой `poll()` — Linux, macOS, BSD, Solaris, Zephyr RTOS и другие. Этот бэкенд также используется как запасной вариант, когда более производительный бэкенд (epoll, kqueue) недоступен.  
> **Назначение:** Реализует API `ioselectors` поверх системного вызова POSIX `poll()`, обеспечивая мультиплексирование I/O-событий — одновременное наблюдение за несколькими файловыми дескрипторами без циклического опроса.

---

## Контекст: что такое poll() и чем он отличается от kqueue/epoll?

`poll()` — наиболее переносимый из трёх основных Unix-интерфейсов мультиплексирования I/O. В отличие от kqueue (BSD) или epoll (Linux), которые поддерживают состояние *внутри ядра* и позволяют добавлять/удалять дескрипторы инкрементально, `poll()` работает иначе:

- Вы поддерживаете **плоский массив структур `pollfd`** в пользовательском пространстве, по одной записи на каждый наблюдаемый дескриптор.
- При каждом вызове `poll()` вы передаёте ядру **весь массив**. Ядро сканирует его, помечает, у каких записей есть активность, и возвращает управление.
- Между вызовами ядро **не хранит никакого состояния**. Каждый вызов `poll()` — это свежее сканирование.

Это принципиальное архитектурное решение влечёт важные последствия:

| Свойство | `poll()` | `epoll`/`kqueue` |
|----------|----------|-----------------|
| Хранение состояния | Массив в пользовательском пространстве | Ядро |
| Масштабируемость | O(n) — ядро сканирует все fd при каждом вызове | O(1) — ядро сообщает только о готовых fd |
| Переносимость | Универсальный POSIX | Зависит от ОС |
| Макс. дескрипторов | Нет жёсткого ограничения (в отличие от `select`) | Нет жёсткого ограничения |
| Типы событий | Только чтение, запись, ошибка | Чтение, запись, таймер, сигнал, процесс, vnode, пользователь |

Для небольшого числа дескрипторов (десятки–сотни) `poll()` вполне достаточен. Для тысяч одновременных соединений предпочтительнее epoll или kqueue.

### Какие типы событий поддерживает этот бэкенд?

Поскольку `poll()` понимает только три состояния (`POLLIN`, `POLLOUT`, `POLLERR`/`POLLHUP`/`POLLNVAL`), этот бэкенд поддерживает только:

- `Event.Read` — доступны данные для чтения
- `Event.Write` — буфер записи имеет свободное место
- `Event.User` — ручной сигнал через `SelectEvent`
- `Event.Error` — ошибка или разрыв соединения на дескрипторе

События таймера, сигнала, процесса и vnode **не реализованы** в этом бэкенде. Их поддержка потребовала бы платформо-зависимых расширений, которых в `poll()` нет.

### Два варианта реализации `SelectEvent`

Модуль имеет ветку на этапе компиляции для механизма, на котором строятся пользовательские события:

- **По умолчанию (на основе пайпа):** `newSelectEvent` создаёт OS-пайп. `trigger` записывает 8 байт в записывающий конец; `poll` срабатывает на `POLLIN` у читающего конца. Потребляются два файловых дескриптора.
- **На основе `eventfd`** (при `defined(zephyr)` или `-d:nimPollHasEventFds`): использует системный вызов Linux `eventfd`. Единственный дескриптор одновременно является и источником, и приёмником, что вдвое снижает потребление fd. Автоматически включается для Zephyr RTOS или вручную через флаг компиляции.

## Типы

### `Selector[T]`

```nim
# однопоточная сборка:
type Selector*[T] = ref SelectorImpl[T]
# потокобезопасная сборка (--threads:on):
type Selector*[T] = ptr SelectorImpl[T]
```

Центральный объект. Поддерживает два параллельных массива:

- `fds: seq/ptr SharedArray[SelectorKey[T]]` — индексируется номером OS-дескриптора, хранит метаданные регистрации и пользовательские данные. Неиспользуемые записи имеют `ident == InvalidIdent`.
- `pollfds: seq/ptr SharedArray[TPollFd]` — **плотный, без пробелов** массив структур `pollfd`, который передаётся непосредственно в системный вызов `poll()`. Массив поддерживается непрерывным: при удалении записи все последующие сдвигаются влево.

Два массива служат разным целям. `fds` позволяет O(1)-поиск по номеру fd. `pollfds` — это реальный аргумент для `poll()`: он должен быть плотным и содержать только активные записи, потому что `poll()` сканирует каждый его элемент.

`pollcnt` отслеживает, сколько записей сейчас в `pollfds`. `count` отслеживает общее число зарегистрированных дескрипторов (в этом бэкенде они всегда равны, но поле является частью общего контракта `ioselectors`).

В потокобезопасной сборке `Selector` — сырой `ptr` в разделяемую память, а мьютекс `Lock` защищает все мутации `pollfds` и `fds`.

### `SelectEvent`

```nim
type SelectEvent* = ptr SelectEventImpl
# SelectEventImpl = object
#   rfd: cint   # читающий конец (или единственный eventfd)
#   wfd: cint   # записывающий конец (совпадает с rfd при использовании eventfd)
```

Уведомление, запускаемое вручную. На системах с пайпом `rfd ≠ wfd`. На системах с eventfd `rfd == wfd` (один fd, действующий и как источник, и как приёмник).

Живёт в разделяемой памяти, чтобы его можно было безопасно передавать между потоками.

## Функции жизненного цикла

### `newSelector`

```nim
proc newSelector*[T](): Selector[T]
```

#### Что делает

Создаёт и инициализирует новый селектор. Шаги проще, чем в kqueue-бэкенде, потому что `poll()` не требует объекта в ядре:

1. Запрашивает максимальное допустимое число файловых дескрипторов процесса через `maxDescriptors()`.
2. Предварительно выделяет `fds` и `pollfds` нужного размера. `fds` индексируется номером fd; `pollfds` начинается пустым (все записи неиспользуемы).
3. Помечает каждую запись в `fds` как `InvalidIdent`.
4. В потокобезопасных сборках: выделяет оба массива в разделяемой памяти и инициализирует мьютекс `Lock`.

Обратите внимание: в отличие от kqueue-бэкенда, **здесь не создаётся никакого объекта в ядре**. Системный вызов `poll()` получает всё необходимое при каждом вызове, поэтому открывать или закрывать что-либо на уровне селектора не нужно.

#### Пример

```nim
import std/ioselectors

let sel = newSelector[string]()
defer: sel.close()
```

### `close` (Selector)

```nim
proc close*[T](s: Selector[T])
```

#### Что делает

В **потокобезопасной сборке** уничтожает мьютекс и освобождает три аллокации разделяемой памяти (`fds`, `pollfds`, сам `SelectorImpl`).

В **однопоточной сборке** эта процедура не делает ничего. Сборщик мусора Nim автоматически освобождает `ref`-типизированный селектор при выходе из области видимости. Никакого ресурса ядра освобождать не нужно (в отличие от `kqFD` в kqueue).

Эта асимметрия означает, что следует всегда вызывать `close` безусловно — для корректности в смешанных кодовых базах и для совместимости с будущими версиями.

#### Пример

```nim
let sel = newSelector[int]()
# ... работа с sel ...
sel.close()
```

---

### `newSelectEvent`

```nim
proc newSelectEvent*(): SelectEvent
```

#### Что делает

Создаёт событие, запускаемое вручную. Реализация зависит от платформы:

**Стандартный путь (пайп):**  
Создаёт OS-пайп через `posix.pipe`. Переводит оба конца в неблокирующий режим. Выделяет `SelectEventImpl` в разделяемой памяти. `rfd` = читающий конец, `wfd` = записывающий конец.

**Путь `eventfd`** (`defined(zephyr)` или `-d:nimPollHasEventFds`):  
Вызывает `eventfd(0, O_NONBLOCK)`, создавая единственный дескриптор, поддерживающий и чтение, и запись. Сохраняет один и тот же fd в обоих полях `rfd` и `wfd`. Экономит один файловый дескриптор по сравнению с пайпом — важно на ресурсно-ограниченных системах вроде Zephyr RTOS.

#### Пример

```nim
let wakeup = newSelectEvent()
defer: wakeup.close()
```

---

### `trigger`

```nim
proc trigger*(ev: SelectEvent)
```

#### Что делает

Сигнализирует о событии, записывая 8 байт (значение `uint64` равное `1`) в `ev.wfd`. На пайпе это заставляет `poll()` сообщить о `POLLIN` на читающем конце. На `eventfd` запись инкрементирует внутренний счётчик, также вызывая срабатывание `POLLIN`.

Безопасно вызывать из любого потока. Это стандартный механизм пробуждения потока, заблокированного в `select`/`selectInto`.

Бросает `IOSelectorsError` при ошибке записи.

#### Пример

```nim
# Из рабочего потока:
wakeupEvent.trigger()

# В потоке событийного цикла, select возвращается с Event.User:
let ready = sel.select(timeout = -1)
for key in ready:
  if Event.User in key.events:
    echo "разбужен другим потоком"
```

---

### `close` (SelectEvent)

```nim
proc close*(ev: SelectEvent)
```

#### Что делает

Закрывает файловый дескриптор (дескрипторы) события:

- **Пайп**: закрывает `rfd` и `wfd` по отдельности.
- **`eventfd`**: закрывает только `rfd` (так как `rfd == wfd`).

Затем освобождает блок разделяемой памяти. Всегда вызывайте `unregister` на селекторе *до* вызова `close` на событии.

Бросает `IOSelectorsError`, если любой из системных вызовов `close` завершается ошибкой.

## Функции регистрации

### `registerHandle`

```nim
proc registerHandle*[T](s: Selector[T], fd: int | SocketHandle,
                        events: set[Event], data: T)
```

#### Что делает

Регистрирует существующий OS-дескриптор в селекторе. Сохраняет `SelectorKey` в `fds[fd]` и, если `events` непусто, вызывает `pollAdd` для добавления новой записи `pollfd` в конец `pollfds`, увеличивая `pollcnt` и `count`.

Осмысленными здесь являются только `Event.Read` и `Event.Write`. Передача других типов событий сохраняет их в метаданных ключа, но они никогда не сработают: `poll()` не умеет их сообщать.

Требует, чтобы дескриптор ещё не был зарегистрирован (`fds[fd].ident == InvalidIdent`). Нарушение вызывает сбой `doAssert`.

#### Пример

```nim
import std/[ioselectors, nativesockets]

let sel = newSelector[string]()
let sock = createNativeSocket()
sock.setBlocking(false)
sel.registerHandle(sock, {Event.Read}, "клиент-1")
```

---

### `updateHandle`

```nim
proc updateHandle*[T](s: Selector[T], fd: int | SocketHandle,
                      events: set[Event])
```

#### Что делает

Изменяет наблюдаемые события для уже зарегистрированного I/O-дескриптора. Обрабатывает три различных перехода:

- **`{}` → непустое**: дескриптор был зарегистрирован, но не наблюдался активно. Вызывает `pollAdd` для вставки в `pollfds`.
- **непустое → непустое (изменённое)**: вызывает `pollUpdate`, выполняющий линейное сканирование `pollfds` для поиска и обновления записи на месте.
- **непустое → `{}`**: вызывает `pollRemove`, выполняющий линейное сканирование для поиска записи и её удаления со сдвигом последующих записей влево.

Как и в kqueue-бэкенде, работает только с обычными I/O-дескрипторами. Отклоняет дескрипторы, зарегистрированные с типами Timer, Signal, Process, Vnode, User, Oneshot или Error.

#### Пример

```nim
sel.registerHandle(sock, {Event.Read}, myData)

# Когда нужно что-то отправить:
sel.updateHandle(sock, {Event.Read, Event.Write})

# После опустошения буфера записи:
sel.updateHandle(sock, {Event.Read})
```

---

### `registerEvent`

```nim
proc registerEvent*[T](s: Selector[T], ev: SelectEvent, data: T)
```

#### Что делает

Регистрирует `SelectEvent` в селекторе. Сохраняет `{Event.User}` в наборе событий ключа в `fds`, затем вызывает `pollAdd` с `{Event.User, Event.Read}` — флаг `POLLIN` и есть то, что реально отслеживает `poll()`; `Event.User` — это маркер на уровне Nim.

Когда `selectInto` позднее обнаруживает `POLLIN` на этом fd, он проверяет, есть ли `Event.User` в событиях ключа, и если да — читает 8 байт из пайпа/eventfd для его опустошения, прежде чем вернуть `Event.User` вызывающему.

#### Пример

```nim
let wakeup = newSelectEvent()
sel.registerEvent(wakeup, data = "канал-пробуждения")
```

## Отмена регистрации

### `unregister` (fd)

```nim
proc unregister*[T](s: Selector[T], fd: int | SocketHandle)
```

#### Что делает

Удаляет файловый дескриптор из селектора. Устанавливает `fds[fd].ident = InvalidIdent`, очищает набор событий и вызывает `pollRemove`, выполняющий **линейное сканирование** `pollfds` для поиска записи и её удаления со сдвигом всех последующих записей на одну позицию влево.

Этот сдвиг — O(n)-стоимость удаления в poll-бэкенде, где n — `pollcnt`. В epoll- и kqueue-бэкендах удаление O(1), так как состояние хранится в ядре.

---

### `unregister` (SelectEvent)

```nim
proc unregister*[T](s: Selector[T], ev: SelectEvent)
```

#### Что делает

Удаляет `SelectEvent` из селектора. Проверяет, что событие зарегистрировано и что `Event.User` присутствует в его ключе. Очищает ключ и вызывает `pollRemove` для удаления читающего fd пайпа из `pollfds`.

## Ожидание событий

### `selectInto`

```nim
proc selectInto*[T](s: Selector[T], timeout: int,
                    results: var openArray[ReadyKey]): int
```

#### Что делает

Сердце модуля. Вызывает `posix.poll()` с текущим массивом `pollfds` и `pollcnt`, блокируется до прихода событий или истечения `timeout`, затем транслирует результаты ядра в значения `ReadyKey`.

**Семантика timeout:**

| Значение | Смысл |
|----------|-------|
| `0` | Вернуться немедленно (неблокирующая проверка) |
| `> 0` | Ждать не дольше указанного числа миллисекунд |
| `-1` | Ждать бесконечно |

**После возврата `poll()`** код сканирует `pollfds[0..pollcnt)` в поиске записей, где `revents != 0`. Для каждой такой записи:

- `POLLIN` → `Event.Read`. Но если в ключе установлен `Event.User`, сначала читается 8 байт из fd для опустошения пайпа/счётчика eventfd. Если чтение возвращает `EAGAIN`, кто-то другой уже потребил событие, и запись пропускается. Иначе `Event.User` заменяет `Event.Read` в результате.
- `POLLOUT` → `Event.Write`
- `POLLERR`, `POLLHUP` или `POLLNVAL` → `Event.Error`

После обработки записи `pollfds[i].revents` сбрасывается в `0`.

`EINTR` (прерван сигналом) заставляет `poll()` вернуть -1; процедура обрабатывает это, возвращая `0` вместо исключения, в соответствии с POSIX-соглашениями.

Сканирование прекращается, когда проверены все `pollcnt` записей, потреблены все `count` готовых записей или достигнут `maxres` (размер `results`).

#### Принципиальное отличие от epoll/kqueue

В epoll и kqueue ядро возвращает *только готовые дескрипторы*. В poll ядро помечает все поля `revents`, но возвращает лишь количество ненулевых. Код пользовательского пространства всё равно должен **сканировать весь массив `pollfds`**, чтобы найти, кто сработал. Это сканирование — O(`pollcnt`), а не O(число готовых событий). При большом числе записей это ощутимо.

#### Пример

```nim
var ready: array[64, ReadyKey]
while true:
  let n = sel.selectInto(timeout = 500, ready)
  for i in 0 ..< n:
    let key = ready[i]
    if Event.Read in key.events:
      echo "fd ", key.fd, " готов к чтению"
    if Event.Error in key.events:
      echo "fd ", key.fd, " ошибка"
```

### `select`

```nim
proc select*[T](s: Selector[T], timeout: int): seq[ReadyKey]
```

#### Что делает

Удобная обёртка над `selectInto`, которая выделяет и возвращает `seq[ReadyKey]`. Внутри предварительно выделяет `MAX_POLL_EVENTS` (64) слотов, вызывает `selectInto` и обрезает результат до реального количества.

Используйте `selectInto` в горячих циклах, где важны выделения памяти. Используйте `select` ради простоты.

#### Пример

```nim
let ready = sel.select(timeout = 1000)
for key in ready:
  echo "fd: ", key.fd, "  события: ", key.events
```

## Функции доступа к данным

### `getData`

```nim
proc getData*[T](s: Selector[T], fd: SocketHandle | int): var T
```

#### Что делает

Возвращает изменяемую ссылку на пользовательские данные `T`, хранящиеся для указанного дескриптора в `fds[fd].data`. Ссылка указывает непосредственно в таблицу, поэтому изменения отражаются сразу.

Если `fd` не зарегистрирован, возвращает значение по умолчанию для `T` (нулевая инициализация) без исключения. Используйте `contains` для предварительной проверки, если значение по умолчанию может ввести в заблуждение.

#### Пример

```nim
sel.registerHandle(sock, {Event.Read}, 0)
sel.getData(sock) += 1   # инкрементируем счётчик
```

---

### `setData`

```nim
proc setData*[T](s: Selector[T], fd: SocketHandle | int, data: T): bool
```

#### Что делает

Заменяет хранимые пользовательские данные для зарегистрированного дескриптора. Возвращает `true`, если дескриптор зарегистрирован и данные обновлены; `false` — иначе.

#### Пример

```nim
if not sel.setData(sock, newState):
  echo "sock не зарегистрирован"
```

---

### `withData` (ветка «найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body: untyped)
```

#### Что делает

Выполняет `body` с `value`, привязанным как указатель на хранимые пользовательские данные, если `fd` зарегистрирован. Если `fd` не зарегистрирован, `body` пропускается. Объединяет проверку регистрации и доступ к данным в один безопасный шаг.

#### Пример

```nim
sel.withData(sock, statePtr):
  statePtr.bytesRead += n
```

### `withData` (ветки «найдено» и «не найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body1, body2: untyped)
```

#### Что делает

Двухветочная версия: `body1` выполняется, если `fd` зарегистрирован; `body2` — если нет. Эквивалент `if/else` вокруг `contains`, но чище, поскольку указатель на данные уже доступен в `body1`.

#### Пример

```nim
sel.withData(sock, statePtr):
  statePtr.bytesRead += n
do:
  echo "fd не в селекторе"
```

## Функции запросов

### `isEmpty`

```nim
template isEmpty*[T](s: Selector[T]): bool
```

#### Что делает

Возвращает `true`, когда `s.count == 0` — нет активно зарегистрированных дескрипторов. Типичное условие завершения событийного цикла: если за ничем не наблюдают, вызывать `select` бессмысленно.

#### Пример

```nim
while not sel.isEmpty:
  let ready = sel.select(timeout = -1)
  # обрабатываем события...
```

---

### `contains`

```nim
proc contains*[T](s: Selector[T], fd: SocketHandle | int): bool {.inline.}
```

#### Что делает

Возвращает `true`, если `fd` в данный момент зарегистрирован. Проверяет `fds[fd.int].ident != InvalidIdent` — O(1).

#### Пример

```nim
if sock in sel:
  sel.updateHandle(sock, {Event.Read, Event.Write})
```

---

### `getFd`

```nim
proc getFd*[T](s: Selector[T]): int
```

#### Что делает

Всегда возвращает `-1`.

В отличие от kqueue (у которого есть реальный fd очереди в ядре) или epoll (у которого есть fd экземпляра epoll), `poll()` не имеет постоянного объекта в ядре. Возвращать нечего. Значение `-1` сигнализирует «неприменимо» и отличает этот бэкенд от остальных, если потребуется ветвление по бэкенду.

#### Пример

```nim
let fd = sel.getFd()
assert fd == -1  # всегда верно для poll-бэкенда
```

## Внутренние шаблоны (не публичные, но важные для понимания)

### `pollAdd`

Добавляет новую запись `pollfd` в позицию `pollfds[pollcnt]` и инкрементирует `pollcnt` и `count`. В потокобезопасных сборках защищён `withPollLock`. Вызывается из `registerHandle` и `registerEvent`.

### `pollUpdate`

Выполняет **линейное сканирование** `pollfds[0..pollcnt)` для поиска записи с совпадающим `fd`, затем обновляет её поле `events` на месте. Вызывается из `updateHandle` при переходе между непустыми наборами событий.

### `pollRemove`

Выполняет **линейное сканирование** для поиска записи с совпадающим `fd`. Удаляет её:
- Если это последняя запись: обнуляет слот.
- Иначе: сдвигает все последующие записи на одну позицию влево, сохраняя массив плотным.

Декрементирует `pollcnt` и `count`. Вызывается из `updateHandle` (при переходе к `{}`), а также из `unregister`.

Именно сдвиговое поведение поддерживает `pollfds` плотным. Это обязательно, потому что `poll()` сканирует ровно `pollcnt` записей — любой пробел привёл бы к лишним итерациям системного вызова.

## Полный пример: минимальный событийный цикл

```nim
import std/[ioselectors, nativesockets, posix]

type ConnData = object
  id: int

let sel = newSelector[ConnData]()
defer: sel.close()

# Следим за stdin
sel.registerHandle(0, {Event.Read}, ConnData(id: 0))

# Ручное пробуждение из другого потока
let wakeup = newSelectEvent()
sel.registerEvent(wakeup, ConnData(id: -1))

# Здесь можно запустить поток, который вызовет wakeup.trigger() через 1 секунду

while not sel.isEmpty:
  let ready = sel.select(timeout = 2000)
  if ready.len == 0:
    echo "таймаут — нет готовых"
    break
  for key in ready:
    if Event.Read in key.events and key.fd == 0:
      echo "stdin содержит данные"
      sel.unregister(0)
    if Event.User in key.events:
      echo "разбужен другим потоком"
      sel.unregister(wakeup)
      wakeup.close()
```

## Сравнение: poll-бэкенд против kqueue-бэкенда

| Аспект | poll-бэкенд | kqueue-бэкенд |
|--------|-------------|---------------|
| Объект в ядре | Отсутствует | fd kqueue (один на Selector) |
| `close` (Selector) | No-op в однопоточной сборке | Должен закрыть kqFD и фиктивный сокет |
| Регистрация | Добавляет запись в массив пользовательского пространства | Отправляет `EV_ADD` kevent в ядро |
| Удаление | Линейный поиск + сдвиг влево в массиве | Отправляет `EV_DELETE` kevent в ядро |
| Обновление | Линейный поиск + обновление на месте | Отправляет `EV_ADD` (новые) + `EV_DELETE` (старые) |
| Ожидание | Передаёт весь массив при каждом вызове | Ядро хранит состояние, сообщает только о готовых |
| Масштабируемость | O(n) на каждый вызов poll | O(готовые события) на каждый вызов kevent |
| Типы событий | Read, Write, User, Error | Read, Write, User, Timer, Signal, Process, Vnode, Error |
| `getFd()` | Всегда `-1` | Возвращает fd kqueue |
| Timer/Signal/Process | Не поддерживаются | Нативная поддержка через EVFILT_* |
| Основа `SelectEvent` | Пайп или `eventfd` | Только пайп |
| Потокобезопасность | Мьютекс вокруг мутаций `pollfds` | Мьютекс вокруг буфера изменений kevent |

## Примечания и подводные камни

- **`poll()` передаёт весь массив при каждом вызове.** Даже если из ста дескрипторов готов один, ядро всё равно сканирует все сто. Для большого числа соединений предпочтительнее epoll или kqueue.
- **`pollRemove` сдвигает записи.** Удаление дескриптора из середины `pollfds` — O(n) от `pollcnt`. Напротив, массив `fds` никогда не уплотняется — записи просто помечаются `InvalidIdent` на месте.
- **Нет событий таймера, сигнала и процесса.** Это фундаментальные ограничения `poll()`. Если они нужны, требуется epoll (Linux) или kqueue (BSD).
- **`getFd()` возвращает `-1`.** Это намеренно и документирует факт отсутствия постоянного объекта в ядре. Код, которому нужно вкладывать селекторы или экспортировать fd селектора наружу, не может использовать этот бэкенд.
- **`count` и `pollcnt` всегда равны.** В отличие от бэкендов, где могут существовать зарегистрированные-но-неактивные дескрипторы, каждый зарегистрированный дескриптор в poll-бэкенде имеет соответствующую запись в `pollfds`.
- **`EINTR` обрабатывается молча.** `poll()`, прерванный сигналом, возвращает `0` событий вместо исключения. Это корректно — обрабатывайте сигнал обычными средствами.
- **Потокобезопасность распространяется только на мутации `pollfds`.** Шаблон `withPollLock` защищает `pollAdd`, `pollUpdate`, `pollRemove` и ожидание в `selectInto`. Записи `fds` защищаются не независимо, а в координации с мутациями `pollfds`.
- **Режим `eventfd` вдвое снижает потребление fd.** На Zephyr или при `-d:nimPollHasEventFds` каждый `SelectEvent` использует один fd вместо двух. Реализация `close` учитывает это, условно пропуская второй вызов `close`.
