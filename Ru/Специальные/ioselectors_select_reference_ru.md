# Справочник по модулю `ioselectors_select`

> **Модуль:** `std/ioselectors` (бэкенд POSIX/Windows `select()`)  
> **Язык:** Nim  
> **Платформа:** **Windows** и любая POSIX-система — единственный бэкенд `ioselectors`, работающий на обеих платформах. Используется как последний запасной вариант, когда epoll, kqueue и poll недоступны.  
> **Назначение:** Реализует API `ioselectors` поверх классического системного вызова `select()`, обеспечивая кроссплатформенное мультиплексирование I/O ценой самых жёстких ограничений масштабируемости среди всех бэкендов.

---

## Контекст: что такое select() и почему это крайний случай?

`select()` — старейший из четырёх Unix-интерфейсов мультиплексирования I/O, появившийся в 4.2BSD в 1983 году. Он предшествует `poll()`, `epoll` и `kqueue`, и его дизайн отражает ограничения той эпохи:

- В пользовательском пространстве поддерживаются **три структуры-битовых маски** (`fd_set`) — одна для чтения, одна для записи, одна для исключений.
- Перед каждым вызовом `select()` все три набора **копируются** во временные переменные и передаются ядру, потому что `select()` **перезаписывает** их результатами.
- Ядро сканирует наборы и сбрасывает биты для дескрипторов, не готовых к работе. Оставшиеся биты указывают на готовые дескрипторы.
- `select()` имеет **жёсткое ограничение** на максимальный номер файлового дескриптора: `FD_SETSIZE`, обычно равный 1024 на POSIX и 64 на Windows (где считаются сокеты, а не номера fd).

Эти ограничения делают `select()` наименее масштабируемым мультиплексором, но зато наиболее переносимым: он работает на любой POSIX-системе и на Windows без дополнительных заголовков.

### Зачем этот бэкенд вообще существует

Бэкенд `select` — кроссплатформенная страховочная сетка `ioselectors`. На Windows нет ни epoll, ни kqueue; `select()` (через Winsock2) — единственный доступный механизм. На экзотических встраиваемых POSIX-системах, где может не быть `poll()`, `select()` служит запасным вариантом. Для любой Nim-программы, которая должна работать на Windows и использовать `ioselectors`, именно этот бэкенд делает это возможным.

### Как здесь работает таблица `fds` — принципиальное отличие

В kqueue- и poll-бэкендах `fds` индексируется **непосредственно по номеру дескриптора** — поиск fd 42 — это O(1): просто обращение к `fds[42]`. Здесь, поскольку дескрипторы сокетов Windows могут быть произвольными большими целыми числами (а не маленькими последовательными), массив `fds` используется как **неупорядоченный список, сканируемый линейно**. Каждый поиск — `getKey`, `setSelectKey`, `delKey`, `contains`, `getData`, `setData`, `withData` — это **линейное сканирование всех слотов `FD_SETSIZE`**. Это фундаментальная O(n)-стоимость каждой операции, а не только ожидания.

## Типы

### `Selector[T]`

```nim
# однопоточная сборка:
type Selector*[T] = ref SelectorImpl[T]
# потокобезопасная сборка (--threads:on):
type Selector*[T] = ptr SelectorImpl[T]
```

Центральный объект. Содержит три объекта `FdSet` — `rSet`, `wSet`, `eSet` — которые являются **мастер-копиями** наборов дескрипторов для чтения, записи и исключений. Они обновляются инкрементально по мере регистрации и отмены регистрации дескрипторов. При вызове `selectInto` они копируются во временные переменные, которые `select()` может испортить, сохраняя мастер-состояние для следующего вызова.

Также содержит `fds` — неупорядоченный массив `SelectorKey[T]` фиксированного размера `FD_SETSIZE`, по которому выполняется линейный поиск при каждом обращении.

`maxFD` отслеживает наибольший номер зарегистрированного дескриптора (только POSIX). Это значение передаётся первым аргументом в `select()` (`nfds = maxFD + 1`), что сообщает ядру, насколько далеко сканировать битовые наборы. На Windows этот аргумент игнорируется `select()`/Winsock, поэтому `maxFD` там не отслеживается.

В потокобезопасных сборках все мутации наборов и `fds` защищены мьютексом `Lock` через `withSelectLock`.

### `SelectEvent`

```nim
type SelectEvent* = ptr SelectEventImpl
# SelectEventImpl = object
#   rsock: SocketHandle   # читающий сокет
#   wsock: SocketHandle   # записывающий сокет
```

Уведомление, запускаемое вручную. В именах полей используется `sock`, а не `fd`, потому что на Windows `select()` работает только с **Winsock-сокетами**, а не с произвольными дескрипторами. Реализация `newSelectEvent` поэтому **принципиально разная** на Windows и POSIX.

## Функции жизненного цикла

### `newSelector`

```nim
proc newSelector*[T](): Selector[T]
```

#### Что делает

Создаёт и инициализирует новый селектор:

1. В потокобезопасной сборке выделяет `SelectorImpl` и массив `fds` в разделяемой памяти и инициализирует мьютекс. В однопоточной — выделяет обычным образом.
2. Помечает каждый слот `fds` как `InvalidIdent`.
3. Нулевая инициализация всех трёх структур `FdSet` через `FD_ZERO`, сбрасывающую каждый бит.

Никакой объект в ядре не создаётся — `select()`, как и `poll()`, не имеет состояния на стороне ядра. Всё живёт в пользовательском пространстве в `SelectorImpl`.

#### Пример

```nim
import std/ioselectors

let sel = newSelector[string]()
defer: sel.close()
```

---

### `close` (Selector)

```nim
proc close*[T](s: Selector[T])
```

#### Что делает

В **потокобезопасной сборке**: освобождает разделяемый массив `fds`, освобождает сам `SelectorImpl` и уничтожает мьютекс. Обратите внимание на порядок: `deinitLock` вызывается *после* освобождения памяти — это означает, что мьютекс не должен быть захвачен в момент вызова.

В **однопоточной сборке**: не делает ничего. Сборщик мусора Nim управляет `ref`-типизированным селектором. Никакого ресурса ядра освобождать не нужно.

#### Пример

```nim
let sel = newSelector[int]()
sel.close()
```

---

### `newSelectEvent` — POSIX

```nim
# POSIX (не Windows):
proc newSelectEvent*(): SelectEvent
```

#### Что делает (POSIX)

На POSIX `select()` умеет наблюдать за любыми файловыми дескрипторами, а не только сокетами. Реализация следует тому же подходу, что и poll-бэкенд: создаёт OS-пайп, переводит оба конца в неблокирующий режим и сохраняет их как `rsock` и `wsock` (имена полей используют «sock» для единообразия с Windows-версией, но хранят обычные fd пайпа, приведённые к `SocketHandle`).

---

### `newSelectEvent` — Windows

```nim
# Windows:
proc newSelectEvent*(): SelectEvent
```

#### Что делает (Windows)

На Windows `select()` работает **только с Winsock-сокетами**, не с пайпами или произвольными дескрипторами. Пайпы нельзя добавить в `fd_set` на Windows. Это вынуждает использовать более сложную реализацию:

1. Создаёт временный **слушающий сокет** (`ssock`), привязанный к `127.0.0.1:0` (ОС сама назначает порт).
2. Создаёт **записывающий сокет** (`wsock`) и подключает его к тому же loopback-адресу и порту.
3. Вызывает `accept()` на слушающем сокете, чтобы получить **читающий сокет** (`rsock`).
4. Закрывает слушающий сокет — он был нужен только для установления соединения.
5. Переводит оба сокета `rsock` и `wsock` в неблокирующий режим через `ioctlsocket(FIONBIO)`.
6. Выделяет `SelectEventImpl` в разделяемой памяти.

В итоге получается пара связанных loopback TCP-сокетов. Когда `trigger` отправляет данные через `wsock`, `rsock` становится читаемым, что `select()` обнаруживает обычным образом.

Это дороже пайпа (создаётся три сокета, один сразу закрывается, выполняется полный TCP-хэндшейк на loopback), но является единственным переносимым механизмом на Windows без дополнительных API.

#### Пример (работает одинаково на обеих платформах)

```nim
let wakeup = newSelectEvent()
defer: wakeup.close()
```

---

### `trigger`

```nim
# POSIX:
proc trigger*(ev: SelectEvent)
# Windows:
proc trigger*(ev: SelectEvent)
```

#### Что делает

Записывает 8 байт (`uint64 = 1`) в `ev.wsock`:

- **POSIX**: использует `posix.write` на fd записывающего конца пайпа.
- **Windows**: использует `winlean.send` на записывающем сокете.

В обоих случаях это делает `ev.rsock` читаемым, что `select()` зафиксирует как активность.

Безопасно вызывать из любого потока.

#### Пример

```nim
# Из фонового потока:
wakeupEvent.trigger()
# selectInto/select основного потока вернётся с Event.User.
```

---

### `close` (SelectEvent)

```nim
# POSIX:
proc close*(ev: SelectEvent)
# Windows:
proc close*(ev: SelectEvent)
```

#### Что делает

Закрывает оба сокета/дескриптора и освобождает разделяемую память:

- **POSIX**: `posix.close` для `rsock` и `wsock`.
- **Windows**: `winlean.closesocket` для обоих.

Всегда вызывайте `unregister` *перед* `close`.

## Функции регистрации

### `registerHandle`

```nim
proc registerHandle*[T](s: Selector[T], fd: int | SocketHandle,
                        events: set[Event], data: T)
```

#### Что делает

Регистрирует OS-дескриптор или сокет в селекторе. Мьютекс удерживается на протяжении всей операции.

1. Вызывает `setSelectKey` — **линейное сканирование** `fds` для поиска свободного слота (`InvalidIdent`) и сохранения ключа. Бросает исключение, если `fds` заполнен.
2. На POSIX обновляет `maxFD`, если `fd > s.maxFD`. На Windows `maxFD` не отслеживается.
3. Если `Event.Read` в `events`: вызывает `IOFD_SET(fd, rSet)` и инкрементирует `count`.
4. Если `Event.Write` в `events`: вызывает `IOFD_SET(fd, wSet)` **и** `IOFD_SET(fd, eSet)`, инкрементируя `count`. Набор исключений (`eSet`) также включается для регистраций с записью — на Windows ошибки на записываемых сокетах сообщаются через набор исключений.

Важно: `maxFD` никогда не уменьшается при отмене регистрации — он только растёт. После отмены регистрации дескриптора с наибольшим номером `select()` продолжит получать `maxFD + 1` как `nfds`, что безвредно, но несколько расточительно.

#### Пример

```nim
import std/[ioselectors, nativesockets]

let sel = newSelector[string]()
let sock = createNativeSocket()
sock.setBlocking(false)
sel.registerHandle(sock, {Event.Read}, "клиент-А")
```

---

### `updateHandle`

```nim
proc updateHandle*[T](s: Selector[T], fd: int | SocketHandle,
                      events: set[Event])
```

#### Что делает

Изменяет наблюдаемые события для уже зарегистрированного I/O-дескриптора. Захватывает мьютекс, вызывает `getKey` (линейный поиск) для нахождения текущего ключа, затем сравнивает старый набор событий с новым:

- `Read` был и убирается: `IOFD_CLR(fd, rSet)`, уменьшить `count`.
- `Write` был и убирается: `IOFD_CLR(fd, wSet)`, `IOFD_CLR(fd, eSet)`, уменьшить `count`.
- `Read` не был и добавляется: `IOFD_SET(fd, rSet)`, увеличить `count`.
- `Write` не был и добавляется: `IOFD_SET(fd, wSet)`, `IOFD_SET(fd, eSet)`, увеличить `count`.

Запрещена для дескрипторов, зарегистрированных с событиями Timer, Signal, Process, Vnode, User, Oneshot или Error.

#### Пример

```nim
sel.registerHandle(sock, {Event.Read}, myData)

# Нужно что-то отправить:
sel.updateHandle(sock, {Event.Read, Event.Write})

# Буфер записи опустел:
sel.updateHandle(sock, {Event.Read})
```

---

### `registerEvent`

```nim
proc registerEvent*[T](s: Selector[T], ev: SelectEvent, data: T)
```

#### Что делает

Регистрирует `SelectEvent` (читающий сокет) в селекторе. Сохраняет `{Event.User}` в ключе, добавляет `ev.rsock` в `rSet` и инкрементирует `count`. На POSIX также обновляет `maxFD`.

Когда `selectInto` позднее обнаруживает, что `ev.rsock` читаем, он идентифицирует ключ как `Event.User` и читает 8 байт через `recv` для опустошения сокета/пайпа, прежде чем вернуть `Event.User` вызывающему.

#### Пример

```nim
let wakeup = newSelectEvent()
sel.registerEvent(wakeup, data = "пробуждение-потока")
```

## Отмена регистрации

### `unregister` (fd)

```nim
proc unregister*[T](s: Selector[T], fd: SocketHandle | int)
```

#### Что делает

Захватывает мьютекс. Вызывает `getKey` (линейный поиск) для нахождения ключа. Удаляет fd из соответствующих наборов:

- Если был зарегистрирован `Read` или `User`: `IOFD_CLR(fd, rSet)`, уменьшить `count`.
- Если был зарегистрирован `Write`: `IOFD_CLR(fd, wSet)`, `IOFD_CLR(fd, eSet)`, уменьшить `count`.

Затем вызывает `delKey` (ещё одно линейное сканирование) для очистки слота в `fds`.

---

### `unregister` (SelectEvent)

```nim
proc unregister*[T](s: Selector[T], ev: SelectEvent)
```

#### Что делает

Захватывает мьютекс. Убирает `ev.rsock` из `rSet`, уменьшает `count` и вызывает `delKey` для очистки слота в `fds`. Всегда вызывайте это перед `ev.close()`.

## Ожидание событий

### `selectInto`

```nim
proc selectInto*[T](s: Selector[T], timeout: int,
                    results: var openArray[ReadyKey]): int
```

#### Что делает

Центральная функция ожидания. Её механика отличается от poll-бэкенда в тонком, но важном аспекте.

**Шаг 1 — копирование мастер-наборов.** Под мьютексом копирует `rSet`, `wSet`, `eSet` в локальные переменные `rset`, `wset`, `eset`. Это обязательно, потому что `select()` *изменяет* переданные ему наборы, сбрасывая биты для неготовых дескрипторов. Если бы мастер-наборы передавались напрямую, каждый вызов разрушал бы состояние регистрации.

**Шаг 2 — вызов `select()`.** Передаёт три скопированных набора вместе с `maxFD + 1` (POSIX) и тайм-аутом. На Windows `EINTR` не подавляется — любая ошибка приводит к немедленному исключению.

**Семантика timeout:**

| Значение | Смысл |
|----------|-------|
| `0` | Неблокирующая проверка |
| `> 0` | Ждать не дольше указанного числа миллисекунд |
| `-1` | Ждать бесконечно (передаётся `nil` как указатель на тайм-аут) |

Примечание: `select()` использует разрешение в микросекундах (`timeval`), тогда как `poll()` — в миллисекундах. Выполняется преобразование `timeout * 1000` для `tv_usec`.

**Шаг 3 — сканирование `fds` для получения результатов.** После возврата `select()` код последовательно сканирует `fds[0..FD_SETSIZE)`, проверяя каждую зарегистрированную запись через `IOFD_ISSET` по трём результирующим наборам:

- `IOFD_ISSET(fd, rset)`: если ключ — `Event.User`, читает 8 байт через `recv` для опустошения сокета/пайпа. При `EAGAIN` кто-то другой уже потребил событие — пропустить. Иначе возвращает `Event.User`. Для обычного дескриптора чтения — возвращает `Event.Read`.
- `IOFD_ISSET(fd, wset)`: добавляет `Event.Write` в результат.
- Если `wset` сработал и ещё `IOFD_ISSET(fd, eset)`: добавляет `Event.Error`.

Это сканирование — **O(`FD_SETSIZE`)** вне зависимости от числа реально готовых дескрипторов: нужно проверить каждую зарегистрированную запись, потому что `select()` не сообщает, *какие именно* битовые позиции он установил — только суммарное число оставшихся битов.

**Обработка EINTR**: на POSIX, если `select()` возвращает -1 с `EINTR`, функция возвращает `0` (не ошибку). На Windows любой отрицательный возврат немедленно бросает исключение.

#### Пример

```nim
var ready: array[64, ReadyKey]
while true:
  let n = sel.selectInto(timeout = 1000, ready)
  for i in 0 ..< n:
    if Event.Read in ready[i].events:
      echo "fd ", ready[i].fd, " готов к чтению"
    if Event.Error in ready[i].events:
      echo "fd ", ready[i].fd, " ошибка"
```

### `select`

```nim
proc select*[T](s: Selector[T], timeout: int): seq[ReadyKey]
```

#### Что делает

Удобная обёртка. Выделяет `seq[ReadyKey]` размером `FD_SETSIZE` (а не `MAX_POLL_EVENTS` как в других бэкендах — здесь результирующий буфер такого же размера, как сам набор fd), вызывает `selectInto` и обрезает результат.

Замечание: поскольку `FD_SETSIZE` на POSIX обычно равен 1024, здесь выделяется больше памяти, чем в 64-слотовых буферах epoll/kqueue/poll-бэкендов.

---

### `flush`

```nim
proc flush*[T](s: Selector[T]) = discard
```

#### Что делает

Пустышка (no-op). Существует для удовлетворения контракта API `ioselectors` — в kqueue-бэкенде `flush` нетривиален при отключённом кэшировании изменений. Здесь нечего сбрасывать: `select()` получает все входные данные при каждом вызове.

## Функции доступа к данным

Принципиальное отличие от других бэкендов: **все функции доступа к данным выполняют линейное сканирование**. Нет O(1)-обращения по номеру fd. Каждый вызов `getData`, `setData`, `contains` и `withData` просматривает до `FD_SETSIZE` слотов. Это цена совместимости с дескрипторами сокетов Windows, не гарантированно являющимися маленькими последовательными числами.

### `getData`

```nim
proc getData*[T](s: Selector[T], fd: SocketHandle | int): var T
```

#### Что делает

Захватывает мьютекс, сканирует `fds[0..FD_SETSIZE)` для поиска слота с `ident == fd` и возвращает изменяемую ссылку на его `.data`. Если fd не найден, возвращает (без значения — это неопределённое поведение, если T не инициализирован по умолчанию). Используйте `contains` для предварительной проверки.

#### Пример

```nim
sel.registerHandle(sock, {Event.Read}, 0)
sel.getData(sock) += 1
```

---

### `setData`

```nim
proc setData*[T](s: Selector[T], fd: SocketHandle | int, data: T): bool
```

#### Что делает

Захватывает мьютекс, сканирует для поиска fd и при нахождении перезаписывает `.data`. Возвращает `true` при успехе, `false` если не найдено.

**Важно: в этой реализации есть скрытая ошибка в цикле.** Пропущен `inc(i)` в ветке несовпадения — цикл завершается только через `break` при совпадении или когда `i` достигает `FD_SETSIZE`... но `i` не инкрементируется в несовпадающих итерациях. На практике функция работает корректно, потому что единственный выход при несовпадении — через `FD_SETSIZE`, но это хрупкая конструкция.

#### Пример

```nim
discard sel.setData(sock, newState)
```

---

### `contains`

```nim
proc contains*[T](s: Selector[T], fd: SocketHandle | int): bool {.inline.}
```

#### Что делает

Захватывает мьютекс и линейно сканирует `fds` для поиска слота с `ident == fd`. Возвращает `true` при нахождении. O(`FD_SETSIZE`).

В отличие от других бэкендов (где это O(1) через прямое индексирование массива), здесь — полное сканирование при каждом вызове.

#### Пример

```nim
if sock in sel:
  sel.updateHandle(sock, {Event.Read, Event.Write})
```

---

### `withData` (ветка «найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body: untyped)
```

#### Что делает

Захватывает мьютекс, сканирует для поиска `fd`. При нахождении привязывает `value` как указатель на `.data` и выполняет `body`. Если не найдено — `body` пропускается. Поскольку мьютекс удерживается на протяжении `body`, будьте осторожны: не вызывайте другие методы селектора внутри `body` в потокобезопасной сборке — это приведёт к дедлоку.

#### Пример

```nim
sel.withData(sock, connPtr):
  connPtr.bytesSent += 512
```

---

### `withData` (ветки «найдено» и «не найдено»)

```nim
template withData*[T](s: Selector[T], fd: SocketHandle | int,
                      value, body1, body2: untyped)
```

#### Что делает

Двухветочная версия: `body1` выполняется при нахождении (с привязанным `value`), `body2` — если не найдено. Применяется то же предупреждение об удержании мьютекса.

#### Пример

```nim
sel.withData(sock, connPtr):
  connPtr.bytesSent += 512
do:
  echo "fd не зарегистрирован"
```

## Функции запросов

### `isEmpty`

```nim
template isEmpty*[T](s: Selector[T]): bool
```

Возвращает `true`, когда `s.count == 0`. Ни один дескриптор не наблюдается.

---

### `getFd`

```nim
proc getFd*[T](s: Selector[T]): int
```

Всегда возвращает `-1`. Как и poll-бэкенд, `select()` не имеет постоянного объекта в ядре.

## Полный пример: кроссплатформенный событийный цикл

```nim
import std/[ioselectors, nativesockets]

type ConnData = object
  id: int

let sel = newSelector[ConnData]()
defer: sel.close()

# Следим за сокетом на чтение
let sock = createNativeSocket()
sock.setBlocking(false)
sel.registerHandle(sock, {Event.Read}, ConnData(id: 1))

# Кросспоточное событие пробуждения (работает на Windows и POSIX)
let wakeup = newSelectEvent()
sel.registerEvent(wakeup, ConnData(id: 0))

while not sel.isEmpty:
  let ready = sel.select(timeout = 2000)
  for key in ready:
    if Event.Read in key.events:
      echo "сокет fd ", key.fd, " содержит данные"
    if Event.Write in key.events:
      echo "сокет fd ", key.fd, " готов к записи"
    if Event.Error in key.events:
      echo "сокет fd ", key.fd, " ошибка"
    if Event.User in key.events:
      echo "разбужен trigger()"
      sel.unregister(wakeup)
      wakeup.close()
      sel.unregister(sock)
```

## Сравнение: select-бэкенд vs poll-бэкенд vs kqueue-бэкенд

| Аспект | select-бэкенд | poll-бэкенд | kqueue-бэкенд |
|--------|--------------|-------------|---------------|
| Платформа | Windows + POSIX | Только POSIX | Только BSD |
| Объект в ядре | Отсутствует | Отсутствует | fd kqueue |
| Поиск в таблице fd | O(n) линейный | O(1) прямой | O(1) прямой |
| `contains` | O(n) линейный | O(1) | O(1) |
| `getData`/`setData` | O(n) линейный | O(1) | O(1) |
| Макс. дескрипторов | `FD_SETSIZE` (жёсткое) | ulimit процесса | ulimit процесса |
| `FD_SETSIZE` типично | 1024 POSIX / 64 Win | Нет такого ограничения | Нет такого ограничения |
| Наборы fd select() | Копируются при каждом вызове | Весь массив передаётся | Состояние в ядре |
| Сканирование после ожидания | O(`FD_SETSIZE`) | O(`pollcnt`) | O(готовые события) |
| Сообщение об ошибках | Запись + набор исключений | Флаг POLLERR | Флаг EV_ERROR |
| `SelectEvent` (POSIX) | Пайп | Пайп или eventfd | Пайп |
| `SelectEvent` (Windows) | TCP-сокеты loopback | Н/П | Н/П |
| Timer/Signal/Process | Не поддерживаются | Не поддерживаются | Нативная поддержка |
| `getFd()` | Всегда -1 | Всегда -1 | fd kqueue |
| `flush()` | No-op | No-op | Может сбрасывать изменения kevent |

---

## Примечания и подводные камни

- **`FD_SETSIZE` — жёсткий потолок.** На большинстве POSIX-систем это 1024 (установлено во время компиляции, а не во время выполнения). На Windows с Winsock — 64. Если процесс открывает больше файловых дескрипторов, чем это значение — даже если в селекторе зарегистрированы лишь некоторые — `select()` может вести себя непредсказуемо при дескрипторе с номером, превышающим `FD_SETSIZE`. Это ограничение уровня ядра, а не только Nim.
- **Каждый поиск — O(n).** В отличие от poll- и kqueue-бэкендов, где `fds[fd]` — прямое обращение по индексу, здесь каждый вызов `contains`, `getData`, `setData`, `withData` и `getKey` сканирует до `FD_SETSIZE` слотов. При большом `FD_SETSIZE` и многих короткоживущих соединениях это накапливается.
- **Наборы fd копируются при каждом вызове `selectInto`.** Три полных копии структур `FdSet` происходят перед каждым системным вызовом `select()`, потому что `select()` разрушает свои входные данные.
- **`maxFD` только растёт, никогда не уменьшается.** После отмены регистрации дескриптора с наибольшим номером `select()` продолжает получать `maxFD + 1` как `nfds`. Это безопасно, но ядро сканирует больше битов, чем строго необходимо.
- **На Windows `EINTR` не перехватывается.** Любой отрицательный возврат из `ioselect` на Windows немедленно бросает исключение. На POSIX -1 с `EINTR` молча возвращается как `0`.
- **На Windows `select()` игнорирует аргумент `nfds`.** Windows-реализация `select()` полностью игнорирует первый аргумент (работает с дескрипторами сокетов, а не номерами fd). Поэтому `maxFD` в Windows-сборке не отслеживается.
- **`newSelectEvent` на Windows выполняет TCP-хэндшейк.** Создание события пробуждения на Windows выделяет три сокета, выполняет последовательность `bind`/`listen`/`connect`/`accept` на loopback, затем уничтожает слушатель. Это дорого по сравнению с пайпом, но неизбежно без `eventfd` или `CreateEvent`.
- **Мьютекс удерживается во время `body` в `withData`.** Шаблоны захватывают мьютекс селектора перед выполнением пользовательского кода. Не вызывайте другие методы селектора изнутри `body` в потокобезопасных сборках — это приведёт к дедлоку.
- **В `setData` скрытая ошибка цикла.** Пропущенный `inc(i)` в ветке несовпадения делает выход из цикла возможным только через `break` при совпадении. На практике функция работает корректно, но конструкция хрупкая.
