# volatile — справочник по модулю

## Обзор

Модуль `volatile` предоставляет две обобщённые процедуры — `volatileLoad` и `volatileStore` — для выполнения **энергозависимого (volatile) обращения к памяти** в Nim. Они предназначены прежде всего для **встраиваемого и системного программирования**, где компилятор не должен оптимизировать, переупорядочивать или кешировать операции чтения и записи в память.

### Что означает «volatile»?

В C и C++ ключевое слово `volatile` у указателя или переменной говорит компилятору: *«не делай никаких предположений о содержимом этой ячейки памяти между обращениями к ней»*. Каждое чтение обязано физически обращаться к аппаратному адресу; каждая запись обязана физически достигать его. Без `volatile` оптимизирующий компилятор может:

- Закешировать значение в регистре и больше не перечитывать его из памяти.
- Исключить «лишнюю», с его точки зрения, запись.
- Переупорядочить чтения и записи ради производительности.

Это катастрофично, когда ячейка памяти является **регистром аппаратного устройства**, **адресом устройства ввода-вывода, отображённого в память (memory-mapped I/O)**, или разделяемой памятью, которую изменяет обработчик прерываний (ISR) или иной контекст исполнения, невидимый компилятору.

### Когда нужен этот модуль?

- Чтение текущего значения **регистра периферийного устройства** (например, флага готовности UART, счётчика таймера).
- Запись команды или значения в **регистр, отображённый в память**.
- Доступ к переменным, разделяемым между **основным кодом и обработчиками прерываний** на bare-metal системах.
- Любая ситуация, в которой компилятор не может видеть полную картину того, кто изменяет память.

### Поведение на разных бэкендах

| Бэкенд | Поведение |
|---|---|
| C / C++ | Генерируется настоящее приведение к `volatile`-указателю; компилятор лишается права кешировать или переупорядочивать обращение. |
| Nim VM (`nimvm`) | Деградирует до обычного разыменования `src[]` / `dest[] = val` — volatile-семантика не имеет смысла во время компиляции. |
| JavaScript | Деградирует до обычного разыменования — движок JS не знает понятия «volatile-память». |

## Экспортируемые процедуры

---

### `volatileLoad`

```nim
proc volatileLoad*[T](src: ptr T): T {.inline, noinit.}
```

Выполняет **volatile-чтение** из адреса памяти, на который указывает `src`, и возвращает хранящееся там значение типа `T`.

Процедура обобщённая: `T` может быть любым типом — `uint8`, `uint32`, `distinct`-типом регистра, `tuple` и т. д. Компилятор генерирует приведение к `volatile`-указателю перед разыменованием, что гарантирует обращение к реальному аппаратному адресу при каждом вызове, а не к закешированной копии.

Прагма `noinit` указывает Nim не инициализировать нулём переменную `result` до того, как volatile-чтение запишет в неё значение — это устраняет лишнюю запись в памяти.

**На C-бэкенде генерируется примерно такой код:**

```c
result = (*( uint32 volatile* ) src);
```

**Пример — чтение регистра состояния аппаратного устройства:**

```nim
import volatile

# Допустим, 0x40013800 — базовый адрес периферийного UART.
# Бит 5 его регистра состояния означает «буфер передачи пуст».
const UART_SR = cast[ptr uint32](0x40013800'u32)

proc uartReady(): bool =
  let status = volatileLoad(UART_SR)
  return (status and (1'u32 shl 5)) != 0
```

Каждый вызов `uartReady` физически перечитает регистр, а не воспользуется значением, закешированным компилятором после предыдущего чтения.

**Пример — чтение байта из памяти, обновляемой обработчиком прерываний:**

```nim
import volatile

var flag {.global.}: uint8 = 0

# Обработчик прерывания асинхронно устанавливает flag = 1.
# В главном цикле нельзя допустить кеширование чтения компилятором:
proc pollFlag(): uint8 =
  volatileLoad(addr flag)
```

---

### `volatileStore`

```nim
proc volatileStore*[T](dest: ptr T, val: T) {.inline.}
```

Выполняет **volatile-запись** значения `val` по адресу памяти, на который указывает `dest`.

Как и `volatileLoad`, процедура обобщённая и работает с любым типом `T`. Компилятор генерирует приведение к `volatile`-указателю перед присваиванием, гарантируя, что запись действительно достигнет целевого адреса и не будет исключена оптимизатором как «лишняя».

**На C-бэкенде генерируется примерно такой код:**

```c
*((uint32 volatile*)(dest)) = val;
```

**Пример — запись команды в регистр управления аппаратным устройством:**

```nim
import volatile

const TIMER_CR1 = cast[ptr uint32](0x40000000'u32)

proc startTimer() =
  # Устанавливаем бит 0 (CEN — включение счётчика) в регистре управления таймером.
  volatileStore(TIMER_CR1, 1'u32)
```

Без `volatile` компилятор мог бы заметить, что из `TIMER_CR1` «нигде не читают», и тихо исключить запись. С `volatileStore` запись гарантированно произойдёт.

**Пример — сигнализация обработчику прерываний через разделяемую память:**

```nim
import volatile

var ready {.global.}: uint8 = 0

proc signalIsr() =
  # Обработчик прерываний читает `ready`; нужно убедиться, что запись видна.
  volatileStore(addr ready, 1'u8)
```

## Полный пример использования

Следующий пример моделирует типичный bare-metal паттерн: опрос регистра состояния периферийного устройства и запись данных в регистр данных после того, как устройство готово.

```nim
import volatile

# Адреса регистров, отображённых в память (условные значения)
const
  SPI_SR   = cast[ptr uint32](0x4001300C'u32)  # Регистр состояния
  SPI_DR   = cast[ptr uint32](0x40013010'u32)  # Регистр данных
  TXE_BIT  = 1'u32 shl 1                        # Флаг "буфер передачи пуст"

proc spiSendByte(data: uint8) =
  # Ждём, пока буфер передачи не освободится — каждый раз читаем заново.
  while (volatileLoad(SPI_SR) and TXE_BIT) == 0:
    discard

  # Записываем байт — запись нельзя оптимизировать.
  volatileStore(SPI_DR, uint32(data))
```

Без volatile-обращений оптимизирующий компилятор мог бы вынести чтение `SPI_SR` за пределы цикла (он не видит обычного кода Nim, который изменяет этот адрес), что привело бы к бесконечному циклу или неверному поведению.

## Важные замечания

- **Используйте `volatileLoad` / `volatileStore` вместо обычных `src[]` / `dest[] = val`** всегда, когда ячейка памяти может измениться или должна измениться вне поля зрения компилятора: аппаратные регистры, переменные, разделяемые с ISR, спин-блокировки и т. д.
- **Volatile — не примитив синхронизации.** На многоядерных системах он не обеспечивает упорядоченность обращений к памяти (memory ordering) и атомарность. Для этих целей используйте атомарные операции или соответствующие барьеры памяти.
- **Эффект есть только на C-подобных бэкендах.** На Nim VM и JS-бэкенде процедуры деградируют до обычного разыменования указателя — volatile-семантика в этих средах просто отсутствует.
- **Процедуры помечены `inline`** — накладных расходов на вызов нет. Они раскрываются непосредственно в точке вызова.
