# `strmantle.nim` — справочник модуля

> **Часть стандартной библиотеки времени выполнения Nim**  
> Внутренний вспомогательный модуль компилятора — пользовательский код его не импортирует.

---

## Общее описание

`strmantle.nim` — это **низкоуровневый модуль поддержки строковых и символьных операций** рантайма Nim. Название «mantle» («мантия», «оболочка») отражает его роль: он обёртывает и поддерживает базовую строковую инфраструктуру, располагаясь между высокоуровневым строковым API и железом.

Каждая процедура здесь помечена либо `{.compilerproc.}`, либо `{.compilerRtl.}` — это значит, что **компилятор Nim генерирует вызовы этих функций автоматически** при компиляции обычных строковых операций: `==`, `<`, `<=`, а также конверсий строки в bool/char. Их никогда не вызывают явно — они вызываются неявно, от вашего имени.

В модуле также содержится **каноническая процедура разбора числа с плавающей точкой** `nimParseBiggestFloat`, от которой в конечном счёте зависит весь числовой парсинг Nim.

### Ключевые ограничения дизайна

- Нулевая зависимость от высокоуровневой стандартной библиотеки Nim — только от `c_memcmp`, `c_strcmp`, `equalMem` и единственного C-импорта (`strtod`).
- Все процедуры сравнения и хеширования используют в точности те же алгоритмы, которые применяет **сам компилятор** при генерации оптимизированных операторов `case` и сравнений строковых литералов. Комментарий *«the compiler needs exactly the same hash function!»* встречается дважды и не является преувеличением.
- Парсер чисел с плавающей точкой реализует **двухпутевую архитектуру**: быстрый путь для чисел, представимых в 53 битах, и медленный путь, делегирующий C-функции `strtod` всё остальное.

## Внутренняя константа: `IdentChars`

```nim
const IdentChars = {'a'..'z', 'A'..'Z', '0'..'9', '_'}
```

Не экспортируется. Набор символов, используемый в `nimParseBiggestFloat` для определения конца специального литерала типа `NaN` или `Inf` — чтобы убедиться, что они не встроены в середину идентификатора (например, `NaNometer` не должен разбираться как `NaN`).

---

## Внутренняя константа: `powtens`

```nim
const powtens = [1e0, 1e1, 1e2, ..., 1e22]
```

Не экспортируется. Таблица степеней десяти от 10⁰ до 10²², используемая быстрым путём `nimParseBiggestFloat`. Верхняя граница 10²² точна: это наибольшая степень десяти, которую можно точно представить в 64-битном числе с плавающей точкой по стандарту IEEE 754. За её пределами точное умножение уже невозможно и необходимо использовать медленный путь через `strtod`.

## Процедуры сравнения строк

Эти четыре процедуры образуют основу всех сравнений строк в Nim. Они помечены `{.compilerproc.}`: компилятор генерирует вызовы к ним для операций `==`, `<` и `<=` над значениями типа `string`.

---

### `cmpStrings`

```nim
proc cmpStrings(a, b: string): int {.inline, compilerproc.}
```

**Фундаментальное сравнение строк.** Возвращает отрицательное целое, если `a < b`, ноль, если `a == b`, и положительное целое, если `a > b`. Все остальные функции сравнения строк (`leStrings`, `ltStrings`, `eqStrings`) построены поверх неё.

Алгоритм следует классическому контракту C-функции `strcmp` с одним важным расширением: он корректно обрабатывает строки, содержащие встроенные нулевые байты (`\0`), потому что сравнивает **байтовые последовательности с префиксом длины**, а не по признаку нуль-терминации.

**Как работает:**

1. Вычисляется `minlen = min(a.len, b.len)`.
2. Если `minlen > 0`, используется `c_memcmp` для сравнения первых `minlen` байт. На большинстве платформ это одна высокооптимизированная машинная инструкция.
3. Если сравнение байт возвращает `0` (все общие байты равны), результатом является `a.len - b.len`. Более короткая строка предшествует более длинной.
4. Если любая из строк пустая (`minlen == 0`), сразу переходим к разнице длин.

```nim
# Соотношения, которые использует компилятор:
# "abc" vs "abd"  → memcmp находит 'c' < 'd' → отрицательный результат
# "abc" vs "abcd" → memcmp находит 3 равных байта → 3 - 4 = -1 (a короче)
# "abc" vs "abc"  → memcmp находит 3 равных байта → 3 - 3 = 0 (равны)
```

---

### `leStrings`

```nim
proc leStrings(a, b: string): bool {.inline, compilerproc.}
```

Возвращает `true`, если `a <= b` лексикографически. Тонкая обёртка: `cmpStrings(a, b) <= 0`.

Комментарий *«required by upcoming backends (NIR)»* указывает, что функция была явно выделена как именованный символ, чтобы новый бэкенд на основе IR (NIR) мог обращаться к ней напрямую по имени, а не воссоздавать логику встроенным кодом.

```nim
# Компилятор преобразует:
#   if s1 <= s2: ...
# в вызов leStrings(s1, s2)
```

---

### `ltStrings`

```nim
proc ltStrings(a, b: string): bool {.inline, compilerproc.}
```

Возвращает `true`, если `a < b` лексикографически. Тонкая обёртка: `cmpStrings(a, b) < 0`.

Та же мотивация, связанная с NIR, что и у `leStrings`. Вместе `ltStrings` и `leStrings` предоставляют бэкенду все необходимые предикаты сравнения без требования знать о `cmpStrings` напрямую.

```nim
# Компилятор преобразует:
#   if s1 < s2: ...
# в вызов ltStrings(s1, s2)
```

---

### `eqStrings`

```nim
proc eqStrings(a, b: string): bool {.inline, compilerproc.}
```

Возвращает `true`, если содержимое `a` и `b` идентично. Оптимизирована специально для случая проверки равенства — быстрее, чем `cmpStrings(a, b) == 0`, поскольку делает несколько ранних выходов:

1. Если `a.len != b.len` → сразу `false`. Сравнение байт не нужно.
2. Если `a.len == 0` → сразу `true`. Две пустые строки всегда равны.
3. Иначе → `equalMem`, сравнивающая сырые байты.

Эта трёхшаговая структура означает, что **неравенство обнаруживается за O(1)** для любых двух строк разной длины, независимо от содержимого.

```nim
# Компилятор преобразует:
#   if s1 == s2: ...
# в вызов eqStrings(s1, s2)
```

## Процедуры хеширования строк

### `hashString`

```nim
proc hashString(s: string): int {.compilerproc.}
```

Вычисляет целочисленный хеш Nim-строки `string`. Это именно та хеш-функция, которую использует компилятор при генерации **эффективных операторов `case` над строковыми литералами**. Вместо цепочки `if s == "x" elif s == "y"...` компилятор может сгенерировать таблицу диспетчеризации на основе хеша. Скомпилированный хеш обязан точно совпадать с этой функцией, иначе таблица диспетчеризации была бы некорректной.

**Алгоритм** — вариант хеша Jenkins one-at-a-time:

```
для каждого символа c:
    h += c
    h += h << 10
    h ^= h >> 6
# финализация:
h += h << 3
h ^= h >> 11
h += h << 15
```

Это обеспечивает хороший лавинный эффект (небольшие изменения входных данных дают большие изменения результата) при минимальном объёме кода — идеально подходит для хеш-таблиц и табличной диспетчеризации.

```nim
# Используется неявно, когда вы пишете:
case myString
of "apple":  ...
of "banana": ...
of "cherry": ...
# Компилятор хеширует каждый литерал при компиляции, а при выполнении
# хеширует входное значение один раз и диспетчеризует по ведру хеша.
```

---

### `hashCstring`

```nim
proc hashCstring(s: cstring): int {.compilerproc.}
```

Тот же хеш Jenkins one-at-a-time, что и в `hashString`, но для нуль-терминированных значений `cstring`. Ключевые отличия:

- Явно обрабатывает `nil`: nil-строка `cstring` хешируется как `0`.
- Итерирует по индексу до тех пор, пока не встретит `'\0'`, вместо использования поля длины.
- Используется компилятором при генерации операторов `case` над литералами `cstring`.

```nim
# nil-строка обрабатывается безопасно:
# hashCstring(nil) → 0
```

Алгоритм хеширования побайтово идентичен `hashString` — две строки с одинаковым содержимым всегда дадут одинаковый хеш, будь то `string` или `cstring`.

## Сравнение C-строк на равенство

### `eqCstrings`

```nim
proc eqCstrings(a, b: cstring): bool {.inline, compilerproc.}
```

Сравнивает два значения `cstring` на равенство с тщательной обработкой nil — то, чего сырой `c_strcmp` сделать не может, поскольку передача `nil` в `strcmp` является неопределённым поведением в C.

Логика имеет три случая, проверяемых по порядку:

1. `pointer(a) == pointer(b)` → оба указывают на одну и ту же память (или оба nil) → `true`. Это быстрая проверка равенства указателей.
2. `a.isNil or b.isNil` → ровно один из них nil, второй нет → `false`.
3. Иначе → `c_strcmp(a, b) == 0`, сравнивающая побайтово до нуль-терминатора.

```nim
# Безопасное сравнение, не вызывающее краша на nil:
var p: cstring = nil
var q: cstring = "hello"
# eqCstrings(p, q)    → false (без краша)
# eqCstrings(nil, nil) → true
# eqCstrings("hi", "hi") → true (одинаковое содержимое, возможно разные указатели)
```

## Разбор чисел с плавающей точкой

### `nimParseBiggestFloat`

```nim
proc nimParseBiggestFloat(s: openArray[char], number: var BiggestFloat): int {.compilerproc.}
```

**Центральная процедура разбора числа с плавающей точкой** для всей стандартной библиотеки Nim. `parseFloat`, `parseBiggestFloat` и преобразование строки в число с плавающей точкой в `strconv` — все они в конечном счёте вызывают именно её. Возвращает количество потреблённых символов из `s` или `0` при ошибке разбора. Разобранное значение записывается в выходной параметр `number`.

Это сложная процедура с несколькими отдельными стадиями. Полное понимание её объясняет, *почему* разбор чисел с плавающей точкой в Nim одновременно быстрый и корректный на всех платформах.

#### Стадия 1 — Знак

Читает необязательный ведущий `+` или `-`. Устанавливает множитель знака.

#### Стадия 2 — Специальные значения (NaN и Inf)

Распознаёт `NaN`/`nan` и `Inf`/`inf` (без учёта регистра). Проверка границы по `IdentChars` принципиальна: она гарантирует, что `NaNometer` будет отклонён (следующий за `NaN` символ `o` входит в `IdentChars`), тогда как `NaN`, стоящий отдельно или перед пробелом/пунктуацией, принимается.

```nim
var f: float
discard nimParseBiggestFloat("NaN", f)   # f = NaN,  возвращает 3
discard nimParseBiggestFloat("inf", f)   # f = +Inf, возвращает 3
discard nimParseBiggestFloat("-Inf", f)  # f = -Inf, возвращает 4
```

#### Стадия 3 — Накопление целой и дробной части

Читает цифры до и после десятичной точки в единый 64-битный аккумулятор `integer` типа `uint64`. Символы-разделители подчёркивания (стиль `1_000_000` в Nim) молча пропускаются. Количество целых цифр (`kdigits`) и дробных цифр (`fdigits`) подсчитывается отдельно для последующего вычисления дробного показателя.

#### Стадия 4 — Показатель степени

Читает необязательные `e` или `E`, за которыми следует необязательный знак и десятичные цифры, также допуская подчёркивания.

#### Стадия 5 — Быстрый путь (≤ 15–16 значащих цифр, |показатель| ≤ 22)

Если число помещается в 53 бита (не более 15 значащих цифр, или 16 цифр начиная с `0`–`8`) **и** итоговый скорректированный показатель степени находится в пределах `[-22, +22]`, результат может быть вычислен точно с помощью целочисленной арифметики и таблицы `powtens`:

```
number = sign × integer × 10^exponent   (используя powtens[exponent])
```

Это «быстрый путь», описанный в [статье на Exploringbinary.com](https://www.exploringbinary.com/fast-path-decimal-to-floating-point-conversion/), на которую ссылается исходный код.

Есть также небольшое расширение: если показатель степени немного превышает 22, процедура пытается поглотить избыток в целую часть путём умножения, используя оставшийся «запас» точных цифр.

#### Стадия 6 — Медленный путь (все остальные случаи)

Если быстрый путь не применим (слишком много цифр, слишком большой/маленький показатель), процедура:

1. Перекодирует всё число без десятичной точки, подчёркиваний и знака во временный буфер из 500 символов `t` в форме `MANTИССА E±ПОК`.
2. Вызывает C-функцию `strtod` для этого буфера.

Делегирование в `strtod` обрабатывает сложные случаи корректно и переносимо — комментарий объясняет, что это позволяет избежать проблем с переносимостью десятичного символа (разные C-библиотеки могут использовать разные разделители в `strtod` в зависимости от локали, но процедура всегда строит буфер с ASCII-символами `.` и `E`).

#### Переполнение и потеря значимости показателя

До принятия решения о быстром/медленном пути, если `absExponent > 999`:
- Если `integer == 0` → результат `0.0` (ноль, умноженный на что угодно = ноль).
- Если показатель отрицательный → результат `0.0 * sign` (потеря значимости до нуля).
- Если показатель положительный → результат `Inf * sign` (переполнение до бесконечности).

```nim
var f: float
discard nimParseBiggestFloat("1.5e10", f)           # быстрый путь: f = 15000000000.0
discard nimParseBiggestFloat("1.23456789012345678", f)  # медленный путь (17 цифр)
discard nimParseBiggestFloat("1e9999", f)           # переполнение: f = +Inf
discard nimParseBiggestFloat("1e-9999", f)          # потеря значимости: f = 0.0
discard nimParseBiggestFloat("3.14_159", f)         # подчёркивания допустимы: f ≈ 3.14159
discard nimParseBiggestFloat("abc", f)              # ошибка: возвращает 0
```

Пара прагм `{.push staticBoundChecks: off.}` / `{.pop.}` вокруг этой процедуры отключает статическую проверку границ массивов Nim для обращений к таблице `powtens`. Индекс всегда находится в пределах `[0, 22]` по построению, поэтому проверка доказуемо излишня и только замедляла бы внутренний цикл.

## Процедуры конверсии строк и символов

### `nimBoolToStr`

```nim
proc nimBoolToStr(x: bool): string {.compilerRtl.}
```

Преобразует `bool` в его каноническое строковое представление. Возвращает `"true"` для `true` и `"false"` для `false`. Это функция, которую компилятор вызывает при написании `$myBool` или интерполяции булева значения в строке формата.

Помечена `{.compilerRtl.}` (runtime library компилятора), а не `{.compilerproc.}` — тонкое различие: символы `compilerRtl` являются частью ABI рантайма и могут вызываться из генерируемого C-кода по имени, тогда как символы `compilerproc` компилятор может агрессивнее встраивать или подставлять.

```nim
# Компилятор преобразует:
#   let s = $true
# в вызов nimBoolToStr(true) → "true"
```

---

### `nimCharToStr`

```nim
proc nimCharToStr(x: char): string {.compilerRtl.}
```

Преобразует одиночный символ `char` в однобуквенную `string`. Выделяет новую `string` длиной 1 и записывает символ в `result[0]`. Это функция, которую компилятор вызывает при написании `$myChar`.

```nim
# Компилятор преобразует:
#   let s = $'A'
# в вызов nimCharToStr('A') → "A"
```

## Статистика памяти (только ARC/ORC)

### `GC_getStatistics*`

```nim
proc GC_getStatistics*(): string
```

*Компилируется только при активном флаге `-d:gcDestructors` (то есть при использовании управления памятью ARC или ORC).*

Возвращает многострочную строку в человекочитаемом виде с базовой статистикой памяти:

```
[GC] total memory: 1048576
[GC] occupied memory: 204800
```

Использует `getTotalMem()` и `getOccupiedMem()` из менеджера памяти. Закомментированная строка (`[GC] cycle collections:`) указывает на то, что статистика сборки циклических ссылок может быть добавлена в будущем.

Это одна из немногих **экспортированных** (`*`) процедур в этом модуле — она является частью публичного API статистики GC в Nim, доступного обычному пользовательскому коду для диагностики и профилирования памяти.

```nim
when defined(gcDestructors):
  echo GC_getStatistics()
  # [GC] total memory: ...
  # [GC] occupied memory: ...
```
