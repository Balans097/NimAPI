# Nim `alloc.nim` — справочник внутреннего аллокатора

> **Замечание об аудитории.** `alloc.nim` — это *внутренний* низкоуровневый аллокатор памяти Nim. Это **не публичный API** — он является частью runtime-библиотеки и компилируется непосредственно в каждую Nim-программу. Функции, описанные здесь, вызываются прозрачно через `system.alloc`, `system.dealloc` и сборщик мусора. Напрямую с этим модулем работают только при разработке самого Nim-runtime, кастомного GC или портирования на встроенную платформу без ОС-аллокатора.

---

## Обзор архитектуры

Аллокатор управляет памятью в виде выровненных по странице блоков, называемых **чанками (chunks)**. Существует два типа чанков:

- **Малые чанки (small chunks)** — ровно одна страница ОС (`PageSize`, обычно 4 КБ). Внутри делятся на *ячейки (cells)* фиксированного размера одного класса. Повторное использование освобождённых ячеек избегает системных вызовов для типичного случая.
- **Большие чанки (big chunks)** — одна и более страниц. Управляются матрицей свободных списков по алгоритму **TLSF** (Two-Level Segregated Fit) для поиска наилучшего соответствия за почти O(1).
- **Огромные чанки (huge chunks)** — больше `MaxBigChunkSize`. Выделяются и освобождаются напрямую у ОС при каждом вызове.

Все выделения находятся внутри `MemRegion` — самодостаточной кучи. Потоко-локальные регионы не разделяют ничего между собой, кроме явной области общей кучи, защищённой мьютексом.

### Три пути малого выделения

```
alloc
 └─► rawAlloc
       ├─ Путь 1: Нет свободного чанка нужного класса → запросить страницу у ОС
       ├─ Путь 2: Чанк есть, аккумулятор не исчерпан → сдвинуть acc-указатель
       └─ Путь 3: Чанк есть, в free-списке есть ячейка → взять ячейку  ← предпочтительный
```

### Два пути малого освобождения

```
dealloc
 └─► rawDealloc
       ├─ Мы владеем этим чанком → вернуть ячейку в free-список, возможно активировать чанк
       └─ Чанк принадлежит другому потоку → поместить в sharedFreeLists (ARC/ORC)
                                          → или отметить в AVL-дереве (legacy GC)
```

## Ключевые константы

| Константа | Смысл |
|---|---|
| `nimMinHeapPages` | Минимальное число страниц, запрашиваемых у ОС за раз (по умолчанию 128 = 0,5 МБ). Переопределяется флагом `-d:nimMinHeapPages=N`. |
| `SmallChunkSize` | Равна `PageSize` (обычно 4096 байт). Максимальный размер малого чанка. |
| `MaxFli` | Максимальный индекс первого уровня матрицы TLSF (30 на 64-бит, 14 на 16-бит). |
| `MaxLog2Sli` | Максимальный log₂ числа ячеек второго уровня (5, то есть 32 подвёдра). |
| `MaxBigChunkSize` | Наибольший размер чанка, вмещающийся в матрицу TLSF. Чанки большего размера становятся *огромными*. |
| `HugeChunkSize` | `MaxBigChunkSize + 1` — порог, выше которого чанк считается огромным. |
| `nimMaxHeap` | Опциональный потолок общего размера кучи в МБ (`-d:nimMaxHeap=N`). Ноль означает «без ограничений». |

## Ключевые типы

### `MemRegion`

Корневая структура данных, представляющая одну кучу целиком. В однопоточной программе — один `MemRegion`. При наличии потоков у каждого потока свой регион; для `allocShared` используется отдельный регион общей кучи.

Важные поля:

| Поле | Назначение |
|---|---|
| `freeSmallChunks[s]` | Массив списков свободных малых чанков, по одному на класс размера (индекс = `size / MemAlign`). Голова — *активный* чанк для данного класса. |
| `sharedFreeLists[s]` | *(только ARC/ORC)* Межпоточные свободные ячейки, ожидающие выборки потоком-владельцем. |
| `flBitmap` / `slBitmap` | Битовые карты TLSF: показывают, какие ячейки матрицы непусты. Позволяют за O(1) найти блок через bit-scan. |
| `matrix` | Матрица свободных списков TLSF `[RealFli][MaxSli]` из двусвязных списков больших чанков. |
| `currMem` / `maxMem` / `freeMem` / `occ` | Учёт: всего получено от ОС, пиковое ОС-количество, свободно от ОС, занято вызывающим кодом. |
| `chunkStarts` | `IntSet` (компактное битовое множество) всех индексов страниц, начинающих чанк. Используется для поиска чанка по указателю. |
| `heapLinks` | Связный список пар `(чанк, размер)` — полная история ОС-аллокаций. Используется в `deallocOsPages` для возврата памяти. |
| `llmem` | Низкоуровневый бамп-аллокатор для внутренних структур данных самого аллокатора (AVL-узлы, trunk-узлы, записи heapLinks). Живёт столько, сколько `MemRegion`. |

---

### `SmallChunk`

Одна страница ОС, разделённая на ячейки одного класса размера.

| Поле | Назначение |
|---|---|
| `size` | Размер ячейки для этого чанка (все ячейки одинакового размера). |
| `acc` | Аккумулятор — байтовое смещение следующей неинициализированной ячейки от `data`. Монотонно возрастает, пока страница не заполнена. |
| `free` | Суммарное число байт, которые ещё может предоставить чанк (через free-список и аккумулятор). Когда `free < size`, чанк исключается из `freeSmallChunks`. |
| `freeList` | Односвязный список переработанных ячеек. Ячейки могут принадлежать как данному чанку, так и быть *иностранными* (переданными из другого чанка или потока). |
| `foreignCells` | Счётчик свободных ячеек в `freeList`, происходящих из другого чанка. Чанк с `foreignCells > 0` **нельзя** освобождать: это привело бы к утере этих ячеек. |
| `owner` | Указатель на `MemRegion`, создавший этот чанк. Используется для обнаружения межпоточных освобождений. |
| `data` | Выровненное начало используемой памяти внутри страницы. |

---

### `BigChunk`

Выровненный по странице блок из одной или нескольких страниц для выделений, не помещающихся в малый чанк. Также служит базой для огромных чанков.

| Поле | Назначение |
|---|---|
| `size` | Полный размер чанка в байтах, включая заголовок `BigChunk`. |
| `prevSize` | Размер *предыдущего* соседнего чанка в виртуальном адресном пространстве; младший бит кодирует, занят ли *данный* чанк. Этот двойственный формат движет логикой слияния TLSF. |
| `prev` / `next` | Указатели двусвязного free-списка. **Пока чанк выделен**, `prev` перезаписывается выровненным указателем на данные (значением, возвращённым пользователю) — так GC и код освобождения находят пользовательский указатель по заголовку чанка. |
| `owner` | Владелец — `MemRegion`. |

---

### `FreeCell`

Переработанная ячейка малого чанка, доступная для повторного использования. Заголовок `FreeCell` **накладывается** поверх ранее хранившихся данных, не занимая дополнительной памяти.

| Поле | Назначение |
|---|---|
| `next` | Следующая ячейка в free-списке. |
| `zeroField` *(не-деструкторы)* | `0` = свободная ячейка, `1` = ручное управление памятью, иначе — указатель на `PNimType`. Используется legacy GC для определения живости ячейки. |
| `alignment` *(ARC/ORC)* | Выравнивающий отступ для совместимости разметки. |

---

### `LLChunk` (низкоуровневый чанк)

Бамп-аллокатор, используемый *только* для служебных структур самого аллокатора (AVL-узлы, trunk-узлы, записи heapLinks). Каждый `LLChunk` — ровно одна страница. Он никогда не возвращается ОС в штатном режиме; `llDeallocAll` возвращает их все при завершении.

---

### `IntSet` / `Trunk`

Компактное битовое множество, отображающее индексы страниц в булевы значения. Реализовано как хеш-таблица узлов `Trunk`, каждый из которых покрывает 256-битное окно адресов. Используется в `chunkStarts` — множестве адресов страниц, начинающих чанк, — что обеспечивает запрос O(1): «Принадлежит ли указатель `p` какому-либо чанку?»

---

### `AvlNode` *(только не-ARC/ORC)*

Узлы AVL-дерева (`root`), отображающего адреса данных больших чанков на диапазоны их размеров. GC использует это дерево при сканировании внутренних указателей: по произвольному указателю внутрь выделения найти его базу.

---

### `HeapLinks`

Связный список массивов, каждый элемент которых хранит `(PBigChunk, размер)` — каждую область памяти, полученную от ОС. Используется `deallocOsPages` для возврата всей ОС-памяти при завершении программы или уничтожении региона.

---

## Основные внутренние функции

### `rawAlloc`
```
proc rawAlloc(a: var MemRegion, requestedSize: int, alignment: int = 0): pointer
```

Центральный рабочий механизм выделения памяти. Все публичные вызовы `alloc`/`alloc0` проходят через него.

**Что делает:**

1. Округляет `requestedSize` до кратного `MemAlign`.
2. **Путь малых выделений** (`size ≤ SmallChunkSize - overhead`, без нестандартного выравнивания): ищет активный чанк в `a.freeSmallChunks[s]`. Если чанк есть и в free-списке есть ячейка — берёт её оттуда. Если ячейки нет — продвигает аккумулятор. Если активного чанка нет вовсе — выделяет новую страницу через `getSmallChunk` и делает её активной.
3. **Путь больших/огромных выделений**: обрабатывает отложенные межпоточные освобождения (`sharedFreeListBigChunks`), затем вызывает `getBigChunk` или `getHugeChunk`. При нестандартном выравнивании вычисляет отступ, сохраняет выровненный указатель данных в `c.prev`.
4. В не-ARC/ORC сборках добавляет диапазон выделения в AVL-дерево (`a.root`) для сканирования внутренних указателей.
5. Обновляет `a.occ`.

**Возвращает** сырой указатель на используемую память. Всё служебное обрамление — ответственность вызывающего кода.

---

### `rawAlloc0`
```
proc rawAlloc0(a: var MemRegion, requestedSize: int): pointer
```

Тонкая обёртка над `rawAlloc`, обнуляющая возвращённую память через `zeroMem`. Используется как реализация `alloc0`.

---

### `rawDealloc`
```
proc rawDealloc(a: var MemRegion, p: pointer)
```

Центральный рабочий механизм освобождения памяти. Определяет, к какому чанку (малому или большому) относится `p`, и выбирает соответствующий путь.

**Путь малых чанков:**

- Если `c.owner == addr(a)` (мы владеем): уменьшает `a.occ`, опционально заполняет `0xFF` (отладочный режим) и добавляет ячейку в free-список *активного* чанка (головы `freeSmallChunks[s]`). Если ячейка принадлежит другому чанку — она отдаётся как иностранная. Если текущий чанк был исчерпан — он повторно активируется.
- Если чанк принадлежит другому потоку (ARC/ORC): добавляет ячейку в `a.sharedFreeLists[s]` для отложенной обработки.

**Путь больших чанков:**

- Если `c.owner == addr(a)`: вызывает `deallocBigChunk`, который удаляет чанк из AVL-дерева, затем — `freeBigChunk` (слияние + возврат в матрицу TLSF) или `freeHugeChunk` (прямое освобождение ОС).
- Если чанк принадлежит другому потоку (ARC/ORC): помещает его в `a.sharedFreeListBigChunks`.

---

### `alloc`
```
proc alloc(allocator: var MemRegion, size: Natural): pointer
```

Публичная функция выделения для legacy-GC сборок. Вызывает `rawAlloc(size + sizeof(FreeCell))`, ставит `zeroField = 1` в заголовке `FreeCell` (помечает ячейку как активную), затем возвращает указатель, сдвинутый за заголовок. Заголовок невидим вызывающему коду.

В сборках ARC/ORC — прямая передача в `rawAlloc` без заголовка.

---

### `alloc0`
```
proc alloc0(allocator: var MemRegion, size: Natural): pointer
```

Как `alloc`, но нулевая инициализация возвращаемой памяти.

---

### `dealloc`
```
proc dealloc(allocator: var MemRegion, p: pointer)
```

Публичная функция освобождения. В режиме legacy GC проверяет заголовок `FreeCell`, отматывает `p` на `sizeof(FreeCell)` назад и вызывает `rawDealloc`. В режиме ARC/ORC вызывает `rawDealloc` напрямую.

---

### `realloc`
```
proc realloc(allocator: var MemRegion, p: pointer, newsize: Natural): pointer
```

Перераспределяет блок по адресу `p` до `newsize` байт. Всегда выделяет новый блок, копирует `min(старый_размер, newsize)` байт, затем освобождает старый блок. Оптимизации изменения размера на месте нет. Если `newsize == 0` и `p != nil` — эквивалентно `dealloc`. Если `p == nil` — эквивалентно `alloc`.

---

### `realloc0`
```
proc realloc0(allocator: var MemRegion, p: pointer, oldsize, newsize: Natural): pointer
```

Как `realloc`, но если `newsize > oldsize`, дополнительные байты (`[oldsize..newsize)`) явно обнуляются.

---

### `getBigChunk`
```
proc getBigChunk(a: var MemRegion, size: int): PBigChunk
```

Ищет или создаёт большой чанк размером не менее `size` байт по алгоритму TLSF:

1. Вызывает `mappingSearch(size, fl, sl)`, округляя `size` вверх до следующей границы ведра TLSF (кратной странице).
2. Вызывает `findSuitableBlock(a, fl, sl)` для поиска свободного блока по битовым картам.
3. Если блок не найден: вызывает `requestOsChunks` для получения свежей памяти у ОС с адаптивной стратегией — начинает с 4 страниц, удваивает запрос вплоть до `MaxBigChunkSize` в зависимости от текущей заполненности кучи.
4. Если найден блок крупнее нужного: вызывает `splitChunk`, возвращая остаток в матрицу.
5. Помечает чанк как «занятый» (устанавливает бит использования в `prevSize`).

---

### `freeBigChunk`
```
proc freeBigChunk(a: var MemRegion, c: PBigChunk)
```

Возвращает большой чанк в свободный пул TLSF. Реализует **слияние по граничным тегам**:

1. Сбрасывает бит использования в `c.prevSize`.
2. Если *левый* сосед свободен и не превышает `MaxBigChunkSize`: удаляет его из матрицы и сливает с `c`.
3. Если *правый* сосед свободен и не превышает `MaxBigChunkSize`: удаляет его из матрицы и сливает с `c`.
4. Вызывает `addChunkToMatrix` для вставки (возможно расширенного) чанка обратно.

---

### `getHugeChunk` / `freeHugeChunk`
```
proc getHugeChunk(a: var MemRegion; size: int): PBigChunk
proc freeHugeChunk(a: var MemRegion; c: PBigChunk)
```

Для выделений больше `MaxBigChunkSize`. Каждый вызов идёт напрямую к ОС (`allocPages` / `osDeallocPages`). Пулинга и слияния нет.

---

### `getSmallChunk`
```
proc getSmallChunk(a: var MemRegion): PSmallChunk
```

Получает свежий `SmallChunk`, выделяя ровно одну страницу через `getBigChunk(a, PageSize)`. Страница затем переинтерпретируется как `SmallChunk` — никакие данные не перемещаются.

---

### `requestOsChunks`
```
proc requestOsChunks(a: var MemRegion, size: int): PBigChunk
```

Запрашивает память у ОС через `osAllocPages` / `osTryAllocPages`. Реализует адаптивную стратегию роста: начинает консервативно с 4 страниц (16 КБ), затем удваивает запрос при каждом вызове вплоть до `MaxBigChunkSize`, масштабируя по текущей занятости кучи. При невозможности удовлетворить увеличенный запрос — откатывается к точному `size`. Записывает выделение в `a.heapLinks`.

---

### `splitChunk` / `splitChunk2`
```
proc splitChunk(a: var MemRegion, c: PBigChunk, size: int)
proc splitChunk2(a: var MemRegion, c: PBigChunk, size: int): PBigChunk
```

Отрезает префикс `size` байт от чанка `c`, оставляя остаток как самостоятельный чанк. `splitChunk2` возвращает остаток без вставки в матрицу; `splitChunk` дополнительно вызывает `addChunkToMatrix` для остатка.

После разбиения обновляется поле `prevSize` следующего соседнего чанка.

---

### `deallocBigChunk`
```
proc deallocBigChunk(a: var MemRegion, c: PBigChunk)
```

Выполняет служебные операции перед освобождением большого чанка:

- Уменьшает `a.occ`.
- В не-ARC/ORC сборках удаляет диапазон адресов чанка из AVL-дерева.
- Сбрасывает `c.prev` (перезаписанный выровненным указателем данных на время жизни чанка).
- Направляет вызов в `freeHugeChunk` или `freeBigChunk` по размеру.

---

### `deallocOsPages`
```
proc deallocOsPages(a: var MemRegion)
```

Возвращает **все** ОС-страницы, удерживаемые регионом, обратно ОС. Вызывается при завершении программы или уничтожении потоковой кучи. Итерирует список `heapLinks` и вызывает `osDeallocPages` для каждой записи, затем вызывает `llDeallocAll` для освобождения страниц бамп-аллокатора.

---

### `llAlloc`
```
proc llAlloc(a: var MemRegion, size: int): pointer
```

Частный бамп-аллокатор для метаданных самого аллокатора (AVL-узлы, trunk-узлы, записи heapLinks). Выделяет из `a.llmem`; при исчерпании запрашивает новую страницу у ОС. Память, выделенная здесь, **никогда не освобождается поштучно** — вся она освобождается разом через `llDeallocAll`.

---

### `llDeallocAll`
```
proc llDeallocAll(a: var MemRegion)
```

Обходит цепочку `llmem` и вызывает `osDeallocPages` для каждой страницы, использованной под внутренние метаданные. Вызывается при уничтожении региона.

---

### `compensateCounters` *(только ARC/ORC)*
```
proc compensateCounters(a: var MemRegion; c: PSmallChunk; size: int)
```

Когда ячейки из `sharedFreeLists` поглощаются чанком (`fetchSharedCells`), эта функция корректирует счётчик `free` чанка и `a.occ`, учитывая восстановленную ёмкость. Также увеличивает `c.foreignCells` для каждой ячейки из другого чанка.

---

### `freeDeferredObjects` *(только ARC/ORC)*
```
proc freeDeferredObjects(a: var MemRegion; root: PBigChunk)
```

Обрабатывает очередь `sharedFreeListBigChunks`, вызывая `deallocBigChunk` для каждой записи. Ограничена `MaxSteps = 20` итерациями за вызов, чтобы не создавать непредсказуемых задержек; оставшиеся элементы повторно ставятся в очередь.

---

### `addToSharedFreeList` / `addToSharedFreeListBigChunks` *(только ARC/ORC)*
```
proc addToSharedFreeList(c: PSmallChunk; f: ptr FreeCell; size: int)
proc addToSharedFreeListBigChunks(a: var MemRegion; c: PBigChunk)
```

Атомарные бесблокировочные операции добавления в начало списка — помещают освобождённую ячейку или большой чанк в очередь отложенного освобождения потока-владельца. При поддержке потоков используют compare-and-swap.

---

## Операции с битовыми картами TLSF

### `msbit` / `lsbit`
```
proc msbit(x: uint32): int
proc lsbit(x: uint32): int
```

Находят позицию старшего и младшего установленного бита в 32-битном слове соответственно. `msbit` использует таблицу `fsLookupTable` (256 элементов) для побайтовой обработки. `lsbit` выводится через `x & (-x)`.

Используются для отображения запрошенного размера в индексы матрицы TLSF (`fl`, `sl`) и для поиска непустого ведра.

---

### `mappingSearch`
```
proc mappingSearch(r, fl, sl: var int)
```

Для запрошенного размера `r` округляет его **вверх** до ближайшей границы ведра TLSF (выровненной по странице), затем вычисляет индекс первого уровня `fl` и второго уровня `sl` в матрице. Это округление «при поиске» гарантирует, что любой найденный блок не меньше исходного запроса.

---

### `mappingInsert`
```
proc mappingInsert(r: int): tuple[fl, sl: int]
```

Для точного размера чанка `r` вычисляет координаты `fl`/`sl` для *вставки* в матрицу без округления. Чанк попадает в ведро, точно соответствующее его размеру.

---

### `findSuitableBlock`
```
proc findSuitableBlock(a: MemRegion; fl, sl: var int): PBigChunk
```

Сканирует битовые карты TLSF для поиска **наименьшего свободного блока** нужного или большего размера. Сначала проверяет точное ведро `(fl, sl)`; если пусто — через bit-scan находит ближайшее непустое ведро выше. Обновляет `fl` и `sl` на найденное ведро.

---

### `addChunkToMatrix` / `removeChunkFromMatrix`
```
proc addChunkToMatrix(a: var MemRegion; b: PBigChunk)
proc removeChunkFromMatrix(a: var MemRegion; b: PBigChunk)
proc removeChunkFromMatrix2(a: var MemRegion; b: PBigChunk; fl, sl: int)
```

Управляют двусвязными списками свободных больших чанков в матрице TLSF. `addChunkToMatrix` добавляет чанк в начало списка нужного ведра `(fl, sl)` и устанавливает биты в `slBitmap`/`flBitmap`. `removeChunkFromMatrix` вырезает чанк из любого места списка, при необходимости сбрасывая биты.

---

## Операции с IntSet (отслеживание чанков)

### `intSetGet` / `intSetPut`
```
proc intSetGet(t: IntSet, key: int): PTrunk
proc intSetPut(a: var MemRegion, t: var IntSet, key: int): PTrunk
```

Поиск и вставка в хеш-таблицу множества `IntSet`. Ключи — индексы страниц, делённые на `TrunkShift`; каждый `Trunk` покрывает `IntsPerTrunk * IntSize` страниц. `intSetPut` выделяет новые узлы `Trunk` через `llAlloc` при первом использовании ведра.

---

### `contains` / `incl` / `excl`
```
proc contains(s: IntSet, key: int): bool
proc incl(a: var MemRegion, s: var IntSet, key: int)
proc excl(s: var IntSet, key: int)
```

Получение/установка/сброс бита в `IntSet`. Каждая операция сначала находит узел `Trunk` через хеш, затем вычисляет битовый индекс внутри массива `bits` трункового узла.

---

## Функции статистики памяти

### `getFreeMem` / `getTotalMem` / `getOccupiedMem`
```
proc getFreeMem(a: MemRegion): int
proc getTotalMem(a: MemRegion): int
proc getOccupiedMem(a: MemRegion): int
```

Прямые аксессоры к полям учёта `MemRegion`:

- **`getFreeMem`** — `a.freeMem`: байты ОС-памяти, не занятые ни одним выделенным чанком (свободные большие чанки в матрице TLSF).
- **`getTotalMem`** — `a.currMem`: суммарно получено от ОС.
- **`getOccupiedMem`** — `a.occ`: байты, переданные вызывающему коду и ещё не освобождённые.

---

### `getMaxMem`
```
proc getMaxMem(a: var MemRegion): int
```

Возвращает **пиковое** ОС-потребление памяти с момента создания региона: `max(a.currMem, a.maxMem)`. `maxMem` обновляется при уменьшении `currMem` (возврате страниц ОС), обеспечивая точный водяной знак.

---

## Шаблон инициализации по региону

### `instantiateForRegion`
```
template instantiateForRegion(allocator: untyped)
```

«Грязный» шаблон, привязывающий весь публичный интерфейс аллокатора к конкретной переменной `MemRegion`. Генерирует следующие свободно-стоящие процедуры для вызывающего модуля:

| Генерируемая процедура | Делегирует |
|---|---|
| `allocImpl(size)` | `alloc(allocator, size)` |
| `alloc0Impl(size)` | `alloc0(allocator, size)` |
| `deallocImpl(p)` | `dealloc(allocator, p)` |
| `reallocImpl(p, newSize)` | `realloc(allocator, p, newSize)` |
| `realloc0Impl(p, oldSize, newSize)` | `realloc0(allocator, p, newSize)` + обнуление |
| `allocSharedImpl(size)` | Аллокация общей кучи (с блокировкой при потоках) |
| `allocShared0Impl(size)` | Аллокация общей кучи с обнулением |
| `deallocSharedImpl(p)` | Освобождение общей кучи |
| `reallocSharedImpl(p, newSize)` | Реаллокация общей кучи |
| `reallocShared0Impl(p, oldSize, newSize)` | Реаллокация общей кучи с обнулением |
| `getFreeMem()` | `allocator.freeMem` |
| `getTotalMem()` | `allocator.currMem` |
| `getOccupiedMem()` | `allocator.occ` |
| `getMaxMem*()` | `getMaxMem(allocator)` |

В многопоточной сборке без ARC/ORC функции `allocShared*`/`deallocShared*` захватывают/освобождают `heapLock` вокруг отдельного `sharedHeap: MemRegion`.

---

## Утилиты проверки указателей *(только не-ARC/ORC)*

### `isAllocatedPtr`
```
proc isAllocatedPtr(a: MemRegion, p: pointer): bool
```

Возвращает `true`, если `p` указывает на *выделенную и живую* ячейку. Проверяет доступность `p` (индекс страницы есть в `chunkStarts`), что чанк не свободен, и что `FreeCell.zeroField > 1` (активный объект GC). Используется GC в assert-проверках.

---

### `interiorAllocatedPtr`
```
proc interiorAllocatedPtr(a: MemRegion, p: pointer): pointer
```

Для указателя `p`, который может указывать в *любое место* внутри выделения (не обязательно на его начало), возвращает **базовый указатель** этого выделения или `nil`, если `p` не попадает ни в одно живое выделение.

Для малых чанков: вычисляет базу ячейки из размера ячейки чанка и смещения `p` внутри страницы.

Для больших чанков: использует `c.prev` (хранящий выровненный указатель данных на время жизни чанка).

Для указателей вне любого известного чанка: обращается к AVL-дереву (`a.root`) через `inRange` — дерево отображает диапазоны адресов на базовые указатели, обеспечивая O(log n) поиск по внутреннему указателю.

---

### `prepareForInteriorPointerChecking`
```
proc prepareForInteriorPointerChecking(a: var MemRegion)
```

Кэширует `lowGauge(a.root)` и `highGauge(a.root)` в `a.minLargeObj`/`a.maxLargeObj` перед циклом GC. Эта проверка диапазона отфильтровывает 99,96%+ запросов по внутренним указателям без спуска по AVL-дереву.

---

### `isAccessible`
```
proc isAccessible(a: MemRegion, p: pointer): bool
```

Возвращает `true`, если страница, содержащая указатель `p`, отслеживается в `a.chunkStarts` — то есть `p` попадает в память, управляемую данным регионом. Это первый фильтр, применяемый перед любыми более глубокими проверками указателей.

---

## Вспомогательные функции для списков чанков

### `listAdd` / `listRemove`
```
proc listAdd[T](head: var T, c: T)
proc listRemove[T](head: var T, c: T)
```

Обобщённые операции над встроенными двусвязными списками. `listAdd` добавляет `c` в начало `head`; `listRemove` вырезает `c`. Обе содержат отладочные проверки через `sysAssert`.

---

### `updatePrevSize`
```
proc updatePrevSize(a: var MemRegion, c: PBigChunk, prevSize: int)
```

После изменения размера или разбиения чанка обновляет поле `prevSize` следующего соседнего чанка в виртуальном адресном пространстве. Это механизм граничных тегов — он позволяет коду слияния в `freeBigChunk` находить и объединять соседние свободные чанки без отдельной структуры данных.

---

## Итерация объектов GC

### `allObjects` *(итератор)*
```
iterator allObjects(m: var MemRegion): pointer
```

Выдаёт сырой указатель на каждый текущий выделенный объект в регионе. Итерирует `m.chunkStarts`, затем для каждого активного чанка:

- **Малый чанк**: шагает от `data` до `data + acc` с шагом `c.size`, выдавая адрес каждой ячейки.
- **Большой чанк**: выдаёт `c.prev` (выровненный указатель данных, сохранённый при выделении).

На время итерации устанавливает `m.locked = true`. Используется в фазах маркировки GC.

---

### `iterToProc`
```
proc iterToProc*(iter: typed, envType: typedesc; procName: untyped)
```

Компиляторная «магия» (`{.magic: "Plugin", compileTime.}`), преобразующая замыкающий итератор в процедуру в стиле коллбэка. Используется внутренне для адаптации `allObjects` под форму, ожидаемую C-интерфейсом GC.

---

## Вспомогательные запросы о чанках

### `ptrSize`
```
proc ptrSize(p: pointer): int
```

Возвращает используемый размер в байтах выделения по адресу `p`. Для малых чанков: `c.size - sizeof(FreeCell)` (legacy GC) или `c.size` (ARC/ORC). Для больших чанков вычитается `bigChunkOverhead()`.

Используется в `realloc` для определения числа копируемых байт.

---

### `isSmallChunk`
```
proc isSmallChunk(c: PChunk): bool
```

Возвращает `true`, если `c.size ≤ SmallChunkSize - smallChunkOverhead()`. Порог — ровно одна страница минус заголовок структуры `SmallChunk`.

---

### `chunkUnused`
```
proc chunkUnused(c: PChunk): bool
```

Возвращает `true`, если бит 0 поля `c.prevSize` равен 0 — чанк свободен (не выделен в данный момент). Двойное использование `prevSize` одновременно для размера предыдущего чанка и бита занятости — классический приём граничных тегов.

---

### `pageIndex` / `pageAddr`
```
proc pageIndex(c: PChunk): int
proc pageIndex(p: pointer): int
proc pageAddr(p: pointer): PChunk
```

Конвертируют между значениями указателей и индексами страниц (сдвиг вправо на `PageShift`). `pageAddr` округляет указатель *вниз* до границы его страницы, возвращая заголовок `PChunk` для любого указателя на этой странице. Постоянно используются по всему аллокатору для перехода между «пользовательским указателем» и «его чанком».

---

## Сводка по потокобезопасности

| Операция | Потокобезопасность |
|---|---|
| Выделение из потоко-локального `MemRegion` | Безопасно — блокировки не нужны |
| Освобождение в потоко-локальный `MemRegion` | Безопасно — бесблокировочный быстрый путь при `c.owner == addr(a)` |
| Освобождение в *чужой* `MemRegion` (ARC/ORC) | Бесблокировочный атомарный push в `sharedFreeLists`/`sharedFreeListBigChunks` |
| Освобождение в *чужой* `MemRegion` (legacy GC) | Не поддерживается для малых чанков; для больших — поддерживается |
| `allocShared` / `deallocShared` (без ARC/ORC) | Защищено `heapLock` |
| `allocShared` / `deallocShared` (ARC/ORC) | Делегирует в потоко-локальный путь (нет разграничения общей кучи) |
| Итерация `allObjects` | Блокирует регион (`m.locked = true`); нельзя выделять/освобождать в процессе |

---
