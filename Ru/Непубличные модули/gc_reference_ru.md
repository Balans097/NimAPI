# gc.nim — Полный справочник

> **Модуль**: `system/gc` — стандартная библиотека Nim  
> **Автор**: Andreas Rumpf  
> **Алгоритм**: Отложенный подсчёт ссылок + обнаружение циклов методом «пометить и замести» (Mark & Sweep)  
> **Область применения**: Это *внутренний* модуль среды выполнения. Большинство функций здесь вызываются исключительно генератором кода компилятора Nim (`compilerproc`) или самой библиотекой (`rtl`). Небольшое число функций, помеченных `*` (экспортируемых звёздочкой), образуют **публичный API GC**, доступный обычным Nim-программам.

---

## Как работает GC — Мысленная модель

Прежде чем разбирать отдельные функции, полезно понять три взаимодействующих механизма, реализованных в этом файле.

### 1. Отложенный подсчёт ссылок (нормальный путь)

Каждый объект, размещённый на куче (строка, последовательность, значение `ref`, замыкание), называется **ячейкой**. Каждая ячейка несёт скрытый заголовок, содержащий счётчик ссылок и указатель на дескриптор типа объекта. Счётчик ссылок хранится сдвинутым на 3 бита влево, чтобы три младших бита можно было использовать для *цвета*, применяемого при обнаружении циклов.

Когда Nim-код присваивает, передаёт или отпускает ссылку, компилятор генерирует вызовы `incRef`/`decRef`. Если счётчик падает до нуля, ячейка добавляется в **ZCT** (таблицу нулевого счёта) для отложенной сборки — она не освобождается немедленно, потому что объект всё ещё может быть достижим через стековую переменную (ссылки на стеке намеренно *не засчитываются* для повышения производительности).

### 2. Сборка ZCT (быстрый путь)

`collectZCT` опустошает ZCT. Для каждого кандидата проверяется, равен ли реальный счётчик ссылок нулю. Если да — вызывается финализатор объекта (если есть), уменьшаются счётчики ссылок всех дочерних объектов (что может добавить их в ZCT), затем физически освобождается память. Весь процесс представляет собой итеративное «глубокое освобождение» без рекурсии.

### 3. Обнаружение циклов методом Mark & Sweep (медленный путь)

Один лишь подсчёт ссылок не может освободить *циклы ссылок* (A → B → A). Когда занятая память кучи превышает `cycleThreshold`, выполняется `collectCycles`: проход «пометить и замести», который помечает каждую ячейку, достижимую со стека, из регистров и глобальных корней, а затем заметает всё непомеченное.

### Структура заголовка ячейки

```
 ┌──────────────────────────────────────────────────┐
 │  Заголовок ячейки (скрытый, по отрицательному   │
 │  смещению от пользовательского указателя)        │
 │  ┌─────────────────────────────────┬──────────┐  │
 │  │  refcount (биты 3…63) │ цвет (биты 0..2)  │  │
 │  └─────────────────────────────────┴──────────┘  │
 │  typ: PNimType                                   │
 └──────────────────────────────────────────────────┘
 ┌──────────────────────────────────────────────────┐
 │  Пользовательские данные  ← указатель для Nim   │
 └──────────────────────────────────────────────────┘
```

Битовые поля цвета кодируют роль ячейки при обнаружении циклов:

| Константа | Биты | Значение |
|-----------|------|----------|
| `rcBlack` | `000` | Используется или свободна — не является кандидатом |
| `rcGray` | `001` | Возможный член цикла |
| `rcWhite` | `010` | Подтверждённый член мусорного цикла |
| `rcPurple` | `011` | Возможный корень цикла |

## Важные константы

| Константа | Значение | Значение |
|-----------|----------|----------|
| `CycleIncrease` | `2` | Мультипликативный коэффициент роста порогов после каждой сборки |
| `InitialCycleThreshold` | `4 МБ` (или `high(int)` при `-d:nimCycleBreaker`) | Уровень занятой памяти, при котором запускается сборка циклов |
| `InitialZctThreshold` | `500` | Количество записей в ZCT, при котором запускается сборка |
| `rcIncrement` | `0b1000` (8) | Шаг, добавляемый/вычитаемый из сырого поля `refcount` на одну ссылку |
| `rcShift` | `3` | Количество бит сдвига вправо для получения целочисленного счётчика ссылок |
| `ZctFlag` | `0b100` | Бит, установленный в `refcount`, пока ячейка находится в ZCT |

## Ключевые внутренние типы

### `GcStat`

Запись статистики, накапливаемой в течение всего времени работы программы.

| Поле | Тип | Описание |
|------|-----|----------|
| `stackScans` | `int` | Сколько раз сканировался C-стек |
| `cycleCollections` | `int` | Сколько полных проходов mark-and-sweep выполнено |
| `maxThreshold` | `int` | Наибольшее значение, которое когда-либо достигал `cycleThreshold` |
| `maxStackSize` | `int` | Наибольший наблюдавшийся размер C-стека |
| `maxStackCells` | `int` | Наибольший `decStack`, наблюдавшийся в ходе сборки |
| `cycleTableSize` | `int` | Пиковый размер таблицы обнаружения циклов |
| `maxPause` | `int64` | Наибольшая наблюдавшаяся пауза GC в наносекундах |

### `GcStack`

Описывает один C-стек (обычно он один; сопрограммы могут иметь несколько).

| Поле | Доступно при | Описание |
|------|-------------|----------|
| `bottom` | всегда | Указатель на нижний используемый адрес стека |
| `pos` | `withRealTime` или `nimCoroutines` | Текущий указатель стека, используемый для ограничения глубины сканирования |
| `bottomSaved` | `withRealTime` | Сохранённый `bottom` во время вызова `GC_step` с нестандартным `stackSize` |
| `prev`, `next` | `nimCoroutines` | Ссылки двусвязного списка для стеков сопрограмм |
| `maxStackSize` | `nimCoroutines` | Максимальный размер стека на сопрограмму |

### `GcHeap`

Центральное состояние GC, хранимое как переменная-per-thread (`gch`).

| Поле | Описание |
|------|----------|
| `stack` | Дескриптор текущего C-стека |
| `cycleThreshold` | Высоководный уровень занятой памяти, при котором запускается сборка циклов |
| `zctThreshold` | Длина ZCT, при которой запускается сборка ZCT |
| `zct` | Таблица нулевого счёта — ячейки, чей счётчик ссылок только что достиг нуля |
| `decStack` | Ячейки, найденные на стеке, счётчик которых нужно уменьшить после сканирования |
| `tempStack` | Вспомогательный стек для устранения рекурсии при пометке |
| `recGcLock` | Счётчик повторного входа; если > 0 — GC отключён (`GC_disable`) |
| `maxPause` | Максимальное время паузы (в наносекундах) в режиме реального времени |
| `region` | Нижележащая область памяти (аллокатор-арена) |
| `stat` | Накопленная статистика |
| `marked` | Набор ячеек, построенный в ходе mark-and-sweep |
| `additionalRoots` | Дополнительные корни GC, зарегистрированные через `GC_ref` / `nimGCref` |
| `toDispose` | Потокобезопасный список утилизации (в многопоточных сборках) |
| `gcThreadId` | Индекс потока, которому принадлежит эта куча (начиная с нуля) |

### `WalkOp`

Перечисление, используемое в качестве второго аргумента каждой функции, обходящей дочерние ячейки.

| Значение | Назначение |
|----------|------------|
| `waMarkGlobal` | Пометить ячейку в ходе глобального (резервного) прохода mark-and-sweep |
| `waMarkPrecise` | Поместить ячейку в `tempStack` в ходе точного прохода пометки |
| `waZctDecRef` | Уменьшить счётчик дочернего объекта при его освобождении в ZCT-сборке |
| `waPush` | Поместить дочернюю ячейку в `tempStack` без дальнейших действий |

### `Finalizer`

```nim
Finalizer = proc (self: pointer) {.nimcall, gcsafe, raises: [].}
```

Коллбэк, хранящийся в дескрипторе типа. Вызывается функцией `prepareDealloc` непосредственно перед физическим освобождением памяти ячейки, пока все дочерние ссылки ещё действительны.

## Публичный API (экспортируется через `*`)

Это единственные функции, которые предполагается вызывать из обычного кода Nim-приложений.

---

### `GC_disable()`

**Увеличивает** счётчик повторного входа `gch.recGcLock`. Пока этот счётчик больше нуля, GC не будет собирать мусор, даже если пороги превышены. Вызовы можно вкладывать друг в друга; каждый `GC_disable` должен соответствовать ровно одному `GC_enable`.

**Когда использовать**: вокруг критических секций, где нужна нулевая активность аллокатора (например, манипуляции с lock-free-структурами данных, измерение производительности, вызов C-кода, не дружественного к GC).

```nim
GC_disable()
# ... критическая секция; сборка мусора здесь не будет происходить ...
GC_enable()
```

> **Предупреждение**: если забыть вызвать `GC_enable`, память будет накапливаться бесконечно, так как ничто никогда не освобождается.

---

### `GC_enable()`

**Уменьшает** `gch.recGcLock`. Если счётчик уже был равен нулю и `nimDoesntTrackDefects` не определён, вызывает `AssertionDefect`, чтобы поймать несбалансированные пары disable/enable.

```nim
GC_disable()
try:
  riskyOperation()
finally:
  GC_enable()  # всегда включаем обратно, даже при исключении
```

---

### `GC_fullCollect()`

Принудительно выполняет **немедленную полную сборку мусора**:
1. Временно устанавливает `cycleThreshold = 0`, чтобы `collectCT` обязательно запустил обнаружение циклов.
2. Вызывает `collectCT`, который в свою очередь опустошает ZCT и запускает `collectCycles`.
3. Восстанавливает исходный порог.

Используйте, когда знаете, что только что освободили большой граф объектов и хотите вернуть память *немедленно*, не дожидаясь следующего автоматического срабатывания.

```nim
# Освобождаем большое дерево
bigTree = nil
# Немедленно возвращаем память
GC_fullCollect()
echo getOccupiedMem()  # должно быть значительно меньше
```

---

### `GC_enableMarkAndSweep()`

Сбрасывает `gch.cycleThreshold` обратно до `InitialCycleThreshold` (4 МБ), повторно включая сборку циклов после её подавления. Не имеет эффекта, если сборка циклов никогда не была отключена.

```nim
GC_disableMarkAndSweep()
# ... работа, которая не может позволить себе паузу сборщика циклов ...
GC_enableMarkAndSweep()
```

---

### `GC_disableMarkAndSweep()`

Устанавливает `gch.cycleThreshold = high(int) - 1`, делая условие запуска сборки циклов никогда не выполнимым. Подсчёт ссылок продолжает работать; подавляется только фаза mark-and-sweep, разрывающая циклы.

**Когда использовать**: в секциях, чувствительных к задержкам, или когда известно, что граф объектов не содержит циклов.

```nim
GC_disableMarkAndSweep()
# Чистые древовидные структуры — циклы невозможны
buildHugeTree()
GC_enableMarkAndSweep()
GC_fullCollect()
```

---

### `GC_getStatistics(): string`

Возвращает читаемую многострочную строку с текущим состоянием GC и накопленной статистикой. Полезна для профилирования производительности и отладки проблем с памятью.

```nim
echo GC_getStatistics()
```

**Пример вывода**:
```
[GC] total memory: 8388608
[GC] occupied memory: 102400
[GC] stack scans: 14
[GC] stack cells: 3
[GC] cycle collections: 1
[GC] max threshold: 4194304
[GC] zct capacity: 64
[GC] max cycle table size: 0
[GC] max pause time [ms]: 0
[GC] max stack size: 16384
```

Каждая строка начинается с `[GC]`, что позволяет фильтровать её из обычного вывода программы.

---

### `GC_setStrategy(strategy: GC_Strategy)`

Заглушка, не выполняющая никаких действий, сохранённая для обратной совместимости. Старый GC Nim поддерживал несколько стратегий; текущая реализация игнорирует эту настройку.

```nim
GC_setStrategy(gcThroughput)  // молча игнорируется
```

---

### `GC_setMaxPause(MaxPauseInUs: int)`

*Доступна только при компиляции с `-d:useRealtimeGC`.*

Устанавливает **максимально допустимую паузу GC** в микросекундах. При установленном лимите `collectZCT` прерывает свою работу и возвращает `false`, если прошедшее время превысило лимит, позволяя вызывающему потоку соблюдать мягкие дедлайны реального времени.

```nim
GC_setMaxPause(500)  // GC не может приостанавливаться более чем на 500 мкс
```

После установки лимита паузы регулярно вызывайте `GC_step` из основного цикла, чтобы распределить работу GC по кадрам.

---

### `GC_step(us: int, strongAdvice = false, stackSize = -1)`

*Доступна только при компиляции с `-d:useRealtimeGC`.*

Просит GC выполнить не более `us` микросекунд работы за этот вызов. GC решает, запускаться ли на самом деле, опираясь на внутренние пороги, — если только `strongAdvice = true` не принудит к сборке независимо от состояния.

`stackSize` позволяет сообщить GC точную глубину текущего стека в байтах. Это полезно в фреймворках с сопрограммами, где размер C-стека заранее известен. Передайте `-1`, чтобы использовать полный стек.

```nim
while true:
  handleEvents()
  GC_step(1000)   // тратить не более 1 мс на GC за кадр
  renderFrame()
```

---

### `GC_collectZct()`

*Экспериментально и нестабильно — не использовать в production-коде.*

Напрямую вызывает `collectCTBody` — основную процедуру опустошения ZCT и опциональной сборки циклов — без проверки каких-либо порогов. Предназначена для тестирования внутренностей GC.

---

### `gcInvariant()`

*Экспортируется для отладочных инструментов.*

Проверяет, что внутренний инвариант аллокатора выполняется (через `sysAssert(allocInv(...))`). При включённом `traceGC` дополнительно запускает `markForDebug`. Используется тестовым набором Nim для верификации корректности кучи между операциями.

```nim
gcInvariant()  // упадёт с отладочным сообщением, если куча повреждена
```

## Функции для компилятора (`compilerproc`)

Эти функции не вызываются пользовательским кодом напрямую — Nim-компилятор генерирует вызовы к ним в своём выходном коде. Они задокументированы здесь, поскольку понимание их работы помогает при чтении сгенерированного C-кода или диагностике тонких ошибок памяти.

---

### `nimGCref(p: pointer)`

**Аналог `GC_ref` для сырых указателей.** Увеличивает счётчик ссылок ячейки по адресу `p` и добавляет её в `gch.additionalRoots`, чтобы фаза mark-and-sweep рассматривала её как корень даже при отсутствии типизированной Nim-ссылки. Высокоуровневая процедура `GC_ref` из стандартной библиотеки вызывает именно это.

```nim
// Nim-код:
var x: ref MyObj = new(MyObj)
GC_ref(x)  // внутренне вызывает nimGCref
```

---

### `nimGCunref(p: pointer)`

**Аналог `GC_unref` для сырых указателей.** Удаляет `p` из `gch.additionalRoots` (линейный поиск) и уменьшает его счётчик ссылок. Парная функция к `nimGCref`.

---

### `nimGCunrefNoCycle(p: pointer)`

Встраиваемая версия `decRef` с проверками инварианта аллокатора. Вызывается компилятором, когда он может доказать на этапе компиляции, что освобождаемый объект не может быть частью цикла ссылок (например, строка или простой нерекурсивный `ref`). Позволяет избежать накладных расходов полного пути `nimGCunref`.

---

### `nimGCunrefRC1(p: pointer)`

Ещё более тонкая обёртка над `decRef`. Используется в ситуациях, когда проверки инварианта аллокатора не нужны (например, в ходе определённых последовательностей финализации).

---

### `asgnRef(dest: PPointer, src: pointer)`

**Стандартное присваивание ссылки.** Вызывается компилятором при присваивании ссылки на кучу в размещённое на куче место (`dest` заведомо *не* на стеке):

1. Если `src != nil`: увеличить счётчик `src`.
2. Если `dest[]` не nil: уменьшить счётчик старого значения.
3. Записать `src` в `*dest`.

Порядок «сначала incRef, потом decRef» критически важен: если оба указателя указывают на один и тот же объект (самоприсваивание), уменьшение первым вызвало бы преждевременное освобождение.

```c
// Сгенерированный C для: node.child = newChild
asgnRef(&node->child, newChild);
```

---

### `asgnRefNoCycle(dest: PPointer, src: pointer)`

Устаревший псевдоним `asgnRef`, сохранённый для совместимости со старым сгенерированным кодом. На практике обе функции идентичны.

---

### `unsureAsgnRef(dest: PPointer, src: pointer)`

**Присваивание ссылки с неопределённым местоположением.** Вызывается, когда компилятор не может определить на этапе компиляции, находится ли `dest` на стеке или на куче (классический случай — параметр `var ref T`). Проверяет `isOnStack(dest)` во время выполнения:

- Если `dest` **на стеке**: пропустить все обновления счётчиков (ссылки на стеке не засчитываются), просто записать `src` в `*dest`.
- Если `dest` **на куче**: выполнить полную логику `asgnRef`.

Чуть дороже `asgnRef` из-за проверки стека, но на типичных платформах с непрерывным стеком проверка — это всего два сравнения.

```c
// Сгенерированный C для: setRef(&r) где r может быть на стеке или куче
unsureAsgnRef(&r, newObj(TNode_TI, sizeof(TNode)));
```

---

### `newObj(typ: PNimType, size: int): pointer`

Выделяет `size` байт обнулённой памяти для объекта типа `typ`, регистрирует его в ZCT со счётчиком 0 и возвращает указатель на область пользовательских данных (т.е. область *после* скрытого заголовка Cell). Компилятор генерирует этот вызов для каждого выражения `new(T)`.

---

### `newObjNoInit(typ: PNimType, size: int): pointer`

Аналог `newObj`, но без вызова `zeroMem`. Используется компилятором, когда он может доказать, что каждое поле будет записано до его чтения (например, в литералах агрегатной конструкции).

---

### `newObjRC1(typ: PNimType, size: int): pointer`

Аналог `newObj`, но инициализирует счётчик ссылок значением **1** вместо 0. Суффикс «RC1» сигнализирует, что результирующий указатель уже владеет одной ссылкой, поэтому компилятор может пропустить первый вызов `incRef` в точке первого присваивания.

---

### `newSeq(typ: PNimType, len: int): pointer`

Выделяет последовательность (динамический массив) из `len` элементов. Вычисляет общий размер как `GenericSeqSize + len × размер_элемента`, вызывает `newObj`, затем устанавливает `len` и `reserved` в заголовке последовательности.

---

### `newSeqRC1(typ: PNimType, len: int): pointer`

Аналог `newSeq`, но предварительно устанавливает счётчик ссылок в 1 (вызывает `newObjRC1`). Используется в оптимизированных путях присваивания.

---

### `growObj(old: pointer, newsize: int): pointer`

Расширяет существующую последовательность или строку до `newsize` байт. Выделяет новую ячейку, копирует существующее содержимое, заполняет нулями новый хвост, обнуляет поле `len` в заголовке старой последовательности (чтобы любая оставшаяся ссылка на `old` видела пустую последовательность) и возвращает пользовательский указатель новой ячейки.

Вызывается компилятором, когда `add`, `setLen` или `&` вызывают перераспределение памяти.

---

### `nimGCvisit(d: pointer, op: int)`

Универсальная точка входа для посещения дочерних ячеек, используемая функциями-маркерами GC. Компилятор генерирует для каждого типа процедуру `marker`, которая вызывает `nimGCvisit` для каждого трассируемого поля. Аргумент `op` — это `WalkOp`, приведённый к `int`, указывающий действие (поместить во временный стек, уменьшить счётчик и т.д.).

---

### `extGetCellType(c: pointer): PNimType`

Возвращает дескриптор типа объекта по адресу `c` времени выполнения. Используется инструментами отладки и профилирования, которым нужно идентифицировать тип объекта по сырому указателю.

---

### `internRefcount(p: pointer): int`

Возвращает текущее количество ссылок на объект по адресу `p`. Экспортируется в C как `getRefcount`. Доступна для диагностических инструментов; не использовать в логике приложения (значение нестабильно между точками GC).

```nim
// Только для низкоуровневой диагностики:
echo internRefcount(addr someObj)
```

---

## Внутренние вспомогательные функции

Эти функции не экспортируются и не вызываются пользовательским кодом, но их понимание необходимо для чтения или модификации самого GC.

---

### `cellToUsr(cell: PCell): pointer`

Преобразует внутренний указатель `PCell` (указывающий на скрытый заголовок) в пользовательский указатель (на сами данные объекта). Это просто `cell + sizeof(Cell)`.

### `usrToCell(usr: pointer): PCell`

Обратная операция к `cellToUsr`. Преобразует пользовательский указатель обратно в заголовок: `usr - sizeof(Cell)`. Каждая функция компилятора, получающая `pointer` из Nim-кода, начинает именно с этого вызова.

### `incRef(c: PCell)`

Добавляет `rcIncrement` (8) к `c.refcount`. Встраиваемая; без проверки кучи в release-сборках.

### `decRef(c: PCell)`

Вычитает `rcIncrement` из `c.refcount`. Если результат падает ниже `rcIncrement` (т.е. реальный счётчик теперь равен нулю), добавляет ячейку в ZCT через `rtlAddZCT`.

### `addZCT(s, c)`

Добавляет ячейку `c` в последовательность ZCT `s`, предварительно установив бит `ZctFlag`. Флаг предотвращает двойную вставку.

### `isOnStack(p: pointer): bool`

Проверяет, попадает ли указатель `p` в границы текущего C-стека. На платформах с непрерывным стеком — это два сравнения. Возвращает `true`, если `p` — стековый адрес.

### `rawNewObj(typ, size, gch)`

Самый низкоуровневый аллокатор: резервирует `size + sizeof(Cell)` байт из `gch.region`, устанавливает заголовок Cell со счётчиком = `ZctFlag` (нулевой счёт, помечен для ZCT) и добавляет ячейку в ZCT. Все публичные процедуры аллокации (`newObj`, `newObjRC1` и т.д.) вызывают именно это.

### `collectZCT(gch): bool`

Опустошает ZCT до полного завершения или (в режиме реального времени) до исчерпания бюджета паузы. Возвращает `true`, если ZCT полностью опустошён, `false` — если прерван.

### `collectCycles(gch)`

Запускает полный сборщик циклов mark-and-sweep: сначала опустошает ZCT (для получения чистых цветов), затем помечает всё достижимое со стека, регистров и глобальных корней, после чего заметает всё непомеченное.

### `collectCT(gch)`

Главный шлюз сборки мусора. Определяет, выполнены ли условия для запуска сборки (размер ZCT, занятая память или флаг отладки `alwaysGC`), и при необходимости вызывает `collectCTBody`. Также адаптивно обновляет `zctThreshold` после каждой сборки.

### `collectCTBody(gch)`

Непосредственное тело одного цикла сборки:
1. Фиксирует максимальный размер стека.
2. Вызывает `markStackAndRegisters` для сканирования C-стека и регистров CPU.
3. Вызывает `collectZCT`.
4. Если ZCT полностью опустошён и память превышает `cycleThreshold`, вызывает `collectCycles`.
5. Вызывает `unmarkStackAndRegisters` для уменьшения счётчиков, повышенных при сканировании.
6. В режиме реального времени — фиксирует длительность как образец времени паузы.

### `markStackAndRegisters(gch)`

Обходит каждое слово C-стека и регистров CPU, вызывая `gcMark` для каждого. Помечена атрибутом `CLANG_NO_SANITIZE_ADDRESS`, чтобы подавить ложные срабатывания ASAN, вызванные консервативным сканированием произвольных слов памяти.

### `gcMark(gch, p)`

Вызывается для каждого потенциального указателя, найденного на стеке. Проверяет, похож ли `p` на корректный адрес кучи (больше `PageSize`), затем использует `interiorAllocatedPtr` для поиска начала ячейки, содержащей `p` (поддержка внутренних указателей). Если ячейка найдена — увеличивает её счётчик ссылок и записывает в `decStack`, чтобы это увеличение было отменено после сканирования.

### `unmarkStackAndRegisters(gch)`

Уменьшает счётчик каждой ячейки из `decStack` (временно увеличенный при сканировании стека), затем очищает `decStack`.

### `markS(gch, c)`

Помечает одну ячейку `c` и все её транзитивные потомки, итеративно извлекая их из `gch.tempStack`. Избегает C-рекурсии, используя `tempStack` как явную очередь обработки.

### `markGlobals(gch)`

Вызывает все зарегистрированные глобальные и поточно-локальные маркерные функции, затем помечает все `additionalRoots`. Вызывается в `collectCycles`, чтобы глобально достижимые ячейки не были заметены.

### `sweep(gch)`

Перебирает каждый объект в `gch.region`. Для каждой ячейки, отсутствующей в `gch.marked` (и не находящейся в данный момент в ZCT), вызывает `freeCyclicCell` для запуска финализатора и освобождения памяти.

### `freeCyclicCell(gch, c)`

Выполняет `prepareDealloc(c)` (финализатор), затем физически освобождает `c` через `rawDealloc`.

### `forAllChildren(cell, op)`

Применяет `op` ко всем GC-трассируемым потомкам ячейки `cell`. Использует функцию `marker` типа при её наличии (генерируется компилятором, быстрый путь), иначе прибегает к рефлексивному пути `forAllChildrenAux`.

### `forAllChildrenAux(dest, mt, op)`

Рефлексивный обходчик потомков. Интерпретирует дескриптор `PNimType` для нахождения трассируемых полей в объектах, кортежах и массивах рекурсивно.

### `forAllSlotsAux(dest, n, op)`

Обходит дерево `TNimNode` дескриптора типа для посещения каждого слота в `object` или `tuple`, обрабатывая узлы `nkSlot`, `nkList` и `nkCase` (вариантные объекты).

### `doOperation(p, op)`

Диспетчерский узел для `WalkOp`. Вызывается по одному разу для каждого дочернего указателя при обходе:
- `waZctDecRef` → `decRef(c)`
- `waPush` / `waMarkPrecise` → `add(gch.tempStack, c)`
- `waMarkGlobal` → `markS(gch, c)`

### `addNewObjToZCT(res, gch)`

Оптимизированная вставка в ZCT для свежевыделенных объектов. Сканирует последние 8 записей ZCT в поисках слота, который уже может быть переиспользован (если `refcount ≥ rcIncrement` — значит, объект «повышен» и слот можно занять). Этот кэш-дружественный поиск успешен примерно в 63% случаев и позволяет избежать лишних добавлений в ZCT.

### `initGC()`

Сбрасывает всё состояние GC к начальным значениям. Вызывается один раз при запуске программы (и по одному разу на каждый поток). Устанавливает пороги, обнуляет статистику, инициализирует все внутренние последовательности и набор ячеек.

### `cellsetReset(s)`

Очищает `CellSet`, вызывая `deinit`, а затем `init`.

## Флаги условной компиляции

Поведение gc.nim управляется несколькими флагами времени компиляции:

| Флаг | Эффект |
|------|--------|
| `-d:useRealtimeGC` | Включает `GC_setMaxPause` и `GC_step` для инкрементальной, ограниченной по времени сборки |
| `-d:nimCycleBreaker` | Устанавливает `InitialCycleThreshold = high(int)`, фактически отключая автоматическую сборку циклов при запуске |
| `-d:logGC` | Включает подробное логирование по каждой ячейке в stderr через `writeCell` |
| `-d:useGcAssert` | Включает проверки `gcAssert` во время выполнения (дорого; для отладки самого GC) |
| `-d:memProfiler` | Вызывает `nimProfile` после каждой аллокации для записи размеров |
| `--threads:on` | Включает разделяемый список `toDispose` для утилизации между потоками |
| `--define:nimCoroutines` | Добавляет поддержку нескольких стеков (по одному на сопрограмму) |
| `--define:traceGC` | Включает трассировку переходов состояний для каждой ячейки |
| `--define:leakDetector` | Записывает имя исходного файла и номер строки в заголовок каждой ячейки |

## Краткий справочник: какая функция вызывается и когда

| Действие в Nim-коде | Вызываемая функция |
|---------------------|-------------------|
| `new(T)` | `newObj` или `newObjRC1` |
| `@[...]` / `newSeq` | `newSeq` или `newSeqRC1` |
| `a = b` (оба на куче) | `asgnRef` |
| `setRef(r)` с `var ref` | `unsureAsgnRef` |
| `add(s, x)` вызывает перераспределение | `growObj` |
| выход из области видимости / последняя ссылка отпущена | `decRef` → добавление в ZCT |
| размер ZCT ≥ порог | `collectZCT` |
| куча ≥ порог циклов | `collectCycles` |
| `GC_fullCollect()` | `collectCT` с порогом = 0 |
| `GC_ref(x)` | `nimGCref` |
| `GC_unref(x)` | `nimGCunref` |
