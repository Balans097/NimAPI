# `dyncalls.nim` — справочник модуля

> **Часть стандартной библиотеки времени выполнения Nim**  
> Внутренний модуль рантайма — пользовательский код его не импортирует.  
> Вдохновлён кодом загрузки динамических библиотек из Lua.

---

## Общее описание

`dyncalls.nim` реализует **интерфейс загрузки динамических библиотек** в Nim — механизм рантайма, позволяющий программам загружать разделяемые библиотеки (`.so`, `.dll`, `.dylib`) и вызывать из них функции во время выполнения, без компоновки с ними на этапе сборки.

Это фундамент под прагмой `dynlib` в Nim. Когда вы пишете:

```nim
proc myFunc(x: int): int {.importc, dynlib: "mylib.so".}
```

…компилятор генерирует код, вызывающий `nimLoadLibrary`, `nimGetProcAddr` и в итоге `nimUnloadLibrary` из этого модуля.

### Платформенная диспетчеризация

Модуль использует ветви `when` времени компиляции для выбора корректной реализации под каждую цель:

| Платформа | Механизм | Используемый C API |
|---|---|---|
| POSIX (Linux, macOS, BSD, Solaris, AIX, HPUX…) | интерфейс `dlfcn` | `dlopen`, `dlsym`, `dlclose`, `dlerror` |
| Windows / DOS | WinAPI | `LoadLibraryA`, `GetProcAddress`, `FreeLibrary` |
| Genode | Заглушка — вызывает `AssertionDefect` | — |
| Nintendo Switch / FreeRTOS / Zephyr / NuttX | Заглушка — пишет в stderr и завершается | — |
| Всё остальное | Ошибка компиляции | — |

Три публичных процедуры (`nimLoadLibrary`, `nimUnloadLibrary`, `nimGetProcAddr`) имеют **одинаковую сигнатуру на всех платформах**. Все платформенные различия скрыты внутри них.

### Ограничения дизайна

Весь модуль компилируется с `{.push stack_trace: off.}`. Это гарантирует, что данные низкоуровневые функции не появляются в трассировках стека Nim — они работают ниже уровня, где трассировка имеет смысл, а сама трассировка потребовала бы аллокаций, которые здесь могут быть небезопасны.

Две процедуры вывода ошибок также написаны с полным **отказом от выделения памяти в куче**. Причина задокументирована в комментариях к коду: в момент, когда библиотека не может быть загружена или символ не найден, программа находится в неопределённом состоянии, и обращение к аллокатору может привести к дедлоку, повреждению состояния или маскировке реальной ошибки. Вывод выполняется через `rawWrite` напрямую в `cstderr`.

## Константы

### `NilLibHandle`

```nim
const NilLibHandle: LibHandle = nil
```

Типизированная nil-константа для `LibHandle`. Используется как сигнальное значение, означающее «ни одна библиотека не загружена». Не экспортируется, но используется внутри как каноническое «пустое» значение дескриптора — дескриптор загруженной библиотеки всегда отличен от nil.

## Процедуры вывода ошибок

### `nimLoadLibraryError`

```nim
proc nimLoadLibraryError(path: string)
```

Вызывается при сбое `nimLoadLibrary` (возврат nil). Выводит диагностическое сообщение в `cstderr` и вызывает `rawQuit(1)` — программа завершается немедленно и безусловно.

Реализация тщательно выстроена, чтобы **не выделять памяти**, поскольку сбой загрузки библиотеки может означать серьёзно повреждённую кучу или среду. Весь строковый вывод выполняется через `rawWrite` напрямую.

Сообщение и поведение различаются по платформам:

**POSIX без `-d:nimDebugDlOpen`:**
```
could not load: /path/to/lib.so
(compile with -d:nimDebugDlOpen for more information)
```

**POSIX с `-d:nimDebugDlOpen`:**  
Строка ошибки от `dlerror()` выводится прямо в ходе вызова `nimLoadLibrary` (до вызова `nimLoadLibraryError`), поэтому пользователь видит полную причину ошибки на уровне ОС.

**Windows — `ERROR_BAD_EXE_FORMAT`:**
```
could not load: mylib.dll
(bad format; library may be wrong architecture)
```
Эта ошибка возникает при попытке загрузить 32-битную DLL из 64-битного процесса (или наоборот) — очень распространённая ошибка.

**GUI-приложения Windows (`-d:guiapp`):**  
Поскольку GUI-приложения не имеют видимой консоли, ошибка отображается как диалоговое окно Windows через `MessageBoxA`. Сообщение формируется в фиксированном стековом буфере на 1000 символов (снова — без аллокации в куче) перед отображением.

```nim
# Вызывается внутри примерно так:
let lib = nimLoadLibrary("mylib.so")
if lib == nil:
  nimLoadLibraryError("mylib.so")  # не возвращается
```

### `procAddrError`

```nim
proc procAddrError(name: cstring) {.compilerproc, nonReloadable, hcrInline.}
```

Вызывается, когда `nimGetProcAddr` не может найти именованный символ в загруженной библиотеке. Выводит диагностику и вызывает `rawQuit(1)`.

Вывод:
```
could not import: my_function_name
```

Прагмы несут конкретный смысл:
- **`{.compilerproc.}`** — компилятор знает эту процедуру по имени и может вызывать её непосредственно из генерируемого кода, минуя обычную диспетчеризацию.
- **`{.nonReloadable.}`** — эта процедура не должна перезагружаться при горячей перезагрузке кода (HCR). Поскольку она сама является частью механизма динамической загрузки, её перезагрузка была бы циклической и опасной.
- **`{.hcrInline.}`** — подсказка системе HCR встроить этот вызов, избегая косвенности через таблицу перезагрузки.

Как и `nimLoadLibraryError`, эта процедура написана без выделения памяти в куче.

```nim
// Вызывается внутри примерно так:
let fn = nimGetProcAddr(lib, "my_func")
// если fn == nil после всех попыток, вызывается procAddrError("my_func")
```

## Основные процедуры динамической загрузки

Эти три процедуры образуют полный публичный интерфейс `dyncalls.nim`. Они имеют единообразные сигнатуры на всех платформах.

---

### `nimLoadLibrary`

```nim
proc nimLoadLibrary(path: string): LibHandle
```

Загружает разделяемую библиотеку по пути `path` и возвращает непрозрачный дескриптор к ней. При сбое возвращает `nil` (вызывающий код должен затем вызвать `nimLoadLibraryError` с этим путём).

Дескриптор — это непрозрачный указатель (`LibHandle`). Его конкретный тип (`THINSTANCE` на Windows, сырой указатель на POSIX) скрыт от вызывающего кода.

**Реализация для POSIX:**

Вызывает `dlopen(path, flags)`, где `flags`:
- `RTLD_NOW` по умолчанию — все символы библиотеки разрешаются немедленно при загрузке, а не лениво. Это означает, что `dlopen` немедленно завершится ошибкой при неразрешимом символе, вместо того чтобы упасть при первом вызове.
- `RTLD_NOW | RTLD_GLOBAL` при компиляции с `-d:globalSymbols` — символы библиотеки также становятся доступными для всех последующих загружаемых библиотек. Полезно для систем плагинов, где разделяемые библиотеки обращаются друг к другу.

С `-d:nimDebugDlOpen` любое сообщение `dlerror()` немедленно выводится в `cstderr`, давая детальную диагностику на уровне ОС (например, о недостающих транзитивных зависимостях или несовместимых версиях библиотек).

На Linux и macOS `RTLD_NOW` является константой времени компиляции (значение `2`); на других POSIX-системах оно импортируется из `<dlfcn.h>`, поскольку значение может различаться.

**Реализация для Windows:**

Вызывает `LoadLibraryA(path)` — ANSI (8-битную) версию загрузчика библиотек Windows. Суффикс `A` означает, что функция принимает путь `char*`, а не широкосимвольный `wchar_t*`. Результат приводится из `THINSTANCE` к `LibHandle`.

```nim
let lib = nimLoadLibrary("libssl.so.1.1")
if lib == nil:
  nimLoadLibraryError("libssl.so.1.1")
# lib теперь — валидный дескриптор
```

### `nimUnloadLibrary`

```nim
proc nimUnloadLibrary(lib: LibHandle)
```

Выгружает ранее загруженную библиотеку, освобождая ресурсы ОС. После этого вызова любые указатели на функции, полученные из `lib` через `nimGetProcAddr`, становятся недействительными и не должны использоваться — это неопределённое поведение.

**Реализация для POSIX:** Вызывает `dlclose(lib)`. ОС уменьшает счётчик ссылок на библиотеку; физическая выгрузка происходит только при достижении нуля (другой код может удерживать ссылки на ту же библиотеку).

**Реализация для Windows:** Вызывает `FreeLibrary(cast[THINSTANCE](lib))`. Та же семантика подсчёта ссылок, что и на POSIX.

**Genode:** Вызывает `raiseAssert` — динамическая выгрузка не поддерживается и вызовет `AssertionDefect`.

**Nintendo Switch / FreeRTOS / Zephyr / NuttX:** Выводит сообщение об ошибке в `cstderr` и вызывает `rawQuit(1)` — динамические операции с библиотеками полностью не поддерживаются на этих платформах.

```nim
nimUnloadLibrary(lib)
// lib теперь недействителен — не используйте никакие указатели функций из него
```

---

### `nimGetProcAddr`

```nim
proc nimGetProcAddr(lib: LibHandle, name: cstring): ProcAddr
```

Ищет экспортированный символ `name` в загруженной библиотеке `lib` и возвращает указатель на функцию. Если символ не найден, вызывает `procAddrError(name)`, завершающую программу — эта процедура **никогда не возвращает nil**.

`ProcAddr` — непрозрачный тип указателя на функцию. Вызывающий код сам отвечает за приведение к правильной сигнатуре функции перед вызовом.

**Реализация для POSIX:**

Вызывает `dlsym(lib, name)`. Если `dlsym` возвращает nil — символ однозначно отсутствует, и `procAddrError` вызывается немедленно.

**Реализация для Windows — откат к декорированным именам:**

Соглашение Windows ABI для функций `__stdcall` (используемых во многих Win32 API) декорирует имена символов: добавляет ведущее подчёркивание и суффикс `@N`, где N — суммарный размер аргументов в байтах (например, `_MyFunc@8` для функции, принимающей 8 байт аргументов). Если обычный вызов `GetProcAddress(lib, name)` не находит символ, код систематически перебирает декорированные имена:

```
_name@0
_name@4
_name@8
_name@12
...
_name@200   (50 попыток, шаг 4 байта до 200)
```

Этот перебор гарантирует корректную работу `nimGetProcAddr` для функций `__stdcall`, даже когда точный размер аргументов неизвестен в месте вызова — типичная ситуация при загрузке системных DLL Windows из Nim.

Декорированное имя собирается в фиксированный стековый буфер из 250 символов (снова — без аллокации в куче).

```nim
type MyFunc = proc(x: int): int {.cdecl.}

let fn = cast[MyFunc](nimGetProcAddr(lib, "my_function"))
let result = fn(42)
```

## Платформенные заглушки

### Genode

```nim
proc nimUnloadLibrary(lib: LibHandle) = raiseAssert("nimUnloadLibrary not implemented")
proc nimLoadLibrary(path: string): LibHandle = raiseAssert("nimLoadLibrary not implemented")
proc nimGetProcAddr(lib: LibHandle, name: cstring): ProcAddr = raiseAssert("nimGetProcAddr not implemented")
```

Genode — это микроядерная ОС на основе возможностей (capability-based) со строгой моделью ресурсов. Загрузка динамических библиотек является привилегированной операцией, не соответствующей программной модели Genode. Любая попытка использовать `dynlib` на Genode вызывает `AssertionDefect` во время выполнения. Код, который должен работать на Genode, обязан компоноваться со всеми библиотеками статически.

### Nintendo Switch / FreeRTOS / Zephyr / NuttX

Эти встроенные/игровые платформы не имеют файловых загрузчиков разделяемых библиотек. Реализации-заглушки выводят сообщение об ошибке в stderr и завершаются:

```
nimLoadLibrary not implemented
```

Обратите внимание: в исходном коде есть опечатка — сообщение об ошибке для `nimGetProcAddr` гласит `"nimGetProAddr not implemented"` (пропущена буква 'c'). Это известная косметическая ошибка в исходнике.

## Флаги времени компиляции

| Флаг | Эффект |
|---|---|
| `-d:nimDebugDlOpen` | На POSIX: выводить вывод `dlerror()` в stderr после каждого вызова `dlopen`, даже при успехе. Незаменим для диагностики сбоев «could not load», вызванных недостающими транзитивными зависимостями. |
| `-d:globalSymbols` | На POSIX: открывать библиотеки с `RTLD_NOW \| RTLD_GLOBAL` вместо только `RTLD_NOW`. Делает символы загруженной библиотеки видимыми для всех последующих загружаемых библиотек — необходимо для систем плагинов, разделяющих код. |
| `-d:guiapp` | На Windows: отображать ошибки загрузки как диалоговое окно `MessageBoxA` вместо вывода в stderr (который невидим в GUI-приложениях). |

## Внутренние POSIX C-импорты

Импортируются из `<dlfcn.h>` и используются только внутри POSIX-ветви. Не экспортируются.

| Функция | Назначение |
|---|---|
| `dlopen(path, mode)` | Загрузить разделяемую библиотеку; возвращает дескриптор или nil |
| `dlsym(lib, name)` | Найти символ по имени; возвращает указатель на функцию или nil |
| `dlclose(lib)` | Уменьшить счётчик ссылок и выгрузить при достижении нуля |
| `dlerror()` | Вернуть человекочитаемую строку с описанием последней ошибки `dl*` |

`RTLD_NOW` — либо константа времени компиляции (Linux/macOS: `2`), либо импортируется из `<dlfcn.h>` на других POSIX-платформах.

---

## Внутренние Windows C-импорты

Импортируются из `<windows.h>` и используются только внутри Windows-ветви. Не экспортируются.

| Функция | Назначение |
|---|---|
| `LoadLibraryA(path)` | Загрузить DLL по пути; возвращает `HINSTANCE` или nil |
| `GetProcAddress(lib, name)` | Найти экспортированный символ; возвращает указатель на функцию или nil |
| `FreeLibrary(lib)` | Уменьшить счётчик ссылок и выгрузить DLL при достижении нуля |
| `GetLastError()` | Получить последний код ошибки Win32 |
| `MessageBoxA(hwnd, text, title, flags)` | Показать модальное диалоговое окно (только для GUI-приложений) |

`THINSTANCE` оборачивает тип Windows `HINSTANCE`. В режиме C++ это структура с полем `pointer` (соответствующее определение C++); в режиме C — просто `pointer`.
