# `excpt.nim` — Справочник по модулю

> **Модуль:** `excpt` — Стандартная библиотека Nim, ядро обработки исключений  
> **Стабильность:** Внутренний / частично публичный. Символы, отмеченные `*`, экспортированы. Многие из них помечены `compilerRtl`, то есть вызываются прежде всего кодогенератором Nim, а не прикладным кодом. Наиболее полезные для разработчиков приложений символы: `onUnhandledException`, `errorMessageWriter`, `getStackTraceEntries` и `stackTraceAvailable`.

---

## Обзор

`excpt.nim` — это сердце механизма обработки исключений в рантайме Nim. Его задачи:

- Ведение на каждый поток **стека вызовов** (call-frame stack), используемого для формирования трассировки стека.
- Ведение на каждый поток **цепочки текущих исключений**.
- Вывод форматированных трассировок в stderr (или пользовательский хук).
- Возбуждение, повторное возбуждение и передача исключений ближайшему обработчику.
- Перехват **сигналов ОС** (SIGSEGV, SIGFPE и др.) и преобразование их в читаемые сообщения об ошибках.

Модуль написан с особой осторожностью в отношении выделения памяти: он должен работать даже тогда, когда куча недоступна (например, в маленьких встраиваемых программах, которые никогда не выделяют память). Многие подпрограммы не требуют кучи совсем.

## Экспортируемые переменные

---

### `errorMessageWriter`

```nim
var errorMessageWriter*: (proc(msg: string) {.tags: [WriteIOEffect], gcsafe, nimcall, raises: [].})
```

**Что делает:**  
Заменяемый callback, который получает каждое диагностическое сообщение рантайма (трассировки стека, отчёты о сбоях, сообщения о сигналах). Когда переменная равна `nil`, рантайм пишет сообщения напрямую в `stderr`.

**Зачем устанавливать:**  
Чтобы перенаправить вывод краш-отчётов в лог-файл, диалоговое окно GUI, сетевой сокет или тестовый стенд, перехватывающий вывод.

**Важно:** Сигнатура callback строгая — он должен быть `gcsafe`, не принимать дополнительных аргументов и никогда не выбрасывать исключений, поскольку может вызываться из обработчиков сигналов и GC-небезопасных контекстов.

```nim
# Перенаправить все сообщения рантайма в файл
var logFile = open("crash.log", fmWrite)

errorMessageWriter = proc(msg: string) =
  logFile.write(msg)
  logFile.flushFile()

# Теперь любое необработанное исключение или краш по сигналу пишет в crash.log
```

---

### `onUnhandledException`

```nim
var onUnhandledException*: (proc (errorMsg: string) {.nimcall, gcsafe.})
```

**Что делает:**  
Вызывается рантаймом, когда исключение распространяется до самого верха потока и не находит ни одного обработчика. Строка `errorMsg` уже полностью отформатирована: содержит трассировку стека, сообщение исключения и имя его типа.

Если переменная равна `nil` (по умолчанию), рантайм пишет `errorMsg` в stderr и вызывает `quit(1)`.

**Зачем устанавливать:**  
- Собственные crash-репортеры (отправка отчёта об ошибке на сервер).
- GUI-приложения, которые должны показывать диалог вместо вывода в stderr.
- Тестовые фреймворки, перехватывающие необработанные исключения.

**Стабильность:** Нестабильный API — может измениться между версиями Nim.

```nim
onUnhandledException = proc(errorMsg: string) =
  # Показать всплывающее окно вместо вывода в консоль
  showDialog("Критическая ошибка", errorMsg)
  quit(1)
```

```nim
# Сохранить текст краша в тестах
var lastCrash = ""
onUnhandledException = proc(errorMsg: string) =
  lastCrash = errorMsg
  quit(1)
```

---

### `frameMsgBuf` *(условный)*

```nim
when NimStackTraceMsgs:
  var frameMsgBuf* {.threadvar.}: string
```

**Что делает:**  
Потокозависимый строковый буфер, который накапливает отладочные сообщения для каждого стекового фрейма, когда активен флаг компилятора `-d:nimStackTraceMsgs`. Каждый фрейм может добавить контекстный текст (например, значения переменных), который появится прямо в трассировке стека при сбое.

**Когда существует:** Только при компиляции с флагом `NimStackTraceMsgs` (т.е. `-d:nimStackTraceMsgs`).

```nim
# Работает только при компиляции с -d:nimStackTraceMsgs
frameMsgBuf.add("userId=" & $userId & " ")
```

## Функции состояния фреймов

Эти функции сохраняют и восстанавливают полное контекстное состояние исключений/фреймов для текущего потока. Используются компилятором при реализации корутин, async/await и замыканий — конструкций, которые требуют приостановки и возобновления контекста выполнения.

---

### `getFrameState`

```nim
proc getFrameState*(): FrameState {.compilerRtl, inl.}
```

**Что делает:**  
Делает снимок всего контекста обработки исключений текущего потока и возвращает его в виде кортежа `FrameState`. В зависимости от режима компиляции (`gotoBasedExceptions` или нет) снимок включает часть или все из следующих полей:

- `framePtr` — указатель на вершину стека фреймов вызовов.
- `currException` — текущее активное исключение (если есть).
- `excHandler` — ближайший активный обработчик блока `try`.
- `gcFramePtr` — указатель стека GC-фреймов.

**Аналогия:** Похоже на `setjmp` — делает «фотографию» состояния управления потоком в данный момент.

```nim
# Сохранить состояние перед входом в тело корутины
let savedState = getFrameState()
# ... выполнить другой код ...
setFrameState(savedState)  # восстановить при возврате
```

---

### `setFrameState`

```nim
proc setFrameState*(state: FrameState) {.compilerRtl, inl.}
```

**Что делает:**  
Обратная операция к `getFrameState`. Восстанавливает все потокозависимые указатели исключений/фреймов из ранее сохранённого снимка `FrameState`.

**Предупреждение:** Неправильное использование (например, восстановление состояния, которое ссылается на уже несуществующие стековые фреймы) приводит к неопределённому поведению. Существует для использования генерируемым кодом, не для прикладной логики.

```nim
let state = getFrameState()
doSomeWork()
setFrameState(state)  // отменить изменения фреймов/исключений, сделанные в doSomeWork
```

---

### `getFrame`

```nim
proc getFrame*(): PFrame {.compilerRtl, inl.}
```

**Что делает:**  
Возвращает `PFrame` — указатель на вершину связанного списка фреймов вызовов текущего потока. Каждый фрейм хранит имя процедуры, имя исходного файла и номер строки, которые появляются в трассировке стека.

**Типичное применение:** Программная инспекция или обход трассировки стека во время выполнения без возбуждения исключения.

```nim
let topFrame = getFrame()
if topFrame != nil:
  echo "Сейчас в: ", topFrame.procname
  echo "  в ", topFrame.filename, ":", topFrame.line
```

---

### `setFrame`

```nim
proc setFrame*(s: PFrame) {.compilerRtl, inl.}
```

**Что делает:**  
Напрямую перезаписывает потокозависимый `framePtr` заданным указателем фрейма `s`. Используется компилятором при восстановлении указателя фрейма после размотки стека на границах корутин и async-кода.

**Осторожно:** Как и `setFrameState`, это низкоуровневая операция. Установка неверного указателя фрейма разрушает трассировки стека.

```nim
// Паттерн, генерируемый компилятором при возобновлении async:
let savedFrame = getFrame()
setFrame(resumptionFrame)
runResumedCode()
setFrame(savedFrame)
```

## Функции GC-фреймов *(только без `gotoBasedExceptions`)*

Четыре функции управляют вторичным связанным списком **GC-фреймов** — структур, сообщающих сборщику мусора, какие ссылки на кучу живы в текущем стеке вызовов. Существуют только когда `gotoBasedExceptions` равен `false` (традиционная модель исключений на основе `setjmp`).

---

### `getGcFrame`

```nim
proc getGcFrame*(): GcFrame {.compilerRtl, inl.}
```

Возвращает указатель на вершину стека GC-фреймов. Используется при сохранении/восстановлении GC-корней через границы корутин или исключений.

---

### `popGcFrame`

```nim
proc popGcFrame*() {.compilerRtl, inl.}
```

Убирает верхний GC-фрейм из стека (эквивалентно `gcFramePtr = gcFramePtr.prev`). Вызывается кодом, генерируемым компилятором при возврате из процедуры.

---

### `setGcFrame`

```nim
proc setGcFrame*(s: GcFrame) {.compilerRtl, inl.}
```

Напрямую устанавливает указатель стека GC-фреймов в `s`. Используется при возобновлении корутины для восстановления видимости GC-корней.

---

### `pushGcFrame`

```nim
proc pushGcFrame*(s: GcFrame) {.compilerRtl, inl.}
```

**Что делает:**  
Помещает новый GC-фрейм на стек. Перед связыванием функция **обнуляет все слоты указателей** внутри фрейма (память сразу после `GcFrameHeader`). Это обнуление критично: оно гарантирует, что GC никогда не увидит устаревших/мусорных указателей, если цикл GC произойдёт до того, как эти слоты будут нормально инициализированы.

```nim
# Код, генерируемый компилятором в начале любой процедуры с локальными ref-переменными:
var myGcFrame: MyGcFrameType
pushGcFrame(addr myGcFrame)
# ... тело процедуры ...
popGcFrame()
```

## Функции трассировки стека

---

### `stackTraceAvailable`

```nim
proc stackTraceAvailable*(): bool
```

**Что делает:**  
Возвращает `true`, если текущая сборка способна формировать полноценные трассировки стека. Результат зависит от флагов компиляции:

| Условие | Результат |
|---|---|
| Скомпилировано с `NimStackTrace` (debug-сборки, по умолчанию) | `true`, если `framePtr != nil` |
| Скомпилировано с `-d:nimStackTraceOverride` | всегда `true` |
| Скомпилировано с `-d:nativeStackTrace` на Linux/macOS | `true` |
| Release-сборки без вышеперечисленного | `false` |

**Почему это важно:** В production (release) сборках трассировки стека обычно отключены ради производительности. Вызов `getStackTraceEntries()` в такой сборке вернёт пустую последовательность. `stackTraceAvailable` позволяет проверить это до попытки сбора данных.

```nim
if stackTraceAvailable():
  let entries = getStackTraceEntries()
  for e in entries:
    echo e.filename, ":", e.line, " в ", e.procname
else:
  echo "(трассировки стека недоступны в этой сборке)"
```

---

### `getStackTraceEntries` *(для исключения)*

```nim
proc getStackTraceEntries*(e: ref Exception): lent seq[StackTraceEntry]
```

**Что делает:**  
Возвращает трассировку стека, захваченную **в момент возбуждения исключения `e`**, в виде последовательности записей `StackTraceEntry`. Каждая запись содержит:

- `procname` — имя процедуры на данном уровне стека.
- `filename` — исходный файл.
- `line` — номер строки.
- `frameMsg` — необязательное сообщение фрейма (при активном `NimStackTraceMsgs`).

Тип возврата — `lent`, то есть заимствованная ссылка внутрь объекта исключения; копия не создаётся.

**Ключевое:** Трассировка зафиксирована в момент `raise`, а не в момент вызова `getStackTraceEntries`. Поэтому функцию можно вызывать из блока `finally` или из подпрограммы формирования отчётов и всё равно получить оригинальное место возбуждения исключения.

```nim
try:
  riskyOperation()
except IOError as e:
  let entries = getStackTraceEntries(e)
  for entry in entries:
    echo entry.filename, "(", entry.line, ") ", entry.procname
```

```nim
# Структурированное логирование: прикрепить трассировку к записи лога
proc logException(e: ref Exception) =
  var record = newLogRecord()
  record.message = e.msg
  record.trace = getStackTraceEntries(e)  # зафиксированная трассировка
  submitLog(record)
```

---

### `getStackTraceEntries` *(текущий стек)*

```nim
proc getStackTraceEntries*(): seq[StackTraceEntry]
```

**Что делает:**  
Захватывает трассировку стека **прямо сейчас** — в точке вызова этой функции — и возвращает её как новую `seq[StackTraceEntry]`. Это вариант без исключения: вам не нужно находиться внутри обработчика исключений.

**Отличие от перегрузки с исключением:** Данная функция создаёт новую `seq` и захватывает живой стек вызовов. Перегрузка с исключением возвращает уже существующую (зафиксированную) трассировку, хранящуюся внутри объекта исключения.

**Примечание:** Возвращает пустую последовательность, когда `stackTraceAvailable()` равно `false`.

```nim
// Логировать текущее местоположение без возбуждения исключения
proc checkpoint(label: string) =
  let trace = getStackTraceEntries()
  debugLog(label, trace)

proc processOrder(id: int) =
  checkpoint("processOrder start")
  # ...
```

```nim
// Убедиться, что функция вызвана из ожидаемого места
proc mustBeCalledFrom(expectedProc: string) =
  let trace = getStackTraceEntries()
  let found = trace.anyIt(it.procname == expectedProc)
  assert found, "Ожидался вызов из " & expectedProc
```

## Платформенные варианты

- **Windows GUI-приложения** (`-d:guiapp`): `writeToStdErr` заменяется на `MessageBoxA`, и сообщения о сбоях появляются в виде диалоговых окон.
- **Genode**: Использует `echo` через сессию LOG Genode, так как stderr недоступен.
- **C++ backend** (`--cc:cpp`): Распространение исключений использует нативный C++ `throw`/`catch`. Устанавливается `std::set_terminate` для обработки неперехваченных C++-исключений.
- **`gotoBasedExceptions`**: Альтернативная модель исключений (используется в ARC/ORC), заменяющая `setjmp`/`longjmp` явными флагами ошибок и `goto` — функции GC-фреймов и `excHandler` в этом режиме не существуют.
- **JS backend**: `getStackTraceEntries` задокументирован как ещё не поддерживаемый.

## Краткая шпаргалка

| Символ | Вид | Назначение |
|---|---|---|
| `errorMessageWriter` | `var` | Перенаправить вывод краш-сообщений |
| `onUnhandledException` | `var` | Обработчик необработанных исключений |
| `frameMsgBuf` | `var` | Отладочные сообщения фреймов (debug-сборки) |
| `getFrameState` / `setFrameState` | proc | Сохранить/восстановить полный контекст исключений |
| `getFrame` / `setFrame` | proc | Получить/установить указатель текущего фрейма |
| `getGcFrame` / `setGcFrame` / `pushGcFrame` / `popGcFrame` | proc | Управление GC-фреймами (не-goto режим) |
| `stackTraceAvailable` | proc | Проверить, доступны ли трассировки в этой сборке |
| `getStackTraceEntries(e)` | proc | Получить зафиксированную трассировку из исключения |
| `getStackTraceEntries()` | proc | Захватить живую трассировку прямо сейчас |
