# `osalloc.nim` — справочник модуля

> **Часть стандартной библиотеки времени выполнения Nim**  
> Компилируется с `{.push raises: [], gcsafe.}`

---

## Общее описание

`osalloc.nim` — это **низкоуровневый, платформозависимый модуль выделения памяти**, расположенный в самом основании менеджера памяти Nim. Его единственная задача — запрашивать у **операционной системы** большие, выровненные по границам страниц блоки «сырой» памяти и возвращать их обратно, когда они больше не нужны.

Модуль не управляет отдельными объектами. Он не хранит метаданные аллокаций. Он лишь служит мостом между высокоуровневым аллокатором Nim и конкретными механизмами ОС (`mmap`, `VirtualAlloc`, `malloc`, bump-pointer и т.д.).

Реализация выбирается на **этапе компиляции** через ветви `when`, ориентированные на следующие платформы:

| Платформа / Флаг | Механизм |
|---|---|
| `-d:nimAllocPagesViaMalloc` (только ARC/ORC) | `c_malloc` + ручное выравнивание по странице |
| Emscripten (WASM) | `mmap` со встроенным блоком метаданных |
| POSIX (Linux, macOS, BSD, Solaris, Haiku…) | `mmap` с флагами `MAP_ANONYMOUS | MAP_PRIVATE` |
| Windows | `VirtualAlloc` / `VirtualFree` |
| Standalone / без ОС | Bump-pointer аллокатор на статическом массиве |

Все три публичные процедуры имеют **одинаковый интерфейс** независимо от платформы — в этом и состоит ключевая цель дизайна модуля.

---

## Внутренний вспомогательный элемент: `roundup`

```nim
proc roundup(x, v: int): int {.inline.}
```

**Не экспортируется**, но является основой для всей работы модуля. Округляет целое число `x` **вверх** до ближайшего кратного числу `v`, где `v` обязательно должно быть степенью двойки. Используется внутри модуля для выравнивания размеров аллокаций по границам страниц памяти.

```nim
roundup(14, 4096)  # → 4096  (одна полная страница)
roundup(15, 8)     # → 16
roundup(65, 8)     # → 72
```

Реализация использует классический трюк с битовой маской: `(x + (v-1)) and not (v-1)`. Это работает без ветвлений и крайне быстро.

## Константа компиляции: `doNotUnmap`

```nim
const doNotUnmap = not (defined(amd64) or defined(i386)) or
                   defined(windows) or defined(nimAllocNoUnmap)
```

Эта константа определяет, будет ли `osDeallocPages` реально возвращать память операционной системе. Значение `true` (то есть **не** выполнять unmap) устанавливается, если:

- Архитектура **не** является x86/amd64. На некоторых платформах (например, Linux/PowerPC) `munmap` одной страницы освобождает весь блок — это ошибка ОС.
- ОС — **Windows**, потому что `MEM_RELEASE` требует параметр размера равный `0` и имеет особую семантику.
- Разработчик явно задал флаг `-d:nimAllocNoUnmap`.

Когда `doNotUnmap` равно `true`, процедура `osDeallocPages` становится пустой операцией (no-op) ради безопасности.

## Экспортируемые процедуры

Модуль экспортирует ровно **три** процедуры. Они образуют минимальный и чистый интерфейс к памяти ОС.

---

### `osAllocPages`

```nim
proc osAllocPages(size: int): pointer {.inline.}
```

#### Что делает

Выделяет не менее `size` байт **«сырой», выровненной по странице памяти** у операционной системы. Выделенная память доступна для чтения и записи. Эта процедура **никогда не возвращает `nil`** — если аллокация по любой причине не удалась (нет памяти, исчерпаны лимиты ресурсов и т.д.), она вызывает `raiseOutOfMem()`, что генерирует `OutOfMemDefect` и завершает программу.

#### Когда использовать

Используйте эту процедуру, когда **без памяти продолжение работы невозможно**. Это основной путь выделения памяти в менеджере Nim. Если ваш код не может разумно продолжить работу без успешной аллокации — выбирайте `osAllocPages`, а не `osTryAllocPages`.

#### Детали по платформам

| Платформа | Вызов под капотом |
|---|---|
| malloc-backed | `c_malloc(size + PageSize - 1 + 4)` + ручное выравнивание |
| Emscripten | `mmap` с дополнительным местом для заголовка метаданных |
| POSIX | `mmap(nil, size, PROT_READ\|PROT_WRITE, MAP_ANON\|MAP_PRIVATE, -1, 0)` |
| Windows | `VirtualAlloc(nil, size, MEM_RESERVE\|MEM_COMMIT, PAGE_READWRITE)` |
| Standalone | Продвижение bump-pointer в статическом массиве `float64` |

#### Пример

```nim
# Выделяем ровно одну страницу «сырой» памяти от ОС
let pageSize = 4096
let mem = osAllocPages(pageSize)
# mem — это выровненный по странице указатель на 4096 записываемых байт.
# Если системная память исчерпана, программа завершится
# до того, как мы доберёмся до этой строки.
defer: osDeallocPages(mem, pageSize)
```

### `osTryAllocPages`

```nim
proc osTryAllocPages(size: int): pointer {.inline.}
```

#### Что делает

Аналогична `osAllocPages`, но с **семантикой мягкого отказа**: если ОС не может предоставить запрошенную память, процедура возвращает `nil` вместо аварийного завершения. Проверка возвращаемого значения — ответственность вызывающего кода.

#### Когда использовать

Используйте эту процедуру, когда у вашего кода есть **план Б** на случай нехватки памяти — например, сброс кэша, уплотнение данных, повторная попытка через некоторое время или корректное сообщение об ошибке пользователю. Это «вежливый» запрос аллокации.

#### Детали по платформам

| Платформа | Поведение при ошибке |
|---|---|
| malloc-backed | Так же, как `osAllocPages` — ошибка `c_malloc` приводит к немедленному исключению |
| Emscripten | Делегирует напрямую в `osAllocPages` |
| POSIX | `mmap` возвращает `MAP_FAILED` (`-1`), что конвертируется в `nil` |
| Windows | `VirtualAlloc` возвращает `nil` при ошибке, это значение передаётся как есть |
| Standalone | Возвращает `nil`, если статическая куча исчерпана |

> **Важно:** На пути с malloc (`-d:nimAllocPagesViaMalloc`) `osTryAllocPages` ведёт себя так же, как `osAllocPages` — ошибка `c_malloc` сразу генерирует исключение. Это известное ограничение данного пути.

#### Пример

```nim
proc tryGrowHeap(size: int): bool =
  let mem = osTryAllocPages(size)
  if mem == nil:
    # ОС отказала. Пробуем уплотнить или сообщаем о проблеме.
    echo "Предупреждение: не удалось расширить кучу на ", size, " байт."
    return false
  # Память получена — регистрируем блок в нашем аллокаторе
  registerChunk(mem, size)
  return true
```

### `osDeallocPages`

```nim
proc osDeallocPages(p: pointer, size: int) {.inline.}
```

#### Что делает

Возвращает блок памяти, ранее полученный через `osAllocPages` или `osTryAllocPages`, обратно операционной системе. Указатель `p` должен быть **точно тем указателем**, который вернул вызов аллокации, а `size` должен совпадать с изначально запрошенным размером.

Передача неверного указателя или неверного размера приводит к **неопределённому поведению** — это низкоуровневый вызов ОС без какой-либо собственной бухгалтерии.

#### Защита через `doNotUnmap`

На платформах, где `doNotUnmap` равно `true`, эта процедура является **полным no-op** — она ничего не делает, и память не возвращается ОС. Это намеренно и безопасно: таким образом избегаются ошибки платформ (Linux/PowerPC) и ограничения `MEM_RELEASE` в Windows. «Утечка» памяти на этих платформах допустима, потому что аллокатор Nim повторно использует освобождённые страницы внутри себя, прежде чем запрашивать новые у ОС.

#### Детали по платформам

| Платформа | Механизм |
|---|---|
| malloc-backed | Считывает 4-байтовое смещение выравнивания перед `p`, вызывает `c_free(p - offset)` |
| Emscripten | Считывает блок метаданных перед `p`, вызывает `munmap(realPointer, realSize)` |
| POSIX | `munmap(p, size)`, если `reallyOsDealloc` равно `true` |
| Windows | `VirtualFree(p, 0, MEM_RELEASE)`, если `reallyOsDealloc` равно `true` |
| Standalone | Уменьшает bump-pointer, если `p` была последней аллокацией (стекообразная деаллокация) |

#### Пример

```nim
let pageSize = 4096
let mem = osAllocPages(pageSize)

# ... используем память ...

# Возвращаем ОС (или no-op, если doNotUnmap == true для данной платформы)
osDeallocPages(mem, pageSize)
# После этого вызова `mem` — висящий указатель. Использовать его нельзя.
```

## Standalone / Bare-Metal аллокатор

Когда `hostOS == "standalone"` или задан флаг `-d:StandaloneHeapSize`, все три процедуры работают со **статически выделенным массивом** значений `float64` (тип выбран для естественного 8-байтового выравнивания). Размер кучи по умолчанию — `1024 * PageSize`, но его можно переопределить:

```nim
# Компиляция с кастомным размером кучи:
# nim c -d:StandaloneHeapSize=65536 myprogram.nim
```

Аллокатор реализован как **bump-pointer**: при каждой аллокации указатель просто сдвигается вперёд. Деаллокация работает только если освобождаемый блок является **самой последней** аллокацией (порядок LIFO). Это делает его чрезвычайно быстрым и пригодным для встраиваемых систем, где фрагментация кучи не является проблемой и ОС отсутствует.

## Принципы дизайна

1. **Нулевые накладные расходы на быстром пути.** Все процедуры помечены `{.inline.}`, поэтому вызовной оверхед полностью исчезает.
2. **Никаких метаданных внутри выделяемого региона.** Возвращаемая память чистая. Путь через malloc хранит 4-байтовое смещение *перед* пользовательским указателем; Emscripten хранит небольшой дескриптор перед указателем — но они невидимы для вызывающего кода.
3. **Быстрый аварийный выход по умолчанию.** `osAllocPages` считает нехватку памяти фатальным условием. Исчерпание памяти на уровне ОС почти всегда означает, что что-то пошло серьёзно не так.
4. **Безопасность платформы важнее экономии памяти.** Механизм `doNotUnmap` жертвует возвратом памяти ОС ради корректности работы на платформах с некорректной семантикой unmap.
