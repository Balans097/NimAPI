# Справочник модуля `cstrutils`

> **Модуль:** `std/cstrutils`  
> **Назначение:** Вспомогательные функции для работы со значениями типа `cstring` напрямую, без преобразования в `string`, чтобы избежать лишних аллокаций памяти.

---

## Общее описание

В Nim тип `cstring` — это «Си-строка»: сырой указатель на последовательность байтов, заканчивающуюся нулевым символом (`'\0'`). Именно этот тип используется в C-библиотеках и в среде выполнения JavaScript. Операции над `cstring` низкоуровневые: стандартный модуль `strutils` их не покрывает, поскольку `string` и `cstring` — разные типы.

Главная проблема `cstring` в том, что преобразование его в `string` (например, `$myCstring`) вызывает аллокацию в куче: все байты копируются в новый объект Nim-строки. В горячих участках кода — плотных циклах, FFI-коллбэках, встраиваемых системах или высоконагруженных серверах — такие накладные расходы нежелательны.

`cstrutils` решает эту проблему, предоставляя типичные операции проверки и сравнения строк, работающие непосредственно с `cstring` без каких-либо аллокаций.

### Когда выбирать `cstrutils`, а не `strutils`

| Ситуация | Используйте |
|---|---|
| Данные пришли из C FFI (`const char*`) | `cstrutils` |
| JS-бэкенд, хотите нативные JS-методы строк | `cstrutils` |
| Нужно избежать любых аллокаций в горячем пути | `cstrutils` |
| Работаете с обычными Nim-строками `string` | `strutils` |
| Нужны богатые операции (split, replace, strip и др.) | Преобразуйте в `string`, используйте `strutils` |

### Кросс-платформенные замечания

Все четыре функции модуля работают в трёх контекстах исполнения:

- **Нативный (C-бэкенд)** — самописные циклы, работающие напрямую с памятью, без аллокаций.
- **JavaScript-бэкенд** — `startsWith` и `endsWith` делегируют нативным методам `String.prototype.startsWith` / `String.prototype.endsWith`; `cmpIgnoreStyle` и `cmpIgnoreCase` используют общую реализацию.
- **`nimvm` (вычисление во время компиляции)** — общая реализация, позволяющая вызывать функции в блоках `static:` и выражениях `const`.

## Экспортируемые символы

---

### `startsWith`

```nim
func startsWith*(s, prefix: cstring): bool
```

#### Что делает

Возвращает `true`, если `cstring` `s` начинается с байтовой последовательности, заданной `prefix`. Сравнение **регистрозависимое** и **побайтовое**.

#### Граничные случаи

- Если `prefix` — пустая строка (`cstring""`), функция всегда возвращает `true`, независимо от содержимого `s`. Это математически корректно: пустая строка является префиксом любой строки.
- Если `s` пуста, а `prefix` — нет, функция возвращает `false`.

#### Как работает (нативный бэкенд)

Реализация параллельно обходит обе строки посимвольно. Как только `prefix` достигает нулевого терминатора (`'\0'`), все символы префикса совпали и функция возвращает `true`. Если какой-либо символ отличается раньше этого момента — немедленно возвращается `false`. Нет вычисления длины, нет аллокации, нет лишних переходов.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `s` | `cstring` | Строка, которую проверяем. |
| `prefix` | `cstring` | Префикс, который ищем в начале `s`. |

#### Примеры

```nim
import std/cstrutils

# Базовое совпадение
assert startsWith(cstring"Hello, World", cstring"Hello")

# Нет совпадения — строка есть, но не в начале
assert not startsWith(cstring"Hello, World", cstring"World")

# Пустой префикс всегда даёт true
assert startsWith(cstring"anything", cstring"")
assert startsWith(cstring"", cstring"")

# Регистрозависимость: "hello" ≠ "Hello"
assert not startsWith(cstring"hello, world", cstring"Hello")
```

```nim
import std/cstrutils

# Практический пример: маршрутизация HTTP-запроса без аллокаций
proc routeRequest(path: cstring) =
  if startsWith(path, cstring"/api/"):
    echo "API-маршрут"
  elif startsWith(path, cstring"/static/"):
    echo "Статический файл"
  else:
    echo "Неизвестный маршрут"

routeRequest(cstring"/api/users")     # → API-маршрут
routeRequest(cstring"/static/app.js") # → Статический файл
```

### `endsWith`

```nim
func endsWith*(s, suffix: cstring): bool
```

#### Что делает

Возвращает `true`, если `cstring` `s` заканчивается байтовой последовательностью, заданной `suffix`. Сравнение **регистрозависимое** и **побайтовое**.

#### Граничные случаи

- Пустой `suffix` всегда даёт `true` — пустая строка является суффиксом любой строки.
- Если `suffix` длиннее `s`, функция возвращает `false`.

#### Как работает (нативный бэкенд)

Реализация сначала вычисляет длины `s` и `suffix`. Затем начинает сравнение с позиции `len(s) - len(suffix)` — то есть ровно с того места в `s`, где должен начинаться суффикс, если он присутствует. Это позволяет пропустить весь начальный участок строки и сразу перейти к проверяемой области. Беззнаковое сравнение `i + j <% slen` корректно обрабатывает граничные случаи с пустыми строками.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `s` | `cstring` | Строка, которую проверяем. |
| `suffix` | `cstring` | Суффикс, который ищем в конце `s`. |

#### Примеры

```nim
import std/cstrutils

# Базовое совпадение
assert endsWith(cstring"image.png", cstring".png")

# Нет совпадения
assert not endsWith(cstring"image.png", cstring".jpg")

# Пустой суффикс всегда даёт true
assert endsWith(cstring"anything", cstring"")

# Регистрозависимость
assert not endsWith(cstring"file.PNG", cstring".png")
```

```nim
import std/cstrutils

# Практический пример: определение типа файла без аллокаций
proc describeFile(name: cstring) =
  if endsWith(name, cstring".nim"):
    echo "Исходный файл Nim"
  elif endsWith(name, cstring".c") or endsWith(name, cstring".h"):
    echo "Исходник или заголовочный файл C"
  elif endsWith(name, cstring".json"):
    echo "Данные JSON"
  else:
    echo "Неизвестный тип файла"

describeFile(cstring"main.nim")    # → Исходный файл Nim
describeFile(cstring"bindings.h")  # → Исходник или заголовочный файл C
```

### `cmpIgnoreCase`

```nim
func cmpIgnoreCase*(a, b: cstring): int
```

#### Что делает

Сравнивает два значения `cstring`, **не учитывая регистр ASCII-букв**. Возвращаемое значение следует стандартному трёхстороннему соглашению, принятому в Nim и C:

| Возвращаемое значение | Смысл |
|---|---|
| `0` | `a` и `b` равны при сравнении без учёта регистра |
| `< 0` | `a` идёт перед `b` лексикографически |
| `> 0` | `a` идёт после `b` лексикографически |

#### Что означает «без учёта регистра»

Приводятся к единому регистру только ASCII-буквы (`A`–`Z` / `a`–`z`) — с помощью `toLowerAscii`. Байты вне диапазона ASCII сравниваются по сырым числовым значениям. Это **не Unicode**-сравнение: для юникодного приведения регистра потребуется отдельная библиотека.

#### Как работает (нативный бэкенд)

Функция обходит обе строки параллельно, приводя каждый символ к нижнему ASCII-регистру перед сравнением. Если символы отличаются — немедленно возвращает их разность. Если оба символа одновременно являются нулевыми терминаторами — строки равны, возвращается `0`.

#### Когда использовать

- Сравнение ключей командной строки, HTTP-заголовков или параметров конфигурации, где регистр не важен, а Unicode-сложность не нужна.
- Сортировка списка `cstring`-значений без учёта регистра.
- Проверка пользовательского ввода против известного ключевого слова.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `a` | `cstring` | Первая строка для сравнения. |
| `b` | `cstring` | Вторая строка для сравнения. |

#### Примеры

```nim
import std/cstrutils

# Равенство несмотря на разный регистр
assert cmpIgnoreCase(cstring"hello", cstring"HeLLo") == 0
assert cmpIgnoreCase(cstring"NIM", cstring"nim") == 0

# Лексикографический порядок: "echo" < "hello" ('e' < 'h')
assert cmpIgnoreCase(cstring"echo", cstring"hello") < 0

# "yellow" > "hello" ('y' > 'h')
assert cmpIgnoreCase(cstring"yellow", cstring"hello") > 0
```

```nim
import std/cstrutils, std/algorithm

# Сортировка seq[cstring] без учёта регистра
var words = @[cstring"Zebra", cstring"apple", cstring"Mango", cstring"banana"]
words.sort(proc(a, b: cstring): int = cmpIgnoreCase(a, b))
# Порядок после сортировки: apple, banana, Mango, Zebra
for w in words: echo w
```

```nim
import std/cstrutils

# Определение HTTP-метода независимо от регистра
proc handleMethod(meth: cstring) =
  if cmpIgnoreCase(meth, cstring"GET") == 0:
    echo "GET-запрос"
  elif cmpIgnoreCase(meth, cstring"POST") == 0:
    echo "POST-запрос"

handleMethod(cstring"get")   # → GET-запрос
handleMethod(cstring"Post")  # → POST-запрос
```

### `cmpIgnoreStyle`

```nim
func cmpIgnoreStyle*(a, b: cstring): int
```

#### Что делает

Сравнивает два значения `cstring`, игнорируя одновременно **регистр ASCII-букв** и **символы подчёркивания**. Семантически эквивалентно `cmp(normalize($a), normalize($b))` из `strutils`, но работает напрямую с `cstring` без каких-либо аллокаций.

Возвращаемое значение использует то же трёхстороннее соглашение, что и `cmpIgnoreCase`:

| Возвращаемое значение | Смысл |
|---|---|
| `0` | `a` и `b` равны при стиле-независимом сравнении |
| `< 0` | `a` идёт перед `b` |
| `> 0` | `a` идёт после `b` |

#### Что означает «игнорировать стиль»

Правила именования идентификаторов Nim допускают взаимозаменяемое использование `camelCase` и `snake_case`. `cmpIgnoreStyle` воплощает это правило: символы подчёркивания полностью игнорируются, а буквы сравниваются без учёта регистра. Поэтому `"helloWorld"`, `"hello_world"`, `"Hello_World"` и `"HELLO_WORLD"` сравниваются как равные.

#### Важное предупреждение: не использовать для сравнения Nim-идентификаторов

Документация модуля явно предупреждает: **не применяйте `cmpIgnoreStyle` для сравнения имён идентификаторов Nim в макросах или кодогенерации**. Для этого используйте `macros.eqIdent`, который корректно обрабатывает дополнительные частные случаи, специфичные для правил Nim.

#### Как работает (нативный бэкенд)

Перед сравнением каждого символа функция независимо пропускает все символы `_` с обеих сторон. Затем приводит оба символа к нижнему ASCII-регистру и вычитает их числовые коды. Если разность ненулевая или символ является нулевым терминатором — цикл завершается.

#### Параметры

| Имя | Тип | Описание |
|---|---|---|
| `a` | `cstring` | Первая строка для сравнения. |
| `b` | `cstring` | Вторая строка для сравнения. |

#### Примеры

```nim
import std/cstrutils

# Подчёркивания и регистр одновременно игнорируются
assert cmpIgnoreStyle(cstring"hello_world", cstring"helloWorld") == 0
assert cmpIgnoreStyle(cstring"hello", cstring"H_e_L_Lo") == 0
assert cmpIgnoreStyle(cstring"MY_CONST", cstring"myConst") == 0

# Строки, различающиеся реальными буквами, всё равно не равны
assert cmpIgnoreStyle(cstring"foo", cstring"bar") != 0
```

```nim
import std/cstrutils

# Сравнение ключей конфигурации в snake_case или camelCase
proc lookupConfig(key: cstring): string =
  let knownKeys = [cstring"max_connections", cstring"server_port", cstring"log_level"]
  let labels    = ["MaxConnections", "ServerPort", "LogLevel"]
  for i, k in knownKeys:
    if cmpIgnoreStyle(key, k) == 0:
      return labels[i]
  return "unknown"

echo lookupConfig(cstring"maxConnections")  # → MaxConnections
echo lookupConfig(cstring"SERVER_PORT")     # → ServerPort
echo lookupConfig(cstring"logLevel")        # → LogLevel
```

```nim
import std/cstrutils

# Убеждаемся, что разные соглашения именования обозначают одно и то же
let snakeCase  = cstring"user_account_id"
let camelCase  = cstring"userAccountId"
let screamCase = cstring"USER_ACCOUNT_ID"

assert cmpIgnoreStyle(snakeCase, camelCase)  == 0
assert cmpIgnoreStyle(snakeCase, screamCase) == 0
```

## Типичные ловушки

**1. Регистрозависимость `startsWith` и `endsWith`**  
Эти две функции строго чувствительны к регистру. `startsWith(cstring"Hello", cstring"hello")` вернёт `false`. Если нужна проверка префикса/суффикса без учёта регистра, преобразуйте в `string` и воспользуйтесь `strutils`, либо вручную примените `cmpIgnoreCase` к нужному участку.

**2. Символы вне ASCII**  
Все четыре функции работают с сырыми байтами. Приведение регистра действует только для ASCII (`A`–`Z`). Если `cstring` содержит многобайтовые UTF-8 символы, функции не применяют юникодные правила регистра — например, `"ñ"` и `"Ñ"` не будут считаться равными.

**3. `cmpIgnoreStyle` не предназначен для сравнения Nim-идентификаторов**  
Несмотря на кажущееся соответствие задаче, модуль явно не рекомендует использовать `cmpIgnoreStyle` для сравнения Nim-идентификаторов в коде макросов. Для этого используйте `macros.eqIdent`.

**4. Безопасность при `nil`**  
`cstring` в Nim может быть `nil`. Передача `nil` в любую из этих функций на нативном бэкенде является неопределённым поведением (разыменование нулевого указателя). Всегда проверяйте, что `cstring`-значения ненулевые, перед вызовом этих функций.

**5. Возвращаемое значение функций сравнения не ограничено значениями -1/0/1**  
`cmpIgnoreCase` и `cmpIgnoreStyle` возвращают сырую разность числовых кодов символов — это может быть любое целое число, а не только -1, 0 или 1. При использовании результата в компараторе сортировки это нормально. При сравнении с конкретными числами (кроме 0) помните: только `== 0` надёжно означает «равны».
