# Справочник модуля `reservedmem`

**Модуль:** `reservedmem` (часть стандартной библиотеки Nim)  
**Статус:** ⚠️ Нестабильный API — интерфейсы могут измениться в следующих версиях Nim.  
**Назначение:** Зарезервировать большой блок виртуального адресного пространства без немедленного потребления физической памяти, а затем фиксировать физические страницы по мере роста буфера.

---

## Основная идея

Современные ОС разделяют два понятия, которые на первый взгляд кажутся одним:

- **Виртуальное адресное пространство** — диапазон адресов памяти, которые процесс может называть. Резервировать его почти бесплатно: ни RAM, ни своп не расходуются.
- **Физическая (зафиксированная) память** — реальная RAM (или своп), стоящая за этими адресами. Чтение или запись зафиксированной страницы и есть то, что стоит реальных ресурсов.

`reservedmem` использует этот разрыв. Вы вызываете `init` один раз, сообщая ОС: *«Мне может понадобиться до N байт, начиная отсюда — пожалуйста, удержи этот диапазон за мной.»* ОС резервирует диапазон, но не выделяет RAM. Позже, по мере роста буфера, модуль фиксирует только те страницы, которые действительно нужны, выровненными блоками. При значительном уменьшении буфера лишние зафиксированные страницы возвращаются ОС.

Ключевое преимущество: **базовый адрес буфера никогда не меняется**, даже при росте, потому что виртуальный диапазон был зафиксирован при создании. Обычный буфер на основе `realloc` такой гарантии дать не может.

```
Виртуальное адресное пространство (зарезервировано, RAM не расходуется):
┌─────────────────────────────────────────────────────────────────┐
│  зафиксир. + используется │ зафиксир., свободно │ только резерв │
└───────────────────────────┴─────────────────────┴───────────────┘
^memStart                   ^usedMemEnd            ^committedMemEnd ^memEnd
```

### Платформенное соответствие

| Платформа | Резервирование | Фиксация | Освобождение |
|---|---|---|---|
| Windows | `VirtualAlloc(MEM_RESERVE)` | `VirtualAlloc(MEM_COMMIT)` | `VirtualFree(MEM_DECOMMIT)` |
| POSIX | `mmap(PROT_NONE)` | `mprotect(accessFlags)` | `posix_madvise(DONTNEED)` |

## Вспомогательные шаблоны

### `distance`

```nim
template distance*(lhs, rhs: pointer): int
```

Возвращает знаковое байтовое смещение от `lhs` до `rhs` как `int`. Эквивалентно `cast[int](rhs) - cast[int](lhs)`. Отрицательный результат означает, что `rhs` находится раньше `lhs` в адресном пространстве.

Используется внутри модуля для вычисления расстояний между закладками в `ReservedMem`, но полезен и в вашем собственном низкоуровневом коде.

```nim
import reservedmem

var a: array[10, byte]
let p = addr a[0]
let q = addr a[7]
echo distance(p, q)   # 7  (q на 7 байт дальше p)
echo distance(q, p)   # -7
```

### `shift`

```nim
template shift*(p: pointer, distance: int): pointer
```

Возвращает новый указатель, сдвинутый на `distance` байт вперёд от `p`. Отрицательное `distance` — движение назад. Это арифметика указателей без ограничений типизированных указателей.

```nim
import reservedmem

var buf: array[16, byte]
let base = addr buf[0]
let mid  = base.shift(8)   # указывает на buf[8]
echo distance(base, mid)   # 8
```

## `MemAccessFlags`

```nim
type MemAccessFlags* = int
```

Целочисленный псевдоним, значения которого — флаги защиты памяти ОС. Вы всегда используете именованные константы (`memRead`, `memReadWrite` и т.д.), а не конструируете сырые числа. Одно и то же символическое имя автоматически отображается в правильную константу ОС на каждой платформе.

По умолчанию для всех вызовов `init` используется `memReadWrite` — страницы, доступные на чтение и запись. `memRead` — для разделяемых данных только для чтения, `memExecReadWrite` — для JIT-буферов машинного кода (куда вы сначала записываете инструкции, а потом их исполняете).

## `ReservedMem`

```nim
type ReservedMem* = object
```

Сырой, нетипизированный, побайтовый буфер, чей виртуальный диапазон адресов фиксируется при создании. Воспринимайте его как низкоуровневый фундамент; для типизированных данных предпочитайте `ReservedMemSeq[T]`.

Внутренние закладки (все приватные):

| Поле | Значение |
|---|---|
| `memStart` | Первый байт зарезервированного диапазона |
| `usedMemEnd` | Байт сразу за последним *используемым* байтом |
| `committedMemEnd` | Байт сразу за последней *зафиксированной* страницей |
| `memEnd` | Байт сразу за всем зарезервированным диапазоном |

---

### `ReservedMem.init`

```nim
proc init*(T: type ReservedMem,
           maxLen: Natural,
           initLen: Natural = 0,
           initCommitLen = initLen,
           memStart = pointer(nil),
           accessFlags = memReadWrite,
           maxCommittedAndUnusedPages = 3): ReservedMem
```

**Конструктор.** Создаёт новый `ReservedMem`, резервируя `maxLen` байт виртуального адресного пространства и опционально фиксируя начальную порцию.

#### Подробное описание параметров

**`maxLen`** — жёсткий потолок максимального размера буфера. ОС немедленно резервирует столько байт виртуального адресного пространства. Большое значение (например, 1 ГБ) совершенно нормально на 64-битных системах, где адресного пространства предостаточно; RAM не расходуется, пока страницы не зафиксированы.

**`initLen`** — сколько байт считаются *используемыми* с самого начала (то есть начальная логическая длина буфера). Должен быть `≤ initCommitLen`. По умолчанию 0.

**`initCommitLen`** — сколько байт физической памяти зафиксировать сразу. Фактический зафиксированный размер округляется вверх до гранулярности выделения (размер страницы на POSIX, гранулярность выделения на Windows). По умолчанию равен `initLen`. Установка значения выше `initLen` «прогревает» память для ожидаемого раннего роста, избегая промахов страниц при первых записях.

**`memStart`** — подсказка ОС о предпочтительном базовом адресе. Передайте `nil`, чтобы ОС выбрала сама (рекомендуется). Передача конкретного адреса может молча завершиться неудачей, если ОС не может его выполнить.

**`accessFlags`** — биты защиты для зафиксированных страниц. По умолчанию: `memReadWrite`.

**`maxCommittedAndUnusedPages`** — сколько страниц запаса остаётся зафиксированными после уменьшения буфера через `setLen`. Большее значение снижает частоту циклов освобождения/фиксации ценой немного большего расхода RAM. По умолчанию: 3.

#### Пример

```nim
import reservedmem

# Зарезервировать 64 МБ адресного пространства; ничего не фиксировать.
var buf = ReservedMem.init(maxLen = 64 * 1024 * 1024)

# Зарезервировать 64 МБ; сразу зафиксировать 4 КБ; логическая длина 1 КБ.
var buf2 = ReservedMem.init(
  maxLen        = 64 * 1024 * 1024,
  initLen       = 1024,
  initCommitLen = 4096
)

echo buf2.len         # 1024
echo buf2.commitedLen # 4096 (или выровненное вверх значение)
echo buf2.maxLen      # 67108864
```

### `ReservedMem.len`

```nim
func len*(m: ReservedMem): int
```

Возвращает количество байт, которые в данный момент считаются *используемыми* — логическую длину буфера. Это расстояние от `memStart` до `usedMemEnd`.

```nim
var buf = ReservedMem.init(maxLen = 1024 * 1024, initLen = 256)
echo buf.len   # 256
```

---

### `ReservedMem.commitedLen`

```nim
func commitedLen*(m: ReservedMem): int
```

Возвращает, сколько байт физической памяти в данный момент зафиксировано (то есть обеспечено реальной RAM или свопом). Всегда `≥ len` и выровнено по гранулярности выделения. Лишние зафиксированные, но неиспользуемые байты — это «запас», удерживаемый для снижения частоты обращений к ОС при повторном небольшом росте.

> **Замечание:** Имя `commitedLen` (с одной буквой 't') — опечатка в оригинальном источнике, сохранённая для стабильности API.

```nim
var buf = ReservedMem.init(maxLen = 1024 * 1024, initLen = 100)
echo buf.len         # 100
echo buf.commitedLen # ≥ 100, выровнено по размеру страницы (например, 4096)
```

---

### `ReservedMem.maxLen`

```nim
func maxLen*(m: ReservedMem): int
```

Возвращает полный размер зарезервированного виртуального диапазона — значение, переданное как `maxLen` в `init`. Это абсолютный максимум, до которого буфер вообще может вырасти без создания нового резервирования.

```nim
var buf = ReservedMem.init(maxLen = 10 * 1024 * 1024)
echo buf.maxLen   # 10485760
```

---

### `ReservedMem.setLen`

```nim
proc setLen*(m: var ReservedMem, newLen: int)
```

Изменяет логическую длину буфера до `newLen` байт. Это центральная операция, которая управляет всем процессом фиксации и освобождения страниц.

**Рост (`newLen > len`):**  
Если `newLen` превышает текущий зафиксированный диапазон, модуль фиксирует дополнительные страницы для покрытия новой длины. Размер расширения округляется вверх до гранулярности выделения, поэтому рост на один байт может зафиксировать целые 4 КБ страницы. На Windows сырые байты в новой зафиксированной области гарантированно обнуляются (`VirtualAlloc`), на POSIX могут содержать мусор.

**Уменьшение (`newLen < len`):**  
Логическая длина отступает назад, но зафиксированные страницы не возвращаются немедленно. Страницы освобождаются только тогда, когда незадействованная зафиксированная область превышает `maxCommittedAndUnusedPages` страниц запаса. Это предотвращает «биение» при осцилляции буфера вокруг граничного размера.

`newLen` должен быть от 0 до `maxLen`. Нарушение этого условия приводит к неопределённому поведению (вызов ОС завершится ошибкой или состояние будет повреждено).

```nim
import reservedmem

var buf = ReservedMem.init(maxLen = 4 * 1024 * 1024)

buf.setLen(1000)
echo buf.len         # 1000
echo buf.commitedLen # 4096 (одна страница зафиксирована)

buf.setLen(5000)
echo buf.len         # 5000
echo buf.commitedLen # 8192 (теперь две страницы)

buf.setLen(100)
echo buf.len         # 100
# зафиксированные страницы могут оставаться 8192, пока запас не превысит порог
```

## `ReservedMemSeq[T]`

```nim
type ReservedMemSeq*[T] = object
```

Типизированная, последовательностеподобная обёртка над `ReservedMem`. Хранит элементы типа `T` непрерывно в памяти, по фиксированному адресу, расширяясь по требованию. Интерфейс близко повторяет встроенный `seq[T]` Nim, но с ограниченной максимальной ёмкостью и стабильным базовым адресом.

Все параметры длины/индексов `ReservedMemSeq` — в единицах **элементов**, а не байт.

---

### `ReservedMemSeq.init`

```nim
proc init*(SeqType: type ReservedMemSeq,
           maxLen: Natural,
           initLen: Natural = 0,
           initCommitLen: Natural = 0,
           memStart = pointer(nil),
           accessFlags = memReadWrite,
           maxCommittedAndUnusedPages = 3): SeqType
```

Создаёт типизированную последовательность, поддержанную виртуальным резервированием памяти. Параметры имеют то же значение, что и в `ReservedMem.init`, но `maxLen`, `initLen` и `initCommitLen` задаются в **элементах** типа `T`, а не в байтах. Процедура внутренне умножает на `sizeof(T)` перед вызовом `ReservedMem.init`.

#### Пример

```nim
import reservedmem

# Последовательность из не более миллиона значений int64.
# Зарезервировать адресное пространство для всех; начально ничего не фиксировать.
var s = ReservedMemSeq[int64].init(maxLen = 1_000_000)

echo s.len      # 0
echo s.maxLen   # 1000000
```

---

### `ReservedMemSeq.[](pos)` — доступ к элементам

```nim
func `[]`*[T](s: ReservedMemSeq[T], pos: Natural): lent T
func `[]`*[T](s: var ReservedMemSeq[T], pos: Natural): var T
func `[]`*[T](s: ReservedMemSeq[T], rpos: BackwardsIndex): lent T
func `[]`*[T](s: var ReservedMemSeq[T], rpos: BackwardsIndex): var T
```

Четыре перегрузки оператора индексирования, покрывающие все четыре комбинации:
- `const` vs `var` получатель (только для чтения `lent T` vs изменяемый `var T`)
- Прямой индекс (`Natural`) vs обратный индекс (`BackwardsIndex`, записывается как `^1`, `^2`, …)

Утверждение `rangeCheck` защищает от выхода за пределы. В сборках без проверки диапазонов это превращается в сырое разыменование указателя.

#### Пример

```nim
import reservedmem

var s = ReservedMemSeq[float32].init(maxLen = 1000, initLen = 3)
# initLen=3 означает, что 3 элемента считаются «используемыми»;
# их память зафиксирована, но содержимое неопределено (POSIX) / обнулено (Windows).

# Запись через изменяемую индексацию
s[0] = 1.5
s[1] = 2.5
s[2] = 3.5

# Чтение через константную индексацию
echo s[0]    # 1.5
echo s[^1]   # 3.5  (последний элемент, обратный индекс)
echo s[^2]   # 2.5
```

### `ReservedMemSeq.len`

```nim
func len*[T](s: ReservedMemSeq[T]): int
```

Количество элементов, в данный момент находящихся в последовательности (логическая длина). Эквивалентно `s.mem.len div sizeof(T)`.

```nim
var s = ReservedMemSeq[int32].init(maxLen = 100)
echo s.len   # 0
s.setLen(5)
echo s.len   # 5
```

---

### `ReservedMemSeq.setLen`

```nim
proc setLen*[T](s: var ReservedMemSeq[T], newLen: int)
```

Изменяет количество элементов в последовательности. Внутренне переводит в байты и делегирует `ReservedMem.setLen`.

> ⚠️ **Деструкторы не вызываются.** Исходный код содержит пометку `TODO`: при уменьшении деструкторы удалённых элементов не вызываются. Для типов с управляемыми ресурсами (строки, seqs, refs) это приводит к утечкам памяти. Используйте эту процедуру безопасно только с типами простых данных (`int`, `float`, `object` без полей-ссылок и т.д.).

```nim
import reservedmem

var s = ReservedMemSeq[int32].init(maxLen = 1000)
s.setLen(10)
echo s.len   # 10
s.setLen(3)
echo s.len   # 3
```

---

### `ReservedMemSeq.add`

```nim
proc add*[T](s: var ReservedMemSeq[T], val: T)
```

Добавляет один элемент в конец последовательности. Вызывает `setLen(len + 1)`, затем записывает `val` на новую последнюю позицию. Если новая длина требует фиксации дополнительных страниц, это происходит прозрачно.

Вызов `add` при `len == maxLen` приведёт к панике или повреждению состояния — всегда отслеживайте свой верхний предел.

```nim
import reservedmem

var s = ReservedMemSeq[int32].init(maxLen = 1_000_000)

for i in 0 ..< 5:
  s.add(i * 10)

echo s.len    # 5
echo s[0]     # 0
echo s[4]     # 40
```

---

### `ReservedMemSeq.pop`

```nim
proc pop*[T](s: var ReservedMemSeq[T]): T
```

Удаляет и возвращает последний элемент. Читает значение, затем вызывает `setLen(len - 1)`.

> ⚠️ **Утверждение:** `assert` защищает от извлечения из пустой последовательности. В сборках без утверждений это приведёт к чтению из недопустимой памяти. В релизном коде всегда проверяйте `len > 0` перед вызовом `pop`.

> ⚠️ **Деструкторы не вызываются** (та же оговорка, что и для `setLen`).

```nim
import reservedmem

var s = ReservedMemSeq[int32].init(maxLen = 100)
s.add(10)
s.add(20)
s.add(30)

echo s.pop()   # 30
echo s.pop()   # 20
echo s.len     # 1
```

---

### `ReservedMemSeq.commitedLen`

```nim
func commitedLen*[T](s: ReservedMemSeq[T]): int
```

Количество элементов, для которых в данный момент зафиксирована физическая память. Всегда `≥ len`. Полезно для диагностики того, сколько RAM фактически использует последовательность в сравнении с её логическим размером.

```nim
var s = ReservedMemSeq[int64].init(maxLen = 100_000)
s.add(42)
echo s.len          # 1
echo s.commitedLen  # зависит от размера страницы; скорее всего 512 (4096 / 8 байт на int64)
```

---

### `ReservedMemSeq.maxLen`

```nim
func maxLen*[T](s: ReservedMemSeq[T]): int
```

Максимальное количество элементов, которое последовательность вообще может хранить (значение, переданное как `maxLen` в `init`). Добавление за этот предел обрабатывается некорректно — всегда отслеживайте верхнюю границу.

```nim
var s = ReservedMemSeq[uint8].init(maxLen = 1024)
echo s.maxLen   # 1024
```

## Полный рабочий пример — буфер журнала с растущим, но стабильным адресом

```nim
# Демонстрирует ключевое свойство: адрес буфера не меняется
# при росте вплоть до зарезервированного максимума.

import reservedmem

const MaxEvents = 1_000_000

type Event = object
  timestamp: int64
  code: int32
  value: float32

var log = ReservedMemSeq[Event].init(
  maxLen                    = MaxEvents,
  maxCommittedAndUnusedPages = 5  # держать 5 страниц запаса при уменьшении
)

# Запоминаем стабильный базовый адрес
let baseAddr = addr log.mem.memStart   # адрес не изменится

echo "Ёмкость журнала: ", log.maxLen, " событий"
echo "Начальный RAM:   ", log.commitedLen * sizeof(Event), " байт"

# Моделируем цикл событий
for i in 0 ..< 10_000:
  log.add(Event(timestamp: i, code: int32(i mod 16), value: float32(i) * 0.1))

echo "После 10 000 событий:"
echo "  len          = ", log.len
echo "  commitedLen  = ", log.commitedLen
echo "  maxLen       = ", log.maxLen

# Доступ к элементам — базовый адрес гарантированно стабилен
echo "Временная метка первого события: ", log[0].timestamp
echo "Временная метка последнего события: ", log[^1].timestamp

# Уменьшение — лишние страницы освобождаются лениво
log.setLen(100)
echo "После уменьшения до 100:"
echo "  len          = ", log.len
echo "  commitedLen  = ", log.commitedLen  # ≥ 100, часть страниц может остаться
```

## Соображения по проектированию и предупреждения

**Стабильность адреса — определяющая особенность.** В отличие от `seq[T]`, который может перераспределить и переместить свой буфер, `ReservedMemSeq` гарантирует, что адреса элементов остаются действительными на протяжении всего времени жизни объекта. Это делает безопасным хранение сырых указателей или ссылок в последовательность.

**`maxLen` должен быть выбран заранее.** После `init` расширить зарезервированный диапазон невозможно. Чрезмерное резервирование на 64-битных системах безвредно (нет расхода RAM), но на 32-битных системах адресное пространство — дефицитный ресурс.

**Деструкторы не вызываются** при `setLen`, `pop` или любой другой операцией модуля. Используйте его безопасно только с типами простых данных, если не управляете очисткой вручную.

**POSIX-фиксация не обнуляет память явно.** `mprotect` лишь меняет защиту на уже `mmap`-ированной памяти, которую ядро обнулило при создании отображения (ядро обнуляет новые страницы до их выдачи процессу). На практике вновь зафиксированные страницы на Linux обнулены, но это гарантия ядра, связанная с `mmap`, а не с `mprotect`. Windows `VirtualAlloc(MEM_COMMIT)` обнуляет явно.

**Это нестабильный API.** Заголовок модуля предупреждает, что интерфейсы могут измениться. Не полагайтесь на него в библиотечном коде без фиксации версии Nim.
