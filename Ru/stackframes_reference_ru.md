# std/stackframes — справочник модуля

> **Входит в стандартную библиотеку Nim** (`std/stackframes`).

## Обзор

`std/stackframes` — низкоуровневый диагностический модуль, предоставляющий прямой доступ с нулевыми накладными расходами к кадру стека вызовов, который Nim поддерживает для каждого активного вызова процедуры. Он предназначен для двух конкретных сценариев:

1. **Интроспекция** — получение имени текущей функции C/C++ во время компиляции или указателя на текущий кадр стека для кода, выполняющего обход стека.
2. **Аннотирование кадров стека сообщениями** — прикрепление произвольных строковых заметок к текущему кадру, чтобы они появлялись рядом с именем процедуры в трассировке стека при распространении исключения.

Все три экспортируемых символа — это **шаблоны**, а не процедуры. Это сделано намеренно: шаблоны встраиваются в точке вызова и тем самым не порождают дополнительного вызова функции и нового кадра стека. Для кода интроспекции стека, призванного описывать своего непосредственного вызывающего, это различие принципиально — процедурная версия `getPFrame` возвращала бы указатель на *свой собственный* кадр, а не на кадр вызывающего.

### Охранники времени компиляции

Два константы уровня модуля управляют условной компиляцией повсюду в модуле:

```nim
const NimStackTrace     = compileOption("stacktrace")
const NimStackTraceMsgs = compileOption("stacktraceMsgs")
```

- `NimStackTrace` равна `true`, когда программа скомпилирована с `--stacktrace:on` (по умолчанию в отладочных сборках).
- `NimStackTraceMsgs` равна `true`, когда скомпилировано с `--stacktraceMsgs:on`.

Когда трассировка стека отключена, эти шаблоны либо возвращают пустое/нулевое значение, либо становятся полными заглушками — они никогда не вызывают ошибки компиляции вне зависимости от конфигурации. Это означает, что их можно свободно использовать в библиотечном коде, не навязывая пользователям обязательных флагов компиляции.

## Шаблоны

### `procName`

```nim
template procName*(): string
```

**Возвращает имя текущей функции C или C++ в виде строки.**

Этот шаблон генерирует обращение к встроенному компилятору `__func__` (стандарт C99 / C++11) — `const char*`, содержащей неискажённое имя объемлющей функции в том виде, в котором компилятор видит её в данной точке. Результат преобразуется в строку Nim и возвращается.

**Доступность:** Работает только когда бэкенд Nim — `c` или `cpp` (то есть при компиляции с `-b:c` или `-b:cpp`, что является значением по умолчанию). На других бэкендах тело шаблона отсутствует, и его вызов вызовет ошибку компиляции — оборачивайте в `when defined(c) or defined(cpp)`, если нужна переносимость.

**Как выглядит имя:** Для процедуры Nim `proc myProc(x: int)` C-бэкенд генерирует функцию C, имя которой подчиняется схеме искажения имён Nim, например `myProc__module_NimInt`. Точное имя зависит от модуля, суффикса перегрузки и версии Nim. Это имя *уровня C*, а не уровня Nim.

**Типичное использование — низкоуровневое логирование и диагностика:**

```nim
import std/stackframes

proc doSomething() =
  when defined(c) or defined(cpp):
    echo "Текущая C-функция: ", procName()
```

**Пример — фиксация источника отладочного сообщения:**

```nim
import std/stackframes

proc criticalSection() =
  when defined(c) or defined(cpp):
    let fn = procName()
    debugLog("[" & fn & "] входим в критическую секцию")
    # ... работа ...
    debugLog("[" & fn & "] выходим из критической секции")
```

### `getPFrame`

```nim
template getPFrame*(): PFrame
```

**Возвращает указатель `PFrame` на кадр стека текущей процедуры без накладных расходов на вызов функции.**

`PFrame` — внутренний тип Nim для записи кадра стека. Каждый вызов процедуры, отслеживаемый системой трассировки стека, имеет одну локальную переменную `FR_` этого типа в стеке C. `getPFrame` напрямую берёт адрес этой локальной переменной через `{.emit: "`framePtr` = &FR_;".}` и возвращает его.

Критическое преимущество перед `getFrame()` (из `std/system`) состоит в том, что `getFrame()` — это *функция*: её вызов помещает новый кадр на стек и возвращает кадр для самой `getFrame`, а не для вашего кода. `getPFrame` — шаблон, поэтому после встраивания `&FR_` относится к кадру *точки вызова*, что почти всегда и нужно.

**Когда трассировка стека отключена** (`--stacktrace:off`): `FR_` не существует, поэтому шаблон вычисляется в нулевой результат блока. Возвращаемое значение в этом случае не следует разыменовывать.

**Типичное использование — собственные обходчики стека и профилировщики:**

```nim
import std/stackframes

proc profiledOperation() =
  let frame = getPFrame()
  when frame != nil:
    echo "Имя процедуры кадра: ", frame.procname
    echo "Файл кадра:          ", frame.filename
    echo "Строка кадра:        ", frame.line
```

**Пример — передача текущего кадра в собственный диагностический приёмник:**

```nim
import std/stackframes

proc logWithFrame(msg: string) =
  let f = getPFrame()
  myDiagnosticSink(f, msg)

proc compute(x: int): int =
  logWithFrame("начало вычисления")
  result = x * x
```

### `setFrameMsg`

```nim
template setFrameMsg*(msg: string, prefix = " ")
```

**Прикрепляет строковое сообщение к текущему кадру стека, чтобы оно появлялось в трассировках стека при исключениях.**

Когда исключение возбуждается и распространяется вверх по стеку вызовов, механизм трассировки стека Nim обычно показывает для каждого кадра только имя процедуры, имя файла и номер строки. `setFrameMsg` позволяет встроить дополнительный контекст времени выполнения — значения переменных, флаги состояния, счётчики циклов — прямо в трассу в той точке, где они важны. Это может кардинально сократить время выяснения *причины*, по которой конкретный путь выполнения привёл к исключению.

Сообщение добавляется к глобальной строке `frameMsgBuf` по смещению текущего кадра. `setFrameMsg` можно вызывать **несколько раз** в одном и том же кадре — каждый вызов добавляет `prefix + msg` к области сообщений кадра. Параметр `prefix` по умолчанию — одиночный пробел, предназначенный для визуального разделения последовательных сообщений.

**Двойной охранник компиляции:** Этот шаблон является **полной заглушкой**, если не активны *оба* флага: `--stacktrace:on` и `--stacktraceMsgs:on`. При любой другой комбинации он компилируется в ничто и не несёт никаких затрат во время выполнения. Его можно свободно вызывать в горячих путях и просто отключать в релизных сборках.

**Типичное использование — аннотирование кадра переменной цикла:**

```nim
import std/stackframes

proc processItems(items: seq[string]) =
  for i, item in items:
    setFrameMsg("item=" & item & " i=" & $i)
    processOneItem(item)   # если бросает исключение, трасса показывает, какой элемент
```

**Пример — несколько аннотаций в одном кадре:**

```nim
import std/stackframes

proc connect(host: string, port: int) =
  setFrameMsg("host=" & host)
  setFrameMsg("port=" & $port)
  # При ошибке соединения трассировка покажет обе аннотации
  rawConnect(host, port)
```

**Пример — аннотирование вычисленным состоянием:**

```nim
import std/stackframes

proc deserialize(data: openArray[byte]) =
  let version = data[0]
  let length  = (int(data[1]) shl 8) or int(data[2])
  setFrameMsg("v=" & $version & " len=" & $length)
  # Любая ошибка разбора далее несёт версию и длину в трассировке
  parseBody(data[3..^1], length)
```

## Как сообщения кадра стека появляются в трассировках

Когда исключение распространяется при включённых `--stacktrace` и `--stacktraceMsgs`, вывод выглядит примерно так:

```
Traceback (most recent call last)
mymodule.nim(42)         processItems  item=broken_value i=7
mymodule.nim(19)         deserialize   v=2 len=512
system.nim(...)          ...
Error: unhandled exception: unexpected token [ParseError]
```

Без `setFrameMsg` та же трассировка показала бы только имена процедур и номера строк, без намёка на то, *какой* элемент или *какая* версия вызвала сбой.

## Краткий справочник по флагам компиляции

| Флаг | Влияние на этот модуль |
|---|---|
| `--stacktrace:on` (по умолчанию в отладке) | Включает `getPFrame` и `setFrameMsg`; включает локальные переменные `FR_` |
| `--stacktrace:off` (по умолчанию в релизе) | `getPFrame` возвращает nil/ноль; `setFrameMsg` — заглушка |
| `--stacktraceMsgs:on` | Включает добавление сообщений в `setFrameMsg` |
| `--stacktraceMsgs:off` (по умолчанию) | `setFrameMsg` — заглушка даже при `--stacktrace:on` |
| `-b:c` или `-b:cpp` (по умолчанию) | Включает `procName` |
| Другие бэкенды | Тело `procName` отсутствует |

|--------|-----|-----------|---------------|
| `procName()` | шаблон | Имя текущей функции C-уровня | Только бэкенд C/C++ |
| `getPFrame()` | шаблон | Указатель на текущий кадр стека (`PFrame`) | `--stacktrace:on` |
