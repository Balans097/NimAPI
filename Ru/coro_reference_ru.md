# Справочник модуля `coro`

**Модуль:** `coro` (часть стандартной библиотеки Nim)  
**Статус:** ⚠️ Нестабильный API — интерфейсы могут измениться в следующих версиях Nim.  
**Назначение:** Кооперативная многозадачность через стековые корутины — несколько контекстов выполнения, явно передающих управление друг другу в рамках одного потока ОС.

---

## Обзор

**Корутина** — это функция, которая может приостановить собственное выполнение в произвольной точке (`suspend`) и позже возобновиться ровно там, где остановилась, со всем стеком вызовов нетронутым. В отличие от потоков ОС, корутины *кооперативны*: планировщик переключает контекст только тогда, когда текущая корутина явно приостанавливается. Вытеснения нет.

Эта модель хорошо подходит для:
- Конечных автоматов, написанных как линейный код вместо явных переменных состояния.
- Конвейеров ввода-вывода, где одна стадия ждёт, пока другая производит данные.
- Игровых циклов, где каждая сущность «спит» между тиками.
- Любых сценариев, где нужна ясность последовательного кода при параллельном исполнении нескольких независимых задач.

### Как течёт выполнение

```
главный поток
│
├── start(taskA)   ← планирует taskA, не запускает сразу
├── start(taskB)   ← планирует taskB, не запускает сразу
└── run()          ← входит в цикл планировщика
      │
      ├── переключается на taskA → taskA работает до suspend()
      │          taskA приостанавливается ──────────────────────┐
      ├── переключается на taskB → taskB работает до suspend()  │
      │          taskB приостанавливается ─────────────────────┤
      ├── возврат к taskA (если сон истёк) ←────────────────────┘
      │   ...
      └── выходит, когда все корутины вернулись
```

### Необходимый флаг компилятора

```sh
nim c -d:nimCoroutines myapp.nim
```

Без `-d:nimCoroutines` модуль отказывается компилироваться.

### Бэкенды переключения контекста

| Бэкенд | Платформа | Флаг |
|---|---|---|
| `ucontext` | Linux, macOS, большинство POSIX *(по умолчанию)* | `-d:nimCoroutinesUcontext` |
| `setjmp` | x86/x86-64 POSIX, Haiku, OpenBSD | `-d:nimCoroutinesSetjmp` |
| `fibers` | Windows *(единственный вариант)* | автоматически |

Бэкенд выбирается автоматически в зависимости от платформы. Флаги нужны только для переопределения дефолта на POSIX.

## Экспортируемый API

| Символ | Вид | Описание |
|---|---|---|
| `CoroutineRef` | ref object | Безопасный дескриптор работающей корутины |
| `start(c, stacksize)` | proc | Поставить корутину в очередь на выполнение |
| `run()` | proc | Запустить планировщик; блокирует до завершения всех корутин |
| `suspend(sleepTime)` | proc | Уступить управление; опционально спать `sleepTime` секунд |
| `alive(c)` | proc | Проверить, выполняется ли ещё корутина |
| `wait(c, interval)` | proc | Заблокировать текущую корутину до завершения другой |

---

## `CoroutineRef`

```nim
type CoroutineRef* = ref object
  coro: CoroutinePtr   # приватное; nil после завершения корутины
```

### Что это такое

`CoroutineRef` — единственный дескриптор, который публичный API даёт вам для работы с корутиной. Это `ref object`, поэтому он участвует в сборке мусора Nim: вы можете хранить его в переменной, передавать куда угодно, и GC будет удерживать его живым, пока есть ссылка.

Внутреннее поле `coro` указывает на фактическую низкоуровневую структуру корутины. Когда корутина завершается и планировщик очищает её, `coro` устанавливается в `nil`. Все публичные процедуры (`alive`, `wait`) сначала проверяют `nil` и корректно завершаются вместо краша.

Такой дизайн разделяет *логический дескриптор* (который вы держите) и *физическую память* (которой управляет планировщик). Вам никогда не нужно вручную освобождать `CoroutineRef`.

### Пример

```nim
import coro

let handle: CoroutineRef = start(proc() =
  echo "Я корутина"
)

run()
# После возврата run() поле handle.coro равно nil — корутина исчезла.
echo alive(handle)   # false
```

## `start`

```nim
proc start*(c: proc(), stacksize: int = defaultStackSize): CoroutineRef {.discardable.}
```

### Что делает

Регистрирует процедуру `c` без аргументов и результата как новую корутину и добавляет её в очередь планировщика. Корутина **не** начинает выполняться немедленно — она работает только тогда, когда цикл планировщика (`run`) передаёт ей управление.

**`c`** — тело корутины. Это обычное замыкание `proc()`, которое может захватывать переменные из внешней области видимости.

**`stacksize`** — размер приватного стека вызовов корутины в байтах. По умолчанию 512 КБ (`512 * 1024`). Каждая корутина получает собственный стек; выбирайте размер, достаточный для самой глубокой цепочки вызовов внутри корутины. Переполнение стека из-за слишком маленького значения — неопределённое поведение.

**Возвращаемое значение** — `CoroutineRef`, который можно передать в `alive` или `wait`. Помечен `discardable`, поэтому его можно игнорировать, если вас интересует только сам факт выполнения корутины.

`start` вызывает `initialize()` при первом использовании, если контекст планировщика ещё не создан — инициализировать ничего вручную не нужно.

### Когда использовать

Вызывайте `start` один раз на каждую конкурентную задачу, до вызова `run`. Можно вызывать `start` *изнутри* работающей корутины для динамического порождения новых задач; новая корутина будет подхвачена планировщиком в следующем цикле.

### Примеры

```nim
import coro

# Пример 1 — базовое планирование, дескриптор игнорируется
start(proc() = echo "задача первая")
start(proc() = echo "задача вторая")
run()

# Пример 2 — захват переменных из внешней области
var results: seq[int]
for i in 1..3:
  let id = i   # захват по значению
  start(proc() =
    suspend(float(id) * 0.001)  # смещаем время старта
    results.add(id)
  )
run()
echo results  # порядок зависит от тайминга, но содержит 1, 2, 3

# Пример 3 — сохраняем дескриптор для отслеживания завершения
let worker = start(proc() =
  for i in 1..5:
    echo "работаю: ", i
    suspend(0.1)
)
run()
echo "worker завершён: ", not alive(worker)

# Пример 4 — пользовательский размер стека для глубоко рекурсивной корутины
start(proc() =
  discard deeplyRecursiveOperation()
, stacksize = 2 * 1024 * 1024)  # 2 МБ
run()
```

## `run`

```nim
proc run*()
```

### Что делает

Входит в цикл планировщика и доводит все зарегистрированные корутины до завершения. `run` **блокирует вызывающий поток** до тех пор, пока каждая корутина, созданная через `start`, не вернётся (или не будет очищена).

#### Алгоритм планирования

Планировщик ведёт двусвязный список корутин. На каждой итерации он:

1. Смотрит на оставшееся время сна текущей корутины (`sleepTime - прошедшее_время`).
2. Если сон истёк (или его не было), переключает выполнение в эту корутину.
3. Когда корутина вызывает `suspend`, управление возвращается сюда.
4. Если корутина завершилась (`state == CORO_FINISHED`), удаляет её из списка и освобождает память.
5. Переходит к следующей корутине в списке (round-robin).
6. Если все корутины спят, вызывает `os.sleep` на кратчайшую оставшуюся задержку перед следующей проверкой.

Алгоритм **невытесняющий**: корутина работает беспрерывно, пока не вызовет `suspend` или не вернётся.

#### Что происходит с памятью

Когда корутина завершается, планировщик:
- Устанавливает `CoroutineRef.coro` в `nil`, чтобы существующие дескрипторы знали, что корутина ушла.
- Удаляет стек из списка отслеживаемых стеков GC (для сборок без ARC/ORC).
- Освобождает объект fiber (Windows) или выделенную память стека (POSIX).
- Освобождает саму структуру корутины.

`run` возвращается, когда список корутин пуст.

### Когда использовать

Вызывайте `run` ровно один раз за программу (или один раз за логическую «сессию» корутин). Все вызовы `start` для начального набора задач должны быть до `run`. Задачи, динамически порождающие потомков, могут вызывать `start` изнутри своего тела — потомки будут обработаны в следующих циклах планировщика.

### Пример

```nim
import coro

proc producer() =
  for i in 1..5:
    echo "производит ", i
    suspend()

proc consumer() =
  for i in 1..5:
    echo "потребляет ", i
    suspend()

start(producer)
start(consumer)
run()
# Печатает перемежающиеся строки produce/consume, затем возвращается.
```

## `suspend`

```nim
proc suspend*(sleepTime: float = 0)
```

### Что делает

Приостанавливает текущую выполняющуюся корутину и передаёт управление обратно планировщику. Корутина не будет возобновлена раньше, чем пройдёт `sleepTime` секунд. Передача `0` (по умолчанию) уступает управление немедленно и позволяет планировщику запустить следующую корутину, но текущая корутина снова становится кандидатом на выполнение уже в следующем цикле планировщика.

Внутренне `suspend` сохраняет текущий контекст выполнения и вызывает `switchTo`, нацеленный на «loop»-корутину самого планировщика. Затем цикл планировщика выбирает следующую корутину.

`suspend` может вызываться только *изнутри* корутины (то есть из кода, выполняющегося под `run`). Вызов из главного потока вне `run` — неопределённое поведение.

### Параметр `sleepTime` подробно

`sleepTime` — тип `float`, единица измерения — **секунды**. Планировщик проверяет прошедшее время с помощью высокоточного счётчика тиков. Фактический сон неточен — корутина становится *кандидатом* на выполнение через `sleepTime` секунд, но реально запустится только когда планировщик дойдёт до неё в цикле round-robin. Маленькие значения (например, `0.001`) создают высокочастотные корутины; большие (например, `5.0`) — низкочастотные.

Отрицательное значение действует так же, как `0`.

### Когда использовать

- Вызывайте `suspend()` (без аргумента) в каждой естественной «точке уступки» в кооперативных задачах — между итерациями цикла обработки, после каждой единицы работы и т.д.
- Вызывайте `suspend(секунды)`, когда корутина должна ждать реального события: опрос датчика каждые 100 мс, анимация со скоростью 60 fps, повторная попытка после задержки отката.
- Вызывайте `suspend()` внутри цикла ожидания, когда нужно многократно проверять условие без активного ожидания.

### Примеры

```nim
import coro

# Пример 1 — уступка на каждой итерации (максимальное чередование)
proc taskA() =
  for i in 1..5:
    echo "A: ", i
    suspend()   # даём taskB шанс запуститься

proc taskB() =
  for i in 1..5:
    echo "B: ", i
    suspend()

start(taskA)
start(taskB)
run()
# Вывод чередуется: A:1, B:1, A:2, B:2, ...

# Пример 2 — тайминговый сон (корутина просыпается через ~0,5 с)
proc timedTask() =
  echo "начинаю"
  suspend(0.5)   # спать 500 мс
  echo "возобновился через 0.5 с"

start(timedTask)
run()

# Пример 3 — цикл опроса с задержкой
proc poller(ready: ptr bool) =
  while not ready[]:
    suspend(0.05)   # проверять каждые 50 мс
  echo "условие выполнено!"
```

## `alive`

```nim
proc alive*(c: CoroutineRef): bool = c.coro != nil and c.coro.state != CORO_FINISHED
```

### Что делает

Возвращает `true`, если корутина, представленная `c`, ещё запланирована и не вернулась; `false`, если она завершилась или если `c.coro` равно `nil` (то есть планировщик уже освободил память корутины).

Проверка — простое двухусловное выражение:
- `c.coro != nil` — планировщик ещё не освободил структуру.
- `c.coro.state != CORO_FINISHED` — функция корутины не вернулась.

`alive` **не** возобновляет корутину и никак не взаимодействует с планировщиком. Это чистое чтение без побочных эффектов.

### Когда использовать

- Как охранник в написанном вами цикле ожидания (когда нужна логика между проверками сложнее, чем предлагает `wait`).
- Для проверки после возврата `run()`, завершилась ли конкретная корутина нормально.
- Для реализации паттерна «запустил и забыл», где вы изредка проверяете, продолжается ли фоновая работа.

### Примеры

```nim
import coro

let h = start(proc() =
  suspend(0.2)
  echo "готово"
)

# Изнутри другой корутины — ручной опрос:
start(proc() =
  while alive(h):
    echo "ещё работает..."
    suspend(0.05)
  echo "завершилась"
)

run()

# После возврата run():
echo alive(h)   # false — планировщик уже очистил
```

## `wait`

```nim
proc wait*(c: CoroutineRef, interval = 0.01)
```

### Что делает

Приостанавливает *вызывающую* корутину и многократно проверяет, завершилась ли корутина `c`, опрашивая каждые `interval` секунд. Возвращается только после того, как `c` перестаёт быть живой.

Внутренне это тонкий цикл:

```nim
while alive(c):
  suspend(interval)
```

`wait` должна вызываться *изнутри* корутины (внутри она вызывает `suspend`). Вызов из главного потока вне контекста `run` приведёт к зависанию.

**`interval`** — частота опроса в секундах. Дефолтное значение `0.01` (10 мс) — разумный баланс между отзывчивостью и нагрузкой на CPU. Используйте меньшее значение для низкой задержки; большее — если ожидаемая задача работает много секунд.

### Когда использовать

Используйте `wait`, когда одна корутина зависит от результата или побочных эффектов другой и не должна продолжаться до её завершения. Это корутинный аналог объединения потоков (`join`).

**Не** вызывайте `wait(c)` на собственном `CoroutineRef` — это создаст бесконечный цикл.

### Примеры

```nim
import coro

# Пример 1 — последовательная зависимость между двумя корутинами
proc setup() =
  echo "инициализация..."
  suspend(0.1)
  echo "инициализация завершена"

proc mainTask(setupRef: CoroutineRef) =
  wait(setupRef)    # блокируем до завершения setup
  echo "главная задача после инициализации"

let s = start(setup)
start(proc() = mainTask(s))
run()
# Гарантированный порядок: "инициализация завершена" раньше "главная задача после инициализации"

# Пример 2 — паттерн fan-out / fan-in
proc worker(id: int) =
  suspend(float(id) * 0.05)
  echo "worker ", id, " завершён"

proc coordinator() =
  var handles: seq[CoroutineRef]
  for i in 1..4:
    handles.add(start(proc() = worker(i)))
  for h in handles:
    wait(h)    # ждём каждого worker по порядку
  echo "все worker'ы завершены"

start(coordinator)
run()
```

## Полный рабочий пример

```nim
# компиляция: nim c -d:nimCoroutines -r pipeline.nim

import coro

# Простой конвейер: производитель → преобразователь → потребитель.
# Каждая стадия — корутина, обмен через общий seq.

var buffer: seq[int]
var transformBuffer: seq[int]
var done = false

proc producer() =
  for i in 1..10:
    buffer.add(i)
    echo "произведено: ", i
    suspend(0.01)   # уступаем после каждого элемента
  done = true

proc transformer() =
  var consumed = 0
  while not done or consumed < buffer.len:
    if consumed < buffer.len:
      let val = buffer[consumed] * buffer[consumed]  # возводим в квадрат
      transformBuffer.add(val)
      inc consumed
    suspend(0.005)

proc consumer() =
  var printed = 0
  while not done or printed < transformBuffer.len:
    if printed < transformBuffer.len:
      echo "потреблено: ", transformBuffer[printed]
      inc printed
    suspend(0.005)

let p = start(producer)
let t = start(transformer)
let c = start(consumer)

# Корутина-наблюдатель ждёт все стадии и сообщает о завершении
start(proc() =
  wait(p)
  wait(t)
  wait(c)
  echo "Конвейер завершён."
)

run()
```

## Ограничения и оговорки

**Однопоточность.** Все корутины, созданные в рамках одного вызова `run`, разделяют один поток ОС. Запустить корутины сразу в нескольких потоках нельзя — `ctx` является thread-local переменной, поэтому каждый поток имеет собственный независимый планировщик и список корутин.

**Только кооперативное планирование.** Корутина, которая никогда не вызывает `suspend`, заблокирует все остальные корутины на неограниченное время, выморив их. Всегда вызывайте `suspend` в любой долго работающей или циклической корутине.

**Размер стека фиксируется при создании.** Расширить стек корутины после `start` невозможно. Переполнение стека — неопределённое поведение (скорее всего, краш). Для корутин с тяжёлыми вызовами передавайте в `start` больший `stacksize`.

**Исключения в корутинах.** Необработанные исключения внутри корутины перехватываются `runCurrentTask`, который печатает сообщение и трассировку стека, затем помечает корутину как завершённую. Исключение **не** распространяется на вызывающую сторону или другие корутины.

**`wait` должна вызываться внутри `run`.** Вызов `wait` (или `suspend`) вне контекста `run` — неопределённое поведение, так как нет цикла планировщика, которому можно вернуть управление.

**Ограничение платформы для бэкенда setjmp.** Бэкенд `setjmp` поддерживает только x86 и x86-64. На других архитектурах используйте `ucontext` (POSIX) или `fibers` (Windows).

**Нестабильный API.** Модуль явно помечен как имеющий нестабильный API. Зафиксируйте версию Nim, если зависите от него.
