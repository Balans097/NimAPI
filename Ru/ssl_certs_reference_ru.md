# Справочник модуля `ssl_certs`

> **Модуль:** `std/ssl_certs`  
> **Назначение:** Поиск файлов и каталогов с доверенными CA-сертификатами SSL/TLS на хосте, чтобы OpenSSL-соединения могли проверять подлинность серверов.

---

## Общее описание

Когда Nim-программа открывает HTTPS-соединение (или любое TLS-соединение), нижележащая библиотека OpenSSL должна убедиться, что сертификат сервера подписан доверенным удостоверяющим центром (Certificate Authority, CA). Для этого ей нужен файл или каталог с доверенными корневыми CA-сертификатами — *хранилище доверия* (trust store).

Проблема в том, что хранилище доверия находится в разных местах в каждой ОС и дистрибутиве Linux. `ssl_certs` решает её: в модуле жёстко прописаны канонические пути поиска для каждой поддерживаемой платформы, а единственный итератор `scanSSLCertificates` обходит их и выдаёт те, которые реально существуют на текущей машине.

Этот модуль используется внутри модулей Nim `httpclient` и `net` для автоматической настройки CA-сертификатов. Вы также можете использовать его напрямую, если строите собственные TLS-соединения.

### Что находит итератор

Итератор выдаёт пути двух видов:

- **Единый файл-бандл** — файл `.pem` или `.crt`, содержащий все доверенные CA-сертификаты, сконкатенированные вместе (наиболее распространённый вид на Linux и macOS).
- **Каталог сертификатов** — каталог, где каждый сертификат хранится отдельным файлом, именованным по его хешу (формат хеш-каталога OpenSSL, используемый в некоторых дистрибутивах Linux).

### Переопределение через переменные окружения

При вызове с `useEnvVars = true` итератор проверяет две стандартные переменные окружения прежде, чем обратиться к встроенным путям:

| Переменная | Действие |
|---|---|
| `SSL_CERT_FILE` | Использовать этот единственный файл как источник сертификатов. Итератор выдаёт его и завершается. |
| `SSL_CERT_DIR` | Сканировать все файлы внутри этого каталога. Итератор выдаёт каждый найденный файл. |

Эти переменные — стандартный механизм для указания OpenSSL-программам нестандартного хранилища доверия: например, внутри контейнера, виртуального окружения или корпоративной сети с собственным CA.

## Пути поиска сертификатов по платформам

Модуль компилирует разный список путей-кандидатов в зависимости от целевой ОС. Каждый путь проверяется во время выполнения; несуществующие пути молча пропускаются. Пути проверяются в порядке их перечисления.

### macOS

| Путь | Источник |
|---|---|
| `/etc/ssl/cert.pem` | Системный бандл |
| `/System/Library/OpenSSL/certs/cert.pem` | Устаревшее расположение OpenSSL |

### Linux

| Путь | Дистрибутивы / примечания |
|---|---|
| `/etc/ssl/certs/ca-certificates.crt` | Debian, Ubuntu, Arch, SUSE, Gentoo, NetBSD |
| `/etc/ssl/ca-bundle.pem` | OpenSUSE |
| `/etc/pki/tls/certs/ca-bundle.crt` | Red Hat 5+, Fedora, CentOS |
| `/usr/share/ssl/certs/ca-bundle.crt` | Red Hat 4 |
| `/etc/pki/tls/certs` | Fedora/RHEL (каталог) |
| `/data/data/com.termux/files/usr/etc/tls/cert.pem` | Termux (Android) |
| `/system/etc/security/cacerts` | Android (каталог) |

### BSD (FreeBSD, OpenBSD, NetBSD)

| Путь | Источник |
|---|---|
| `/etc/ssl/certs/ca-certificates.crt` | BSD-дистрибутивы на основе Debian, NetBSD |
| `/usr/local/share/certs/ca-root-nss.crt` | FreeBSD (пакет `security/ca-root-nss`) |
| `/etc/ssl/cert.pem` | OpenBSD, симлинк FreeBSD |
| `/usr/local/share/certs` | FreeBSD (каталог) |
| `/etc/openssl/certs` | NetBSD (каталог) |

### Windows

На Windows модуль **не** использует системное хранилище сертификатов Windows. Вместо этого он ищет файл с именем `cacert.pem`:

1. Сначала в каталоге запускаемого исполняемого файла (`getAppDir()`).
2. Затем в каждом каталоге из переменной окружения `PATH` (снимая окружающие кавычки у записей в кавычках).

Файл `cacert.pem` — это обычно [Mozilla CA bundle](https://curl.se/docs/caextract.html), распространяемый вместе со многими сборками OpenSSL для Windows (например, через curl, Git for Windows или загруженный вручную). Если файл не найден, итератор ничего не выдаёт.

### Haiku

Использует нативный API Haiku `find_paths_etc` (из `<FindDirectory.h>`) для поиска `ssl/CARootCertificates.pem` в системном каталоге данных. Память, возвращённая API, освобождается автоматически через `defer`.

## Экспортируемые символы

---

### `scanSSLCertificates`

```nim
iterator scanSSLCertificates*(useEnvVars = false): string
```

#### Что делает

Итератор, выдающий пути к файлам CA-сертификатов SSL/TLS (а для хранилищ в формате хеш-каталога — также сам путь к каталогу, за которым следуют пути к отдельным файлам внутри него), которые существуют на текущей системе. Вызывающий код получает один путь за каждый `yield` и может передать его в OpenSSL или любой TLS API, принимающий путь к бандлу сертификатов.

Итератор **не проверяет** синтаксическую корректность выданных файлов как PEM/DER-данных — он только проверяет существование файла или каталога. При подаче повреждённого файла в OpenSSL ошибку выдаст уже OpenSSL.

#### Дерево решений

Итератор следует этой логике по порядку, останавливаясь на первой применимой ветке:

```
useEnvVars = true И SSL_CERT_FILE установлена?
  → выдать SSL_CERT_FILE, завершить.

useEnvVars = true И SSL_CERT_DIR установлена?
  → выдать каждый файл внутри SSL_CERT_DIR/*, завершить.

Windows (переменные окружения не сработали)?
  → искать cacert.pem в каталоге приложения, затем в PATH.

Haiku?
  → вызвать find_paths_etc, выдать результаты.

Всё остальное (Linux, macOS, BSD)?
  → для каждого пути из встроенного списка certificatePaths:
      если заканчивается на .pem или .crt и файл существует → выдать.
      если это каталог:
        если в нём есть файлы *.0 → выдать путь к каталогу.
        выдать каждый файл в каталоге.
```

#### Параметры

| Имя | Тип | Умолч. | Описание |
|---|---|---|---|
| `useEnvVars` | `bool` | `false` | При `true` проверяет переменные окружения `SSL_CERT_FILE` и `SSL_CERT_DIR` перед обращением к встроенным путям. При `false` переменные окружения полностью игнорируются. |

#### Выдаваемое значение

`string` — путь к файлу сертификата в файловой системе, или (для хранилищ в формате хеш-каталога) путь к каталогу, после которого выдаются пути к отдельным файлам внутри него.

#### Примеры

```nim
import std/ssl_certs

# Вывести все источники CA-сертификатов на этой системе
for path in scanSSLCertificates():
  echo path
# Пример вывода на Ubuntu:
# /etc/ssl/certs/ca-certificates.crt
```

```nim
import std/ssl_certs

# Учитывать переменные окружения — полезно в контейнерах и CI
for path in scanSSLCertificates(useEnvVars = true):
  echo path
# Если SSL_CERT_FILE=/etc/my-custom-ca.pem установлена:
# /etc/my-custom-ca.pem
```

```nim
import std/ssl_certs

# Найти первый доступный бандл сертификатов и остановиться
var firstCert = ""
for path in scanSSLCertificates(useEnvVars = true):
  firstCert = path
  break

if firstCert.len > 0:
  echo "Используем CA-бандл: ", firstCert
else:
  echo "Предупреждение: CA-сертификаты не найдены на этой системе!"
```

```nim
import std/ssl_certs, std/net

# Ручная настройка TLS-контекста с использованием первого найденного CA-бандла
# (примерно так это делает httpclient внутри)
let ctx = newContext(caFile = block:
  var f = ""
  for path in scanSSLCertificates(useEnvVars = true):
    f = path
    break
  f
)
```

```nim
import std/ssl_certs

# Собрать все найденные пути сертификатов в последовательность
var certs: seq[string]
for path in scanSSLCertificates():
  certs.add(path)

echo "Найдено источников сертификатов: ", certs.len
for c in certs:
  echo "  ", c
```

```nim
import std/ssl_certs, std/os

# Паттерн переопределения через переменную окружения — удобно для тестов с локальным CA
putEnv("SSL_CERT_FILE", "/tmp/test-ca.pem")
for path in scanSSLCertificates(useEnvVars = true):
  echo path   # → /tmp/test-ca.pem
```

## Типичные сценарии использования

### Совместная работа с `httpclient`

`httpclient` Nim вызывает `scanSSLCertificates` автоматически при создании SSL-контекста по умолчанию. Обычно вам не нужно вызывать итератор напрямую — только если вы создаёте собственный `SslContext`:

```nim
import std/[httpclient, net, ssl_certs]

# Ручная настройка контекста — найти сертификаты, затем настроить
let sslCtx = newContext()
for certPath in scanSSLCertificates(useEnvVars = true):
  sslCtx.wrapSocket(...)  # платформо-зависимая обвязка
  break
```

### Диагностика ошибок TLS-сертификатов

Если программа падает с ошибкой TLS-верификации и вы подозреваете отсутствие CA-сертификатов, переберите `scanSSLCertificates`, чтобы увидеть, что находит библиотека:

```nim
import std/ssl_certs

echo "Источники сертификатов, видимые Nim на этой системе:"
var count = 0
for p in scanSSLCertificates(useEnvVars = true):
  echo "  [", count, "] ", p
  inc count
if count == 0:
  echo "  (ничего не найдено — установите пакет CA-сертификатов или задайте SSL_CERT_FILE)"
```

### Контейнеры и минимальные окружения

Docker-образы на основе `alpine`, `scratch` и других минималистичных баз часто не содержат CA-сертификатов. Рекомендуемое решение — установить пакет `ca-certificates` в `Dockerfile`. Альтернатива — примонтировать бандл с хоста и указать на него:

```sh
SSL_CERT_FILE=/host-certs/ca-bundle.pem ./your_nim_binary
```

## Замечания по дизайну и типичные ловушки

**1. По умолчанию `useEnvVars = false`**  
Переменные окружения намеренно игнорируются, пока вызывающий код явно не включит их. Это предотвращает скрытое влияние пользовательского окружения на программы, которые не ожидают этого. Всегда передавайте `useEnvVars = true` в долгоживущих сервисах и инструментах, где оператор должен иметь возможность переопределить хранилище доверия.

**2. Итератор может не выдать ни одного пути**  
На только что установленной минимальной системе (или внутри контейнера без пакетов CA) ни один из встроенных путей может не существовать. Итератор молча завершится без единого `yield`. Всегда проверяйте, что получен хотя бы один путь, перед выполнением TLS-операций.

**3. На Windows системное хранилище сертификатов не используется**  
Модуль не интегрируется с хранилищем сертификатов Windows (`CertOpenSystemStore` / `CertEnumCertificatesInStore`). Этот закомментированный код присутствует в источнике, но не компилируется. Вы должны предоставить файл `cacert.pem` рядом с исполняемым файлом или где-либо в `PATH`, либо задать `SSL_CERT_FILE`. Это известное ограничение.

**4. Для каталогов выдаются и путь к каталогу, и пути к файлам**  
Для хранилищ в формате хеш-каталога (например, `/etc/pki/tls/certs` на Fedora/RHEL) итератор сначала выдаёт сам путь к каталогу (если в нём есть файлы `*.0`), а затем — каждый файл внутри каталога. Потребители должны корректно обрабатывать и строку с путём к каталогу, и строки с путями к файлам.

**5. Существование файла проверяется; содержимое — нет**  
Итератор проверяет `fileExists` и `dirExists`, но не открывает, не разбирает и не валидирует данные сертификата. Файл нулевого размера или повреждённый PEM будет выдан, а ошибка возникнет только тогда, когда OpenSSL попытается его загрузить.

**6. `SSL_CERT_DIR` и `SSL_CERT_FILE` взаимно исключают друг друга**  
При `useEnvVars = true` переменная `SSL_CERT_FILE` имеет абсолютный приоритет. Если она установлена, `SSL_CERT_DIR` не проверяется вовсе — даже если тоже установлена. Если вам нужны и каталог, и файл, объедините их в один файл-бандл и укажите на него через `SSL_CERT_FILE`.
