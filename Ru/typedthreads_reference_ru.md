# Справочник модуля `typedthreads`

> **Назначение модуля:** `typedthreads` — базовый низкоуровневый модуль управления потоками в стандартной библиотеке Nim. Он предоставляет тип `Thread[TArg]` и все процедуры, необходимые для создания потоков, получения информации об их состоянии, синхронизации и привязки к ядрам процессора.
>
> 💡 **Рекомендация самого модуля:** Для параллелизма на уровне приложений предпочтительнее использовать высокоуровневые библиотеки: `malebolgia`, `taskpools` или `weave`. `typedthreads` используется напрямую, когда нужен тонкий контроль или создаётся собственная абстракция над параллелизмом.

---

## Критически важные правила безопасности памяти

Прежде чем использовать потоки в Nim, необходимо понять два правила:

**Правило 1 — Нельзя делиться GC-управляемой памятью между потоками (кроме `arc`/`orc`/`boehm`).**
Сборщик мусора Nim по умолчанию (`orc`) поддерживает разделяемую кучу, но старые стратегии вроде `refc` используют локальные кучи для каждого потока. Передача значений `ref`, `seq`, `string` и других объектов, управляемых GC, между потоками при компиляции с `refc` — **неопределённое поведение**. Используйте глобальные переменные, сырые указатели или память, выделенную через `allocShared`.

**Правило 2 — Аргументы потока копируются.**
Аргумент `param`, переданный в `createThread`, копируется во внутреннее хранилище потока. Изменения аргумента внутри потока не затрагивают оригинал. Для доступа к разделяемому изменяемому состоянию передавайте `ptr` на общую память.

## Справочник по типам

### `Thread[TArg]`

```nim
type Thread[TArg] = object  # реэкспортируется из std/private/threadtypes
```

Представляет один поток ОС. Это обобщённый тип, параметризованный `TArg` — типом данных, передаваемых в процедуру-точку входа потока. `TArg` может быть `void`, если поток не нуждается в аргументе.

Значение `Thread` должно быть объявлено как `var` (изменяемое). Его следует рассматривать как непрозрачный дескриптор: не копировать и не перемещать после вызова `createThread`.

```nim
var t: Thread[int]                    # поток, получающий один int
var t2: Thread[tuple[a, b: int]]      # поток, получающий кортеж
var t3: Thread[void]                  # поток без аргумента
```

## Справочник по процедурам

---

### `createThread` (с аргументом)

```nim
proc createThread*[TArg](t: var Thread[TArg],
                          tp: proc (arg: TArg) {.thread, nimcall.},
                          param: TArg)
```

#### Что делает

Создаёт новый поток ОС, сохраняет его дескриптор в `t` и немедленно запускает параллельное выполнение `tp(param)`. Вызов возвращается сразу после успешного запуска потока — он не ожидает завершения `tp`.

`param` **копируется** во внутреннее хранилище потока до его начала. Если ОС не может создать поток (нехватка ресурсов), выбрасывается `ResourceExhaustedError`.

#### Прагма `{.thread.}`

Процедура-точка входа `tp` **обязана** быть аннотирована прагмой `{.thread.}`. Она инициализирует потоко-локальное состояние GC, указывает правильное соглашение о вызове для платформы (`stdcall` на Windows, `noconv` на POSIX) и запрещает вызов как обычного Nim-proc. Отсутствие `{.thread.}` — ошибка компиляции.

#### Описание параметров

| Параметр | Тип | Описание |
|---|---|---|
| `t` | `var Thread[TArg]` | Объект потока для инициализации. Должен быть объявлен как `var`. |
| `tp` | `proc (arg: TArg) {.thread, nimcall.}` | Процедура-точка входа. Обязательна прагма `{.thread.}`. |
| `param` | `TArg` | Аргумент, передаваемый в `tp`. Копируется в хранилище потока. |

#### Примеры

Запуск потока, получающего простое значение:

```nim
import std/typedthreads

proc worker(n: int) {.thread.} =
  echo "Поток получил: ", n

var t: Thread[int]
createThread(t, worker, 42)
joinThread(t)
```

Запуск нескольких потоков с кортежами:

```nim
import std/typedthreads, std/locks

var L: Lock
initLock(L)

proc rangeEcho(interval: tuple[a, b: int]) {.thread.} =
  for i in interval.a .. interval.b:
    withLock L: echo i

var threads: array[5, Thread[tuple[a, b: int]]]
for i in 0 .. high(threads):
  createThread(threads[i], rangeEcho, (i * 10, i * 10 + 5))
joinThreads(threads)
deinitLock(L)
```

Разделение изменяемых данных через указатель (требуется `arc` или `orc`):

```nim
import std/typedthreads, std/locks

var L: Lock
initLock(L)

proc accumulate(s: ptr seq[int]) {.thread.} =
  withLock L:
    for i in 0 ..< 10:
      s[].add(i)

var data = newSeq[int]()
var threads: array[3, Thread[ptr seq[int]]]
for i in 0 .. high(threads):
  createThread(threads[i], accumulate, data.addr)
joinThreads(threads)
echo data
deinitLock(L)
```

### `createThread` (void, без аргумента)

```nim
proc createThread*(t: var Thread[void], tp: proc () {.thread, nimcall.})
```

#### Что делает

Удобная перегрузка для потоков, которым не нужен никакой аргумент. Эквивалентна `createThread[void](t, tp)`, но не требует явного указания параметра типа `[void]`.

#### Пример

```nim
import std/typedthreads

proc background() {.thread.} =
  echo "Выполняется в фоне без аргументов"

var t: Thread[void]
createThread(t, background)
joinThread(t)
```

### `joinThread`

```nim
proc joinThread*[TArg](t: Thread[TArg]) {.inline.}
```

#### Что делает

Блокирует вызывающий поток до тех пор, пока поток `t` не завершит выполнение. После возврата из `joinThread` все записи, сделанные потоком `t`, гарантированно видны вызывающему коду (устанавливается отношение happens-before).

Внутри использует `WaitForSingleObject` на Windows и `pthread_join` на POSIX.

Всегда соединяйте созданные потоки, прежде чем они (или связанные с ними данные) выйдут из области видимости. Уничтожение объекта `Thread` без предварительного `joinThread`, пока поток ещё выполняется, — неопределённое поведение.

#### Пример

```nim
import std/typedthreads

proc slowWork(n: int) {.thread.} =
  for i in 0 ..< n: discard

var t: Thread[int]
createThread(t, slowWork, 1_000_000)
# ... можно делать другие вещи здесь ...
joinThread(t)
echo "Поток завершён"
```

### `joinThreads`

```nim
proc joinThreads*[TArg](t: varargs[Thread[TArg]])
```

#### Что делает

Ожидает завершения **каждого** потока из списка `varargs`. Эквивалентна последовательному вызову `joinThread` для каждого потока, но на Windows использует `WaitForMultipleObjects` батчами по 64 потока — это эффективнее последовательных ожиданий при большом числе потоков.

На Windows API `WaitForMultipleObjects` обрабатывает не более 64 дескрипторов за раз (`MAXIMUM_WAIT_OBJECTS = 64`). `joinThreads` автоматически разбивает большие массивы на батчи, поэтому передавать массивы любого размера безопасно.

#### Пример

```nim
import std/typedthreads

proc task(id: int) {.thread.} =
  echo "Задача ", id, " выполнена"

var workers: array[10, Thread[int]]
for i in 0 .. high(workers):
  createThread(workers[i], task, i)

joinThreads(workers)
echo "Все задачи выполнены"
```

### `running`

```nim
proc running*[TArg](t: Thread[TArg]): bool {.inline.}
```

#### Что делает

Возвращает `true`, если поток `t` в данный момент выполняется (его процедура-точка входа была установлена и ещё не вернула управление). Возвращает `false`, если поток не был запущен или уже завершился.

Проверка не является точкой синхронизации — это лёгкий опрос внутреннего поля `dataFn` потока. Из-за особенностей планировщика и видимости памяти результат `false` не гарантирует, что побочные эффекты потока видны вызывающему. Для этой гарантии используйте `joinThread`.

#### Пример

```nim
import std/typedthreads

proc longTask(n: int) {.thread.} =
  for i in 0 ..< n: discard

var t: Thread[int]
createThread(t, longTask, 100_000_000)

if running(t):
  echo "Поток ещё выполняется..."

joinThread(t)
echo "Теперь завершён. running = ", running(t)  # false
```

### `handle`

```nim
proc handle*[TArg](t: Thread[TArg]): SysThread {.inline.}
```

#### Что делает

Возвращает внутренний дескриптор ОС (`SysThread`) для потока `t`. На Windows это `HANDLE`, на POSIX — `pthread_t`.

Это **аварийный выход** для взаимодействия с платформенно-специфичными API потоков, не охваченными данным модулем — например, для установки приоритетов реального времени, запроса идентификаторов потока через системные вызовы или передачи дескриптора в стороннюю C-библиотеку. В обычных обстоятельствах `handle` не нужен.

#### Пример

```nim
import std/typedthreads

proc work(n: int) {.thread.} =
  for i in 0 ..< n: discard

var t: Thread[int]
createThread(t, work, 1_000_000)

let h = handle(t)
# h — SysThread (HANDLE на Windows, pthread_t на POSIX)
# передаём в платформенно-специфичные вызовы ОС при необходимости...

joinThread(t)
```

### `pinToCpu`

```nim
proc pinToCpu*[Arg](t: var Thread[Arg]; cpu: Natural)
```

#### Что делает

Устанавливает **привязку к CPU** (CPU affinity) потока `t` к логическому ядру процессора с номером `cpu` (нумерация с нуля). После этого вызова планировщик ОС будет запускать поток только на указанном ядре.

Привязка к ядру может улучшить локальность кэша (рабочий набор потока остаётся в кэше L1/L2 одного ядра) и снизить джиттер в приложениях, чувствительных к задержкам. При неосторожном использовании может нарушить балансировку нагрузки планировщика.

#### Поведение на разных платформах

| Платформа | Механизм | Примечания |
|---|---|---|
| Windows | `SetThreadAffinityMask` | `cpu` отображается в один бит маски привязки |
| Linux | `pthread_setaffinity_np` через `cpuset` | Полное управление `cpu_set_t` |
| macOS | Заглушка (no-op) | macOS не предоставляет привязку потоков на этом уровне |
| Genode | Заглушка (выдаётся hint) | Привязка задаётся при создании потока, а не после |

#### Пример

```nim
import std/typedthreads

proc worker(n: int) {.thread.} =
  for i in 0 ..< n: discard

var t: Thread[int]
createThread(t, worker, 500_000_000)
pinToCpu(t, 0)   # привязать к ядру 0
joinThread(t)
```

> ⚠️ Используйте `pinToCpu` только если вы измерили, что эффекты кэша или джиттер планировщика действительно являются проблемой. Необоснованная привязка нередко *ухудшает* производительность, не позволяя ОС перемещать потоки на простаивающие ядра.

## Настройка размера стека потоков

На ресурсоограниченных платформах (Zephyr, FreeRTOS, NuttX, 8-/16-битные CPU) размеры стека переопределяются при компиляции:

| Определение при компиляции | По умолчанию | Описание |
|---|---|---|
| `-d:nimThreadStackSize=N` | `8192` | Полный размер стека в байтах для встроенных/RTOS платформ |
| `-d:nimThreadStackGuard=N` | `128` | Размер защитной страницы стека в байтах |

На стандартных настольных/серверных платформах размер стека составляет приблизительно **2 МБ** на поток на 64-битных системах (1 МБ на 32-битных), за вычетом 4 КБ защитной страницы.

## Управление памятью и потоки

| Стратегия | Разделяемая куча? | Передача `ref`/`seq`/`string`? | Примечания |
|---|---|---|---|
| `orc` (по умолчанию) | ✅ Да | ✅ Безопасно | Подсчёт ссылок с обнаружением циклов |
| `arc` | ✅ Да | ✅ Безопасно | Подсчёт ссылок без обнаружения циклов |
| `boehm` | ✅ Да | ✅ Безопасно | Консервативный GC |
| `refc` | ❌ Нет | ❌ Небезопасно | Локальные кучи для каждого потока; использовать `ptr` + `allocShared` |

## Полный пример

```nim
import std/typedthreads, std/locks

var
  results: seq[int]
  L: Lock

initLock(L)

proc computeRange(bounds: tuple[lo, hi: int]) {.thread.} =
  var localSum = 0
  for i in bounds.lo .. bounds.hi:
    localSum += i
  withLock L:
    results.add(localSum)

const nThreads = 4
var threads: array[nThreads, Thread[tuple[lo, hi: int]]]

for i in 0 ..< nThreads:
  createThread(threads[i], computeRange, (lo: i * 250, hi: (i + 1) * 250 - 1))
  pinToCpu(threads[i], i)

joinThreads(threads)
deinitLock(L)

echo "Частичные суммы: ", results
```

## Сводная таблица

| Процедура | Назначение | Блокирует? |
|---|---|---|
| `createThread` | Запустить поток с типизированным аргументом | Нет |
| `createThread` (void) | Запустить поток без аргумента | Нет |
| `joinThread` | Дождаться завершения одного потока | Да |
| `joinThreads` | Дождаться завершения всех потоков массива | Да |
| `running` | Опросить, выполняется ли поток | Нет |
| `handle` | Получить сырой дескриптор потока ОС | Нет |
| `pinToCpu` | Привязать поток к конкретному ядру CPU | Нет |

## Важные замечания

- Объекты `Thread` нельзя копировать после вызова `createThread`. Обращайтесь с ними как с уникальными, неперемещаемыми дескрипторами на протяжении всего времени жизни потока.
- Публичной процедуры `destroyThread` нет. Реализация существует в исходном коде, но отключена (`when false`), поскольку принудительное завершение потока принципиально небезопасно. Проектируйте потоки так, чтобы они завершались естественным образом.
- Для межпоточного взаимодействия используйте `Channel` из `system` или высокоуровневый примитив. Никогда не читайте и не записывайте глобальную изменяемую переменную из нескольких потоков без `Lock` — это гонка данных.
- Этот модуль недоступен при компиляции с `--os:any`. Попытка его импортировать в таком контексте приводит к ошибке компиляции.
