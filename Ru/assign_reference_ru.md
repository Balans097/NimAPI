# Справочник модуля `assign`

> **Модуль:** `assign.nim` — стандартная библиотека Nim  
> **Назначение:** Низкоуровневый движок для копирования, инициализации и сброса значений Nim любой сложности. Каждый оператор присваивания в вашей программе — `a = b` — в итоге вызывает одну из процедур этого модуля, если тип содержит управляемую память (строки, последовательности, ссылки, объекты с вариантными полями). Именно этот модуль стоит в основе гарантий безопасной работы с памятью в Nim.

---

## Общая картина

В Nim простые типы-значения (целые числа, числа с плавающей точкой, `char`) можно скопировать единственным вызовом `copyMem`. Но как только тип содержит `string`, `seq`, `ref` или вариантный объект (`case`), среда исполнения должна сделать больше: глубоко скопировать данные на куче, уведомить сборщик мусора о новых ссылках, корректно сбросить прежнее содержимое памяти, а также безопасно обработать самоприсваивание (`x = x`).

`assign.nim` реализует эту логику обобщённо, опираясь на информацию о типах в режиме исполнения (RTTI, представленную в виде `PNimType` / `TNimNode`). Компилятор вставляет вызовы этих процедур автоматически — вы никогда не вызываете их из прикладного кода.

## Экспортируемые (компилятор-ориентированные) процедуры

---

### `genericAssign`

```nim
proc genericAssign(dest, src: pointer, mt: PNimType)
```

**Что делает:**  
Выполняет **глубокое копирование** значения из `src` в память по адресу `dest`, руководствуясь дескриптором типа `mt`. «Глубокое» означает, что каждый компонент на куче (строки, последовательности, вложенные ссылки) независимо дублируется на куче — после копирования `dest` и `src` не разделяют никаких данных на куче.

Именно эту процедуру компилятор вызывает для обычного оператора `=`, когда тип требует нетривиального копирования. Она полностью делегирует работу `genericAssignAux` с параметром `shallow = false`.

**Как работает внутри:**  
`genericAssign` переключается по `mt.kind`:
- `tyString` — копирует буфер строки на куче (через `copyString` / `nimAsgnStrV2`)
- `tySequence` — выделяет новый payload, затем рекурсивно копирует каждый элемент
- `tyObject` — обходит цепочку наследования, копирует каждое поле, затем записывает указатель на статический тип в заголовок объекта
- `tyTuple` — копирует каждое поле
- `tyArray` — копирует каждый элемент по отдельности
- `tyRef` — копирует саму ссылку (указатель), уведомляя GC через `unsureAsgnRef`
- Всё остальное — сырой `copyMem` (тип не содержит управляемых указателей)

**Почему важно:**  
Без семантики глубокого копирования изменение последовательности в одной переменной молча испортило бы другую, присвоенную из первой. `genericAssign` обеспечивает семантику значений для всех типов Nim.

**Пример (с точки зрения программиста на Nim):**
```nim
var a = @[1, 2, 3]
var b = a           # компилятор вызывает genericAssign
b.add(4)
echo a  # @[1, 2, 3]  — a не изменилась; b получила собственную копию на куче
echo b  # @[1, 2, 3, 4]
```

### `genericShallowAssign`

```nim
proc genericShallowAssign(dest, src: pointer, mt: PNimType)
```

**Что делает:**  
Выполняет **поверхностное копирование** значения из `src` в `dest`. Для строк и последовательностей вместо дублирования данных на куче копируется только заголовок (указатель + длина) — в итоге `dest` и `src` разделяют один и тот же буфер на куче.

Это процедура, вызываемая при использовании `shallowCopy` или прагмы `shallow` на типе. Делегирует в `genericAssignAux` с `shallow = true`.

**Принципиальный компромисс:**  
Поверхностное присваивание значительно быстрее для больших последовательностей и строк (нет выделения памяти, нет поэлементного копирования), но создаёт алиасинг: изменение последовательности через одну переменную изменяет и другую. Это сознательно, и пользователь должен явно это запросить.

**Когда среда исполнения сама делает присваивание поверхностным:**  
Строка или последовательность, у которой в поле `reserved` установлен флаг `seqShallowFlag`, всегда копируется поверхностно даже через `genericAssign`. Это происходит со строковыми литералами и compile-time константами — они находятся в памяти только для чтения и не должны дублироваться.

**Пример:**
```nim
var a = @[1, 2, 3]
shallowCopy(b, a)  # компилятор вызывает genericShallowAssign
# b.p указывает на тот же payload, что и a.p
# Изменение b также изменит a — используйте осторожно
```

### `genericSeqAssign`

```nim
proc genericSeqAssign(dest, src: pointer, mt: PNimType)
```

**Что делает:**  
Специализированная обёртка над `genericAssign` для ситуаций, когда значение последовательности передаётся по значению (не через указатель) — например, когда компилятор держит значение последовательности на стеке, а не за указателем. Перед вызовом `genericAssign` берёт адрес `src`, устраняя несоответствие.

Комментарий в исходнике (`# ugly, but I like to stress the parser sometimes :-)`) от Андреаса Румпфа намекает, что это обходной манёвр для особенностей генерации C-кода в некоторых контекстах.

**Почему важно:**  
Без этого адаптера обобщённая машинерия присваивания (всегда работающая через указатели) не смогла бы принять голое значение последовательности. Это делает диспетчеризацию из сгенерированного C-кода единообразной.

**Пример (прозрачно для программиста на Nim):**
```nim
proc take(s: seq[int]) =
  var local = s  # в зависимости от контекста кодогенерации может быть вызван genericSeqAssign
  echo local
```

### `genericAssignOpenArray`

```nim
proc genericAssignOpenArray(dest, src: pointer, len: int, mt: PNimType)
```

**Что делает:**  
Копирует `len` элементов из открытого массива, начинающегося по адресу `src`, в память по адресу `dest`, элемент за элементом, используя `genericAssign` для каждого. Информация о типе элементов берётся из `mt.base`.

`openArray[T]` во время исполнения — это просто указатель + длина. Данная процедура умеет обходить эти элементы и глубоко копировать каждый из них.

**Почему важно:**  
Когда вы присваиваете или передаёте `openArray`, элементы которого являются управляемыми типами (например, `openArray[string]`), каждую строку необходимо скопировать индивидуально. Сырой `copyMem` по всему блоку продублировал бы заголовки-указатели, но не данные на куче — именно такой небезопасный алиасинг предотвращает эта функция.

**Пример:**
```nim
proc copyStrings(dest: var seq[string], src: openArray[string]) =
  dest.setLen(src.len)
  # внутри компилятор может вызвать genericAssignOpenArray
  # для копирования каждой строки из src в dest
  for i, s in src:
    dest[i] = s
```

### `objectInit`

```nim
proc objectInit(dest: pointer, typ: PNimType)
```

**Что делает:**  
Инициализирует только что выделенный объект или массив до нулевого/дефолтного состояния, с одним важным дополнительным шагом: записывает корректный **указатель на тип** в заголовок объекта. Именно этот указатель обеспечивает работу оператора `of` (проверка типа во время исполнения) и динамического диспетчеризирования методов.

Обрабатывает:
- `tyObject` — записывает указатель на информацию о типе в заголовок объекта, затем рекурсивно инициализирует все поля через `objectInitAux`
- `tyTuple` — рекурсивно инициализирует все поля
- `tyArray` — инициализирует каждый элемент рекурсивно
- Все остальные типы — ничего делать не нужно (нулевая память уже корректна)

**Почему важно:**  
Простого обнуления памяти недостаточно для объектов. Нулевой заголовок объекта означает «тип неизвестен» — любая последующая проверка `of` или диспетчеризация метода либо завершится ошибкой, либо даст неверный результат. `objectInit` гарантирует, что объект находится в корректном, типизированном состоянии с самого первого момента существования.

**Пример (прозрачно для программиста Nim, но принципиально важно):**
```nim
type Animal = ref object of RootObj
  name: string

type Dog = ref object of Animal
  breed: string

var d = new Dog
# После вызова `new` выполняется objectInit, который записывает
# указатель на тип Dog в заголовок объекта, так что `d of Dog`
# немедленно возвращает true
echo d of Dog     # true
echo d of Animal  # true (Dog наследует от Animal)
```

### `genericReset`

```nim
proc genericReset(dest: pointer, mt: PNimType)
```

**Что делает:**  
Сбрасывает значение по адресу `dest` до нулевого/дефолтного состояния, корректно освобождая все управляемые ресурсы, которые оно содержит. Именно эту процедуру вызывает стандартная библиотечная функция `reset(x)`.

Логика сброса по виду типа:
- `tyRef` — вызывает `unsureAsgnRef(dest, nil)`, сообщая GC об исчезновении ссылки
- `tyString` — освобождает буфер на куче (путь nimSeqsV2) или обнуляет GC-указатель
- `tySequence` — освобождает payload (путь nimSeqsV2) или обнуляет GC-указатель
- `tyTuple` — рекурсивно сбрасывает каждое поле через `genericResetAux`
- `tyObject` — рекурсивно сбрасывает каждое поле, затем **также обнуляет указатель на тип** в заголовке объекта (критично для вариантных объектов: сброс дискриминатора должен аннулировать старую ветку)
- `tyArray` — сбрасывает каждый элемент
- Всё остальное — `zeroMem` (сырое обнуление)

**Почему указатель на тип должен быть обнулён для объектов:**  
Вариантные (`case`) объекты в Nim могут менять активную ветку путём переприсваивания поля-дискриминатора. Прежде чем безопасно записать новую ветку, поля старой ветки должны быть сброшены. `genericReset` справляется с этим, обнуляя указатель на тип вместе с данными, оставляя объект в чистом, бестипном состоянии, готовом к повторной инициализации.

**Пример:**
```nim
var s = "hello, world"
reset(s)        # вызывает genericReset; буфер на куче освобождается
echo s == ""    # true — s теперь пустая строка по умолчанию

var nums = @[1, 2, 3]
reset(nums)
echo nums.len   # 0 — payload освобождён, len обнулён
```

### `FieldDiscriminantCheck`

```nim
proc FieldDiscriminantCheck(oldDiscVal, newDiscVal: int,
                            a: ptr array[0x7fff, ptr TNimNode],
                            L: int)
```

**Что делает:**  
Применяет правило безопасности Nim для **вариантных объектов**: нельзя присвоить новое значение дискриминатору, которое переключит объект на другую ветку `case`, если текущая ветка уже содержит ненулевые данные. Нарушение этого правила создаёт путаницу типов — поля старой ветки были бы переинтерпретированы как поля новой.

Проверка сравнивает, какой ветке соответствует `oldDiscVal` и какой — `newDiscVal`, используя `selectBranch`. Если они различаются, во время исполнения возбуждается `FieldDefect`.

**Аварийный люк `nimOldCaseObjects`:**  
С флагом `-d:nimOldCaseObjects` проверка немного мягче: исключение поднимается только если `oldDiscVal != 0` (ненулевое старое значение) *и* ветка меняется. Без этого флага (современный Nim) любая смена ветки вызывает исключение, даже из нуля — если предварительно не выполнить `reset` объекта.

**Почему важно:**  
Этот барьер обеспечивает безопасность памяти для вариантных объектов. Без него следующий код молча испортил бы память:

```nim
type Shape = object
  case kind: ShapeKind
  of Circle:
    radius: float      # 8 байт по некоторому смещению
  of Rect:
    width, height: float  # другие поля, та же область памяти

var s = Shape(kind: Circle, radius: 3.14)
s.kind = Rect  # ← здесь FieldDiscriminantCheck выбросит исключение!
               # биты radius были бы переинтерпретированы как width/height
```

**Правильный шаблон — сначала reset, потом присваивание:**
```nim
reset(s)                                       # очищает radius, обнуляет тип
s = Shape(kind: Rect, width: 10, height: 5)   # безопасная повторная инициализация
```

## Внутренние вспомогательные функции (не экспортируются, важны для понимания)

| Функция | Роль |
|---|---|
| `genericAssignAux(dest, src, mt, shallow)` | Центральный рекурсивный диспетчер: выбирает стратегию копирования по виду типа |
| `genericAssignAux(dest, src, n, shallow)` | Узловой диспетчер: обходит дерево `TNimNode` (слоты, списки, ветки case) |
| `deepSeqAssignImpl` | Шаблон, выделяющий новый payload последовательности и копирующий элементы; устраняет дублирование кода между двумя вариантами последовательностей |
| `genericResetAux` | Узловой диспетчер сброса, зеркально повторяющий `genericAssignAux` для пути сброса |
| `objectInitAux` | Узловой диспетчер инициализации, зеркально повторяющий `genericAssignAux` для пути инициализации |
| `selectBranch(discVal, L, a)` | По значению дискриминатора и таблице ветвей возвращает активный узел `TNimNode` (или ветку `else`) |

## Ключевые принципы проектирования

**Диспетчеризация через RTTI:** Все процедуры принимают `PNimType` (указатель на дескриптор типа в режиме исполнения) и переключаются по `mt.kind`. Компилятор генерирует и компонует эти дескрипторы для каждого типа в вашей программе. Никаких обобщений и шаблонов на стороне вызывающего кода — только помеченный типом указатель и большой оператор `case`.

**Сотрудничество с GC через `unsureAsgnRef`:** При каждом сохранении `ref` в новое место GC должен быть уведомлён, чтобы обновить свои счётчики ссылок или метки для прометки. `unsureAsgnRef` — единственный безопасный барьер записи для этой цели. «Unsure» (неопределённость) означает, что функция корректно обрабатывает как размещённые на куче, так и на стеке адреса назначения.

**Флаг shallow в заголовках последовательностей:** Бит `seqShallowFlag` в поле `reserved` заголовка последовательности сообщает среде исполнения: «этот payload не принадлежит нам — не копировать, не освобождать». Строковые литералы используют это, чтобы их можно было дёшево присваивать без выделения памяти на куче.

**Указатель типа в заголовке объекта:** Каждое значение `object` (на куче или на стеке) начинается с указателя на его дескриптор `PNimType` / `PNimTypeV2`. Именно это обеспечивает корректную работу `x of SomeType`, диспетчеризации `method` и `reset`. Как `objectInit`, так и `genericReset` поддерживают этот инвариант.

**Безопасность вариантных объектов:** Комбинация `FieldDiscriminantCheck` + `genericResetAux` обеспечивает безопасное с точки зрения памяти переназначение дискриминатора. Правило простое: сначала сброс, затем присваивание новой ветке. Nim применяет это правило во время исполнения (а компилятор нередко может обнаружить нарушения статически).
