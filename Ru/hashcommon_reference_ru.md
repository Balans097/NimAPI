# hashcommon — Справочное руководство

## Общее описание

`hashcommon.nim` — это **внутренний `include`-файл**, совместно используемый модулями `std/tables`, `std/sets` и `std/sharedtables`. Он не импортируется напрямую — он вставляется в подключающий модуль дословно во время компиляции через директиву `include`, делая все свои определения доступными как если бы они были написаны непосредственно в том файле.

Файл реализует **низкоуровневый движок хеш-таблицы с открытой адресацией**, который лежит в основе каждого хеш-контейнера стандартной библиотеки Nim. Понимание его механизмов объясняет поведение этих контейнеров: как выполняется зондирование слотов, когда запускается перехеширование, как обрабатываются нулевые хеш-коды, и почему одна функция поиска одновременно служит и как «найти ключ», и как «найти место для вставки».

### Открытая адресация в одном абзаце

Хеш-таблица с открытой адресацией — это плоский массив слотов. Чтобы найти ключ, вычисляется его хеш — начальный индекс слота — а затем происходит последовательный обход (зондирование) вперёд, пока не будет найден либо сам ключ, либо пустой слот (означающий отсутствие ключа). Указателей и отдельных цепочек нет — всё хранится в одном непрерывном массиве. Компромисс: массив никогда не должен быть слишком заполнен; при высоком коэффициенте загрузки коллизии становятся дорогостоящими. Логика роста и перехеширования в этом файле управляет данным ограничением.

## Константы

### `growthFactor`

```nim
const growthFactor = 2
```

**Что это:** множитель, применяемый к длине внутреннего массива данных каждый раз, когда запускается перехеширование. При нехватке места массив удваивается.

**Почему 2:** удвоение — канонический выбор для таблиц с открытой адресацией. Оно обеспечивает амортизированную стоимость вставок O(1) — каждый элемент перемещается O(log n) раз за всё время своего существования. Меньший множитель снижал бы пиковые расходы памяти, но увеличивал бы частоту перехеширований; больший — наоборот.

**Практическое следствие:** в худшем случае `Table` или `HashSet` занимают примерно вдвое больше памяти, чем нужно для хранения текущих элементов. Точное соотношение зависит от того, когда срабатывает `mustRehash`.

## Внутренние процедуры и шаблоны

### `isEmpty`

```nim
proc isEmpty(hcode: Hash): bool {.inline.}
```

**Назначение:** проверяет, является ли слот в массиве данных незанятым. Возвращает `true` тогда и только тогда, когда `hcode == 0`.

**Контракт кодирования:** хеш-таблица кодирует «пустой слот» как хранимое значение `hcode`, равное ровно нулю. Настоящие ключи никогда не могут иметь хеш равный нулю — шаблон `genHashImpl` переотображает любой нулевой хеш в ненулевую константу ещё до попадания в таблицу. Это означает, что одно целочисленное поле (`hcode`) выполняет две роли одновременно: фактический хеш ключа (когда не нуль) и сигнальная метка «пусто» (когда нуль). Дополнительного булева поля «занято» не требуется, что экономит память.

---

### `isFilled`

```nim
proc isFilled(hcode: Hash): bool {.inline.}
```

**Назначение:** логическое дополнение `isEmpty`. Возвращает `true`, если слот содержит настоящий ключ (то есть `hcode != 0`).

Используется в циклах зондирования: цикл обходит слоты, пока `isFilled` — остановка на первом пустом слоте означает, что ключ не может находиться дальше по цепочке зондирования (цепочка непрерывна).

---

### `nextTry`

```nim
proc nextTry(h, maxHash: Hash): Hash {.inline.}
```

**Назначение:** продвигает индекс зондирования на один шаг вперёд с переносом на границе массива.

**Формула:** `(h + 1) and maxHash`

Это **линейное зондирование** — простейшая стратегия разрешения коллизий. После коллизии в слоте `h` следующим кандидатом является слот `h + 1`. Часть `and maxHash` заменяет операцию взятия остатка от деления: поскольку длина массива всегда является степенью двойки, `maxHash = len - 1` имеет все младшие биты установленными, и `and maxHash` эквивалентно `mod len`, но быстрее.

**Почему линейное зондирование:** оно кешелюбиво. Последовательные слоты смежны в памяти, поэтому зондирование является последовательным сканированием, которое механизм предвыборки процессора обрабатывает эффективно. Недостаток — *первичная кластеризация*: серии занятых слотов разрастаются, увеличивая среднюю длину зондирования. Для типичных распределений ключей линейное зондирование на практике всё равно быстрее цепочечных или квадратичных альтернатив из-за кеш-эффектов.

**Пример прохода:**

```
Длина массива = 8, maxHash = 7
Ключ хешируется в слот 5, слот 5 занят.
nextTry(5, 7) = (5 + 1) & 7 = 6   → проверяем слот 6
nextTry(6, 7) = (6 + 1) & 7 = 7   → проверяем слот 7
nextTry(7, 7) = (7 + 1) & 7 = 0   → перенос на слот 0
```

### `mustRehash`

```nim
proc mustRehash[T](t: T): bool {.inline.}
```

**Назначение:** решает, переполнена ли таблица настолько, что следующая вставка деградирует производительность. Возвращает `true`, если перед вставкой следующего элемента нужно запустить перехеширование.

**Условие (любое из двух запускает перехеширование):**
- `t.dataLen < t.counter + t.counter div 2` — в массиве слотов меньше, чем 1,5× текущего числа элементов (коэффициент загрузки выше ~67%)
- `t.dataLen - t.counter < 4` — осталось менее 4 пустых слотов (защита от вырождения при малых таблицах)

**Почему ~67% коэффициент загрузки:** производительность открытой адресации деградирует по мере приближения коэффициента загрузки к 1. При 67% среднее число зондирований при поиске ещё невелико (около 1,5–2 для случайных ключей при линейном зондировании). Выше этого значения длина зондирования растёт сверхлинейно. Минимальная защита в 4 слота обрабатывает маленькие таблицы, где процентные пороги оставляли бы слишком мало места.

**`assert`:** `mustRehash` утверждает, что `dataLen > counter`, прежде чем выполнять проверку. Это инвариант безопасности — таблица никогда не должна быть полностью заполнена. Иначе поиск отсутствующего ключа завис бы в бесконечном цикле (нет пустого слота, где можно остановиться).

**Практический пример:**

```
Таблица с dataLen=8, counter=4:
  counter + counter div 2 = 4 + 2 = 6
  dataLen (8) < 6? Нет.
  dataLen - counter = 4. 4 < 4? Нет.
  → mustRehash = false, можно вставлять.

Таблица с dataLen=8, counter=6:
  counter + counter div 2 = 6 + 3 = 9
  dataLen (8) < 9? Да.
  → mustRehash = true, нужно перехешировать перед вставкой.
```

### `slotsNeeded`

```nim
proc slotsNeeded(count: Natural): int {.inline.}
```

**Назначение:** по желаемому числу элементов `count` возвращает минимальную длину массива (число слотов), которая удержит коэффициент загрузки ниже порога `mustRehash`. Используется при инициализации таблицы с известной подсказкой о ёмкости.

**Формула:** `nextPowerOfTwo(count div 2 + count + 4)`

Упрощается до `nextPowerOfTwo(count * 3 / 2 + 4)` — это точная обратная функция к порогу `mustRehash`: вычисляется наименьшая степень двойки `L`, при которой `mustRehash` вернул бы `false` для `counter = count`.

**Почему степень двойки:** шаг зондирования `nextTry` использует `and maxHash` (битовую маску) вместо взятия остатка, что работает только когда длина массива является степенью двойки.

**Пример:**

```nim
slotsNeeded(10)
  = nextPowerOfTwo(10 div 2 + 10 + 4)
  = nextPowerOfTwo(5 + 10 + 4)
  = nextPowerOfTwo(19)
  = 32

# Таблица, предварительно рассчитанная на 10 элементов,
# получает 32 слота — с хорошим запасом до первого перехеширования.
```

**Замечание о синхронизации:** комментарий в исходнике подчёркивает, что `slotsNeeded` и `mustRehash` должны оставаться синхронизированными. Если изменить одно, нужно обновить другое, иначе таблица, инициализированная через `initTable(n)`, может запустить немедленное перехеширование при самой первой вставке.

### `genHashImpl` (шаблон)

```nim
template genHashImpl(key, hc: typed)
```

**Назначение:** вычисляет хеш `key`, сохраняет его в `hc` и переотображает результат, если он оказался нулём. Это единственная точка, где обеспечивается инвариант «hcode 0 = пустой слот».

**Что делает:**
1. Вызывает `hash(key)` и сохраняет результат в `hc`.
2. Если `hc == 0` (почти никогда для реальных ключей), заменяет его фиксированной ненулевой константой:
   - На 16-битных платформах: `31415`
   - На всех остальных: `314159265` (усечённое π × 10⁸)

**Почему переотображаем ноль:** кодирование пустоты слота использует `hcode == 0` как сигнальную метку. Если бы реальный ключ имел хеш ноль и был сохранён, слот выглядел бы пустым и ключ был бы невидим для всех поисков. Переотображение в фиксированную константу безопасно, потому что сам ключ тоже хранится в слоте — ложная коллизия между переотображённым хешем и хешем какого-то другого ключа будет разрешена последующей проверкой равенства ключей.

**Почему почти никогда ноль:** хорошая хеш-функция равномерно распределяет результаты по всем возможным значениям типа `Hash`. Вероятность любого конкретного результата (включая ноль) составляет приблизительно 1/2^64 на 64-битных системах. Ветка намеренно описана как «почти никогда не выполняемая», чтобы подсказать предсказателю переходов: всегда предсказывать «не выполнять», сохраняя общий путь быстрым.

### `genHash` (шаблон)

```nim
template genHash(key: typed): Hash
```

**Назначение:** удобная обёртка вокруг `genHashImpl`, возвращающая хеш как выражение, а не сохраняющая его в переменную, переданную вызывающим кодом. Используется когда результат хеша нужен встроенно без предварительно объявленной переменной.

**Пример (концептуально):**
```nim
let h = genHash(myKey)    # эквивалентно:
# var h: Hash
# genHashImpl(myKey, h)
```

### `rawGetKnownHCImpl` (шаблон)

```nim
template rawGetKnownHCImpl() {.dirty.}
```

**Назначение:** основной цикл поиска слота. По ключу и его уже вычисленному хешу `hc` просматривает массив данных и возвращает либо индекс найденного слота (≥ 0), либо отрицательное значение, кодирующее индекс, куда ключ следует вставить при добавлении.

**Прагма `{.dirty.}`:** «грязные» шаблоны разворачиваются без гигиенического переименования — переменные внутри шаблона (`h`) разделяют пространство имён вызывающего кода. Это позволяет одному шаблону быть включённым как в `rawGet`, так и в `rawGetKnownHC` без дублирования кода, при этом изменяя общую локальную переменную.

**Кодирование возвращаемого значения:**
- `≥ 0` — индекс слота, где найден ключ. Вызывающий может читать или обновлять `t.data[result]`.
- `< 0` — ключ отсутствует. Индекс для вставки: `(-1 - result)`, то есть `-1 - rawGetKnownHCImpl()`. Это двойное использование возвращаемого значения исключает второй проход по цепочке зондирования при вставке: поиск уже нашёл лучший слот.

**Логика цикла зондирования:**

```
h = hc and maxHash(t)            # начальный слот (быстрое взятие остатка)
пока слот h занят:
    если hcode совпадает И ключ совпадает:
        вернуть h                # нашли
    h = nextTry(h, maxHash(t))   # линейный шаг вперёд
вернуть -1 - h                   # не найден; h — первый увиденный пустой слот
```

**Двухэтапное сравнение (сначала hcode, затем ключ):** цикл сначала сравнивает хранимый `hcode` с запрошенным `hc`, и только потом сравнивает ключи. Это обеспечивает короткое замыкание для большинства шагов зондирования при отсутствующих ключах: хеш-коллизии редки, поэтому дорогостоящая проверка равенства ключей (которая может включать сравнение строк, структур и т. д.) обычно пропускается. Для присутствующих ключей сравнение hcode добавляет одно дешёвое целочисленное сравнение на шаг зондирования — оправданный компромисс для нетривиальных типов ключей.

**Защита от пустой таблицы:** если `t.dataLen == 0`, функция сразу возвращает `-1`, избегая деления на ноль или бесконечного цикла при вычислении `and maxHash(t)`.

### `rawGetKnownHC`

```nim
proc rawGetKnownHC[X, A](t: X, key: A, hc: Hash): int {.inline.}
```

**Назначение:** типизированная процедура-обёртка вокруг `rawGetKnownHCImpl`. Принимает предварительно вычисленный хеш `hc` (уже переотображённый из нуля при необходимости) и выполняет цикл зондирования. Используется когда хеш уже вычислен вызывающим кодом и его выгодно сохранить для повторного использования — например, при вставке после неудачного поиска.

**Параметры:**
- `t` — таблица или множество (любой тип с `data`, `dataLen` и подходящей структурой слотов)
- `key` — ключ для поиска
- `hc` — предварительно вычисленный, переотображённый из нуля хеш `key`

**Возвращает:** то же двойное целое число, что и `rawGetKnownHCImpl`.

---

### `rawGetImpl` (шаблон)

```nim
template rawGetImpl() {.dirty.}
```

**Назначение:** составной шаг «вычислить хеш, затем зондировать». Вызывает `genHashImpl` для вычисления и сохранения хеша в переменной `hc` вызывающего кода, затем немедленно выполняет `rawGetKnownHCImpl`. Используется в `rawGet` для объединения обоих шагов в один вызов, когда хеш ещё не вычислен.

---

### `rawGet`

```nim
proc rawGet[X, A](t: X, key: A, hc: var Hash): int {.inline, outParamsAt: [3].}
```

**Назначение:** основная точка входа для поиска, когда хеш ещё не вычислен. Вычисляет хеш, сохраняет его в `hc` (выходной параметр) и возвращает индекс слота (или отрицательный индекс вставки).

**Параметры:**
- `t` — таблица или множество
- `key` — ключ для поиска
- `hc` — выход: принимает вычисленный, переотображённый из нуля хеш `key`

**Прагма `outParamsAt: [3]`:** сообщает компилятору, что аргумент 3 (`hc`) — чистый выходной параметр: он записывается до чтения. Это включает определённые оптимизации и подавляет предупреждения «неинициализированная переменная» в местах вызова.

**Почему возврат `hc` важен:** после неудачного поиска (ключ не найден) у вызывающего кода уже есть готовый хеш. Если затем нужно вставить ключ, можно вызвать `rawGetKnownHC(t, key, hc)` напрямую — повторно используя уже вычисленный хеш — вместо его пересчёта. Этот паттерн широко применяется в `tableimpl` для эффективных операций «вставить или обновить».

**Паттерн использования (концептуально):**
```nim
var hc: Hash
var idx = rawGet(t, key, hc)
if idx >= 0:
    # ключ найден в t.data[idx]
else:
    # ключ отсутствует; вставить в -1 - idx
    # hc уже вычислен, повторно используем его:
    rawInsert(t, t.data, key, val, hc, -1 - idx)
```

## Как части складываются вместе

Следующая диаграмма показывает поток вызовов для типичной операции поиска и вставки в `Table`:

```
t[key] = val
  │
  ├─ rawGet(t, key, hc)
  │     ├─ genHashImpl(key, hc)          # вычислить и переотобразить хеш
  │     └─ rawGetKnownHCImpl()           # цикл зондирования → индекс слота или отрицательное
  │
  ├─ если слот найден (idx ≥ 0):
  │     t.data[idx].val = val            # обновить на месте
  │
  └─ если не найден (idx < 0):
        mustRehash(t)?
          да → enlarge(t)               # удвоить массив, перевставить все элементы
               rawGetKnownHC(t, key, hc) # повторно зондировать в новом массиве
          нет → использовать -1 - idx   # вставить в слот, найденный при поиске
        rawInsert(t, t.data, key, val, hc, insertIdx)
```

Изящество подхода в том, что один проход зондирования одновременно отвечает на вопросы «присутствует ли ключ?» и «куда его вставить, если нет?» — без двойного зондирования.
