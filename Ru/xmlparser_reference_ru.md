# Справочник по модулю `xmlparser`

> **Модуль:** `std/xmlparser`  
> **Язык:** Nim  
> **Назначение:** Разбирает XML-документ из строки, потока или файла и строит дерево `XmlNode` в памяти, которым можно управлять средствами `std/xmltree`.

---

## Обзор

Модуль `xmlparser` — это мост между сырым XML-текстом и структурированным деревом `XmlNode` в Nim. Его задача проста, но принципиальна: принять XML откуда угодно (строковый литерал, сетевой поток, файл на диске) и вернуть дерево узлов, по которому можно ходить и которое можно запрашивать.

### Зависимости, которые важно знать

| Модуль | Роль |
|--------|------|
| `std/streams` | Абстрагирует источник данных (файл, строка, сеть…) |
| `std/parsexml` | Низкоуровневый токенизатор; порождает события «открыт элемент», «встречен атрибут» и т.д. |
| `std/xmltree` | Определяет `XmlNode` и структуру дерева, которое вы получаете на выходе |
| `std/strtabs` | Используется внутри для хранения атрибутов элементов в виде строковой таблицы |

### Два подхода к обработке ошибок

Каждая публичная функция представлена в **двух вариантах**, отличающихся только тем, как сообщаются ошибки:

- **Вариант с `errors: var seq[string]`** — мягкие ошибки. Проблемы разбора накапливаются в последовательности строк с человекочитаемыми сообщениями, которую вызывающий код проверяет после завершения. Разбор продолжается насколько возможно; вы получаете (возможно частичное) дерево даже при наличии ошибок.
- **Выбрасывающий вариант** — строгие ошибки. Если ошибки накопились, сразу после разбора вызывается `raiseInvalidXml`, бросающий исключение `XmlError`. Частичное дерево отбрасывается.

Выбирайте мягкий вариант, когда хотите показать диагностику, мириться с незначительными дефектами или логировать проблемы без падения. Выбирайте строгий вариант, когда неверный XML — это всегда ошибка программирования или фатальное условие.

## `XmlError`

```nim
type XmlError* = object of ValueError
  errors*: seq[string]
```

### Что это такое

`XmlError` — тип исключения, которое бросают строгие варианты `parseXml` и `loadXml` при обнаружении некорректного XML. Оно наследует от `ValueError`, который, в свою очередь, наследует от `CatchableError`, поэтому участвует в стандартном механизме `try/except`.

### Поля

| Поле | Тип | Описание |
|------|-----|----------|
| `msg` | `string` | (унаследовано) Первое сообщение об ошибке — именно его видно в обработчике исключений по умолчанию |
| `errors` | `seq[string]` | **Все** сообщения об ошибках, собранные за время разбора, а не только первое |

### Почему `errors` — это последовательность?

Реальный XML-документ может содержать сразу несколько проблем: непарный закрывающий тег, некорректный атрибут, посторонний символ. Поле `XmlError.errors` сохраняет каждую из них, чтобы перехватывающий код мог показать полный диагностический отчёт, а не только первый сбой.

### Пример

```nim
import std/xmlparser

try:
  let tree = parseXml("<root><unclosed>")
except XmlError as e:
  echo "Первая ошибка: ", e.msg
  echo "Все ошибки:"
  for err in e.errors:
    echo "  - ", err
```

## `parseXml` (поток + имя файла + ошибки)

```nim
proc parseXml*(s: Stream, filename: string,
               errors: var seq[string],
               options: set[XmlParseOption] = {reportComments}): XmlNode
```

### Что делает

Это **основная перегрузка** — все остальные варианты `parseXml` и `loadXml` в конечном счёте делегируют именно ей. Она открывает `XmlParser` на переданном потоке, прогоняет его токен за токеном, строит дерево `XmlNode` и дописывает все встреченные проблемы в `errors`. По завершении парсер всегда закрывается.

Строка `filename` выполняет сугубо диагностическую функцию: она фигурирует в сообщениях об ошибках (например, `"myfile.xml(12, 5) ошибка: …"`), помогая точно указать место проблемы в исходнике. Когда вы разбираете строку, а не файл, другие перегрузки передают сюда `"unknown_xml_doc"`.

### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `s` | `Stream` | Любой читаемый поток: `FileStream`, `StringStream`, сетевой поток и т.д. |
| `filename` | `string` | Имя, используемое в сообщениях об ошибках для идентификации источника |
| `errors` | `var seq[string]` | Накапливает строки с сообщениями об ошибках; уже имеющееся содержимое сохраняется |
| `options` | `set[XmlParseOption]` | Опции парсера из `std/parsexml`. По умолчанию: `{reportComments}` |

### Возвращаемое значение

Корневой `XmlNode` разобранного дерева, или `nil`, если корневого элемента не было обнаружено вовсе (например, пустой поток).

### Опции `XmlParseOption`

Берутся из `std/parsexml`. Наиболее значимые:

| Опция | Эффект |
|-------|--------|
| `reportComments` | Включать узлы-комментарии в дерево (по умолчанию включено) |
| `reportWhitespace` | Включать текстовые узлы, содержащие только пробелы |
| `allowUnquotedAttribs` | Принимать атрибуты без кавычек (мягкий режим) |
| `allowParentClose` | Принимать теги, закрытые родительским элементом (мягкий режим) |

### Пример

```nim
import std/[xmlparser, streams]

let xml = """
  <library>
    <book id="1">
      <title>Моби Дик</title>
    </book>
  </library>
"""

var errors: seq[string] = @[]
let stream = newStringStream(xml)
let tree = parseXml(stream, "inline_doc", errors)

if errors.len > 0:
  for e in errors: echo "Ошибка разбора: ", e
else:
  echo tree.tag        # "library"
```

## `parseXml` (поток, выбрасывающий)

```nim
proc parseXml*(s: Stream,
               options: set[XmlParseOption] = {reportComments}): XmlNode
```

### Что делает

Удобная обёртка над основной перегрузкой. Разбирает поток без необходимости управлять последовательностью ошибок. Если ошибки накопятся — бросает `XmlError`. Имя файла в сообщениях будет `"unknown_xml_doc"`.

Используйте этот вариант, когда у вас есть поток и вы хотите получать исключения при невалидном XML.

### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `s` | `Stream` | Любой читаемый поток |
| `options` | `set[XmlParseOption]` | Опции парсера. По умолчанию: `{reportComments}` |

### Возвращаемое значение

Корневой `XmlNode`, или исключение `XmlError`.

### Пример

```nim
import std/[xmlparser, streams]

let s = newStringStream("<note><body>Привет</body></note>")
let tree = parseXml(s)    # бросит XmlError при некорректном XML
echo tree[0].innerText    # "Привет"
```

## `parseXml` (строка, выбрасывающий)

```nim
proc parseXml*(str: string,
               options: set[XmlParseOption] = {reportComments}): XmlNode
```

### Что делает

Наиболее удобная перегрузка: принимает обычную Nim-строку с XML и возвращает разобранное дерево, бросая `XmlError` при любой ошибке. Внутри оборачивает строку в `StringStream` и делегирует потоковой перегрузке.

Это правильный выбор для разбора XML, живущего в строковых литералах, ответах API, уже загруженных в память, или вывода шаблонов — ситуаций, где открывать файл или вручную управлять потоком излишне.

### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `str` | `string` | Содержимое XML в виде Nim-строки |
| `options` | `set[XmlParseOption]` | Опции парсера. По умолчанию: `{reportComments}` |

### Возвращаемое значение

Корневой `XmlNode`, или исключение `XmlError`.

### Примеры

```nim
import std/xmlparser

# Простейшее использование
let tree = parseXml("<config><timeout>30</timeout></config>")
echo tree.tag                    # "config"
echo tree[0].tag                 # "timeout"
echo tree[0].innerText           # "30"

# С атрибутами
let xml = """<user id="42" role="admin"/>"""
let user = parseXml(xml)
echo user.attr("id")             # "42"
echo user.attr("role")           # "admin"

# Обработка ошибок
try:
  let bad = parseXml("<broken>")
except XmlError as e:
  echo "Ошибка XML: ", e.msg
```

## `loadXml` (путь + ошибки)

```nim
proc loadXml*(path: string,
              errors: var seq[string],
              options: set[XmlParseOption] = {reportComments}): XmlNode
```

### Что делает

Открывает файл по пути `path` на чтение и разбирает его как XML, собирая ошибки в переданную последовательность. Это файловый аналог потоковой перегрузки `parseXml` с параметром `errors`.

Если файл не удаётся открыть (не существует, нет прав и т.д.), немедленно бросается `IOError` — ещё до начала XML-разбора. Это важное разграничение: проблемы ввода-вывода и проблемы валидности XML сообщаются через разные механизмы.

В сообщениях об ошибках в качестве имени файла будет фигурировать сама строка `path`, что даёт точные позиции в исходнике.

### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `path` | `string` | Путь к XML-файлу в файловой системе |
| `errors` | `var seq[string]` | Накапливает сообщения об ошибках; можно безопасно переиспользовать между вызовами |
| `options` | `set[XmlParseOption]` | Опции парсера. По умолчанию: `{reportComments}` |

### Возвращаемое значение

Корневой `XmlNode`, или `nil`, если файл пуст или не содержит корневого элемента. Ошибки дописываются в `errors`.

### Пример

```nim
import std/xmlparser

var errors: seq[string] = @[]
let tree = loadXml("config.xml", errors)

if errors.len > 0:
  echo "Загружено с предупреждениями:"
  for e in errors: echo "  ", e
  # решаем: продолжать несмотря на ошибки или нет
else:
  echo "Загружено без ошибок: ", tree.tag
```

## `loadXml` (путь, выбрасывающий)

```nim
proc loadXml*(path: string,
              options: set[XmlParseOption] = {reportComments}): XmlNode
```

### Что делает

Строгий вариант загрузки файла. Читает и разбирает XML-файл по пути `path`, и если в процессе обнаружились ошибки разбора — бросает `XmlError`. Как и мягкий вариант, бросает `IOError`, если файл не удаётся открыть.

Это правильный выбор при загрузке конфигурационного файла, файла данных или любого ресурса, где некорректный XML означает, что программа просто не может продолжать работу.

### Параметры

| Параметр | Тип | Описание |
|----------|-----|----------|
| `path` | `string` | Путь к XML-файлу в файловой системе |
| `options` | `set[XmlParseOption]` | Опции парсера. По умолчанию: `{reportComments}` |

### Возвращаемое значение

Корневой `XmlNode`, или исключение `XmlError` / `IOError`.

### Пример

```nim
import std/xmlparser

# Загружаем обязательный конфиг — неверный XML смертелен
try:
  let config = loadXml("settings.xml")
  echo "Корневой тег: ", config.tag
except IOError as e:
  echo "Файл не найден или недоступен: ", e.msg
  quit(1)
except XmlError as e:
  echo "Неверный XML в settings.xml:"
  for err in e.errors:
    echo "  ", err
  quit(1)
```

## Как выбрать нужную перегрузку

Модуль предоставляет четыре публичные функции, расположенные по двум осям:

```
                       Источник
                  ┌────────────┬──────────┐
                  │   строка   │   файл   │
   ┌──────────────┼────────────┼──────────┤
О  │ мягкие ошибки│ parseXml*  │ loadXml* │   * вариант поток+имяФайла+errors
ш  │   (seq)      │ (поток)    │          │     для parseXml
и  ├──────────────┼────────────┼──────────┤
б  │  бросает     │ parseXml   │ loadXml  │
к  │  XmlError    │ (строка)   │          │
и  └──────────────┴────────────┴──────────┘
```

**Краткое руководство по выбору:**

- Разбор строкового литерала или ответа API в памяти → `parseXml(str)`
- Загрузка обязательного конфига или файла данных → `loadXml(path)`
- Хотите показать все предупреждения, но не падать → используйте вариант с `errors: var seq[string]`
- Ошибка XML = фатальное условие → используйте выбрасывающий вариант (без параметра `errors`)

## Какие узлы строит парсер?

Парсер распознаёт все стандартные конструкции XML и отображает их на типы `XmlNode` из `std/xmltree`:

| XML-конструкция | Пример | Создаваемый `XmlNode` |
|-----------------|--------|----------------------|
| Элемент с дочерними узлами | `<tag>…</tag>` | `newElement("tag")` |
| Самозакрывающийся элемент | `<br/>` | `newElement("br")` |
| Текстовое содержимое | `привет мир` | `newText("привет мир")` |
| Комментарий | `<!-- примечание -->` | `newComment(" примечание ")` |
| Секция CDATA | `<![CDATA[…]]>` | `newCData("…")` |
| Ссылка на сущность | `&amp;` | `newEntity("amp")` |
| Атрибуты | `id="42"` | хранятся в строковой таблице `node.attrs` |
| Инструкции по обработке | `<?xml … ?>` | **молча игнорируются** |
| XML-специальные (`<!…>`) | `<!DOCTYPE …>` | **молча игнорируются** |

> **Важно:** Инструкции по обработке (`<?…?>`) и специальные объявления (`<!DOCTYPE …>`) **не представлены** в выходном дереве — парсер молча пропускает их.

## Примечания и подводные камни

- **`nil` в качестве корня возможен.** Если поток или файл пуст, или содержит только пробелы, комментарии и инструкции без единого элемента, `parseXml` и `loadXml` вернут `nil`. Всегда проверяйте это перед обращением к `.tag` или дочерним узлам.
- **Комментарии включаются по умолчанию.** Опция `{reportComments}` по умолчанию означает, что комментарии появляются в дереве. Передайте `{}` или уберите `reportComments` из опций, если хотите дерево без комментариев.
- **Частичные деревья при ошибках.** Мягкие варианты (с параметром `errors`) могут вернуть частичное дерево вместе с ошибками — узлы, разобранные до ошибки, сохраняются. Строгие (выбрасывающие) варианты тоже строят частичное дерево внутри, но отбрасывают его при броске исключения.
- **Сущности сохраняются, не раскрываются.** `&amp;` становится узлом-сущностью, а не символом `&`. Для работы с содержимым сущностей используйте утилиты `std/xmltree`.
- **Поток файла закрывается автоматически.** `loadXml` открывает `FileStream` внутри; парсер закрывает его по завершении (или при ошибке).
